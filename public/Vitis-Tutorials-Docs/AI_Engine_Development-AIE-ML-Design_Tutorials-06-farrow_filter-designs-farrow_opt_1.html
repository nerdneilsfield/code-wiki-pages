farrow_opt_1 模块技术深度解析

一句话概括

farrow_opt_1 是 Farrow 分数延迟滤波器的第一个优化实现版本，它将原本串行执行的 FIR 滤波和多项式插值计算拆分为三个并行内核，通过流水线化设计充分利用 AIE-ML 的向量处理能力，在保持算法精度的同时显著提升吞吐量。

问题空间：为什么需要这个模块？

分数延迟滤波的挑战

在数字信号处理中，分数延迟滤波器（Fractional Delay Filter）是一个经典难题。想象你有一个音频采样序列，现在需要获取两个采样点之间某个任意位置的值——比如第 5.3 个采样点的幅度。你无法直接读取这个位置，因为它不存在于离散采样中。

Farrow 结构是解决这个问题的优雅方案：它用一组并行的 FIR 子滤波器配合一个可变的多项式插值器，将延迟参数 \(\mu\) 作为输入，实时计算出对应延迟位置的输出。这就像是给信号处理系统装了一个"连续变焦镜头"——你可以平滑地在采样点之间滑动，而不需要重新设计整个滤波器。

从 Baseline 到 Opt_1：架构演进的必然性

要理解 farrow_opt_1 的设计意图，必须先看它从何处演化而来。对比 farrow_port_initial（初始移植版本），我们可以清晰地看到优化的脉络：

Baseline 架构（单核串行）：

// farrow_port_initial/farrow_graph.h
class farrow_graph : public graph {
    kernel farrow_kernel1_ins;  // 单个内核：完成全部4个子滤波器
    kernel farrow_kernel2_ins;  // 单个内核：完成多项式插值
    // ...
    connect<> ( sig_i[0], farrow_kernel1_ins.in[0] );
    // Kernel1 输出4路结果到 Kernel2
    connect<> ( farrow_kernel1_ins.out[0], farrow_kernel2_ins.in[1] ); // y3
    connect<> ( farrow_kernel1_ins.out[1], farrow_kernel2_ins.in[2] ); // y2
    connect<> ( farrow_kernel1_ins.out[2], farrow_kernel2_ins.in[3] ); // y1
    connect<> ( farrow_kernel1_ins.out[3], farrow_kernel2_ins.in[4] ); // y0
};

Opt_1 架构（双核并行 FIR）：

// farrow_opt_1/farrow_graph.h
class farrow_graph : public graph {
    kernel farrow_kernel1_ins;  // 处理 taps1_2 → 输出 y3, y2
    kernel farrow_kernel2_ins;  // 处理 taps3_4 → 输出 y1, y0（复用 kernel1 类）
    kernel farrow_kernel3_ins;  // 多项式插值聚合
    // ...
    // sig_i 广播到两个 FIR 内核
    connect<> ( sig_i[0], farrow_kernel1_ins.in[0] );
    connect<> ( sig_i[0], farrow_kernel2_ins.in[0] );
};

核心洞察：

  1. 计算密度不均：在 baseline 中，farrow_kernel1_ins 需要顺序计算 4 个 8-tap FIR 滤波器（共 32 次乘累加/样本），而 farrow_kernel2_ins 仅需 3 次 MAC 和若干加法。前者成为瓶颈，后者却空闲等待。

  2. 并行化的数学基础：Farrow 结构的四个子滤波器 \(f_0, f_1, f_2, f_3\) 完全独立——它们接收相同的输入信号，只是系数不同。这意味着它们可以同时启动、并行执行，无需任何数据依赖。

  3. 资源利用率的跃升：AIE-ML 的核心优势在于 256-bit SIMD 向量单元。将 FIR 计算拆分到两个内核，意味着可以同时利用两组向量寄存器和乘累加单元，理论峰值吞吐提升近 2 倍。

这就是 farrow_opt_1 的设计原点：不是简单的代码重构，而是对算法并行结构的重新发现与映射。

心智模型：如何理解这个设计？

类比：专业化分工的厨房

想象一家高级餐厅的后厨：

  • 主厨（farrow_kernel1_ins 和 farrow_kernel2_ins）同时开工，各自负责两道菜的预处理。他们使用相同的食材（sig_i），但按照不同的配方（taps1_2 vs taps3_4）切配出不同的半成品（y3/y2 和 y1/y0）。

  • 调味师（farrow_kernel3_ins）接收所有半成品，根据每桌客人的个性化口味要求（del_i 中的延迟值 \(\mu\)），按特定比例混合，完成最终的摆盘。

  • 传菜员（dut_graph 中的 PLIO）负责把外部订单送进来，再把成品送出去。

关键洞察在于：主厨们不需要等待彼此完成，他们可以同时处理同一批食材的不同部分。只有当所有半成品都准备好后，调味师才开始工作——这种** fork-join 模式**正是数据流架构的核心优势。

各组件角色

1. dut_graph —— 设备接口适配层

位于 farrow_app.cpp，是顶层测试封装。它的职责很纯粹：

  • 条件编译区分仿真与硬件：#ifdef AIE_SIM_ONLY 下使用文件 I/O 进行离线仿真；否则配置为硬件 PLIO 接口
  • 端口实例化：创建 2 个输入 PLIO（64-bit 位宽）和 1 个输出 PLIO
  • 流连接：将外部 PLIO 连接到内部 farrow_graph 的端口

这是一个典型的网关模式（Gateway Pattern）——对外暴露标准化的 AIE 接口，对内转发到具体的计算图。

2. farrow_graph —— 计算编排层

位于 farrow_graph.h，是整个设计的核心编排器。

关键设计决策：

决策 选择 理由
内核数量 3 个 Kernel 1 复用两次处理对称的抽头组
运行时比例 0.9 每个内核占用 90% 的处理器周期，预留余量应对突发
栈位置 绑定到内核位置 减少内存访问延迟，利用局部性

抽头系数组织：

// taps1_2: 用于计算 y3, y2（高阶子滤波器）
std::vector<int16> taps1_2 = {206,-1264,6606,-14835,14835,-6606,1264,-206,
                               906,-3543,10352,-7628,-7628,10352,-3543,906};

// taps3_4: 用于计算 y1, y0（低阶子滤波器）  
std::vector<int16> taps3_4 = {-51,316,-1652,20093,-20093,1652,-316,51,
                              -226,886,-2588,10099,10099,-2588,886,-226};

注意到这些系数具有对称性（sym），这是 Farrow 结构的数学特性，也是代码注释中标注 "sym" 的原因。

3. farrow_kernel1 —— FIR 计算引擎

位于 farrow_kernel1.h/cpp，是可复用的 FIR 滤波内核。

类型系统设计：

typedef cint16  TT_SIG;   // 复数信号类型：16-bit 实部 + 16-bit 虚部
typedef cacc64  TT_ACC;   // 累加器类型：64-bit 精度防止溢出
static constexpr unsigned BUFFER_SIZE = 1024;  // 缓冲区大小
static constexpr unsigned DNSHIFT = 14;        // 定点归一化位移

状态管理：

alignas(32) TT_SIG f_state[STATE_LEN];  // 8 个复数状态的延迟线
alignas(32) int16 (&f_taps)[16];        // 16-tap 滤波器系数（引用绑定）

alignas(32) 确保数据对齐到 256-bit 边界，这是 AIE-ML 向量加载指令的最优对齐方式。

核心算法（run 方法）：

// 双循环结构：分别生成 y3 和 y2
for (rr = 0; rr < BUFFER_SIZE/32; rr++) {
    // 使用 sliding_mul_ops 进行向量化 FIR 计算
    acc_f3 = aie::sliding_mul_ops<16,8,1,1,1,int16,cint16>::mul(f_coeffs,0,v_buff,25);
    *p_y3++ = acc_f3.to_vector<TT_SIG>(DNSHIFT);  // 定点归一化输出
}

sliding_mul_ops 是 AIE API 的关键原语，它在一个时钟周期内完成 16 个系数的乘累加。模板参数含义：

  • 16：系数向量长度
  • 8：数据滑动步长
  • 1,1,1：乘法、加法、输出的级联配置

状态保存：

*(v8cint16*)f_state = v_buff.extract<8>(3);  // 保存最后 8 个样本供下次迭代

这是重叠相加法（Overlap-Add）的实现，保证块处理之间的连续性。

4. farrow_kernel2 —— 多项式插值聚合器

位于 farrow_kernel2.h/cpp，负责将四个子滤波器的输出合成为最终结果。

Horner 法则实现：

Farrow 结构的插值公式为：

\[y(n) = y_0 + \mu \cdot (y_1 + \mu \cdot (y_2 + \mu \cdot y_3))\]

代码将其展开为三阶段流水线：

// Stage 1: z = y2 + μ·y3
acc_x = aie::mac(aie::from_vector<TT_ACC>(*p_y2++,DNSHIFT), *p_y3++,del);
*p_z2++ = acc_x.to_vector<TT_SIG>(DNSHIFT);

// Stage 2: z' = y1 + μ·z  
acc_y = aie::mac(aie::from_vector<TT_ACC>(*p_y1++,DNSHIFT), *p_z2++,del);
*p_z1++ = acc_y.to_vector<TT_SIG>(DNSHIFT);

// Stage 3: out = y0 + μ·z'
acc_z = aie::mac(aie::from_vector<TT_ACC>(*p_y0++,DNSHIFT), *p_z1++,del);
*p_sig_o++ = acc_z.to_vector<TT_SIG>(DNSHIFT);

延迟值处理：

del = aie::vector_cast<int16>(*p_del_i++); *p_del_i++;

注意这里每次读取 4 个延迟值（begin_vector<4>），但只使用其中 2 个转换为 int16。这是因为延迟值以 32-bit 存储（TT_DEL = int32），而 MAC 运算需要 16-bit 系数。

数据流追踪

让我们跟随一个采样块从输入到输出的完整旅程：

1. 输入阶段（dut_graph）

// 仿真模式：从文件读取测试向量
sig_i[0] = input_plio::create("PLIO_i_0", plio_64_bits, "data/sig_i.txt");
del_i[0] = input_plio::create("PLIO_i_1", plio_64_bits, "data/del_i_optimized.txt");

// 硬件模式：连接到 PL 逻辑
sig_i[0] = input_plio::create("PLIO_i_0", plio_64_bits);

2. 广播分发（farrow_graph）

// sig_i 被广播到两个 FIR 内核
connect<> (sig_i[0], farrow_kernel1_ins.in[0]);
connect<> (sig_i[0], farrow_kernel2_ins.in[0]);

// del_i 只送到聚合内核
connect<> (del_i[0], farrow_kernel3_ins.in[0]);

3. 并行 FIR 计算（farrow_kernel1 × 2）

每个内核处理 1024 个复数样本，产生两组输出：

  • Kernel 1（taps1_2）：输出 y3, y2
  • Kernel 2（taps3_4）：输出 y1, y0

4. 流式聚合（farrow_kernel2）

// 四个输入流汇聚
connect<> (farrow_kernel1_ins.out[0], farrow_kernel3_ins.in[1]); // y3
connect<> (farrow_kernel1_ins.out[1], farrow_kernel3_ins.in[2]); // y2
connect<> (farrow_kernel2_ins.out[0], farrow_kernel3_ins.in[3]); // y1
connect<> (farrow_kernel2_ins.out[1], farrow_kernel3_ins.in[4]); // y0

5. 输出阶段

connect<>(farrow_kernel3_ins.out[0], sig_o[0]);

设计权衡与决策分析

1. 并行度 vs 资源消耗

选择：将 4 个子滤波器合并为 2 个内核（每核处理 2 个子滤波器），而非 4 个独立内核。

理由：

  • AIE-ML 内核有显著的启动开销（代码加载、堆栈初始化）
  • 单个内核处理 2 个 8-tap FIR 仍有充足的周期余量（runtime ratio 0.9）
  • 减少内核间通信的流缓冲需求

代价：如果未来需要更高并行度（如 8-tap 分解为 4×2-tap），当前结构需要重构。

2. 定点精度 vs 动态范围

选择：使用 DNSHIFT = 14 进行定点归一化。

理由：

  • FIR 累加器 64-bit → 输出 16-bit 需要合理的缩放
  • 14-bit 右移对应 \(2^{14} = 16384\) 的缩放因子，平衡了数值范围和精度保留
  • 与抽头系数的量级匹配（最大系数约 20000）

风险：如果输入信号幅度过大，可能发生饱和。代码设置了饱和模式：

aie::set_saturation(aie::saturation_mode::saturate);

3. 同步 vs 异步数据流

选择：使用 ADF 的 connect<> 默认流语义（阻塞式）。

理由：

  • Farrow 算法要求 y3, y2, y1, y0 和 del_i 严格对齐
  • 阻塞流自动提供反压（backpressure）机制，防止缓冲区溢出
  • 简化了内核代码，无需显式同步原语

替代方案：如果使用乒乓缓冲（ping-pong buffer）或异步 FIFO，可以提高吞吐但会增加复杂度。

4. 代码复用 vs 特化优化

选择：farrow_kernel1 被实例化两次，通过不同的抽头参数区分。

理由：

  • 单一源码文件（farrow_kernel1.cpp）维护简单
  • 模板/参数化设计允许编译器针对具体系数做常量传播优化
  • 如果需要对特定子滤波器做特化优化，可以派生新类

5. 为什么不是 4 个独立内核？

一个自然的疑问是：既然四个子滤波器完全独立，为什么不创建 4 个 farrow_kernel1 实例，每个处理一个子滤波器？这样并行度不是更高吗？

决策背后的工程考量：

方案 内核数 理论并行度 实际瓶颈 资源开销
单核串行 (baseline) 2 1× FIR 计算 低
双核并行 (opt_1) 3 ~2× 内存带宽 中等
四核并行 (假设) 5 ~2.5× 调度开销、流缓冲 高

关键限制因素：

  1. AIE-ML 的内存带宽：每个内核需要独立的输入数据流。虽然 sig_i 可以广播，但输出流 y0-y3 都需要独立的缓冲区。过多的并发流会争用片上存储器（Tile Memory）。

  2. 内核启动开销：每个 AIE 内核都有固定的启动时间（加载代码、初始化堆栈）。当粒度太细时，启动开销会吞噬并行收益。

  3. 编译器优化窗口：当前 farrow_kernel1 中的循环已经被充分展开和流水线化。拆分成更小的内核反而可能破坏编译器的优化机会。

验证思路：如果你怀疑当前并行度不是最优，可以通过以下指标验证：

make get_II  # 查看各内核的 Initiation Interval
make throughput  # 测量端到端吞吐

如果 FIR 内核的 II 已经接近理论下限（通常为 1），而聚合内核还有余量，说明当前平衡是合理的。

边缘情况与陷阱

1. 状态连续性

farrow_kernel1 在构造函数中清零状态数组，并在每次 run 后保存最后 8 个样本。这意味着：

  • 首次调用：前 8 个输出样本基于零初始状态，可能不准确
  • 连续调用：状态自动延续，无需外部干预
  • 中断恢复：如果内核被抢占，状态会丢失（AIE 不支持抢占式多任务）

2. 延迟值范围

代码假设 del_i 中的值已经归一化到适合 16-bit 有符号整数的范围。如果延迟值超出 \([-32768, 32767]\)，vector_cast<int16> 会产生未定义行为（实际表现为截断）。

3. 缓冲区对齐

ADF 的 input_buffer/output_buffer 要求缓冲区地址 32-byte 对齐。如果主机端分配的内存未对齐，会在 init() 阶段报错。

4. 仿真与硬件的行为差异

#ifdef AIE_SIM_ONLY
    // 文件 I/O 模式
#else
    // 硬件 PLIO 模式
#endif

两种模式下，数据格式必须完全一致。常见的错误是在仿真时使用文本格式的十六进制数，但在硬件端期望二进制格式。

5. 内核代码共享的副作用

farrow_kernel1 被两个实例共享：

farrow_kernel1_ins = kernel::create_object<farrow_kernel1>(taps1_2);
farrow_kernel2_ins = kernel::create_object<farrow_kernel1>(taps3_4);  // 注意：仍是 farrow_kernel1
source(farrow_kernel2_ins) = "farrow_kernel1.cpp";  // 相同源文件

如果修改 farrow_kernel1.cpp，两个实例都会受影响。如需差异化行为，应使用模板参数或运行时分支。

C++ 技术深度剖析

内存所有权与生命周期模型

1. 抽头系数的引用语义

// farrow_kernel1.h
alignas(32) int16 (&f_taps)[16];  // 引用绑定，非拥有

// farrow_kernel1.cpp
farrow_kernel1::farrow_kernel1(int16 (&taps)[16]) : f_taps(taps) {}  // 引用初始化

关键洞察：f_taps 是一个引用成员，这意味着：

  • 所有权：farrow_graph 中的 std::vector<int16> taps1_2 拥有系数数据的生命周期
  • 借用：farrow_kernel1 只是借用了这个数组的引用，用于向量加载操作
  • 风险：如果 farrow_graph 在 farrow_kernel1 之前销毁（实际上不会发生，因为图是静态实例化的），将形成悬空引用

为什么不用 std::array 或值拷贝？

  • AIE 内核运行在独立的 Tile 上，需要显式的 DMA 传输来移动数据
  • 引用允许编译器直接生成从全局内存加载的向量指令，避免中间拷贝
  • 16 个 int16 只有 32 字节，但乘以数百万次迭代，拷贝开销不可忽视

2. 状态数组的 RAII 管理

alignas(32) TT_SIG f_state[STATE_LEN];  // 值语义，内核私有

// 构造函数清零
for(int i=0; i<STATE_LEN; i++) {
    f_state[i] = (cint16){0,0};
}

// 每次 run() 后保存状态
*(v8cint16*)f_state = v_buff.extract<8>(3);

这里采用了简化的 RAII：没有动态分配，没有智能指针，只有栈上的固定大小数组。这是嵌入式 DSP 编程的典型模式——在编译期确定所有资源，运行期零开销。

3. 缓冲区契约与对齐要求

void run(input_buffer<TT_SIG, adf::extents<BUFFER_SIZE>>& __restrict sig_i,
         output_buffer<TT_SIG, adf::extents<BUFFER_SIZE>>& __restrict y3,
         output_buffer<TT_SIG, adf::extents<BUFFER_SIZE>>& __restrict y2);

__restrict 关键字的作用： 告诉编译器这些指针不会别名（alias），即 sig_i、y3、y2 指向的内存区域互不重叠。这允许编译器生成更激进的向量化代码，无需考虑读写依赖。

alignas(32) 的工程意义：

alignas(32) TT_SIG f_state[STATE_LEN];

AIE-ML 的向量加载指令（如 vload）在 256-bit（32-byte）对齐时性能最优。未对齐访问会导致：

  • 额外的指令周期（可能需要两次加载 + 移位拼接）
  • 或者硬件异常（取决于具体配置）

类型系统的精密设计

信号类型的层次结构

类型别名 实际类型 位宽 用途
TT_SIG cint16 32-bit 输入/输出信号，平衡精度与存储
TT_ACC cacc64 64-bit FIR 累加器，防止溢出
TT_DEL int32 32-bit 延迟值，高精度表示分数延迟

为什么 FIR 用 64-bit 累加器，而插值只用 48-bit？

// farrow_kernel1.h
typedef cacc64 TT_ACC;  // 64-bit for FIR

// farrow_kernel2.h
typedef cacc48 TT_ACC;  // 48-bit for interpolation

FIR 滤波涉及 8-tap 乘累加，动态范围更大；插值阶段的 MAC 运算规模较小，48-bit 已足够。这种精细的类型选择直接影响：

  • 功耗：更宽的累加器消耗更多能量
  • 指令延迟：某些宽累加操作需要额外周期
  • 寄存器压力：影响编译器能否有效调度循环

定点数缩放的艺术

static constexpr unsigned DNSHIFT = 14;
acc_f3.to_vector<TT_SIG>(DNSHIFT);  // 右移 14 位归一化

为什么是 14 而不是 15 或 16？

抽头系数的最大绝对值约为 20000（20093 来自 taps3_4）。8-tap FIR 的理论最大输出为：

\[20000 \times 32767 \times 8 \approx 5.2 \times 10^9\]

这需要约 33 bit 来表示。64-bit 累加器可以容纳，但输出到 16-bit 时需要缩放：

  • 右移 14 位 ≈ 除以 16384
  • 上述最大值缩放后：\(5.2 \times 10^9 / 16384 \approx 3.2 \times 10^5\)，仍在 16-bit 有符号范围外

矛盾？ 实际上，Farrow 结构的系数具有特殊性质（反对称/对称），实际动态范围远小于理论最坏情况。14-bit 位移是通过仿真和实测验证的安全选择。

并发安全与数据竞争

单生产者-多消费者模式

// sig_i 被广播到两个内核
connect<> (sig_i[0], farrow_kernel1_ins.in[0]);
connect<> (sig_i[0], farrow_kernel2_ins.in[0]);

ADF 框架保证这种广播连接是线程安全的：

  • 底层实现可能是硬件多播（multicast）或乒乓缓冲复制
  • 从 kernel 视角看，每个消费者收到的是独立的数据副本
  • 无需互斥锁或原子操作

内核间的 happens-before 关系

// Kernel 1 的输出连接到 Kernel 3 的输入
connect<> (farrow_kernel1_ins.out[0], farrow_kernel3_ins.in[1]); // y3
connect<> (farrow_kernel1_ins.out[1], farrow_kernel3_ins.in[2]); // y2

ADF 的流语义自动建立同步点：Kernel 3 在消费 y3 之前，必须等待 Kernel 1 产生该数据。这种隐式同步消除了显式信号量的需要，但也意味着：

  • 如果 Kernel 1 stall，Kernel 3 必然 stall
  • 死锁可能发生在循环依赖中（本设计无循环，故安全）

向量指令的微观架构

sliding_mul_ops 的硬件映射

acc_f3 = aie::sliding_mul_ops<16,8,1,1,1,int16,cint16>::mul(f_coeffs,0,v_buff,25);

这条看似简单的 C++ 语句，实际上映射到 AIE-ML 的级联乘累加单元（Cascaded MAC Array）的复杂控制：

模板参数解码：

参数 值 含义
Lanes 16 每次处理 16 个系数
Points 8 滑动窗口大小（输出点数）
CoeffStep 1 系数步进（连续访问）
DataStepX 1 数据 X 维度步进
DataStepY 1 数据 Y 维度步进

为什么偏移量是 25 和 9？

// 第一组输出
acc_f3 = aie::sliding_mul_ops<...>::mul(f_coeffs,0,v_buff,25);  // offset 25
// 第二组输出  
acc_f3 = aie::sliding_mul_ops<...>::mul(f_coeffs,0,v_buff,9);   // offset 9

v_buff 是一个 32-element 的向量寄存器，布局如下：

索引: [0-7]  [8-15]  [16-23]  [24-31]
      ^新数据  ^历史数据          ^状态保存区
  • Offset 25 指向第 3 组的末尾（用于计算 y3）
  • Offset 9 指向第 1 组的中间（用于计算 y2）

这种精心设计的偏移量使得单次向量加载可以服务两次 FIR 计算，最大化寄存器复用。

Horner 法则的向量化实现

// Stage 1: z = y2 + μ·y3
acc_x = aie::mac(aie::from_vector<TT_ACC>(*p_y2++,DNSHIFT), *p_y3++,del);

这里涉及多个类型转换：

  1. *p_y2++ 读取 cint16 向量
  2. from_vector<TT_ACC>(..., DNSHIFT) 转换为 cacc48，同时应用定点缩放
  3. mac(..., del) 执行复数乘累加
  4. to_vector<TT_SIG>(DNSHIFT) 缩放回 16-bit 输出

关键优化：aie::vector_cast<int16>() 将 32-bit 延迟值转换为 16-bit，允许使用更高效的乘法指令。

编译器提示与软件流水线

for (unsigned rr=0; rr < BUFFER_SIZE/32; rr++) 
    chess_loop_range(BUFFER_SIZE/32,BUFFER_SIZE/32)  // 精确循环次数
    chess_prepare_for_pipelining                    // 启用软件流水
{
    // ...
}

Chess 编译器专属 pragma：

  • chess_loop_range(min,max)：告诉编译器循环的迭代次数范围。当 min=max 时，编译器可以生成最紧凑的循环 prologue/epilogue。

  • chess_prepare_for_pipelining：请求编译器进行模调度（Modulo Scheduling），将循环体重叠执行以隐藏延迟。

软件流水线的直观理解：

想象一个工厂装配线，每个工人负责一道工序。传统执行是"所有工人完成一批后再开始下一批"；软件流水线则是"工人 A 处理批次 N 的同时，工人 B 处理批次 N-1，工人 C 处理批次 N-2"。

在 AIE 中，这对应于：

  • 加载下一组数据（内存操作，高延迟）
  • 同时执行当前数据的乘法（计算操作）
  • 同时写回上一组数据的累加结果（存储操作）
