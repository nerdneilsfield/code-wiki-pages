Host Control 子模块详解

概述

host.cpp 是系统的中央控制器，运行在 ARM Cortex-A72（或 x86 仿真环境）上，负责：

  1. 加载 FPGA 比特流（xclbin）
  2. 协调 PL Kernels 和 AIE Graph 的启动顺序
  3. 注入异步数组 RTP 参数
  4. 验证计算结果的正确性

这是软件与硬件交互的唯一入口，理解其工作原理对系统级调试至关重要。

核心代码解析

Phase 1: 设备初始化

// 打开设备索引 0（VCK190 上的唯一设备）
auto device = xrt::device(0);

// 加载编译好的比特流
auto uuid = device.load_xclbin(xclbinFilename);

xrt::device 的底层含义

XRT（Xilinx Runtime）是 AMD/Xilinx 提供的用户态驱动库。xrt::device(0) 实际上：

  1. 打开 /dev/dri/renderD128（DRM 渲染节点）
  2. 通过 ioctl 与内核驱动通信
  3. 建立用户态与 FPGA 之间的命令通道

UUID 的作用

load_xclbin() 返回的 uuid 是本次加载的唯一标识符。后续所有 kernel/graph 句柄都必须绑定到这个 UUID，确保使用正确的硬件配置。

Phase 2: 获取硬件句柄

// PL Kernel 句柄 —— 用于控制 datagen 和 s2mm
auto s2mm = xrt::kernel(device, uuid, "s2mm");
auto datagen = xrt::kernel(device, uuid, "datagen");

// AIE Graph 句柄 —— 用于控制 rtpgraph
auto ghdl = xrt::graph(device, uuid, "gr");

命名匹配规则

句柄类型 构造函数参数 对应源代码位置
xrt::kernel "s2mm" pl_kernels/s2mm.cpp 中的 extern "C" void s2mm(...)
xrt::kernel "datagen" pl_kernels/datagen.cpp 中的 extern "C" void datagen(...)
xrt::graph "gr" aie/graph.cpp 中的全局变量 rtpgraph gr;

常见错误：如果名称不匹配（如大小写错误），会在运行时抛出 xrt::system_error。

Phase 3: 内存分配

// 计算缓冲区大小：256 元素/轮 × 4 字节/元素 × 4 轮 = 4096 字节
size_t output_size_in_bytes = ELEM_per_iter * 4 * ITERATION;

// 分配设备内存（通过 XRT Buffer Object）
auto out_bo = xrt::bo(device, output_size_in_bytes, s2mm.group_id(0));

// 映射到主机虚拟地址空间
auto host_out = out_bo.map<int*>();

xrt::bo（Buffer Object）详解

这是 XRT 的统一内存抽象，封装了复杂的物理内存管理：

主机虚拟地址 (host_out)
       │
       ▼
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Page Table │────▶│   IOMMU     │────▶│  DDR Memory │
│  (Linux)    │     │  (SMMU)     │     │  (Device)   │
└─────────────┘     └─────────────┘     └─────────────┘
       │
       └── 由 XRT 自动管理，对用户透明

s2mm.group_id(0) 的含义

每个 kernel 的参数都有对应的 memory group，指示应使用哪个 DDR bank：

// s2mm.cpp 中的接口声明
void s2mm(ap_int<32>* mem, ...)  // mem 是第一个参数（索引 0）
#pragma HLS INTERFACE m_axi port=mem ... bundle=gmem

group_id(0) 查询第一个参数（mem）应该分配到哪个存储组。这允许：

  • 多 DDR bank 并行访问
  • NUMA-aware 的内存放置
  • 避免 bank 冲突导致的性能下降

Phase 4: 启动 PL Kernels

// 启动 s2mm（非阻塞）
auto s2mm_run = s2mm(out_bo, nullptr, ELEM_per_iter * ITERATION);

// 启动 datagen（非阻塞）
auto datagen_run = datagen(nullptr, ELEM_per_iter * ITERATION, 0);

调用约定分析

s2mm 调用：

s2mm(out_bo,           // arg 0: mem → 传入 buffer object
     nullptr,          // arg 1: 未使用（保留）
     total_size)       // arg 2: size → 总共接收的元素数

datagen 调用：

datagen(nullptr,        // arg 0: 未使用（无内存输出）
        total_size,     // arg 1: size → 总共产生的元素数
        0)              // arg 2: init_val → 序列起始值

非阻塞语义

operator() 返回 xrt::run 对象，表示已提交但未完成的执行实例：

auto s2mm_run = s2mm(...);  // 立即返回，不等待 kernel 完成
// ... 可以执行其他工作 ...
s2mm_run.wait();            // 显式阻塞，直到 kernel 完成

这种设计允许流水线并行：在 PL kernel 运行的同时，主机可以准备下一批数据或处理上一批结果。

Phase 5: AIE Graph 控制与 RTP 注入

// 准备 RTP 参数数组
int value[256];
for(int i = 0; i < 256; i++) {
    value[i] = i;  // value[j] = j
}

// 启动 AIE Graph，运行 4 轮迭代
ghdl.run(ITERATION);  // ITERATION = 4

// 异步注入 RTP 参数
ghdl.update("gr.k.in[1]", value);

// 结束 Graph 执行
ghdl.end();

ghdl.run(4) 的精确语义

这不是简单的"启动"，而是带有迭代计数的运行：

AIE Graph 状态机：
IDLE ──run(4)──▶ RUNNING ──▶ 完成 4 轮迭代 ──▶ IDLE
                    │
                    └── 每轮迭代：从 in[0] 读 256 个元素
                                    从 in[1] 读 RTP 数组
                                    计算并写入 out[0]

ghdl.update() 的字符串路径

ghdl.update("gr.k.in[1]", value);
//          └────┬────┘
//               │
//               └── 层次化路径：graph.kernel.port

路径组件解析：

组件 对应代码 说明
gr rtpgraph gr; graph 实例名（graph.cpp）
k adf::kernel k; kernel 对象名（graph.h）
in[1] k.in[1] kernel 的第二个输入端口

为什么用字符串而非强类型？

这是 XRT API 的设计权衡：

  • 优点：灵活，可以在运行时动态选择目标
  • 缺点：无编译期检查，拼写错误导致运行时失败

生产建议：将路径定义为常量，集中管理：

constexpr const char* RTP_PORT = "gr.k.in[1]";
ghdl.update(RTP_PORT, value);

异步语义的时序考量

ghdl.run(4);           // T0: 启动 Graph
ghdl.update(...);      // T1: 注入 RTP（T1 ≈ T0 + 几微秒）

由于两者都是非阻塞的，存在竞态条件：

  • 如果 Graph 启动极快，第一轮可能读取到默认值 0
  • 实际观察到的行为取决于：AIE 频率、DDR 延迟、XRT 开销

确定性方案：

// 方案 1：先更新后启动（推荐）
ghdl.update("gr.k.in[1]", value);
ghdl.run(4);

// 方案 2：使用 initial_value 约束（在 graph.h 中）
initial_value(k.in[1]) = default_value;

Phase 6: 同步与验证

// 等待 s2mm 完成（阻塞）
s2mm_run.wait();

// 将设备内存同步到主机（DMA 传输）
out_bo.sync(XCL_BO_SYNC_BO_FROM_DEVICE);

// 验证结果
int match = 0;
for (int i = 0; i < ITERATION; i++) {
    for(int j = 0; j < ELEM_per_iter; j++) {
        int expected = value[j] + i * ELEM_per_iter + j;
        if(host_out[ELEM_per_iter * i + j] != expected) {
            match = 1;
        }
    }
}

sync() 的必要性

现代 SoC 有复杂的缓存层次：

CPU Core
   │
   ├──▶ L1 Cache ──▶ L2 Cache ──▶ L3 Cache
   │                                   │
   └──▶ 写缓冲区                          │
                                       ▼
                                 CCI/CMN（一致性互联）
                                       │
                                       ▼
                                 DDR Controller ──▶ DDR Memory

xrt::bo 分配的内存通常是缓存一致的，但显式 sync() 确保：

  1. 刷新 CPU 写缓冲区（如果是 TO_DEVICE）
  2. 使 CPU 缓存失效（如果是 FROM_DEVICE）
  3. 触发 DMA 引擎进行高效的数据搬运

验证逻辑推导

来源 第 i 轮第 j 个元素的值
datagen 输出 i * 256 + j（递增序列）
RTP 参数 j（初始化时设置）
期望输出 input + RTP = (i*256+j) + j = i*256 + 2*j

验证公式：

expected = value[j] + i * ELEM_per_iter + j;
//         └─┬─┘     └──────┬──────┘   └┬┘
//           │              │           │
//      RTP[j] = j    轮次偏移    轮内索引

错误处理策略

int main(int argc, char* argv[]) {
    try {
        auto match = run(argc, argv);
        std::cout << "TEST " << (match ? "FAILED" : "PASSED") << "\n";
        return (match ? EXIT_FAILURE : EXIT_SUCCESS);
    } catch (std::exception const& e) {
        std::cout << "Exception: " << e.what() << "\n";
        std::cout << "FAILED TEST\n";
        return 1;
    }
}

总结

Host Control 子模块展示了 Versal ACAP 的软件控制范式：

  1. 设备管理：通过 XRT 加载比特流、获取硬件句柄
  2. 内存管理：使用 xrt::bo 进行异构内存分配和同步
  3. 执行控制：协调 PL kernels 和 AIE graph 的启动顺序
  4. 参数注入：通过 xrt::graph::update() 实现异步 RTP 更新
  5. 结果验证：回读 DDR 数据并进行数值校验

掌握这些模式后，你可以构建更复杂的异构计算应用，充分利用 Versal 的 PS+PL+AIE 三位一体架构。
