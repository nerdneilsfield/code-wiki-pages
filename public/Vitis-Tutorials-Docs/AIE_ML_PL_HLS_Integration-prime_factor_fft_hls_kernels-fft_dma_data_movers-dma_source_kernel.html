dma_source_kernel 模块深度解析

一句话概括

dma_source_kernel 是 Prime Factor FFT 流水线的数据入口闸门——它负责将存储在 DDR4 中的原始 FFT 输入数据，以精确的时序和格式推送到后续的 PL HLS 重排核（permutation kernels）和 AIE 计算阵列。你可以把它想象成一个精密的自动售货机出货口：它不会一次性倾倒所有货物，而是按照下游机器的节奏，一个节拍、一个样品地稳定输出，同时还要处理流水线启动时的"空泡"填充。

问题空间：为什么需要这个模块？

核心挑战：异构系统的数据鸿沟

在 Versal ACAP 架构中，我们面临一个根本性的数据流断层：

层级 时钟域 数据宽度 访问模式 特性
DDR4 ~300 MHz 512-bit (典型) Burst/随机 高延迟、高带宽
PL HLS 312.5 MHz 128-bit 流式 确定性时序
AIE Array 1250 MHz 32-bit × 4 流式 超低延迟、并行

这种差异带来三个具体问题：

  1. 速率不匹配：DDR4 的突发传输（burst）与 AIE 的细粒度流式消费无法直接对接
  2. 时序对齐：FFT 流水线有严格的启动延迟要求（latency padding），必须在数据前插入空周期
  3. 重复测试：验证阶段需要多次循环发送同一组数据，而无需每次都重新从 DDR4 加载

朴素方案的陷阱

如果让 AIE 直接访问 DDR4：

  • AIE 的流式接口没有地址生成能力，无法发起突发传输
  • 每次读取的延迟不可预测，会破坏 FFT 的确定性流水线
  • 频繁的跨时钟域握手会严重降低吞吐量

如果简单地一次性读完所有数据再推送：

  • 1008 点 FFT × 4 路并行 = 4032 个复数样本，片上 BRAM 可能不足以缓冲完整帧
  • 缺乏 latency padding 会导致后续 permutation kernel 的数据错位

设计洞察

解决方案的核心在于双缓冲 + DATAFLOW 流水线：

  • 用 load_buffer() 从 DDR4 批量加载到片上 BRAM（高带宽突发）
  • 用 transmit() 从 BRAM 流式输出到 AXI Stream（低延迟流式）
  • 两个函数通过 #pragma HLS DATAFLOW 实现时空重叠——当 transmit 输出第 N 帧时，load_buffer 可以并行预取第 N+1 帧

代码深度剖析

1. 类型系统与数据布局 (pfa1008_dma_src.h)

namespace pfa1008_dma_src {
  static constexpr unsigned NFFT  = 1008;   // FFT 点数
  static constexpr unsigned DEPTH = 1008;   // 缓冲区深度（4 transforms * 1008 / 4-samples）
  static constexpr unsigned NBITS = 128;    // 总线位宽
  typedef ap_uint<NBITS>      TT_DATA;      // 128-bit 打包数据（4×32bit 复数样本）
  typedef ap_uint<NBITS/4>    TT_SAMPLE;    // 32-bit 单个样本
  typedef hls::stream<TT_DATA> TT_STREAM;   // AXI Stream 通道
  typedef ap_uint<11>         TT_ADDR;      // 地址类型（log2(1008) ≈ 10 bits）
};

设计要点解读：

  • 128-bit 数据宽度：这是关键的架构决策。注释中提到 "32-bit I/O's @ 1250 MHz AIE clock or 128-bit I/O's @ 312.5 MHz"，意味着 PL 侧使用更宽的数据通路、更低的时钟频率，与 AIE 的高频窄带接口形成等效带宽匹配：

    \[\text{Bandwidth} = 128 \text{ bits} \times 312.5 \text{ MHz} = 40 \text{ Gbps} = 32 \text{ bits} \times 1250 \text{ MHz}\]

  • TT_DATA 打包格式：每个 128-bit 字包含 4 个 32-bit 复数样本（实部 16-bit + 虚部 16-bit）。这与后续 pfa1008_permute_i 中的解包逻辑 (permute_i[3], permute_i[2], permute_i[1], permute_i[0]) = sig_i.read() 严格对应。

  • DEPTH = 1008：不是巧合。1008 = 16 × 9 × 7，这是 Prime Factor Algorithm (PFA) 的分解基础。每帧恰好 1008 个样本，4 路并行时每周期处理 4 个样本，共需 252 个周期完成一帧。

2. 顶层封装：pfa1008_dma_src_wrapper

void pfa1008_dma_src_wrapper(
    pfa1008_dma_src::TT_DATA mem[pfa1008_dma_src::DEPTH],
    int loop_cnt,
    pfa1008_dma_src::TT_STREAM& sig_o )
{
#pragma HLS interface m_axi      port=mem         bundle=gmem    offset=slave   depth=DEPTH
#pragma HLS interface axis       port=sig_o
#pragma HLS interface s_axilite  port=loop_cnt    bundle=control
#pragma HLS interface s_axilite  port=mem         bundle=control
#pragma HLS interface s_axilite  port=return      bundle=control
#pragma HLS DATAFLOW

  TT_DATA buff[DEPTH];           // 内部 BRAM 缓冲区
  load_buffer(mem, buff);        // Stage 1: DDR4 → BRAM
  transmit(buff, sig_o, loop_cnt); // Stage 2: BRAM → AXIS (含重复和 padding)
}

接口协议分析：

Pragma 含义 硬件映射
m_axi port=mem bundle=gmem DDR4 内存接口 AXI4-Full master，支持突发传输
axis port=sig_o 流式输出 AXI4-Stream master，连接到下游 permutation kernel
s_axilite port=loop_cnt 标量控制寄存器 AXI4-Lite slave，Host 可配置循环次数
DATAFLOW 任务级并行 load_buffer 和 transmit 作为独立流水线阶段并发执行

关键设计决策：

  • offset=slave：表示 Host 通过 AXI-Lite 写入内存指针偏移量，kernel 从该偏移处开始读取。这支持动态缓冲区定位。

  • 双 s_axilite 绑定：loop_cnt 和 mem 都绑定到 control bundle，意味着它们共享同一个 AXI4-Lite 接口，Host 可以通过连续的寄存器写操作完成配置。

3. 数据加载阶段：load_buffer()

void load_buffer(TT_DATA mem[DEPTH], TT_DATA (&buff)[DEPTH])
{
LOAD_BUFF: for (int mm=0; mm < DEPTH; mm++) {
#pragma HLS PIPELINE II=1
    buff[mm] = mem[mm];
  }
}

性能特征：

  • II=1 (Initiation Interval)：每个时钟周期启动一次迭代，理论吞吐量为 128 bits/cycle

  • 在 312.5 MHz 时钟下，加载 1008 个 128-bit 字需要：

    \[T_{load} = 1008 \times 3.2 \text{ ns} = 3.225 \text{ μs}\]

  • Burst 优化：虽然代码看起来是逐元素访问，但 HLS 工具会自动识别连续地址模式，将其合并为 AXI burst 传输，最大化 DDR4 带宽利用率。

4. 流传输阶段：transmit() —— 最精妙的部分

void transmit(TT_DATA (&buff)[DEPTH], TT_STREAM& sig_o, const int& loop_cnt)
{
  static constexpr int LATENCY = NFFT/2;  // 504 cycles padding
  
  REPEAT: for (int ll=0; ll < loop_cnt; ll++) {
#pragma HLS LOOP_TRIPCOUNT min=1 max=8
    RUN_DEPTH: for (int dd=0; dd < DEPTH; dd++) {
#pragma HLS PIPELINE II=1
      sig_o.write(buff[dd]);
    }
  }
  
  RUN_LATENCY: for (int dd=0; dd < LATENCY; dd++) {
#pragma HLS PIPELINE II=1
    sig_o.write(TT_DATA(0));  // Zero-padding for pipeline latency
  }
}

Latency Padding 的物理意义：

这是理解整个 FFT 流水线时序的关键。注释明确说明了 latency 的来源：

// Push out dummy writes to account for latency involved in PL HLS data path:
//    pfa1008_permute_i:   NFFT/4+0 cycles  (252 cycles)
//    pfa1008_permute_o:   NFFT/4+0 cycles  (252 cycles)
//                Total:   NFFT/2+0 cycles  (504 cycles)

想象 FFT 流水线是一根长长的水管：

  • transmit() 是水源，开始注水
  • pfa1008_permute_i 是第一个弯道，水流经过需要 252 个周期
  • pfa1008_permute_o 是第二个弯道，再需 252 个周期
  • AIE 计算核在水管末端等待

如果不注入额外的 504 个零值"空泡"，当最后一帧真实数据进入水管后，水管内仍有正在处理的前序数据，AIE 会提前收到 EOF 信号，导致最后几帧数据被截断。这些零值就是确保所有有效数据都能完整流过整个管道的"护送者"。

循环重复机制：

loop_cnt 参数允许 Host 在不重新触发 DMA 的情况下，多次发送同一组数据。这对以下场景至关重要：

  • 算法验证：对比不同迭代次数下的输出一致性
  • 性能测试：测量持续吞吐量而非单次传输开销
  • 调试：隔离特定数据帧进行波形分析

LOOP_TRIPCOUNT min=1 max=8 是综合提示，帮助 HLS 估算资源用量，实际运行时可配置为任意正整数。

数据流全链路追踪

让我们跟踪一个样本从 DDR4 到 AIE 的完整旅程：

时钟周期 0:     Host 配置 loop_cnt=2，触发 kernel

周期 0-1007:    load_buffer() 从 DDR4 突发读取 1008 个 128-bit 字到 buff[]
                ↓ DATAFLOW 启动，transmit() 开始并行执行

周期 0:         transmit() 开始第 1 次循环 (ll=0)
周期 0-1007:    RUN_DEPTH 循环，sig_o.write(buff[dd]) 每周期输出 1 个样本
                → 下游 permute_i 接收并开始处理

周期 1008:      transmit() 开始第 2 次循环 (ll=1)
周期 1008-2015: 再次输出相同的 1008 个样本

周期 2016:      RUN_LATENCY 开始，输出 504 个零值
周期 2016-2519: sig_o.write(TT_DATA(0))

周期 2520:      kernel 完成，等待下一次 Host 触发

时序关键点：

  • load_buffer 和 transmit 的第一次循环在时间上重叠，因为 DATAFLOW 允许生产者-消费者并行
  • 由于 transmit 的输出速率（II=1）与 load_buffer 的读取速率相同，且 transmit 有额外 504 个周期的零值输出，load_buffer 实际上会在 transmit 完成前结束
  • 这意味着没有双缓冲乒乓机制——当前实现是单缓冲，下一帧必须等到当前帧完全发送后才能开始加载。如需连续流，需要扩展为 ping-pong 双缓冲。

设计权衡与决策分析

1. 单缓冲 vs 双缓冲

当前选择：单缓冲

TT_DATA buff[DEPTH];  // 仅一个缓冲区
load_buffer(mem, buff);
transmit(buff, sig_o, loop_cnt);

权衡分析：

方案 面积 (BRAM) 连续流传输 复杂度 适用场景
单缓冲 1008 × 128b ≈ 16 BRAM ❌ 帧间有间隙 低 验证/调试，非连续数据
双缓冲 (ping-pong) 32 BRAM ✅ 无缝衔接 中 生产环境，持续数据流

为何当前采用单缓冲？

查看整个 Prime Factor FFT 系统架构可以发现，pfa1008_dma_src 旁边还有 pfa1008_dma_snk（DMA Sink）。这是一个开环测试系统：

  • Source 发送已知测试向量
  • FFT 流水线处理
  • Sink 捕获结果并与黄金参考比对

在这种模式下，帧间的小间隙不影响功能正确性验证，而节省的 BRAM 资源可以用于其他逻辑。

2. Latency Padding 的计算依据

static constexpr int LATENCY = NFFT/2;  // 504

为什么是 NFFT/2 而不是其他值？这需要结合下游 pfa1008_permute_i 的实现理解：

在 pfa1008_permute_i.cpp 中：

static ap_uint<10> startup = 0; // Count off first block of "don't cares"
...
if (startup == ap_uint<10>(NFFT/4-1)) {  // 251
  running = 1;
}

permute_i 有一个 252 周期的启动预热期（startup），在此期间它从 buffer 读取但不输出（running==0）。同理 permute_o 也有 252 周期。两者串联形成 504 周期的端到端延迟。

关键洞察：transmit() 的 latency padding 必须与下游所有 kernel 的 startup delay 之和精确匹配，否则会导致：

  • Padding 不足：最后几帧数据被截断，Sink 捕获不完整
  • Padding 过量：浪费带宽，延长测试时间，但功能正确

3. 128-bit 数据通路的时钟域选择

注释提到两种等效方案：

  • 32-bit @ 1250 MHz (AIE 时钟)
  • 128-bit @ 312.5 MHz (PL 时钟)

为何选择后者？

  1. 时序收敛：312.5 MHz 对 PL 逻辑更容易满足时序，尤其是复杂的 permutation 逻辑
  2. 布线资源：更宽的数据通路减少信号数量，降低布线拥塞
  3. 功耗：低频运行降低动态功耗
  4. HLS 友好：ap_uint<128> 在 HLS 中有原生支持，而多路 32-bit 流需要显式并行化

4. restrict 关键字的缺失

注意到代码中没有使用 restrict 或 #pragma HLS DEPENDENCE：

void load_buffer(TT_DATA mem[DEPTH], TT_DATA (&buff)[DEPTH])
// 对比：void load_buffer(TT_DATA* restrict mem, TT_DATA* restrict buff)

影响分析：

  • mem 和 buff 是两个不同的数组，HLS 工具通常能推断出它们不别名
  • 但在复杂情况下（如通过指针算术访问），缺少 restrict 可能导致保守的依赖分析，限制并行度
  • 当前实现中，数组索引都是简单的线性递增（mm），HLS 可以安全地推断无别名

依赖关系与系统集成

上游调用者

根据依赖图，dma_source_kernel 由以下模块实例化和配置：

  • prime_factor_fft_system_integration：顶层系统集成，定义 PL DMA 端点与 AIE 图的连接
  • pl_dma_source_and_sink_endpoints：具体实例化配置，包括地址映射和流连接

下游调用链

pfa1008_dma_src_wrapper (this module)
    ↓ axis (128-bit stream)
pfa1008_permute_i (input side permutation)
    ↓ axis (128-bit stream)
pfa1008_permute_o (output side permutation)
    ↓ axis (32-bit × 4 parallel streams)
AIE FFT Kernels (DFT7, DFT9, DFT16)
    ↓
pfa1008_dma_snk (DMA Sink for result capture)

与 dma_sink_kernel 的对称关系

pfa1008_dma_src 和 pfa1008_dma_snk 构成一对源-汇镜像：

特性 Source (本模块) Sink (对应模块)
数据方向 DDR4 → AIE AIE → DDR4
主要操作 load_buffer() + transmit() capture_streams() + read_buffer()
Latency 处理 输出末尾加零值 (push) 跳过启动期的无效数据 (skip)
循环控制 loop_cnt (重复发送次数) loop_sel (选择捕获哪一次) + loop_cnt

Sink 模块的 loop_sel 参数允许选择性捕获某一次迭代的结果，这对于多轮测试中的特定帧分析非常有用。

使用指南与配置

HLS 编译配置 (hls.cfg)

[hls]
flow_target=vitis
clock=3.2ns                    # 312.5 MHz
syn.top=pfa1008_dma_src_wrapper
package.output.file=pfa1008_dma_src_wrapper.xo
package.output.format=xo

  • clock=3.2ns：对应 312.5 MHz，与 AIE 的 1250 MHz 形成 1:4 的时钟比，简化跨时钟域设计
  • vivado.flow=impl：执行 Out-of-Context 综合和实现，提供准确的时序和资源报告

边缘情况与注意事项

1. Loop Count 边界条件

// 危险：loop_cnt = 0
REPEAT: for (int ll=0; ll < loop_cnt; ll++)  // 不会执行任何循环

如果 Host 错误地配置 loop_cnt=0，kernel 会直接跳到 RUN_LATENCY 阶段，输出 504 个零值后结束。这不会产生错误，但会浪费时间和带宽。

2. 内存对齐要求

m_axi 接口假设 mem 指针指向的 DDR4 区域满足 AXI 协议的对齐要求（通常 64-byte 对齐）。未对齐访问可能导致：

  • 性能下降（非对齐突发被拆分为多个事务）
  • 某些平台上的总线错误

3. 数据持久性契约

load_buffer 完成后，mem 指向的 DDR4 数据即可被 Host 释放或修改，因为数据已被复制到内部 buff。但注意：这是单缓冲实现，如果在 transmit 完成前修改 mem，不会影响已加载的数据，但可能影响下一帧的加载（如果实现了双缓冲）。

4. 测试台验证策略 (tb_wrapper.cpp)

// 生成递增序列作为测试向量
for (int mm=0,dd=0; mm < DEPTH; mm++,dd+=4) {
   ddr4[mm]  = ( TT_SAMPLE(dd+3), TT_SAMPLE(dd+2), TT_SAMPLE(dd+1), TT_SAMPLE(dd+0) );
}

测试台使用简单的递增序列验证数据完整性。注意这里的打包顺序：dd+3 在最高位，dd+0 在最低位。这与 pfa1008_permute_i 中的解包顺序 (permute_i[3], permute_i[2], permute_i[1], permute_i[0]) 严格对应。

5. 时钟域交叉隐式处理

代码中没有显式的 CDC（Clock Domain Crossing）逻辑，因为：

  • m_axi 接口由 HLS 自动生成适配逻辑
  • axis 输出连接到同频（312.5 MHz）的下游 PL 逻辑
  • 到 AIE 的 1250 MHz 域跨越由 AXI Stream Interconnect 的异步 FIFO 处理

总结

dma_source_kernel 是一个看似简单但设计精巧的模块。它的核心价值不在于复杂的算法，而在于精确的数据搬运和时序控制。理解它的关键在于把握三点：

  1. 分层数据移动：DDR4 (burst) → BRAM (random) → AXIS (stream)，每一层解决不同的问题
  2. Latency Padding 的必要性：504 个零值不是浪费，而是保证流水线完整性的"护送者"
  3. 单缓冲的设计意图：在验证场景中优先节省资源，而非追求极致吞吐量

对于新加入团队的开发者，建议从修改 loop_cnt 和观察波形开始，直观感受 latency padding 对下游数据完整性的影响。
