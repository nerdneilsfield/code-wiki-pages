Emulation Waveform Analysis Pipeline 技术深度解析

一句话概括

这是一个教学性质的硬件仿真调试平台，通过将 PL（可编程逻辑）数据生成器、AI Engine FIR 滤波器和 DDR 存储控制器串联成一个完整的数据流水线，让开发者能够在 Vivado XSIM 波形查看器中实时观察 Versal SoC 内部各组件之间的信号交互——就像给芯片装上了一台"X光机"，让你能看到数据是如何从 PL 内核流向 AI Engine，再经过滤波处理后写回 DDR 的每一个时钟周期。

问题空间：为什么需要这个模块？

背景痛点

在 Versal 自适应 SoC 的开发中，开发者面临一个核心挑战：系统由多个异构计算单元组成（ARM CPU + 可编程逻辑 PL + AI Engine），它们通过复杂的 NoC（片上网络）和 AXI 总线互联。当设计出现问题时：

  1. 难以定位问题根源：是 PL 内核代码错误？AI Engine 核函数逻辑问题？还是数据在传输过程中丢失或损坏？
  2. 缺乏可视化手段：传统的软件打印日志无法捕获硬件级别的时序问题和总线事务细节。
  3. 软硬件协同调试困难：CPU 程序、PL HLS 内核、AI Engine 核函数三者运行在不同的执行域，它们的交互时机和顺序难以追踪。

解决方案

本模块提供了一个最小可运行的端到端参考设计，专门用于演示如何在硬件仿真（Hardware Emulation）环境中使用 Vivado XSIM 波形工具进行调试。它就像一个"显微镜下的标本"——足够简单以至于可以看清每一个细节，又足够完整以至于覆盖了典型的数据流场景。

核心抽象与心智模型

类比：医院检验科的工作流程

想象这个系统是一个医院的检验科：

  • random_noise 内核 = 血液采样机器人（PL 侧）

    • 负责持续不断地生成"样本"（随机噪声数据）
    • 通过专用管道（AXI4-Stream）将样本送往化验室

  • fir24_sym AI Engine 核函数 = 化验分析仪（AI Engine 侧）

    • 接收样本并进行"过滤分析"（FIR 滤波）
    • 可以通过"参数接口"（RTP - Run-Time Parameters）动态调整分析方法（切换窄带/宽带滤波器系数）
    • 处理后的结果通过另一条管道送出

  • s2mm 内核 = 档案归档员（PL 侧）

    • 接收化验结果并写入"病历档案库"（DDR 内存）
    • 使用 DMA 方式高效批量存储

  • Host 程序 = 科室主任

    • 启动整个流程
    • 中途可以下达指令更换分析方法（更新滤波器系数）
    • 最后读取档案库验证结果是否正确

关键抽象概念

概念 说明
Transaction Level Modeling (TLM) CIPS、NoC、AI Engine 等模块使用 SystemC 进行事务级建模，波形上显示的是高层事务而非门级信号
AXI4-Stream 单向数据流接口，用于 random_noise → AI Engine → s2mm 的数据通路
AXI4-Lite 控制寄存器接口，Host 通过它配置 PL 内核和 AI Engine 的参数
RTP (Run-Time Parameter) AI Engine 特有的运行时参数更新机制，允许 Host 在不重启图的情况下动态修改核函数参数
PLIO AI Engine Graph 中用于连接 PL 世界和 AI Engine 世界的端口适配器

端到端数据流详解

Phase 1: 系统启动与配置

    ↓ xrt::device(0), load_xclbin()

    ↓ 
Graph Initialization (ghdl = xrt::graph)
    ↓ ghdl.update("gr.fir24.in[1]", narrow_filter)

关键细节：

  • Host 使用 XRT (Xilinx Runtime) API 与设备交互
  • xrt::graph 对象代表 AI Engine 图实例，支持 init()、run()、wait()、end() 等生命周期管理
  • RTP 更新使用 update() 方法，指定目标端口路径（如 "gr.fir24.in[1]"）和数据

Phase 2: 数据生成与处理

random_noise kernel (PL)
    ↓ hls::stream<std::complex<short>>
Generates noise samples (-127 to 128 range)
    ↓ 
adaptive_graph (AI Engine PLIO)
    ↓ 
fir24_sym kernel (AI Engine)
    ↓ 
Applies 24-tap symmetric FIR filter
    ↓ 
adaptive_graph output PLIO
    ↓ 
s2mm kernel (PL)
    ↓ m_axi burst write
DDR Memory

关键细节：

  • random_noise 使用简单的线性同余发生器（LCG）生成伪随机数：next = next * 1103515245 + 12345
  • 数据类型为 std::complex<short>，实部和虚部分别占用 16 位
  • FIR 滤波器采用对称结构，12 个系数对应 24 抽头（利用系数对称性减少乘法器用量）
  • s2mm 以 II=1（Initiation Interval = 1）的速率将 stream 数据写入 DDR

Phase 3: 动态重配置（演示 RTP 特性）

Host
    ↓ ghdl.wait()  // 等待第一批次完成
    ↓ ghdl.update("gr.fir24.in[1]", wide_filter)
Switch to wide-band filter coefficients
    ↓ ghdl.run(16)
Process second batch with new coefficients

关键细节：

  • RTP 更新可以在图运行期间异步进行（async 端口修饰符）
  • 本例使用同步模式：先 wait() 确保前一阶段完成，再更新系数，然后重新启动
  • 系数回读端口（coefficients_readback）用于验证写入是否成功

组件详解

1. PL Kernels

random_noise —— 数据生成器

extern "C" void random_noise(hls::stream<std::complex<short> > & out, int size) {
#pragma HLS INTERFACE axis port=out           // AXI4-Stream 输出
#pragma HLS INTERFACE s_axilite port=return bundle=control  // 控制接口
#pragma HLS interface s_axilite port=size bundle=control    // 标量参数
    for(int i=0;i<size;i++){
        std::complex<short> sample;
        sample.real((my_rand() % 256) - 127);  // 范围: -127 ~ 128
        sample.imag((my_rand() % 256) - 127);
        out.write(sample);
    }
}

设计要点：

  • 使用 hls::stream 作为数据输出接口，映射到 AXI4-Stream
  • s_axilite 接口用于接收 Host 传来的 size 参数和控制信号
  • 自定义 my_rand() 避免标准库依赖，确保可综合性和确定性行为

s2mm —— Stream-to-Memory 转换器

void s2mm(ap_int<32>* mem, hls::stream<ap_axis<32, 0, 0, 0> >& s, int size) {
#pragma HLS INTERFACE m_axi port=mem offset=slave bundle=gmem  // DDR 访问
#pragma HLS interface axis port=s                              // Stream 输入
#pragma HLS PIPELINE II=1                                      // 每周期处理一个样本
    for(int i = 0; i < size; i++) {
        ap_axis<32, 0, 0, 0> x = s.read();
        mem[i] = x.data;
    }
}

设计要点：

  • m_axi 接口直接访问 DDR，offset=slave 表示地址由 Host 传入
  • ap_axis<32, 0, 0, 0> 是 AXI4-Stream 协议的结构化类型，32 位数据宽度
  • PIPELINE II=1 确保每个时钟周期处理一个样本，达到最大吞吐

2. AI Engine Components

adaptive_graph —— AI Engine 图定义

class adaptive_graph : public graph {
public:
    input_plio in;                    // PL 输入接口
    port<direction::in> coefficients; // RTP 输入端口（系数更新）
    port<direction::inout> coefficients_readback; // RTP 回读端口
    output_plio dataout;              // PL 输出接口
    kernel fir24;                     // FIR 核函数实例

    adaptive_graph() {
        fir24 = kernel::create(fir24_sym);
        
        // PLIO 创建：名称、位宽、仿真文件
        in = input_plio::create("Datain0", plio_32_bits, "data/input.txt");
        dataout = output_plio::create("Dataout0", plio_32_bits, "data/output.txt");
        
        runtime<ratio>(fir24) = 0.5;    // 内核占用 50% 计算资源
        source(fir24) = "kernels/hb24.cc";
        
        // 连接关系定义
        connect<parameter>(coefficients, async(fir24.in[1]));
        connect<parameter>(async(fir24.inout[0]), coefficients_readback);
        connect<>(in.out[0], fir24.in[0]);
        connect<>(fir24.out[0], dataout.in[0]);
    }
};

设计要点：

  • input_plio/output_plio 是 AI Engine 与 PL 世界的桥梁，自动处理时钟域转换和数据格式转换
  • runtime<ratio> 设置内核的计算资源占用率，0.5 表示使用半个 AI Engine tile
  • async() 修饰符允许 RTP 端口进行异步数据传输

fir24_sym —— 对称 FIR 滤波器核函数

void fir24_sym(input_buffer<cint16,extents<64>,margin<24>> &iwin,
               output_buffer<cint16,extents<64>> &owin,
               const int32 (&coeffs)[12],
               int32 (&coeffs_readback)[12])

算法核心：

  • 使用 AIE API 的向量操作实现高效 FIR 计算
  • sliding_mul_sym 和 sliding_mac_sym 利用 AI Engine 的 SIMD 乘累加单元
  • 输入缓冲区带有 24 样本的 margin（重叠区），支持滑动窗口处理
  • 每轮循环处理 16 个输入样本，产生 16 个输出样本

性能特征：

  • 向量化宽度：4 和 8（aie::begin_vector_random_circular<4/8>）
  • 累加器精度：cacc48（48 位复数累加器）
  • 输出缩放：右移 9 位（to_vector<cint16>(9)）

设计权衡与决策分析

1. 为什么选择最简单的 FIR 滤波器作为示例？

权衡：教学清晰度 vs. 实际应用复杂度

决策：使用 24 抽头对称 FIR，原因如下：

  • 算法简单易懂，便于理解数据流
  • 足够展示 RTP 动态重配置特性
  • 计算负载适中，不会成为性能瓶颈
  • 有明确的数学预期（golden 参考数据可用于验证）

2. 为什么使用独立的 random_noise PL 内核而不是从文件读取？

权衡：可控性 vs. 真实性

决策：使用伪随机数生成器，原因如下：

  • 无需外部文件 I/O，简化部署
  • 可重复（固定 seed），便于调试和问题复现
  • 可持续生成任意数量的样本
  • 展示了 PL 内核作为数据源的典型用法

3. 为什么 RTP 端口使用 async 修饰符？

权衡：灵活性 vs. 确定性

决策：使用异步 RTP，原因如下：

  • 允许 Host 在图运行期间动态注入新系数
  • 符合实际信号处理系统中"热更新"滤波器系数的场景
  • 虽然本例使用了同步模式（wait→update→run），但保留了异步能力

4. 为什么 s2mm 使用 ap_axis<32, 0, 0, 0> 而不是 std::complex<short>？

权衡：硬件精确性 vs. 代码可读性

决策：使用原始 AXI 流类型，原因如下：

  • 明确控制 sideband 信号（TKEEP、TSTRB、TLAST 等）
  • <32, 0, 0, 0> 表示 32 位数据，无用户信号，无 keep/strobe
  • 与 AI Engine 输出的 32 位打包数据格式精确匹配

新贡献者注意事项

常见陷阱

  1. 波形查看时机

    • 必须在仿真开始前添加所有要观察的信号，仿真开始后添加的信号不会有历史数据
    • 使用 add_waveforms.tcl 脚本预先配置好常用信号组

  2. TLM 信号解读

    • TLM 波形显示的是事务块，不是单个时钟周期
    • 需要放大（zoom in）才能看到具体的事务细节
    • 颜色块宽度不代表单个事务时长，而是时间轴压缩后的显示效果

  3. RTP 更新时序

    • graph.update() 是异步操作，不保证立即生效
    • 如需确认更新已应用，应调用 graph.wait() 或使用回读端口验证

  4. 数据格式转换

    • AI Engine 使用 cint16（16 位实部 + 16 位虚部）
    • PL 内核使用 std::complex<short> 或 ap_int<32>
    • 两者在总线上都是 32 位打包格式，但语义解释不同

调试技巧

  1. 验证数据通路完整性

    # 在 XSIM Tcl 控制台执行
source ../../../../tcl/pl_to_aie.tcl
source ../../../../tcl/aie_to_ddr.tcl

  2. 检查 RTP 写入

    • 关注 AIENGINE/S00_AXI 接口的写事务
    • 预期看到 12 个连续的写事务（对应 12 个系数）
    • 数据值应与 Host 代码中的 narrow_filter/wide_filter 数组一致

  3. 性能分析

    • 使用 Vitis Analyzer 打开生成的 .vcd 文件
    • 查看 AI Engine 内核的 stall 情况
    • 检查 PL 内核的 II（Initiation Interval）是否符合预期

扩展建议

如需基于此设计开发实际应用：

  1. 替换数据源：将 random_noise 替换为实际的 ADC 数据采集逻辑
  2. 增加处理链：在 FIR 后级联更多 AI Engine 内核（FFT、均衡器等）
  3. 优化系数更新：实现基于反馈的自适应算法（LMS/RLS）动态调整系数
  4. 多通道扩展：复制数据通路实现多通道并行处理
