External Traffic Generator PL Kernel Configs 模块深度解析

一句话概括

本模块是 Versal ACAP 平台上 外部流量生成器(External Traffic Generator, ETG) 与 AI Engine 集成的参考实现，通过 PL(Programmable Logic) 侧的 HLS 内核实现数据搬移(mm2s/s2mm)和信号处理(polar_clip)，为 AI Engine 应用提供可编程的数据源和验证手段。

问题空间：我们为什么要做这个？

在传统的 SoC 开发中，验证 AI Engine 设计的最大痛点是 "鸡生蛋蛋生鸡" 的困境：

  1. 硬件未就绪时无法验证算法：AI Engine 应用需要外部数据输入（如 ADC 采样、网络包、传感器数据），但开发阶段往往没有真实的数据源硬件。

  2. 仿真速度慢到无法接受：纯软件仿真(VPSS)虽然可以验证功能，但对于包含数百个 AI Engine tile 的设计，仿真速度可能慢到实时交互成为不可能。

  3. 硬件验证依赖外部设备：一旦进入硬件验证阶段，如果没有配套的外部数据源设备，就无法向 AI Engine 注入测试激励。

外部流量生成器(ETG) 正是为解决这些问题而生。它允许在 硬件仿真(HW Emulation) 模式下，使用运行在 x86 主机上的软件生成数据，通过 PL 侧的 DMA 内核将数据注入 AI Engine 图；同时也可以在 硬件(HW) 模式下使用真实的外部数据源。

本模块提供了 ETG 架构中 PL 侧的内核实现，包括：

  • Ingress DMA (mm2s)：从 DDR 内存读取数据，转换为 AXI Stream 输出
  • Polar Clip 处理核：对复数信号执行 CFR(Crest Factor Reduction) 峰值削减
  • Egress DMA (s2mm)：将 AXI Stream 输入写入 DDR 内存

核心抽象：如何理解这个模块？

类比：数据中心的水泵站

想象一个城市供水系统：

  1. 水源(DDR 内存)：就像水库，存储着大量的原始数据（水）。

  2. 进水泵站(mm2s)：从水库抽水，通过管道(AXI Stream)输送到用户(AI Engine)。泵站需要控制流量，不能一下子把水全部倒出来，要持续稳定地供应。

  3. 水处理厂(polar_clip)：用户收到的水可能有杂质（信号峰值过高），需要在这里进行过滤/软化处理，确保输出在安全范围内。

  4. 排水泵站(s2mm)：用户使用后的废水（处理后的数据）通过管道回流，排水泵站将其泵回污水处理厂或排放到下游。

关键洞察：PL 内核在这里扮演的是 数据搬运工和预处理工 的角色，它们本身不实现复杂的 AI 算法，但为 AI Engine 提供了干净、格式正确的数据流接口。

数据流抽象：AXI Stream 协议

本模块中的所有内核都使用 AXI4-Stream 协议进行通信。可以把 AXI Stream 想象成一条 单向传送带：

  • TDATA：传送带上的货物（32位数据，本模块中通常是 16-bit I + 16-bit Q 复数）
  • TVALID：送货人说"我有货"
  • TREADY：接收人说"我准备好收"
  • TLAST：这是这批货的最后一个（标记帧边界）

只有当 TVALID 和 TREADY 同时为高时，数据才被视为成功传输（"握手"成功）。这种握手机制天然地实现了 背压(back-pressure) 控制：如果下游处理速度慢，TREADY 会持续拉低，上游就会暂停发送，避免数据丢失。

本模块包含三个核心子模块，它们通过 AXI Stream 接口连接，形成完整的数据通路。

数据流通路详解

阶段 1：DDR → mm2s (Ingress DMA)

  • 接口：AXI4 Memory-Mapped (主) + AXI4-Stream (从)
  • 功能：从 DDR 内存的指定地址读取数据，转换为 AXI Stream 格式输出
  • 关键参数：DATA_WIDTH=32 定义了数据总线宽度为 32 位
  • HLS 配置：
    syn.top=mm2s
syn.file_cflags=mm2s.cpp,-DDATA_WIDTH=32

阶段 2：mm2s → polar_clip (信号预处理)

  • 输入：hls::stream<ap_axis<32, 0, 0, 0>> - 32 位 AXI Stream
  • 数据格式：低 16 位为实部(I)，高 16 位为虚部(Q) 的复数样本
  • 处理逻辑：
    1. 使用 CORDIC 算法 计算复数模值和相位
    2. 与阈值 CFR_THRESHOLD (11626) 比较
    3. 若超过阈值，削减信号幅度（峰值削减）
    4. 输出处理后的复数样本

阶段 3：polar_clip → s2mm (Egress DMA)

  • 功能：接收 AXI Stream 数据，执行突发写入(Burst Write)到 DDR 内存
  • 接口：AXI4-Stream (主) + AXI4 Memory-Mapped (从)
  • 配置：与 mm2s 对称，同样使用 32 位数据宽度

内核间的握手与反压

所有三个内核通过 AXI Stream 的 TVALID/TREADY 握手机制实现同步：

  1. mm2s 从 DDR 读取数据后，置起 tvalid，等待下游 tready
  2. polar_clip 只有在输入 tvalid 为高且自身准备好处理时，才置起输入 tready 和输出 tvalid
  3. s2mm 只有在 DDR 写入通道可用且接收到有效数据时，才置起 tready

这种机制确保了如果 polar_clip 处理速度慢于 mm2s 的读取速度，反压信号会通过 tready 传递回 mm2s，自动降低数据读取速率，避免 FIFO 溢出。

设计决策与权衡分析

1. HLS vs RTL 设计选择

决策：使用 Vitis HLS (C/C++) 而非手工编写 RTL (Verilog/VHDL) 实现 PL 内核。

权衡分析：

维度 HLS (当前选择) RTL (替代方案)
开发效率 高 - C++ 抽象，快速迭代 低 - 手动管理时序、流水线
性能天花板 中等 - 依赖 HLS 工具优化 高 - 手工优化可达理论极限
面积效率 中等 - 通用性牺牲面积 高 - 精确控制资源使用
调试难度 中等 - C 仿真 + 联合仿真 高 - 波形分析为主
可移植性 高 - 跨器件重编译 低 - 器件依赖强

为何选择 HLS： 本模块作为 Feature Tutorial，首要目标是展示 ETG 与 AIE 的集成方法，而非追求极致的 PL 性能。HLS 允许开发者专注于算法逻辑（如 CORDIC 实现），而非底层的流水线时序控制，显著降低了学习门槛。

2. 32-bit 数据宽度选择

决策：使用 DATA_WIDTH=32 作为 AXI Stream 数据宽度。

权衡分析：

本模块处理的是复数信号（I/Q 数据），每个样本包含 16-bit 实部和 16-bit 虚部，共 32-bit。选择 32-bit 数据宽度的原因：

  • 自然对齐：一个 AXI Stream beat 正好携带一个复数样本，无需处理跨 beat 的数据拼接逻辑。
  • 吞吐量匹配：对于典型的信号处理应用（如 100MSPS 采样率），32-bit @ 300MHz 时钟可提供 9.6Gbps 带宽，远超需求。
  • 资源效率：相比 64-bit 或 128-bit，32-bit 数据路径消耗的 DSP slice 和布线资源更少。

潜在限制： 如果处理的是更高精度的数据（如 24-bit I/Q），32-bit 会显得捉襟见肘，需要扩展到 64-bit 或采用自定义打包格式。

3. 纯 PL 处理 vs AIE+PL 协作

决策：将信号处理逻辑（polar_clip）放在 PL 侧而非 AIE 侧。

权衡分析：

维度 PL 处理 (当前选择) AIE 处理 (替代方案)
计算密度 中等 - LUT/DSP 实现 高 - 向量 SIMD 单元
数据局部性 高 - 直接连接 DMA 中等 - 需通过 PL 路由
灵活性 高 - 支持复杂控制流 中等 - 数据流架构限制
开发难度 低 - HLS C++ 中等 - AIE 特定 API
功耗 较高 - 通用逻辑 较低 - 专用 DSP 架构

为何选择 PL： polar_clip 是一个控制流相对复杂的算法（涉及 CORDIC 迭代、阈值比较、条件分支），且数据吞吐量相对适中。使用 HLS 在 PL 侧实现，可以利用成熟的 C/C++ 工具链，便于调试和验证。此外，本模块作为教程，展示 PL-AIE 集成比单纯展示 AIE 计算更有教学价值。

何时选择 AIE： 如果处理的是高吞吐量（>1GSPS）、向量化程度高的算法（如大规模 FFT、矩阵乘法），AIE 的 SIMD 向量单元将提供远超 PL 的计算效率。

4. 静态配置 vs 动态重配置

决策：使用编译时静态配置（如 CFR_THRESHOLD 宏定义）而非运行时寄存器配置。

权衡分析：

当前 polar_clip 内核使用宏定义配置关键参数：

#define CFR_THRESHOLD (11626)

优势：

  • 资源效率：常量在编译期确定，HLS 工具可以进行激进的常量传播和死代码消除，生成的硬件逻辑更精简。
  • 时序优化：消除运行时比较和分支，可能实现更高的时钟频率。
  • 确定性：行为在编译期完全确定，避免运行时配置错误。

劣势：

  • 灵活性缺失：修改阈值需要重新编译整个内核，无法根据运行场景动态调整。
  • 多实例困难：如果同一设计中需要不同参数的多路实例，必须维护多份源代码。

替代方案： 使用 AXI4-Lite 接口 (#pragma HLS INTERFACE s_axilite) 暴露配置寄存器，允许主机通过 XRT API 动态修改参数。适合需要根据信道条件实时调整 CFR 门限的应用。

当前选择原因： 作为 Feature Tutorial，保持实现简洁是首要目标。静态配置减少了接口复杂性，让开发者专注于核心算法（CORDIC 和 CFR 逻辑）而非寄存器映射设计。

关键实现细节与注意事项

1. CORDIC 算法实现细节

polar_clip.cpp 中的 cos_sin_mag 函数使用 CORDIC (COordinate Rotation DIgital Computer) 算法计算复数的幅度和相位。这是数字信号处理中的经典算法，完全由移位和加法操作组成，特别适合 FPGA 硬件实现。

实现特点：

  • 迭代结构：使用 6 步迭代 (nsteps=6)，在精度和延迟之间取得平衡。
  • 查找表优化：使用预计算的 rotation_cos_lut 和 rotation_sin_lut 存储旋转角度，避免运行时计算 arctan。
  • 纯整数运算：使用 ap_int<32> 定点数，无浮点单元消耗，节省 DSP 资源。

关键参数：

  • P_SCALE = 512：幅度计算的比例因子。
  • SHIFT_BITS = 9：右移位数，用于归一化。
  • CFR_THRESHOLD = 11626：峰值削减的门限值，超过此幅度的样本将被削减。

2. 复数数据打包格式

PL 内核间传输的复数样本采用 32-bit 打包格式：

// 输入解析
value_real = short(in_x.data & 0xFFFF);           // 低 16-bit: 实部(I)
value_imag = short((in_x.data >> 16) & 0xFFFF);   // 高 16-bit: 虚部(Q)

// 输出打包
out_x.data = ( (( ((short)ovalue.real) & 0x0000FFFF) << 0) | 
               (( ((short)ovalue.imag) & 0x0000FFFF) << 16) );

设计考虑：

  • 对齐友好：32-bit 正好是 AXI4-Stream 接口的常见数据宽度，单个 beat 传输一个样本，无需跨 beat 组装。
  • 精度权衡：16-bit I/Q 是通信系统的标准位宽，提供足够的动态范围（~90dB）同时节省带宽。
  • 符号扩展：从 ap_int<16> 扩展到 ap_int<32> 进行计算时，需要注意符号位的正确扩展，避免无符号扩展导致的错误。

3. AXI Stream 接口配置

所有 PL 内核都使用 AXI4-Stream 协议进行数据交换，HLS 中的接口配置如下：

// polar_clip.cpp 接口配置
#pragma HLS INTERFACE ap_ctrl_none port=return    // 无块级控制协议，自由运行
#pragma HLS INTERFACE axis port=out_sample        // AXI Stream 输出
#pragma HLS INTERFACE axis port=in_sample         // AXI Stream 输入

关键设计决策：

  1. ap_ctrl_none：内核以 自由运行(free-running) 模式工作，没有启动/停止控制。只要有输入数据，内核就会持续处理。这适合数据流架构，避免了控制开销。

  2. AXIS 接口：使用 hls::stream<ap_axis<32, 0, 0, 0>> 类型，其中：

    • 32：数据位宽
    • 0：用户位宽 (TUSER)
    • 0：保持位宽 (TKEEP)
    • 0：目的地位宽 (TDEST)

  3. 隐式协议：虽然代码中没有显式操作 tvalid 和 tready，HLS 工具会自动生成符合 AXI Stream 协议的 RTL，包含正确的握手机制。

4. HLS 编译配置分析

每个内核都有对应的 .cfg 配置文件，指导 Vitis HLS 的编译行为：

mm2s.cfg / s2mm.cfg：

[hls]
flow_target=vitis              # 目标流程为 Vitis 集成流程
syn.file=mm2s.cpp              # 源文件
syn.cflags=-I.                 # 包含路径
syn.file_cflags=mm2s.cpp,-DDATA_WIDTH=32   # 文件特定宏定义
syn.top=mm2s                   # 顶层函数
package.ip.name=mm2s           # 生成的 IP 名称
package.output.syn=true        # 输出综合结果
package.output.format=xo       # 输出格式为 Xilinx Object (.xo)
package.output.file=mm2s.xo    # 输出文件名

关键配置解读：

  1. flow_target=vitis：区别于纯 HLS 流程，Vitis 流程生成的 .xo 文件可直接被 Vitis Linker (v++ --link) 集成到平台设计中，与 AI Engine 图进行连接。

  2. syn.file_cflags：通过 -DDATA_WIDTH=32 在编译期定义数据宽度常量，允许同一份 C++ 代码通过不同配置生成不同位宽的内核实例，提高代码复用性。

  3. package.output.format=xo：.xo 格式是 Vitis 平台的内核容器，包含 RTL 网表、元数据、接口定义等，是连接 HLS 世界和系统集成世界的桥梁。

5. 模块边界与依赖关系

本模块是更大系统的一部分，其边界和依赖如下：

本模块提供的接口：

  • mm2s.xo：PL kernel 对象，提供 s_axis 输出接口
  • polar_clip.xo：PL kernel 对象，提供 s_axis 输入和输出接口
  • s2mm.xo：PL kernel 对象，提供 s_axis 输入接口

本模块依赖的上游模块（外部定义）：

  • AI Engine 图定义：本模块假设存在某个 AI Engine 图（通常在 aie/ 目录中定义），该图包含输入/输出 PLIO 端口，用于接收来自 mm2s 的数据，并向 s2mm 发送数据。
  • 外部流量生成器(ETG)软件：在 HW Emulation 模式下，需要主机端的 xtg_aie.py 脚本或类似工具驱动数据生成。
  • XRT 运行时：硬件执行时需要 Xilinx Runtime 来管理内核启动和内存分配。

隐式契约：

  1. 时钟域：所有 PL 内核假设运行在相同的时钟域（通常由 Vitis 平台自动分配，如 100MHz 或 300MHz）。
  2. 复位策略：采用 ap_ctrl_none 意味着内核无显式复位端口，依赖全局复位网络。
  3. 缓冲区对齐：mm2s 和 s2mm 假设 DDR 缓冲区至少对齐到 4KB 边界（通常由 XRT 分配器保证）。

子模块组织

本模块包含三个逻辑子模块，每个对应一个特定的 PL 内核功能：

1. Ingress DMA Traffic Source Kernel (子模块文档)

职责：实现内存到流(Memory-to-Stream)的数据搬移，作为外部流量生成器的硬件端点。

核心组件：mm2s HLS 内核及其配置文件。负责从 DDR 读取数据，转换为 AXI4-Stream 协议输出。

2. Polar Clip Processing Kernel and Types (子模块文档)

职责：执行数字预失真(DPD)中的 CFR(Crest Factor Reduction) 算法，使用 CORDIC 进行极坐标变换和峰值削减。

核心组件：polar_clip HLS 内核、复数类型定义、CORDIC 算法实现。这是本模块中唯一的"有状态计算"内核，而非单纯的数据搬移。

3. Egress DMA Traffic Sink Kernel (子模块文档)

职责：实现流到内存(Stream-to-Memory)的数据搬移，接收处理后的数据并写回 DDR。

核心组件：s2mm HLS 内核及其配置文件。与 mm2s 对称，完成数据通路的闭环。

使用模式与集成指南

典型使用流程

在 Vitis 设计流程中，本模块的内核通常按以下方式集成：

  1. 编译内核：使用 vitis_hls 或 v++ -c 编译每个 .cpp 文件，生成 .xo 文件：

    v++ -c -t hw_emu -k mm2s mm2s.cpp -o mm2s.xo
v++ -c -t hw_emu -k polar_clip polar_clip.cpp -o polar_clip.xo
v++ -c -t hw_emu -k s2mm s2mm.cpp -o s2mm.xo

  2. 链接到平台：使用 v++ -l 将内核链接到目标平台，与 AI Engine 图进行连接：

    v++ -l -t hw_emu mm2s.xo polar_clip.xo s2mm.xo aie_graph.xo -o system.xclbin

    链接阶段会解析配置文件中的接口定义，确保 mm2s 的输出连接到 polar_clip 的输入，polar_clip 的输出连接到 AIE 图的输入。

  3. 主机程序：使用 XRT API 分配缓冲区、启动内核：

    // 分配输入/输出缓冲区
xrt::bo in_buf = xrt::bo(device, size, mm2s.group_id(0));
xrt::bo out_buf = xrt::bo(device, size, s2mm.group_id(0));

// 启动内核（ap_ctrl_none 模式下实际是自动启动，但 XRT 仍需调用 run()）
mm2s.run();
polar_clip.run();
s2mm.run();

配置参数调整

本模块的关键可配置参数包括：

  1. CFR_THRESHOLD (polar_clip.cpp)：

    • 默认：11626
    • 调整建议：根据输入信号的 PAPR(Peak-to-Average Power Ratio) 调整。较高的阈值允许更大的峰值通过，较低的阈值更激进地削减峰值但可能增加 EVM(Error Vector Magnitude)。

  2. DATA_WIDTH (mm2s.cfg, s2mm.cfg)：

    • 当前：32（对应 16-bit I/Q）
    • 扩展选项：可改为 64 支持 32-bit I/Q 或双通道并行传输。

  3. CORDIC 迭代次数 (polar_clip.cpp)：

    • 当前：nsteps=6
    • 权衡：更多迭代提高相位精度，但增加延迟和资源消耗。对于 CFR 应用，6 步迭代通常提供足够的精度（~0.01 rad）。

潜在陷阱与调试指南

1. AXI Stream 协议违规

症状：仿真挂起、数据丢失、TLAST 信号错位。

常见原因：

  • 握手违规：在 tvalid 为高时改变 tdata，或在 tready 为低时拉低 tvalid（违反 AXI Stream 协议）。
  • TLAST 丢失：帧的最后一个样本未正确标记 tlast，导致下游内核无法识别帧边界。
  • 死锁：两个内核互相等待对方（如上游等待 tready，下游等待 tvalid）。

调试建议：

  • 使用 Vivado Simulator 或 Vitis Analyzer 查看波形，重点关注 tvalid、tready、tdata、tlast 信号。
  • 检查 polar_clip 中的 in_sample.read() 和 out_sample.write() 调用是否成对出现，避免阻塞。

2. CORDIC 精度与溢出

症状：CFR 后的信号出现明显失真、EVM 恶化或意外削波。

常见原因：

  • 溢出：ap_int<32> 在计算 value_real * value_real 时，如果输入接近 16-bit 满量程（32767），平方结果约为 1e9，接近 32-bit 有符号整数上限（2.147e9）。如果输入信号增益较大，可能发生溢出。
  • LUT 索引越界：rotation_index 计算错误可能导致访问 rotation_cos_lut 越界。
  • 定点精度损失：CORDIC 的移位操作引入的舍入误差累积。

调试建议：

  • 在 HLS 仿真阶段使用 printf 或 std::cout 打印关键中间变量（mag_sq、rotation_index、magout），验证数值范围。
  • 使用 ap_fixed（定点数）替代 ap_int 进行算法验证，确认数值范围后再转换为整数优化。
  • 检查输入信号的统计特性（峰值、RMS），确保 CFR_THRESHOLD 设置合理。

3. 内存对齐与缓冲区大小

症状：mm2s 或 s2mm 出现非法内存访问、DMA 错误中断、数据错位。

常见原因：

  • 对齐违规：XRT 分配的缓冲区通常需要至少 4KB 对齐。如果主机程序使用 malloc 而非 xrt::bo 分配内存，可能导致 DMA 无法正确访问。
  • 大小不匹配：mm2s 配置的传输大小与实际缓冲区大小不匹配，导致读取越界或提前结束。
  • 缓存一致性：主机与设备之间未正确同步缓存（bo.sync(XCL_BO_SYNC_BO_TO_DEVICE)），导致 DMA 读取到旧数据。

调试建议：

  • 始终使用 XRT 的 Buffer Object API (xrt::bo) 分配设备内存，确保满足对齐要求。
  • 在 Host 代码中显式调用 sync 进行主机到设备（H2D）和设备到主机（D2H）的数据同步。
  • 在仿真波形中监控 AXI4-MM 接口的 araddr（读地址）和 awaddr（写地址），确认访问范围在分配的缓冲区之内。

4. ap_ctrl_none 的隐式契约

症状：内核似乎不工作、输出始终为零、仿真波形中没有活动。

原因分析： 本模块的所有内核都使用 #pragma HLS INTERFACE ap_ctrl_none port=return，这意味着：

  • 内核 没有块级控制信号（如 ap_start、ap_done、ap_idle）。
  • 内核在复位后开始自动运行，持续处理数据流。
  • XRT 的 run() 方法对这种内核实际上只是增加引用计数，真正的启动依赖于数据流到达。

潜在问题：

  • 如果上游数据源（如 mm2s）因为 DDR 读取问题没有数据输出，下游的 polar_clip 会在 in_sample.read() 处阻塞等待。
  • 在纯仿真环境中，如果主机程序没有正确初始化并启动数据注入，整个管道会呈现"死锁"状态。

调试建议：

  • 确保在 Host 程序中按照正确顺序初始化所有内核（虽然 ap_ctrl_none 下 run() 是轻量的，但仍需调用）。
  • 在波形中检查 mm2s 的 AXI4-MM 读通道活动，确认有数据从 DDR 读出。
  • 在 polar_clip 中添加 ap_wait() 或轻量级调试输出（仅在仿真中）确认执行到达关键路径。

扩展与定制指南

添加新的处理内核

如果需要在本模块中添加额外的信号处理步骤（如滤波、增益控制），建议遵循以下模式：

  1. 接口一致性：新内核应使用与 polar_clip 相同的 AXI Stream 接口定义：

    void new_kernel(hls::stream<ap_axis<32, 0, 0, 0>> &in_sample,
                hls::stream<ap_axis<32, 0, 0, 0>> &out_sample)

  2. 控制协议：为保持与现有内核一致，使用 ap_ctrl_none 实现自由运行模式。

  3. 数据格式：保持 32-bit 打包的 16-bit I/Q 格式，确保与上下游兼容。

  4. 配置机制：如果新内核需要可配置参数，参考现有的宏定义方式（编译期静态配置）或考虑添加 AXI4-Lite 接口（运行时动态配置）。

集成到更大的系统

本模块的内核通常与 AI Engine 图集成。典型的系统集成步骤：

  1. 定义 AIE 图接口：在 AIE 图定义（通常使用 ADF/C++）中声明 PLIO 端口，指定位宽（32-bit）和 PL 内核连接点。

  2. Vitis 链接配置：使用 system.cfg 或命令行参数指定连接关系：

    [connectivity]
nk=mm2s:1:mm2s_1
nk=polar_clip:1:polar_clip_1
nk=s2mm:1:s2mm_1
stream_connect=mm2s_1.s:polar_clip_1.s
stream_connect=polar_clip_1.s:aie_input.plin
stream_connect=aie_output.plout:s2mm_1.s

  3. 内存分配：在 Host 代码中使用 XRT API 分配与 mm2s 和 s2mm 连接的 DDR 缓冲区，并确保缓冲区大小、对齐方式符合内核预期。

  4. 执行流程：

    • 准备输入数据并写入 DDR 缓冲区
    • 同步缓冲区到设备（H2D）
    • 启动所有 PL 内核（run()）
    • 等待处理完成（通过中断或轮询）
    • 同步结果回主机（D2H）
    • 验证输出数据

总结

external_traffic_generator_pl_kernel_configs 模块作为 Versal ACAP 平台外部流量生成器架构的关键组成部分，提供了从主机内存到 AI Engine 的完整数据通路实现。通过 mm2s、polar_clip 和 s2mm 三个 HLS 内核的协作，该模块展示了：

  1. PL-AIE 集成模式：如何通过 AXI Stream 接口在 PL 可编程逻辑和 AI Engine 阵列之间建立高效的数据传输通道。

  2. HLS 设计最佳实践：从接口定义（ap_ctrl_none、axis）到编译配置（xo 生成），展示了 Vitis HLS 工具链的典型使用模式。

  3. 信号处理算法硬件化：通过 CORDIC 算法的定点实现，展示了如何将复杂的数学运算（极坐标转换、幅度计算）映射到高效的硬件逻辑。

对于新加入团队的开发者，理解本模块的关键在于把握 "数据流" 和 "接口契约" 两个核心概念。数据在模块中的流动是单向、连续的，由 AXI Stream 的握手机制自然调节流速；每个内核都有明确的输入输出接口定义，这些接口是系统集成时的契约，任何修改都需要在系统层面（如 system.cfg）进行相应调整。

该模块作为 Feature Tutorial，其价值不仅在于提供可运行的代码，更在于展示了一种可复制的架构模式：如何在外部主机和 AI Engine 之间建立桥梁，让数据能够自由流动，让算法能够高效执行。
