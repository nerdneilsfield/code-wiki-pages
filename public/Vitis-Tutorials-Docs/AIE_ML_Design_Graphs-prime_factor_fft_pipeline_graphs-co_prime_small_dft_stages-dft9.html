dft9 模块技术深度解析

概述

dft9 模块是 AMD Versal AIE-ML 架构上的一个9点离散傅里叶变换（DFT）加速器，专为素因子 FFT（Prime Factor Algorithm, PFA）流水线设计。想象一下，它就像一条精密装配线上的一个工作站——接收来自上游的原始信号数据，执行高度优化的矩阵乘法运算，然后将中间结果传递给下游组件进行进一步处理。

这个模块的核心价值在于：将数学上复杂的 9点 DFT 计算分解为可在 AIE-ML 向量处理器上高效执行的 SIMD 操作。通过利用 AIE-ML 的 mmul（矩阵乘法）指令和级联（cascade）数据流架构，它实现了比传统 Cooley-Tukey FFT 更高效的计算密度，特别是在处理非 2 的幂次长度的变换时。

问题空间与设计动机

为什么需要专门的 9点 DFT？

在数字信号处理中，FFT 长度通常选择为 2 的幂次（如 1024、2048），因为基-2 FFT 算法实现简单。然而，这种选择往往并非最优：

  • 资源浪费：当应用需要 1000 点变换时，必须向上取整到 1024，造成约 2.4% 的计算浪费
  • 延迟增加：额外的零填充增加了不必要的处理延迟
  • 内存占用：更大的缓冲区需求

素因子算法（PFA） 提供了一种替代方案：如果变换长度 \(N\) 可以分解为互质的因子 \(N = N_1 \times N_2 \times ... \times N_k\)，则可以将一维 DFT 分解为多维 DFT。对于本教程中的 PFA-1008 变换：

\[1008 = 16 \times 9 \times 7\]

其中 16、9、7 两两互质，因此可以构建一个三级流水线，每级处理一个因子。

为什么 9点 DFT 需要特殊处理？

9 不是 2 的幂次，无法直接使用基-2 蝶形运算。传统的 DFT 定义：

\[X[k] = \sum_{n=0}^{8} x[n] \cdot e^{-j2\pi kn/9}, \quad k = 0, ..., 8\]

直接实现需要 81 次复数乘法和 72 次复数加法。但观察发现，这本质上是一个矩阵-向量乘法：

\[\mathbf{X} = \mathbf{W} \cdot \mathbf{x}\]

其中 \(\mathbf{W}\) 是 \(9 \times 9\) 的旋转因子矩阵。这正是 AIE-ML 的 mmul 指令擅长的工作。

核心抽象与心智模型

理解 dft9 模块的最佳方式是将其想象为一个四段式装配流水线：

┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ Stage 0 │───→│ Stage 1 │───→│ Stage 2 │───→│ Stage 3 │

数据类型约定

typedef cint16 TT_DATA;   // 输入/输出数据：16位实部 + 16位虚部
typedef cint16 TT_TWID;   // 旋转因子：同上
typedef cacc64 TT_ACC;    // 累加器：48位实部 + 48位虚部（防止溢出）

设计洞察：使用 48 位累加器（cacc64）而非 32 位，是因为多次 MAC（乘累加）操作可能导致中间结果溢出。这类似于财务计算中使用高精度中间表示来避免舍入误差。

Kernel 分工详解

Stage 0: dft9_0（启动者）

这是流水线的起点。它接收原始输入信号，执行初始矩阵乘法，并将部分和通过 cascade 传递。

关键设计决策：

  • 使用 mul 而非 mac（乘累加），因为它是第一阶段，没有前序累加值需要合并
  • 每个时钟周期处理 8 个变换（因为 \(9 \times 8 = 72\) 是向量对齐的最小公倍数）
  • 输出通过 writeincr(acc_o, ...) 写入 cascade 流
// 伪代码示意
for each batch of 8 transforms:
    load 72 samples into vector registers
    compute0.mul(data_slice_0, twiddles_0)  // 首次乘法
    compute1.mul(data_slice_0, twiddles_1)
    ...
    write results to cascade

Stage 1 & 2: dft9_1 / dft9_2（累加者）

这两个 stages 是结构对称的接力者。它们从 cascade 读取前序阶段的累加值，从输入端口读取新的数据切片，执行 MAC（乘累加）操作，再将结果传递下去。

为什么需要两个累加阶段？

9点 DFT 的完整计算需要多个数据切片的贡献。由于 AIE-ML 的向量宽度限制（每次最多处理 8 个元素），9点变换被拆分为多个步骤：

  • Step 0: 处理样本 0-3（由 kk0 完成）
  • Step 1: 处理样本 1-4（由 kk1 完成，累加到 kk0 的结果）
  • Step 2: 处理样本 2-5（由 kk2 完成，继续累加）
  • ...以此类推

shuffle_down 的作用：

compute2.mac((aie::shuffle_down(data0,1)).extract<4>(2), vc0);

这行代码执行了关键的数据重排：shuffle_down(data0, 1) 将向量循环下移 1 个位置，使得原本位于索引 1 的元素移动到索引 0。这是实现 DFT 矩阵不同行的"滑动窗口"访问模式所必需的。

Stage 3: dft9_3（终结者）

最后一个 stage 负责收集所有累加结果，执行最终定标，并重新排序输出。

定标（Down-shift）：

data0 = readincr_v<8>(acc_i).to_vector<TT_DATA>(DNSHIFT);  // DNSHIFT = 15

由于中间累加使用了 48 位精度，最终需要右移 15 位（相当于除以 \(2^{15} = 32768\)）将结果归一化回 16 位范围。这对应于 Q15 定点数格式。

输出重排：

*itw++ = aie::shuffle_up_fill(data2, aie::shuffle_down_fill(data0, data1, 1), 1);

这行复杂的 shuffle 操作将分散在多个向量寄存器中的 9点变换结果重新组合成连续的输出序列。

代码层面的设计决策

1. 循环展开与向量化

for (unsigned rr = 0; rr < NUM_FFT/8; rr++)
    chess_prepare_for_pipelining
{
    // 8个 MMUL 实例并行工作
    MMUL compute0, compute1, ..., compute7;
    // ...
}

为什么是 8？

  • AIE-ML 的向量寄存器宽度为 512 位（32 个 16 位元素）
  • 每次 mmul<1,4,8> 消耗 4 个输入数据和 8 个系数
  • 为了最大化吞吐量，同时实例化 8 个 MMUL 对象，形成软件流水线

chess_prepare_for_pipelining 是编译器提示，指示工具链尝试将循环体展开为流水线化的硬件状态机。

2. 内存访问模式优化

TT_DATA* __restrict ptr = sig_i.data();
// ...
data0.insert<8>(0, aie::load_v<8>((ptr)));
data0.insert<8>(1, aie::load_v<8>((ptr+8)));
// ...
ptr += 72;  // 跳过 9 samples × 8 transforms

使用 __restrict 关键字告知编译器指针不会别名（alias），允许激进的向量化。insert 操作将 8 元素向量插入到 32 元素寄存器的指定槽位，这是一种显式的寄存器分配策略，避免了对 scratch memory 的依赖。

3. 旋转因子的静态配置

kk0 = kernel::create_object<dft9_0>(
    std::vector<typename dft9_0::TT_TWID>{DFT9_TWID0}
);

旋转因子在 graph 构造时作为模板参数传入，而不是在运行时从内存加载。这是因为：

  • DFT-9 的旋转因子是固定的数学常数
  • 编译时已知允许工具链进行常量传播和死代码消除
  • 减少运行时的内存带宽压力

4. Tile 位置的显式映射

location<kernel>(dft9.kk0) = tile(X+0, Y+1);
location<kernel>(dft9.kk1) = tile(X+1, Y+1);
location<kernel>(dft9.kk2) = tile(X+2, Y+1);
location<kernel>(dft9.kk3) = tile(X+3, Y+1);

四个 kernels 被映射到同一行（Y+1）的相邻 tiles 上。这种布局确保了：

  • Cascade 连接走线最短（水平相邻 tiles 有专用 cascade 总线）
  • 共享输入信号 sig_i 可以通过广播网络高效分发
  • 符合 AIE-ML 阵列的物理拓扑约束

依赖关系与集成

上游调用者

dft9 模块本身是一个独立的 graph 组件，但在完整的 PFA-1008 系统中，它被以下模块调用：

  • permute_i：输入置换模块，对数据进行索引映射后送入 dft9
  • transpose0：矩阵转置模块，在两级 DFT 之间重排数据布局

下游消费者

  • transpose1：接收 dft9 的输出进行下一轮维度变换
  • permute_o：输出置换，完成最终的索引逆映射

同级模块

在同一 PFA-1008 流水线中，dft9 与以下模块协同工作：

  • dft7：处理 7点 DFT（互质因子之一）
  • dft16：处理 16点 DFT（基-16 实现）

这三者的组合构成了完整的 1008 = 16 × 9 × 7 变换。

设计权衡与取舍

1. 精度 vs 性能

选择：使用 16 位输入/输出（Q15 格式）配合 48 位中间累加器。

权衡分析：

  • ✅ 充分利用 AIE-ML 的向量宽度（512 位 = 32 × 16 位）
  • ✅ 48 位累加器消除了溢出问题，无需动态范围调整
  • ❌ 相比浮点实现，动态范围受限（约 90 dB SNR）
  • ❌ 最终右移 15 位引入量化噪声

适用场景：通信系统基带处理、雷达信号处理等对实时性要求高于绝对精度的应用。

2. 面积 vs 吞吐量

选择：使用 4 个 kernels 串联，每个 kernel 实例化 8 个 MMUL 对象。

权衡分析：

  • ✅ 单周期处理 8 个变换，峰值吞吐量为 8 × 9 = 72 复数采样/周期
  • ❌ 占用了 4 个 AIE-ML tiles，对于小规模部署可能过度
  • ⚠️ 90% 的 runtime ratio 预留了 10% 的余量用于调度抖动

替代方案：可以减少 MMUL 实例数量以降低面积，但会线性降低吞吐量。

3. 灵活性 vs specialization

选择：硬编码 9点变换，旋转因子作为编译时常量。

权衡分析：

  • ✅ 编译器可进行激进优化（常量折叠、循环展开）
  • ✅ 无运行时系数查找开销
  • ❌ 无法支持可变长度 DFT
  • ❌ 修改变换长度需要重新编译

这与 dft7 的设计哲学一致：在 PFA 框架下，每个小因子 DFT 都是专门优化的"ASIC-like"组件。

4. 级联 vs 内存缓冲

选择：使用 cascade 连接相邻 kernels，而非通过共享内存。

权衡分析：

  • ✅ Cascade 提供专用的片上 FIFO，延迟可预测（约 1-2 周期）
  • ✅ 避免了 L1/L2 缓存的竞争和一致性协议开销
  • ✅ 自动流控（backpressure）简化了同步逻辑
  • ❌ Cascade 深度有限（默认 4 或 8 个元素），突发流量可能阻塞
  • ❌ 数据必须在 consumer 准备好时立即消费，缺乏缓冲弹性

代码中显式设置了 FIFO 深度来缓解这一问题：

fifo_depth(n0) = 32;
fifo_depth(n1) = 32;
fifo_depth(n2) = 32;

新贡献者须知

常见陷阱

1. 误解 NUM_FFT 的含义

static constexpr unsigned NUM_FFT = 4*7*16;  // = 448

这不是说 kernel 只处理 448 个变换，而是指单次 graph 运行需要处理的变换总数。对于 PFA-1008，这个数字来源于：

  • 16 点 DFT 产生 16 组输出
  • 每组需要 7 × 9 = 63 个 9点 DFT
  • 但出于数据对齐考虑，实际分配了 4 × 7 × 16 = 448

调试建议：如果输出结果错位，检查是否理解了数据打包格式。

2. 忽略 __restrict 的契约

void run(input_buffer<TT_DATA, extents<NSAMP_I> >& __restrict sig_i, ...)

__restrict 承诺编译器：sig_i 指向的内存不会通过其他指针访问。违反此契约会导致未定义的向量化行为。

安全做法：不要在 kernel 内部创建指向同一缓冲区的别名指针。

3. 混淆 shuffle_down 和 shuffle_up

  • shuffle_down(vec, n)：元素向索引减小的方向移动，索引 0 的元素被丢弃，新元素从上方填充
  • shuffle_up(vec, n)：相反方向

在 dft9_mmul3.cpp 中，复杂的嵌套 shuffle 用于将 9点变换的 9 个输出（分布在多个向量中）重新排列成连续存储：

*itw++ = aie::shuffle_up_fill(data2, aie::shuffle_down_fill(data0, data1, 1), 1);

理解技巧：画图！画出 8 元素向量的索引，手动跟踪 shuffle 操作。

4. 忘记设置舍入和饱和模式

dft9_0::dft9_0(...) {
    aie::set_rounding(aie::rounding_mode::symmetric_inf);
    aie::set_saturation(aie::saturation_mode::saturate);
}

如果不设置这些，AIE-ML 可能使用默认的截断（truncate）和无饱和模式，导致数值精度问题和溢出崩溃。

扩展指南

添加新的数据类型支持

如果需要支持 32 位复数（cint32）：

  1. 修改类型定义：
typedef cint32 TT_DATA;
typedef cint32 TT_TWID;
typedef cacc80 TT_ACC;  // 80-bit for cint32 accumulation
  1. 调整 MMUL 配置：
using MMUL = aie::mmul<1, 2, 8, TT_DATA, TT_TWID>;  // 向量宽度减半
  1. 更新 DNSHIFT 以匹配新的动态范围。

移植到其他变换长度

假设需要实现 dft5：

  1. 重新计算旋转因子矩阵（使用 Python/MATLAB 生成）
  2. 确定最优的数据切片策略（5 不是 8 的约数，需要 padding 或特殊处理）
  3. 可能需要调整 kernel 数量（当前 4 级流水线针对 9点优化）
  4. 更新 NUM_FFT 以匹配新的批处理大小

性能调优

使用 Makefile 中的分析目标：

make profile      # 生成 Vitis Analyzer 可用的性能数据
make loop_II      # 检查循环的 Initiation Interval
make throughput   # 测量实际吞吐量

关键指标：

  • II (Initiation Interval)：理想情况下应为 1（每周期启动一次迭代）
  • Memory stall cycles：如果过高，检查数据对齐和访问模式
  • Cascade stall cycles：如果过高，可能需要增加 fifo_depth
