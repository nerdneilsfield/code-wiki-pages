PL内核层：packet_sender 与 packet_receiver

概述

本模块包含两个核心的Vitis HLS内核，负责在PL(可编程逻辑)侧实现数据包的路由功能。它们是连接DDR内存、AIE阵列和主机软件的关键桥梁。

packet_sender：数据包封装器

职责

将来自DMA的原始数据流转换为带头的包交换格式，为AIE阵列提供可路由的数据。

代码详解

// packet_sender.hpp
#define PACKET_NUM 4 
#define PKTTYPE 0 
#define PACKET_LEN 224
#define BURST_SIZE PACKET_LEN * PACKET_NUM // 896 words

typedef ap_uint<32> data_t;
typedef ap_axiu<32, 0, 0, 0> axis_pkt;

static const unsigned int packet_ids[PACKET_NUM]={in_i_0, in_i_1, in_i_2, in_i_3};

关键设计点：

  • packet_ids数组定义了4个包的标识符，这些宏由AIE编译器生成
  • axis_pkt使用AXI4-Stream协议，32位数据宽度

头部生成逻辑

ap_uint<32> generateHeader(unsigned int pktType, unsigned int ID){
#pragma HLS inline
    ap_uint<32> header=0;
    header(4,0)=ID;           // Packet ID (destination)
    header(11,5)=0;           // Reserved
    header(14,12)=pktType;    // Packet type
    header[15]=0;             // Reserved
    header(20,16)=-1;         // Source row (broadcast)
    header(27,21)=-1;         // Source column (broadcast)
    header(30,28)=0;          // Reserved
    header[31]=header(30,0).xor_reduce()?(ap_uint<1>)0:(ap_uint<1>)1; // Parity
    return header;
}

为什么这样设计头部：

  • Bit[4:0] ID：支持最多32个不同的目标通道
  • Bit[14:12] Type：预留用于不同数据类型(控制/数据/状态)
  • Source行列设为-1：表示广播源地址，简化路由表
  • 奇偶校验位：简单的错误检测机制

主函数流程

void packet_sender(hls::stream<axis_pkt>& rx, hls::stream<axis_pkt>& tx)
{
#pragma HLS interface axis port=tx
#pragma HLS interface axis port=rx
#pragma HLS interface s_axilite port=return bundle=control  
   
    axis_pkt x;
    
    for (int h = 0; h < PACKET_NUM; h++) { 
        unsigned int ID = packet_ids[h];
        ap_uint<32> header = generateHeader(PKTTYPE, ID);
        ap_axiu<32,0,0,0> tmp;
        tmp.data = header;
        tmp.keep = -1;      // All bytes valid
        tmp.last = 0;       // Not the last word
        tx.write(tmp);      // Send header
        
        for(int i = 0; i < PACKET_LEN; i++) {
#pragma HLS PIPELINE II=1
            x = rx.read();
            x.last = (i == PACKET_LEN-1);  // Mark end of packet
            tx.write(x);
        }
    }
}

HLS优化指令解析：

指令 作用
#pragma HLS interface axis 声明AXI4-Stream接口，用于高速数据传输
#pragma HLS interface s_axilite 声明AXI4-Lite接口，用于内核控制(启动/停止)
#pragma HLS PIPELINE II=1 流水线化内层循环，目标启动间隔为1周期

性能特征：

  • 每周期输出一个32位字
  • 400MHz时钟下，理论带宽 = \(400M \\times 4B = 1.6 GB/s\)
  • 每个包225个周期(1头 + 224数据)，共4包 = 900周期完成一次burst

packet_receiver：数据包分发器

职责

解析来自AIE的包交换数据流，根据包头信息将数据分发到对应的输出通道。

头部解析逻辑

unsigned int getPacketId(ap_uint<32> header){
#pragma HLS inline
    ap_uint<32> ID=0;
    ID(4,0)=header(4,0);  // Extract lower 5 bits
    return ID;
}

主函数流程

void packet_receiver(hls::stream<axis_pkt>& rx, 
                     hls::stream<axis_pkt>& tx0,
                     hls::stream<axis_pkt>& tx1,
                     hls::stream<axis_pkt>& tx2,
                     hls::stream<axis_pkt>& tx3)
{
#pragma HLS INTERFACE axis port=rx
#pragma HLS INTERFACE axis port=tx0..tx3
#pragma HLS interface s_axilite port=return bundle=control  
  
    axis_pkt x;
    
    for (int h = 0; h < PACKET_NUM; h++) { 
        // Read and parse header
        ap_axiu<32,0,0,0> tmp = rx.read();
        unsigned int ID = getPacketId(tmp.data);
        unsigned int channel = packet_ids[ID];  // Map ID to output channel

        // Route data words based on channel
        for(int i = 0; i < PACKET_LEN; i++) {
#pragma HLS PIPELINE II=1
            x = rx.read();
            switch(channel){
                case 0: tx0.write(x); break;
                case 1: tx1.write(x); break;
                case 2: tx2.write(x); break;
                case 3: tx3.write(x); break;
            }
        }
    }
}

路由决策机制：

Packet ID → Lookup in packet_ids[] → Channel Number → Output Stream
   in_i_0  →        0              →      0         →    tx0
   in_i_1  →        1              →      1         →    tx1
   in_i_2  →        2              →      2         →    tx2
   in_i_3  →        3              →      3         →    tx3

switch语句的硬件实现：

  • HLS综合器将switch转换为多路选择器(multiplexer)网络
  • 无分支预测开销，纯组合逻辑决策
  • 所有输出端口并行准备，选择信号决定哪个端口实际写入

内存模型与资源利用

缓冲区分类

缓冲区 类型 大小 用途
rx/tx streams hls::stream 深度由工具推断 AXI4-Stream FIFO接口
tmp 寄存器 1 word 临时头部存储
x 寄存器 1 word 当前数据字

无片上BRAM使用

这两个内核是**流式处理(streaing)**设计：

  • 数据从输入直接流向输出，不缓存整个包
  • 仅需1-2个字的寄存器缓冲
  • 资源占用极低，主要为LUT和FF

时序与性能分析

关键路径

rx.read() → getPacketId() → switch decision → txN.write()
   ↑                                              ↓
   └─────────────── 1 cycle ──────────────────────┘

吞吐量保证

#pragma HLS PIPELINE II=1
  • II (Initiation Interval) = 1：每周期可以开始一次新迭代
  • 这意味着内核可以连续接收/发送数据，无气泡(bubble)
  • 实际吞吐量受限于上下游的握手信号(TVALID/TREADY)

延迟特性

操作 延迟
Header read to first data out 2 cycles
Per data word 1 cycle
Total per packet (224 words) ~226 cycles
Total per burst (4 packets) ~904 cycles

错误处理与鲁棒性

当前设计的假设

  1. 输入数据完整性：假设DMA提供正确数量的数据字
  2. 包头有效性：假设AIE生成的包头包含有效的ID字段
  3. 无溢出保护：如果下游阻塞，上游会自然反压(backpressure)

潜在风险

风险 影响 缓解措施
包头损坏 数据路由到错误通道 奇偶校验检测(但无自动纠正)
数据字数不匹配 包边界错乱 依赖AIE的正确行为
下游挂起 整个流水线停滞 XRT超时机制

扩展指南

增加包数量

如需支持更多计算单元(如x10设计的10个实例)：

// 修改前
#define PACKET_NUM 4
static const unsigned int packet_ids[PACKET_NUM]={in_i_0, in_i_1, in_i_2, in_i_3};

// 修改后
#define PACKET_NUM 10
static const unsigned int packet_ids[PACKET_NUM]={
    in_i_0, in_i_1, in_i_2, in_i_3, in_i_4,
    in_i_5, in_i_6, in_i_7, in_i_8, in_i_9
};

修改包长度

// 例如改为每包处理64粒子
#define NUM_I 64
#define PACKET_LEN (NUM_I * 7)  // 448 words

注意：需同步更新AIE内核的缓冲区大小。
