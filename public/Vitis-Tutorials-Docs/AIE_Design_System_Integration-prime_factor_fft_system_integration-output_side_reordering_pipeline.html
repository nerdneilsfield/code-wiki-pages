输出端重排序流水线 (Output Side Reordering Pipeline)

一句话概括

这是素因子FFT系统的最后一道数据整形关卡，负责将AIE计算出的频域结果从"算法友好的乱序"还原为"内存友好的自然顺序"——就像把图书馆里按主题分类的书籍重新按书架编号排列，方便读者（DMA）高效取走。

问题空间：为什么需要这个模块？

素因子FFT的数学本质

Prime Factor Algorithm (PFA) FFT 的核心思想是将一个大的DFT分解为互质的小DFT的级联。对于 N = 1008 = 7 × 9 × 16 的情况：

原始时域序列 → [DFT-7] → [转置] → [DFT-9] → [转置] → [DFT-16] → 频域结果

每一次小DFT运算后，数据的排列方式都会发生变化。这种变化是算法层面的必然——为了最大化AIE阵列的并行效率，数据必须以特定的"乱序"方式流动。

输出端的困境

当数据完成所有DFT阶段后，它处于以下状态：

  1. 维度交错：原本连续的时域样本现在分散在不同的存储bank中
  2. 位反转/乱序索引：频域输出的索引遵循复杂的数学映射关系
  3. 多路复用格式：AIE以双路128-bit AXI4-Stream输出，需要解复用

如果直接将这种格式的数据写入DDR，后续软件处理将面临噩梦般的索引计算。因此，必须在PL层进行一次彻底的数据重排。

设计动机：硬件 vs 软件的权衡

方案 优点 缺点
AIE内部重排 零外部延迟 消耗宝贵的AIE计算资源
PL HLS重排 专用硬件加速，AIE专注计算 增加PL资源占用和流水线延迟
软件重排 零硬件成本 CPU负担重，破坏实时性

本模块选择PL HLS重排方案，体现了"让专业的硬件做专业的事"的设计哲学。

核心抽象：理解这个模块的思维方式

类比：快递分拣中心

想象一个大型快递分拣中心：

  • 输入：来自不同区域的包裹（AIE DFT-16的输出），已经按某种内部逻辑排序
  • 转置阶段：将包裹从"区域编码"转换为"路线编码"（矩阵转置，改变数据访问模式）
  • 置换阶段：按照最终投递地址的邮政编码重新排序（查表置换，恢复自然顺序）
  • 输出：按街道顺序排列的包裹流（DMA可高效写入DDR的连续地址）

两个关键操作的数学直觉

1. 转置 (Transpose)：改变数据的"视角"

输入视角：按列存储 (DFT-9的输出维度优先)
┌───┬───┬───┐
│ 0 │ 3 │ 6 │
├───┼───┼───┤
│ 1 │ 4 │ 7 │
├───┼───┼───┤
│ 2 │ 5 │ 8 │
└───┴───┴───┘

输出视角：按行存储 (DFT-16的输入维度优先)
┌───┬───┬───┐
│ 0 │ 1 │ 2 │
├───┼───┼───┤
│ 3 │ 4 │ 5 │
├───┼───┼───┤
│ 6 │ 7 │ 8 │
└───┴───┴───┘

转置的本质是内存访问模式的转换：从stride-N读取变为stride-1读取。

2. 置换 (Permutation)：应用数学映射

置换操作基于预计算的查找表(LUT)，实现公式：

\[X[k] = x[\text{permute\_table}[k]]\]

其中 permute_table 是素因子FFT算法推导出的特定索引映射，用于消除中国剩余定理(CRT)带来的索引缠绕。

组件深度解析

1. pfa1008_permute_o（输出置换核）

职责：执行最终的输出索引置换，将DFT-16的结果从算法顺序转换为自然顺序。

核心机制：

// 四级存储结构：ping-pong × 4 copies × 4 banks × 252 entries
static TT_DATA buff[2][4][4][NFFT/4];

为什么是4份数据拷贝？因为置换操作需要从同一个polyphase读取多个不连续地址。单端口BRAM无法在一个周期内提供8个独立读地址，因此通过数据复制实现空间并行度。

读写模式：

  • 写操作：按DFT-16的自然顺序写入（stride-63寻址）
  • 读操作：查LUT获取置换后的地址，从4个copy中选择正确的bank

关键代码片段（简化）：

// 写地址生成：三维嵌套循环的展开
TT_ADDR wr_addr_use = TT_ADDR(wr_addr + offW);  // offW = ss * 63

// 读地址：LUT查表
TT_ADDR rd_addr_use = permute_lut[ss>>1][rd_addr+ss];

// 多路选择：根据地址低位选择哪个copy
switch ( rd_off[ss] ) {
    case 0 : permute_o[ss] = rd_data0[ss]; break;
    case 1 : permute_o[ss] = rd_data1[ss]; break;
    // ...
}

2. pfa1008_transpose2（第二级转置核）

职责：在置换之后执行最后一次矩阵转置，确保数据以DMA友好的连续突发(burst)模式输出。

复杂度的来源：

输入数据流具有特殊的交织模式——来自DFT-9的两个stream（ss0, ss1）各自携带4个样本，但这些样本并非简单的时序排列。pfa1008_unpack函数中的9状态FSM正是为了处理这种非规则的打包格式。

状态机设计（9状态循环）：

static ap_uint<4> fsm_state = 0;  // 状态0-8循环

// 每个状态定义如何从save0/save1/inss0/inss1中提取8个样本
// 这是一种"旋转缓冲区"模式，处理输入流的相位偏移

双缓冲(Ping-Pong)策略：

static TT_DATA buff[2][4][NFFT/4];  // ping-pong × 4 banks × 252
static bool ping = 0;

// 写入当前ping bank，从上一周期的!ping bank读取
buff[ping][...][...] = ...;           // 写
read_data = buff[!ping][...][...];    // 读

这实现了无气泡流水线——读写可以全速并行，每周期处理8个样本。

设计决策与权衡

决策1：为什么在PL做重排序而不是AIE？

选择的理由：

  • AIE的计算资源（MAC单元、向量寄存器）比PL的LUT/BRAM更宝贵
  • 重排序是纯数据搬运操作，没有算术计算，适合PL的并行存储架构
  • PL可以实现更复杂的寻址模式（如LUT-based permutation）

代价：

  • 增加了芯片间数据移动（AIE→PL）
  • 引入了额外的流水线延迟
  • 消耗了PL的BRAM资源（约 ~200Kb 用于buffer + LUT）

决策2：双缓冲(Ping-Pong) vs 单缓冲+停顿

选择的理由：

  • 维持II=1的吞吐量是必须的（系统瓶颈约束）
  • 双缓冲的硬件成本（2x BRAM）可以接受

替代方案：

  • 单缓冲+读写仲裁：会降低吞吐量，不满足系统要求
  • 更大的单缓冲+批量处理：会增加端到端延迟

决策3：4份数据拷贝 vs 多端口BRAM

在pfa1008_permute_o中，为了实现每周期8个独立读操作，选择了数据空间复制而非真正的多端口BRAM。

原因：

  • Xilinx BRAM原生只有2个端口
  • 构建8端口存储需要复杂的仲裁逻辑或大量的BRAM实例化
  • 对于1008点FFT，4×252 = 1008个entry的复制是可接受的（~16KB per copy）

决策4：LUT-based置换 vs 运行时计算

置换地址通过预计算的C数组(PERM_O_ADDR)硬编码，而非运行时计算CRT索引。

优势：

  • 零计算延迟，确定性时序
  • HLS可以优化LUT的物理布局

劣势：

  • 仅支持固定的1008点FFT
  • 修改FFT大小需要重新生成LUT（见gen_permute_tables.m）

新贡献者必读：陷阱与注意事项

⚠️ 隐式契约

  1. 输入数据格式假设

    • permute_o期望输入是DFT-16的直接输出，样本按特定polyphase排列
    • 如果上游DFT-16的实现改变，这里的unpack逻辑可能需要同步修改

  2. 启动延迟(startup counter)

    static ap_uint<10> startup = 0;
if ( startup == ap_uint<10>(NFFT/8-1) ) { running = 1; }
    • 前NFFT/8个周期的输出被故意丢弃
    • 修改buffer大小必须同步调整startup阈值

  3. Stream阻塞语义

    ( inss0[3], inss0[2], inss0[1], inss0[0] ) = sig_i[0].read();
    • hls::stream::read()是阻塞调用
    • 如果上游AIE产生数据不均匀，可能导致死锁

🔧 调试技巧

  1. 启用仿真打印

    // 取消注释这些行查看中间状态
// cout << "stream inputs: ss0: " << ... << endl;

  2. 验证LUT正确性

    • 参考gen_permute_tables.m脚本理解LUT生成逻辑
    • 可用MATLAB/Octave重新生成并对比

  3. 检查地址越界

    • TT_ADDR是10-bit（覆盖0-1023）
    • NFFT=1008，理论上不会溢出，但需确保stride计算不环绕

📝 扩展指南

如需修改FFT大小（例如改为504点）：

  2. 更新头文件中的NFFT常量

总结

输出端重排序流水线是素因子FFT系统的数据整形专家。它不执行任何数学运算，却承担着将AIE的计算结果转化为可用形式的关键职责。

理解这个模块的关键在于把握三个转换：

  1. Stream解包/打包：AXI4-Stream ↔ 并行样本数组
  2. 置换：算法索引 ↔ 自然索引（通过LUT）
  3. 转置：列优先存储 ↔ 行优先存储（通过stride寻址）

这三个转换层层递进，最终将AIE友好的数据布局转换为DDR友好的连续突发传输——这正是异构计算系统中"软硬件协同设计"的典型范例。
