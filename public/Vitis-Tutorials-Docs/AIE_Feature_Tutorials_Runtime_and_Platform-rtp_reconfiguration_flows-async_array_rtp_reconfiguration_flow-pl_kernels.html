PL Kernels 子模块详解

概述

PL（Programmable Logic）Kernels 是可编程逻辑侧的数据搬运组件，负责在 DDR 内存和 AIE 阵列之间建立数据通路。本模块包含两个轻量级 HLS 内核：

  • datagen：数据生成器，产生测试序列
  • s2mm：Stream-to-Memory，将 AXI-Stream 写入 DDR

这两个内核共同构成了 AIE 系统的数据 I/O 边界。

设计哲学

为什么选择 PL Data Mover 而非 AIE DMA？

方案 优势 劣势 本模块选择
AIE DMA 高带宽、低延迟、硬件自动调度 配置复杂、调试困难、灵活性受限 ❌
PL Data Mover 灵活可控、易于调试、与外部 IP 集成方便 占用 PL 资源、需要手动编写 HLS ✅

对于教学目的，datagen 和 s2mm 提供了清晰的数据流边界，便于理解和调试。

datagen.cpp：数据生成器

功能规格

extern "C" void datagen(
    hls::stream<ap_axis<32,0,0,0>> & out,  // AXI-Stream 输出
    int size,                               // 生成元素个数
    int init_val = 0                        // 初始值偏移
);

算法：产生递增序列 init_val, init_val+1, init_val+2, ..., init_val+size-1

HLS 接口配置

#pragma HLS INTERFACE axis port=out           // AXI-Stream 主设备
#pragma HLS INTERFACE s_axilite port=return   // 控制接口（启动/完成）
#pragma HLS INTERFACE ap_ctrl_hs port=return  // 握手控制模式
#pragma HLS interface s_axilite port=size     // 标量参数（通过 AXI-Lite）
#pragma HLS interface ap_none port=size       // 无握手（由 s_axilite 保证稳定）
#pragma HLS interface s_axilite port=init_val // 初始值参数
#pragma HLS interface ap_none port=init_val   // 无握手

接口类型解析

Pragma 含义 用途
INTERFACE axis AXI4-Stream 协议 高速数据流传输
INTERFACE s_axilite AXI4-Lite 寄存器接口 主机配置参数、启动内核
INTERFACE ap_ctrl_hs 块级握手控制 start/done/idle 信号
INTERFACE ap_none 无时序约束（组合逻辑） 配合 s_axilite 使用，表示值已稳定

实现细节

for(int i = 0; i < size; i++) {
    ap_axis<32,0,0,0> tmp;
    tmp.data = i + init_val;   // 数据负载
    tmp.keep = -1;             // 所有字节有效（0xFF...F）
    tmp.last = 0;              // 非包尾（连续流传输）
    out.write(tmp);            // 阻塞写入 FIFO
}

ap_axis<32,0,0,0> 模板参数

template<int D, int U, int TI, int TD>
struct ap_axis {
    ap_int<D> data;      // D = 数据位宽（32-bit）
    ap_int<U/8> keep;    // U = 用户位宽（0，即无 user 信号）
    ap_int<1> last;      // TLAST 信号
    ap_int<TI> id;       // TID（0，即无 ID）
    ap_int<TD> dest;     // TDEST（0，即无路由）
};

简化后的等效结构：

struct ap_axis_32 {
    ap_int<32> data;     // 实际数据
    ap_int<1> last;      // 帧结束标志（本例始终为 0）
};

tmp.keep = -1 的含义

keep 是字节使能信号：

  • -1（即全 1）表示所有 4 个字节都有效
  • 对于 32-bit 数据，等价于 0b1111 或 0xF

在连续的向量数据流中，通常所有字节都有效，因此设为 -1。

时序特性

sequenceDiagram participant Host as Host (PS) participant CTRL as s_axilite
Control participant DG as datagen participant AIE as AIE Kernel Host->>CTRL: 写 size=1024 Host->>CTRL: 写 init_val=0 Host->>CTRL: 写 start=1 CTRL->>DG: ap_start=1 loop 1024 cycles DG->>AIE: TDATA=0, TLAST=0 DG->>AIE: TDATA=1, TLAST=0 DG->>AIE: ... end DG->>CTRL: ap_done=1 CTRL->>Host: 中断/轮询完成

关键观察：datagen 是阻塞式的——它会持续运行直到产生 size 个数据，期间无法被中断。

s2mm.cpp：Stream-to-Memory

功能规格

extern "C" void s2mm(
    ap_int<32>* mem,                          // DDR 内存指针
    hls::stream<ap_axis<32, 0, 0, 0>> & s,    // AXI-Stream 输入
    int size                                  // 接收元素个数
);

算法：从 AXI-Stream 读取 size 个元素，顺序写入 DDR

HLS 接口配置深度解析

#pragma HLS INTERFACE m_axi port=mem offset=slave bundle=gmem
//          │         │      │   │            │
//          │         │      │   │            └── 内存接口捆绑名（连接到同一总线）
//          │         │      │   └── 偏移模式：slave（主机提供物理地址）
//          │         │      └── 端口名
//          │         └── 接口类型：m_axi（AXI4-Full，支持突发传输）
//          └── 编译指示

#pragma HLS interface axis port=s
// 数据流接口：AXI4-Stream

#pragma HLS INTERFACE s_axilite port=mem bundle=control
#pragma HLS INTERFACE s_axilite port=size bundle=control
#pragma HLS interface s_axilite port=return bundle=control
// 控制接口：AXI4-Lite，用于传递标量参数和状态

为什么需要 offset=slave？

// Host 代码中的调用方式
auto s2mm_run = s2mm(out_bo, nullptr, ELEM_per_iter*ITERATION);
//                     │     │      │
//                     │     │      └── size（通过 s_axilite）
//                     │     └── 保留参数（本例未使用）
//                     └── mem（通过 m_axi 访问）

offset=slave 表示：

  • 主机通过 s_axilite 接口写入一个指针值（Buffer Object 的设备地址）
  • HLS 内核将该值作为 mem 的基地址
  • 这种间接寻址允许运行时动态指定内存位置

流水线优化

for(int i = 0; i < size; i++) {
    #pragma HLS PIPELINE II=1
    ap_axis<32, 0, 0, 0> x = s.read();   // 从 Stream 读取
    mem[i] = x.data;                      // 写入 DDR
}

PIPELINE II=1 的目标

  • II (Initiation Interval) = 1：每个时钟周期开始一次新的循环迭代
  • 理论吞吐率：每周期 4 字节（32-bit @ 300MHz = 1.2 GB/s）

实际性能瓶颈

实际吞吐率受限于：

  1. DDR 带宽：VCK190 的 DDR4 可提供约 25 GB/s，远未达到
  2. AXI-Stream 速率：取决于上游 AIE 内核的产生速度
  3. 突发长度：当前实现每次循环写入单个元素，未利用突发传输优势

优化建议：使用 memcpy 风格的分块处理，启用更长的突发传输。

config.cfg：HLS 综合配置

[hls]
freqhz=400000000          # 目标频率：400 MHz
syn.top=s2mm              # 顶层函数名
syn.file=s2mm.cpp         # 源文件
flow_target=vitis         # 目标流程：Vitis 统一平台
package.output.format=xo  # 输出格式：Xilinx Object (.xo)
package.output.file=s2mm.xo

频率选择的权衡

频率 优势 劣势
300 MHz 更容易满足时序、功耗更低 吞吐率较低
400 MHz（本模块） 平衡性能和资源 需要仔细优化流水线
500+ MHz 最高吞吐率 时序收敛困难、功耗增加

400 MHz 是 VCK190 平台上 PL kernel 的常见选择，与 AIE 的 1 GHz 形成合理的异步时钟域比例。

内存所有权模型

Buffer Object 生命周期

// host.cpp —— 内存分配与所有权
auto out_bo = xrt::bo(device, output_size_in_bytes, s2mm.group_id(0));
//  │            │      │                         │
//  │            │      │                         └── 关联到 s2mm 的 m_axi 接口
//  │            │      └── 大小：256 * 4 * 4 = 4096 字节
//  │            └── XRT Buffer Object（设备内存句柄）
//  └── 分配在设备的 DDR 上

auto host_out = out_bo.map<int*>();   // 主机映射指针（用于初始化/验证）

所有权转移：

  1. 分配：xrt::bo 构造函数在 DDR 上分配物理内存
  2. 映射：map<>() 返回主机可访问的虚拟地址
  3. 传递：s2mm(out_bo, ...) 将设备地址传递给 PL kernel
  4. 同步：out_bo.sync(XCL_BO_SYNC_BO_FROM_DEVICE) 确保数据一致性
  5. 释放：out_bo 析构时自动回收内存

并发安全

s2mm 内核在写入 DDR 期间，主机不应通过 host_out 指针访问同一块内存。这是典型的 生产者-消费者 模式：

时间线：
T0: s2mm 开始写入 DDR[0], DDR[1], ...
T1: s2mm 完成写入
T2: sync() 确保缓存一致性
T3: 主机安全读取 host_out[0], host_out[1], ...

违反此顺序会导致数据竞争和未定义行为。

设计模式与最佳实践

模式 1：预启动 PL Kernel

// 先启动 PL kernels（它们会阻塞等待数据）
auto s2mm_run = s2mm(out_bo, nullptr, total_size);
auto datagen_run = datagen(nullptr, total_size, 0);

// 再启动 AIE graph（此时上下游都已就绪）
ghdl.run(ITERATION);

好处：避免 AIE 产生的首批数据因下游未就绪而丢失。

模式 2：显式依赖管理

// 等待 PL kernel 完成（阻塞）
s2mm_run.wait();

// 然后同步数据到主机
out_bo.sync(XCL_BO_SYNC_BO_FROM_DEVICE);

wait() 确保硬件已完成所有 DDR 写入，sync() 确保 CPU 缓存看到最新数据。

模式 3：错误处理

try {
    auto s2mm_run = s2mm(out_bo, nullptr, size);
    // ... 其他操作
} catch (const std::exception& e) {
    std::cerr << "PL kernel error: " << e.what() << std::endl;
    // 清理资源、记录日志
}

XRT API 在底层错误（如总线超时、非法地址）时会抛出异常。

扩展方向

1. 双向数据传输（MM2S + S2MM）

当前 datagen 是纯生成器，可以扩展为 MM2S（Memory-to-Stream）：

void mm2s(ap_int<32>* mem, hls::stream<ap_axis<32>>& out, int size) {
    #pragma HLS INTERFACE m_axi port=mem
    #pragma HLS INTERFACE axis port=out
    for(int i = 0; i < size; i++) {
        #pragma HLS PIPELINE II=1
        ap_axis<32> tmp;
        tmp.data = mem[i];
        tmp.keep = -1;
        tmp.last = (i == size-1) ? 1 : 0;  // 标记最后一拍
        out.write(tmp);
    }
}

2. 数据格式转换

在 PL 侧添加预处理/后处理：

  • 定点数转浮点数
  • 数据重排（transpose、interleave）
  • 压缩/解压缩

3. 多通道并行

实例化多个 datagen/s2mm 内核，连接 AIE 的多个输入/输出：

[connectivity]
nk=datagen:2:datagen_0,datagen_1
nk=s2mm:2:s2mm_0,s2mm_1
stream_connect=datagen_0.out:ai_engine_0.Datain0
stream_connect=datagen_1.out:ai_engine_0.Datain1
stream_connect=ai_engine_0.Dataout0:s2mm_0.s
stream_connect=ai_engine_0.Dataout1:s2mm_1.s

总结

PL Kernels 子模块展示了如何用 HLS 构建高效的数据搬运组件：

  1. 接口设计：合理选择 axis、m_axi、s_axilite 满足不同需求
  2. 流水线优化：PIPELINE II=1 最大化吞吐率
  3. 系统集成：通过 system.cfg 声明连接关系，XRT 管理执行

理解这些基础组件的工作原理，是构建复杂异构系统的前提。
