多速率信道化器与数字下变频（DDC）原语

概述

本模块提供了在 AMD Versal AI Engine 上实现高性能多速率信号处理的基础构建块，专注于两个核心应用场景：多相滤波器组信道化器（Polyphase Channelizer）和数字下变频链（Digital Down-Conversion Chain）。这些设计展示了如何在 AI Engine 阵列上高效实现计算密集型的多速率 DSP 算法，充分利用 AIE 的 SIMD 向量处理能力和专用数据移动硬件。

想象一下你正在处理一个宽带射频信号——就像在一个拥挤的房间里同时有多个人在不同频率上说话。信道化器就像是给每个人分配一个专属的"听觉通道"，让你能够同时监听所有对话；而 DDC 链则像是精密的调谐器，将高频信号逐步下变频到基带，同时滤除不需要的频率成分。这两个组件是现代软件定义无线电（SDR）、5G/6G 基站和雷达系统的核心构建块。

问题空间与设计动机

为什么要专门设计这个模块？

在无线通信系统中，工程师经常面临以下挑战：

  1. 宽带信号处理：现代系统需要同时处理数百 MHz 甚至 GHz 级别的带宽，这对计算资源提出了极高要求。

  2. 多通道并行性：MIMO 系统可能需要同时处理数十个天线通道的数据，每个通道都需要独立的信号处理链。

  3. 严格的实时约束：射频前端的数据流是连续的，任何处理延迟都会导致数据丢失或系统失步。

  4. 功耗与面积限制：在边缘设备中，必须在有限的功耗预算和芯片面积内完成这些计算密集型任务。

传统的 CPU 或 GPU 实现虽然灵活，但难以满足上述所有约束。FPGA 提供了更好的实时性能，但对于复杂的 DSP 算法，开发周期长且优化困难。AMD Versal AI Engine 架构通过提供专用的向量处理器阵列和高效的片上数据移动网络，为这些问题提供了一个独特的解决方案。

核心设计理念

本模块的设计遵循以下原则：

  • 计算与数据移动协同设计：不仅仅是优化单个内核的计算效率，而是将整个数据流路径（从 PL 接口到 AIE 内核再到输出）作为一个整体进行优化。

  • 模板化与参数化：使用 C++ 模板允许在编译时确定关键参数（如通道数、采样率、滤波器阶数），从而生成高度优化的代码。

  • 显式状态管理：由于 AIE 内核是长期运行的，必须仔细管理滤波器状态和历史缓冲区，确保跨数据块的连续性。

  • SSR（Super Sample Rate）架构：通过并行处理多个样本，突破单核吞吐量的限制。

数据流详解

信道化器数据流

信道化器采用经典的多相滤波器组架构，其核心思想是将一个高阶 FIR 滤波器分解为多个并行的低阶子滤波器（多相分量）：

  1. 输入阶段：数据通过 input_plio 从可编程逻辑（PL）侧进入 AI Engine 阵列。PLIO 接口配置为 32-bit 位宽，支持高速数据传输。

  2. 滤波处理：每个 polyphase_fir_graph 实例包含一个 polyphase_fir 内核，该内核使用 AIE 专用的向量指令（如 mul4、mac4）执行 SIMD 乘累加运算。内核维护内部状态（state0、state1）以支持连续数据流的处理。

  3. 模式选择：通过模板参数 PATTERN，内核可以在 8 种不同的数据处理模式之间切换（REPEATED_D0 到 HOLDOVER_D0）。这种设计用于处理非整数倍抽取率（oversampling ratio P/Q = 8/7）带来的复杂数据对齐问题。

  4. 输出阶段：处理后的数据通过 output_plio 返回 PL 侧，供后续处理或传输。

DDC 链数据流

DDC 链是一个更复杂的多级处理流水线，专为蜂窝通信标准（LTE/NR）设计：

  1. 解交织（Interleaving）：ddcinlv_kernel 负责将输入数据重新排列，为后续处理准备合适的数据布局。

  2. 第一级半带滤波（HBF47）：fir_47t_sym_hb_2d 实现 47-tap 对称半带滤波器，执行 2 倍抽取。半带滤波器的特殊结构（近一半系数为零）被充分利用以减少计算量。

  3. 数字混频（Demixing）：demixer_5c_kernel0 将信号从载波频率下变频到基带。

  4. 第二级半带滤波链：hbd_chain_kernel 进一步降低采样率。

  5. 信道 FIR：ddc_chain_kernel 执行最终的信道滤波，使用 buffer_internal 结构管理复杂的循环缓冲区操作。

核心抽象与心智模型

1. Graph-Kernel 层次结构

AI Engine 编程采用两级抽象：

  • Graph 层：描述数据流拓扑和连接关系。graph 类（如 my_graph、m16_ssr8_filterbank_graph）定义了内核之间的数据路径和与外部世界的接口（PLIO）。这类似于电路设计中的原理图——它告诉你什么连接到什幺，但不涉及具体实现细节。

  • Kernel 层：实现具体的计算逻辑。kernel 对象（如 polyphase_fir、ddc_chain_kernel）包含实际的算法代码，使用 AIE 向量指令进行优化。这是"晶体管级别"的实现。

理解这种分离至关重要：Graph 层关注数据如何流动，Kernel 层关注数据如何被处理。

2. 循环缓冲区抽象（buffer_internal）

DDC 链中的 FIR 滤波器需要维护历史样本（overlap/save 方法）。buffer_internal 结构提供了一种高效管理这些缓冲区的机制：

struct buffer_internal {
    buffer_datatype * restrict head;  // 缓冲区起始地址
    buffer_datatype * restrict ptr;   // 当前读写位置
};

想象这是一个环形跑道：head 是起跑线，ptr 是当前跑者的位置。cyclic_add 函数确保当跑者到达终点时自动回到起点。这种设计避免了昂贵的内存拷贝操作，只需要调整指针即可。

3. SSR（Super Sample Rate）处理

当单核的处理速度无法满足采样率要求时，SSR 架构通过空间并行解决问题。在 m16_ssr8_filterbank_graph 中，8 个 polyphase_fir_graph 实例并行工作，每个处理数据流的一个子集。这就像工厂里的装配线——不是让一个工人做所有工作，而是让多个工人各自负责一部分，共同提高吞吐量。

4. 模板参数化设计

代码大量使用 C++ 模板来实现编译时多态：

template<unsigned PATTERN, unsigned NSAMP>
class polyphase_fir_graph : public graph {
    // ...
};

这种设计的优势在于：

  • 零运行时开销：所有参数在编译时确定，生成的代码是针对特定配置高度优化的。
  • 类型安全：错误的参数组合会在编译期被发现，而不是运行时才暴露。
  • 代码复用：同一套模板可以实例化为多种配置，适应不同的系统需求。

代价是编译时间增加和代码可读性降低——你需要理解模板展开后的实际行为。

关键设计决策与权衡

1. 硬编码参数 vs. 运行时配置

在 polyphase_fir.h 中，许多系统参数被硬编码为 static constexpr：

static constexpr unsigned MCHAN = 16;           // 总通道数
static constexpr unsigned HOP = 14;             // 循环缓冲区步进
static constexpr unsigned TAPS_PER_PHASE = 8;   // 每相抽头数

为什么选择硬编码？

  • AIE 内核对性能极其敏感，编译时常量允许编译器进行激进的优化（如循环展开、常量传播）。
  • 这些参数通常由硬件架构和算法规格决定，在设计阶段就已固定。

代价是什么？

  • 灵活性降低：修改参数需要重新编译整个设计。
  • 代码复用受限：无法在不修改源码的情况下适配不同配置。

2. 显式状态管理 vs. 自动状态保存

FIR 滤波器需要在多次调用之间保持状态（历史样本）。本模块选择了显式状态管理：

// 构造函数初始化状态
polyphase_fir(TT_COEFF (&taps0_i)[...], TT_COEFF (&taps1_i)[...])
    : taps0(taps0_i), taps1(taps1_i),
      state0(aie::zeros<TT_DATA,TAPS_PER_PHASE>()), 
      state1(aie::zeros<TT_DATA,TAPS_PER_PHASE>())
{}

// run_i 方法在开始时加载状态，结束时保存状态
void run_i(input_stream<TT_DATA>* __restrict sig_i,
           output_stream<TT_DATA>* __restrict sig_o) {
    buff0 = upd_w(buff0, 0, *his0);  // 加载历史状态
    // ... 处理逻辑 ...
    *his0 = ext_w(buff0, 0);         // 保存状态供下次使用
}

为什么这样做？

  • AIE 内核可能被调度器暂停和恢复，必须确保状态的持久性。
  • 显式控制允许优化状态加载/存储的时机，减少内存访问开销。

替代方案：使用 ADF 框架的自动状态保存机制，但这会增加运行时开销且降低可控性。

3. 宏定义的 SIMD 模式 vs. 通用循环

polyphase_fir.h 中使用宏定义了 5 种不同的 SIMD 计算模式（REPEATED_D0、NORMAL_FIR 等）：

#define REPEATED_D0(BUFF,COEFF,PTR,ACC) \
    BUFF = shft_elem(BUFF,readincr(sig_i)); \
    discard = readincr(sig_i); \
    // ... 更多向量操作

为什么使用宏？

  • 这些模式对应于非整数倍抽取率下的不同数据对齐情况，每种模式都有特定的寄存器使用模式。
  • 宏在预处理阶段展开，消除了函数调用开销，允许编译器进行跨模式的优化。

风险与注意事项：

  • 宏不进行类型检查，错误的使用可能导致难以调试的问题。
  • 代码可读性降低：需要理解宏展开后的实际效果。

4. 内存对齐与 Bank 冲突避免

代码中随处可见 alignas(32) 修饰符：

alignas(32) TT_COEFF (&taps0)[NUM_POLY*TAPS_PER_PHASE];
alignas(32) aie::vector<TT_DATA,TAPS_PER_PHASE> state0;

为什么 32 字节对齐？

  • AIE 的向量加载/存储指令通常操作 256-bit（32 字节）的向量寄存器。
  • 对齐访问可以避免跨缓存行（cache line）的访问，显著提升内存带宽利用率。

在 ddc_subsystem.h 中，还显式指定了数据存放的 memory bank：

location<buffer>(rx_hbf47.hbf47d.in[0]) = bank(xoff, yoff, 3);
single_buffer(rx_hbf47.demixer.in[0]);  
location<buffer>(rx_hbf47.demixer.in[0]) = bank(xoff, yoff, 2);

这种精细的内存布局控制是为了避免bank 冲突——当多个并发访问指向同一 memory bank 时产生的性能瓶颈。

子模块详解

本模块包含两个主要的设计示例，分别针对不同的应用场景：

1. 多相滤波器组信道化器 (polyphase_fir_app)

位于 04-Polyphase-Channelizer/aie/m16_ssr8_filterbank/ 目录，这是一个 16 通道的多相滤波器组实现。

核心组件：

  • my_graph: 顶层 Graph 类，实例化 PLIO 接口和 polyphase_fir_graph
  • polyphase_fir_graph: 封装单个 FIR 内核的 Graph 类
  • polyphase_fir: 实际的 FIR 滤波器内核实现

关键技术点：

  • SSR8 架构：8 个并行处理单元，每个处理 2 个通道
  • 非整数倍抽取率支持（P/Q = 8/7）
  • 基于 AIE 向量指令的高性能 MAC 运算

2. 数字下变频链 (ddc_chain)

位于 09-ddc_chain/src/ 目录，这是一个面向蜂窝通信的多级 DDC 实现。

核心组件：

  • TEST_32ant: 32 天线配置的顶层测试 Graph
  • ddc_32ant: 32 通道 DDC 阵列
  • ddc: 单通道 DDC 实现，包含多级处理
  • ddc_chain_kernel: 信道 FIR 滤波器内核
  • buffer_internal: 循环缓冲区管理结构

关键技术点：

  • 多级抽取架构（HBF47 -> Demixer -> HB Chain -> Channel FIR）
  • 复杂的循环缓冲区管理，支持大抽头数 FIR（89-tap、199-tap）
  • 显式的 Tile 位置约束，优化数据局部性

  • prime_factor_fft_decomposition_graphs：DDC 链中的某些处理阶段可能涉及频域转换，该模块提供了优化的 FFT 实现。

  • farrow_filter_design_evolution：与本模块同属多速率信号处理范畴，Farrow 滤波器提供了另一种分数倍采样率转换的实现方式。

  • polyphase_channelizer_system_integration：展示了如何将本模块的信道化器集成到完整的 Vitis 系统设计中，包括与 HLS 内核和 DMA 的交互。

新贡献者指南

常见陷阱与注意事项

  1. 模板参数的理解

    • PATTERN 参数（0-7）决定了数据处理的时序模式，错误的模式会导致输出数据错位。
    • NSAMP 必须是 4 的倍数（由 static_assert 保证），这是向量指令处理宽度的要求。

  2. 状态管理的生命周期

    • FIR 内核的状态（state0、state1）在构造函数中初始化，在每次 run_i 调用时加载和保存。
    • 如果内核被重置或重新初始化，状态会丢失，可能导致输出异常。

  3. Memory Bank 约束

    • 在 ddc_subsystem.h 中，buffer 的位置被显式分配到特定 bank。
    • 修改这些约束时需要考虑 bank 冲突问题，不当的布局会显著降低性能。

  4. 循环缓冲区的边界条件

    • buffer_internal 使用模运算实现循环寻址，需要确保 cyclic_add 的参数正确。
    • 缓冲区大小必须是 2 的幂次（16、32 等），以便使用位掩码代替昂贵的模运算。

  5. 编译器提示的使用

    chess_prepare_for_pipelining
chess_loop_range(8,)

    这些 Chess 编译器专用 pragma 指导编译器进行循环优化。删除或修改它们可能影响生成的代码质量。

调试建议

  • 使用 x86simulation：在部署到实际硬件前，先使用 x86 仿真验证功能正确性。
  • 检查数据文件格式：PLIO 接口期望特定格式的文本数据文件（如 data/sig0_i.txt），格式错误会导致仿真失败。
  • 验证系数初始化：滤波器系数在构造时传入，确保系数数组的生命周期覆盖整个 Graph 运行期。

扩展方向

如果你需要修改或扩展本模块：

  1. 增加通道数：修改 MCHAN 常量，并相应调整 SSR 并行度。
  2. 改变抽取率：更新 HOP 常量并重新计算对应的 PATTERN 序列。
  3. 替换滤波器系数：修改 taps_M16_init.h 或相应的系数初始化文件。
  4. 集成到更大系统：参考 polyphase_channelizer_system_integration 模块了解与 PL 侧的交互模式。
