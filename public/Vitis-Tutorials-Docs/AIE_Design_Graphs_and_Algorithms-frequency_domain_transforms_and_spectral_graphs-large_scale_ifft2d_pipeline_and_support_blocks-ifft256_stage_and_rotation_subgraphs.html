IFFT256 阶段与旋转子图 (ifft256_stage_and_rotation_subgraphs)

一句话概括

本模块是 64K点 2D-IFFT 高性能流水线 的核心计算单元，通过将大规模 FFT 分解为两个 256点变换的级联，配合"旋转因子乘法"(Twiddle Rotation)，实现了在 AI Engine 阵列上以 2 Gsps 吞吐量处理超大规模频域转换的目标。你可以把它想象成一条精密装配线：前端负责"横向扫描"数据并施加相位校正，后端负责"纵向扫描"完成最终合成，中间的零填充设计则巧妙地简化了跨域数据传输的复杂度。

问题空间与设计动机

为什么需要这个模块？

在 AI Engine 上实现大规模 FFT（如 64K 点）时，传统的 Stockham 算法虽然在小规模（N < 4096）表现优异，但当采样率超过 1 Gsps（SSR > 1 场景）时，会遇到严重的资源瓶颈：

  1. 蝴蝶路由开销过大：传统 Cooley-Tukey 算法的位反转和蝶形运算需要复杂的 tile 间通信
  2. 内存需求爆炸：单 tile 无法容纳大点数的中间结果
  3. 吞吐量难以扩展：随着点数增加，tile 资源呈非线性增长

2D 分解策略的智慧

本模块采用分而治之的 2D 架构思想：将 \(N = 65536\) 点的 IFFT 分解为两个 \(256 \times 256\) 的 2D 变换：

\[ \text{IFFT}_{64K}(x) = \text{IFFT}_{256}^{col}\left( \text{Twiddle} \odot \text{IFFT}_{256}^{row}(X) \right) \]

这种分解的优势在于：

  • 降低 tile 内存压力：每个维度仅需处理 256 点
  • 减少蝴蝶路由：依赖更规则的行列访问模式
  • 并行度可控：可通过实例化多个子图横向扩展吞吐量

零填充的工程妥协

为了支持 5 路并行流（满足 2 Gsps 目标），设计将 \(256 \times 256\) 的数据立方体零填充为 \(260 \times 260\)。这看似浪费，实则带来了关键收益：

  • 使每 tile 处理 52 个变换（260/5 = 52）
  • 简化 PL 端的存储器转置单元设计（5-bank 架构无冲突访问）
  • 允许统一的 5 流 I/O 接口

旋转因子乘法的本质

在 2D-FFT 算法中，两次 1D-FFT 之间需要逐点乘以旋转因子矩阵：

\[ W_{N}^{rc} = e^{+j2\pi rc/N}, \quad r,c \in [0, 255] \]

ifft256p4_rot 的创新之处在于将旋转因子计算内嵌到内核中，利用 AI Engine 的硬件 sincos 加速器实时生成相位值，而非预存庞大的查找表。这就像一个实时调谐的振荡器，根据当前处理的行号动态生成所需的相位斜坡。

流水线节拍器

想象一个交响乐团：

  • ifft256p4（后端）是低音部，节奏较快（~600 Msps），需要 4 个声部才能达到总音量
  • ifft256p4_rot（前端）是中音部，节奏较慢（~430 Msps），需要 5 个声部
  • 指挥家（系统架构师）决定统一使用 5 个声部，让后端偶尔"轮休"以保持同步

这种不均衡的吞吐率匹配是通过时间复用实现的：5 个前端实例和 5 个后端实例共享处理 256 个行列变换的任务。

旋转因子计算的精妙之处 (ifft256p4_rot.cpp):

// 计算初始相位：COUNT 是当前处理的行号偏移
unsigned int phase = COUNT << ALIGN;  // ALIGN=16, Q1.31 格式归一化

// 预计算 8 步相位斜坡
for (unsigned ii=0; ii < 8; ii++) {
    twid.push(aie::sincos_complex(phase_step));
    phase_step += phase;
}

// 主循环：粗相位 + 细相位混合生成
for (int ii=0; ii < NFFT_1D/8; ii++) {
    // coarse phase * fine phase，实现任意相位精度的旋转因子
    twid_use = (aie::mul(aie::reverse(twid), aie::sincos_complex(phase))).to_vector<TT_TWID>(15);
    phase += phase_step;
    
    // 8点并行复数乘法
    buff.insert(0,*itr++);
    buff.insert(1,*itr++);
    *itw++ = (aie::mul(buff.extract<4>(0),twid_use.extract<4>(0))).to_vector<TT_DATA>(15);
    *itw++ = (aie::mul(buff.extract<4>(1),twid_use.extract<4>(1))).to_vector<TT_DATA>(15);
}

模板参数的设计意图

ifft256p4_rot 使用模板参数 NSTREAM 和 PHASE，这是关键的多实例协调机制：

参数 含义 典型值
NSTREAM 并行流数量 5（决定行号递增步长）
PHASE 初始相位偏移 0..4（各实例不同，确保覆盖所有行）

行号计数器更新逻辑：

COUNT = ( COUNT >= NSAMP-NSTREAM ) ? PHASE : COUNT+NSTREAM;

这实现了循环轮转调度：5 个实例分别处理行号 0,5,10,...、1,6,11,... 等，确保 260 行被均匀分配。

关键设计决策与权衡

1. Stockham vs Cooley-Tukey 算法选择

选择：Stockham 自排序算法
权衡分析：

  • ✅ 优势：无需位反转置换，天然适合 SIMD 向量化；双缓冲策略避免 in-place 运算的复杂性
  • ❌ 代价：需要额外的临时缓冲区（tbuff），增加内存占用
  • 🤔 为何适合此处：AI Engine 的向量架构与 Stockham 的规整访存模式完美契合，缓存友好

2. 旋转因子实时计算 vs 预存 LUT

选择：硬件 sincos 加速器实时生成
权衡分析：

  • ✅ 优势：节省大量 tile 内存（256×256 旋转因子矩阵约需 128KB）
  • ❌ 代价：增加计算延迟，需要精细的流水线编排
  • 🤔 工程智慧：利用 aie::sincos_complex() 的硬件加速，将"内存瓶颈"转化为"计算并行"

3. 零填充的位置与大小

选择：每个 256点变换后填充 4 个零样本（NSAMP = 260）
权衡分析：

  • ✅ 优势：
    • 使 260 可被 5 整除，简化 5-bank 存储器设计
    • 统一前后端接口（都是 260 样本/变换）
    • 为 PL 端的乒乓缓冲提供额外裕量
  • ❌ 代价：约 1.5% 的带宽浪费（4/260）
  • 🤔 系统视角：微小的带宽损失换取架构简洁性，完全可接受

4. 单 tile 实现 vs 多 tile 拆分

选择：ifft256p4_rot 单 tile，ifft256p4 单 tile（但内存溢出到邻接 tile）
权衡分析：

  • ✅ 优势：最小化 tile 间通信，最大化计算密度
  • ❌ 代价：ifft256p4 需要借用邻接 tile 的内存（见物理布局图）
  • 🤔 资源规划：10 个计算 tile + 17 个缓冲 tile 的总成本，远低于纯 Stockham 方案的估计

5. 定点数缩放策略

选择：每级移位 15 位（SHIFT_TW = SHIFT_DT = 15）
权衡分析：

  • ✅ 优势：防止 FFT 过程中的数值溢出，保持 32-bit 动态范围
  • ❌ 代价：精度损失，需要仔细的位真建模验证
  • 🤔 验证保障：Matlab 参考模型（ifft64k_256x256_model.m）确保数值一致性

内存所有权与生命周期

缓冲区管理

缓冲区 类型 所有者 生命周期 对齐要求
tbuff 临时 FFT 缓冲区 内核对象（构造时注入） 与内核实例相同 32-byte (alignas(32))
sig_i / sig_o I/O 窗口 ADF 运行时管理 单次 run() 调用 由 ADF 保证

关键内存契约

// 构造函数：接收外部分配的临时缓冲区引用
ifft256p4( TT_DATA (&tbuff_i)[NFFT] ) : tbuff(tbuff_i)

重要：tbuff 是引用成员，必须在构造时绑定，且其生命周期必须长于内核实例。这由 graph 层的 std::vector 分配保证：

// Graph 层确保 tbuff 的生命周期
k_kernel = kernel::create_object<ifft256p4>(
    std::vector<ifft256p4::TT_DATA>(ifft256p4::NFFT)  // 堆分配，随 graph 销毁
);

数据类型约定

typedef cint32 TT_DATA;   // 32-bit 实部 + 32-bit 虚部 = 64-bit/样本
typedef cint16 TT_TWID;   // 16-bit 实部 + 16-bit 虚部 = 32-bit/旋转因子

精度权衡：数据通路使用 32-bit 保持动态范围，旋转因子使用 16-bit 节省内存和乘法器资源。

并发与执行模型

AI Engine 执行特性

  • 单线程内核：每个 ifft256p4/ifft256p4_rot 实例运行在单个 AIE tile 上
  • 无抢占式调度：内核通过 run() 方法完成一次完整变换后返回
  • 静态比率分配：runtime<ratio>(dut.k_kernel) = 0.9 表示该 tile 90% 时间用于此内核

新贡献者须知

常见陷阱

  1. COUNT 溢出风险

    COUNT = ( COUNT >= NSAMP-NSTREAM ) ? PHASE : COUNT+NSTREAM;

    如果 NSTREAM 或 PHASE 配置错误，可能导致行号追踪混乱，产生错误的旋转因子相位。

  2. 缓冲区别名混淆

    aie::fft_dit_r4_stage<16>(tbuff, ..., ibuff);  // 输出写回 ibuff
aie::fft_dit_r4_stage< 4>(ibuff, ..., tbuff);  // 下一级从 ibuff 读

    注意 ping-pong 模式的缓冲区切换，调试时打印指针地址确认。

  3. 旋转因子相位精度

    static constexpr unsigned ALIGN = 16;
// aie::sincos(arg) 期望 arg 为 Q1.31 格式，归一化到 π

    ALIGN = 16 来自 (31-15)，即 Q 格式转换需要的位移。修改 SHIFT_TW 时必须同步调整。

  4. 零填充位置

    itw += NFFT/4;  // 跳过 256/4 = 64 个向量 = 256 个样本
*itw++ = aie::zeros<TT_DATA,4>();  // 写入 4 个零

    零填充发生在输出缓冲区的第 256-259 位置，不是末尾追加！

调试建议

  • 数值验证：始终与 Matlab 参考模型对比，特别注意旋转因子的相位角
  • 吞吐量测量：使用 Vitis Analyzer 检查 II（Initiation Interval），目标应为连续启动
  • 内存检查：验证 tbuff 的对齐（32-byte）和大小（256 元素）

扩展点

如需修改变换点数（如改为 512点）：

  1. 更新 NFFT、NFFT_1D 常量
  2. 重新生成旋转因子表（使用 gen_vectors.m）
  3. 调整 DIT R4 阶段的 stride 值（当前为 64→16→4→1，对应 \(4^4 = 256\)）
  4. 重新计算零填充大小以满足并行度要求
