Vitis Export to Vivado: 自定义 IP 与多实例连接设计

一句话总结

本模块演示了 Vitis Export to Vivado 流程的高级用法——如何在保持软件定义数据流（AIE Graph + HLS Kernels）的同时，在 Vivado 中灵活集成自定义 RTL IP，并支持通过配置驱动的多实例拓扑扩展。这是 AMD Versal 自适应计算平台"软硬件协同设计"方法论的关键实践。

问题空间：为什么需要这个流程？

传统开发模式的痛点

在传统的 Versal ACAP 开发中，硬件工程师和软件工程师面临一个结构性矛盾：

  1. Vitis 统一流程的限制：Vitis 的 v++ 工具链擅长处理 AIE Graph 和 HLS Kernel 的自动化链接，但对自定义 RTL IP 的支持有限。如果硬件团队有一个精心优化的 Verilog 模块（如专用的 FEC 编码器、定制 DMA），很难直接插入 Vitis 的数据流。

  2. Vivado 原生流程的局限：纯 Vivado 流程虽然可以任意集成 RTL，但 AIE 的编译和 PL-AIE 接口的配置极其繁琐，容易出错。

  3. 迭代效率问题：一旦 AIE-PL 接口发生变化（比如增加一个 PLIO 端口），传统流程需要重新走完整的 Vitis 编译 → Vivado 综合 → 实现的漫长周期。

解决方案：双向移交（Bidirectional Hand-off）

Vitis Export to Vivado 流程引入了 Vitis Metadata Archive (VMA) 机制，实现了工具链间的"软边界"：

┌─────────────────┐     v++ --export_archive      ┌─────────────────┐
│   Vitis World   │ ─────────────────────────────→ │   Vivado World  │
│                 │         design.vma            │                 │
│  • AIE Graph    │                               │  • Block Design │
│  • HLS Kernels  │ ←─────────────────────────────│  • Custom RTL   │
│  • Connectivity │    vitis::import_archive      │  • Timing Closure│
└─────────────────┘                               └─────────────────┘

这种设计的精妙之处在于：Vitis 负责它擅长的（数据流编程、AIE 编译），Vivado 负责它擅长的（RTL 集成、时序优化），两者通过 VMA 格式交换元数据。

核心抽象：理解三个关键概念

1. SPTAG（Stream Port TAG）—— 平台的"插座"

SPTAG 是平台对外暴露的 AXI Stream 端口的逻辑名称。在 system_custom_ip.cfg 中：

sc=AIE_IN:ai_engine_0.DataIn1

这里的 AIE_IN 就是一个 SPTAG。它相当于在平台上预留了一个"插座"，Vivado 中的 RTL IP 可以通过这个插座向 AIE 注入数据。

类比：想象你家的电源插座。装修时（平台设计阶段）预留了位置和规格（SPTAG），入住后（应用开发阶段）可以插不同的电器（RTL IP 或 HLS Kernel）。

2. VMA（Vitis Metadata Archive）—— 软硬件的"契约书"

VMA 文件是 v++ -l --export_archive 生成的压缩归档，包含：

  • AIE Graph 的编译产物 (libadf.a)
  • PL Kernel 的 .xo 对象文件
  • 连接关系图（connectivity graph）
  • 时钟和复位配置

关键洞察：VMA 不是最终比特流，而是一种"可重定位的硬件描述"。它允许 Vivado 在不重新编译 AIE/PL 代码的情况下，调整物理实现细节。

3. Instance Naming Convention —— 多实例的"身份证"

在 system_new.cfg 中可以看到多实例的命名模式：

nk=stream_in:2:stream_in1.stream_in2
nk=polar_clip:2:polar_clip1.polar_clip2

这里 nk=<kernel>:<count>:<inst1>.<inst2> 的语法定义了：

  • 同一个 kernel 逻辑被实例化多次
  • 每个实例有独立的后缀标识（_1, _2）
  • 连接声明中使用这些后缀来区分不同实例的端口

架构详解：两种配置模式的对比

本模块提供了三种系统配置，展示了从简单到复杂的演进路径：

模式一：基础 HLS 流程 (system.cfg)

MM2S] -->|DataIn1| AIE[AI Engine] AIE -->|clip_in| PC[polar_clip] PC -->|clip_out| AIE AIE -->|DataOut1| SI[stream_in
S2MM] end

所有 PL 内核都是 HLS 生成，完全在 Vitis 生态内。

关键变化：stream_out 不再是 HLS kernel，而是 Vivado 中手动添加的 RTL IP。Vitis 通过 AIE_IN 这个 SPTAG "看不到" 数据源的实现细节，只关心数据格式和时钟域。

模式三：多实例扩展 (system_new.cfg)

flowchart TB subgraph "双实例并行处理" direction TB IN[aie_stream_in] -->|DataIn_1| G1[Graph Instance 1] IN -->|DataIn_2| G2[Graph Instance 2] G1 -->|clip_in_1| PC1[polar_clip1] PC1 -->|clip_out_1| G1 G1 -->|DataOut_1| SI1[stream_in1] G2 -->|clip_in_2| PC2[polar_clip2] PC2 -->|clip_out_2| G2 G2 -->|DataOut_2| SI2[stream_in2] end

设计意图：展示如何通过简单的配置修改实现计算单元的横向扩展。两个 graph 实例共享相同的 aie_stream_in 数据源，但独立处理并输出到各自的 stream_in 接收端。

数据流追踪：端到端的信号旅程

以 system_custom_ip.cfg 为例，跟踪一个数据样本的完整生命周期：

阶段 1：RTL IP 产生数据（Vivado 域）

// custom_ip/stream_out/hdl/verilog/stream_out.v
module stream_out (
    input  ap_clk,
    input  ap_rst_n,
    output [31:0] s_TDATA,    // 复数样本 {imag[15:0], real[15:0]}
    output        s_TVALID,
    input         s_TREADY,
    ...
);

这是一个自由运行（free-running）的 RTL 模块，从内部 ROM 读取预置的测试向量，通过 AXI Stream 协议输出。

阶段 2：平台接口路由（Platform Layer）

在 Vivado Block Design 中，stream_out 的 s 端口连接到标记为 AIE_IN 的平台端口。这个连接在 bd.tcl 中建立：

# Platform Setup 步骤中设置
set_property PFM.AXIS_PORT {AIE_IN {memport "axis"}} [get_bd_cells /some_interface]

阶段 3：Vitis 链接的连通性解析

v++ -l 读取 system_custom_ip.cfg：

sc=AIE_IN:ai_engine_0.DataIn1

这条语句告诉链接器：将平台端口 AIE_IN 连接到 AIE array 的 DataIn1 PLIO。

阶段 4：AIE Graph 内部处理

// aie/graph.h
in = adf::input_plio::create("DataIn1", adf::plio_32_bits, "data/input.txt");
connect(in.out[0], interpolator.in[0]);

数据进入 clipped graph，经过：

  1. Interpolator (hb27_2i.cc): 27-tap FIR 滤波器，2x 插值上采样
  2. Polar Clip (外部 HLS kernel): CORDIC 算法计算幅度/相位，应用门限
  3. Classifier (classify.cc): 象限分类，输出 0-3 的类别标签

阶段 5：返回 PL 并写入内存

out = adf::output_plio::create("DataOut1", adf::plio_32_bits, "data/output.txt");
connect(classify.out[0], out.in[0]);

对应配置：

sc=ai_engine_0.DataOut1:stream_in.s

stream_in HLS kernel 将 AXI Stream 转换为 AXI-MM 写入 DDR。

设计决策与权衡分析

决策 1：为什么选择 ap_ctrl_none 控制模式？

观察 polar_clip.cpp：

#pragma HLS INTERFACE ap_ctrl_none port=return

这表示该 kernel 是自由运行的（free-running），没有 start/done 控制信号。

权衡考量：

  • ✅ 优点：数据流驱动，无需 host 干预启动；适合流水线式处理
  • ❌ 代价：无法单独复位或调试单个 kernel；必须保证上游持续供数，否则会死锁

适用场景：数据流连续、无反馈环路的信号处理链路。

决策 2：单端口 vs 多端口存储器访问

stream_in 和 aie_stream_in 都使用 m_axi 接口：

#pragma HLS INTERFACE m_axi port=mem offset=slave bundle=gmem

但 stream_out（RTL 版本）使用内部 ROM，不访问外部 DDR。

设计意图分离：

  • 输入侧：可以从 DDR 加载动态数据（aie_stream_in）或使用固定测试向量（RTL stream_out）
  • 输出侧：必须写入 DDR 供 host 验证结果

决策 3：PL 频率选择的工程含义

freqhz=200MHz:stream_in.ap_clk
[clock]
freqHz=100000000:polar_clip.ap_clk

注意 stream_in 运行在 200MHz，而 polar_clip 只有 100MHz。

原因分析：

  • stream_in 是纯数据传输（AXI-MM → AXI-Stream），逻辑简单，可以跑高速
  • polar_clip 包含 CORDIC 迭代计算（6 步旋转），组合逻辑深，降低频率以保证时序收敛

这体现了异构计算的分频策略：不同计算特性的模块绑定到不同的时钟域，通过 CDC（Clock Domain Crossing）FIFO 桥接。

新贡献者必读：隐性契约与常见陷阱

🔴 陷阱 1：SPTAG 名称不匹配导致静默失败

如果在 Vivado 中将平台端口命名为 AIE_IN，但在 cfg 文件中拼写为 AIE_IN_1，v++ 链接不会报错——它会创建一个未连接的悬空端口，直到硬件仿真时才发现数据从未到达 AIE。

防御措施：始终使用 v++ --save-temps 保留中间文件，检查 _x/link/vivado/vpl/prj/prj.gen/sources_1/bd/*/bd.tcl 中的实际连接。

🔴 陷阱 2：多实例命名后缀的语法严格性

# 正确：点号分隔，无空格
nk=stream_in:2:stream_in1.stream_in2

# 错误：逗号分隔（会被解析为单个实例名）
nk=stream_in:2:stream_in1,stream_in2

这种语法来自 Vitis 链接器的早期设计选择，文档中往往语焉不详。

🔴 陷阱 3：AIE Graph 模板参数与实例名的耦合

在 multi_graph.h 中：

template <int N>
class clipped : public graph {
    in = adf::input_plio::create("DataIn_"+std::to_string(N), ...);
};

clipped<1> clipgraph1;  // 创建 DataIn_1
clipped<2> clipgraph2;  // 创建 DataIn_2

模板参数 N 不仅用于生成唯一的 PLIO 名称，还必须与 cfg 中的连接声明严格对应：

sc=AIE_IN:ai_engine_0.DataIn_1  # 对应 clipgraph1
sc=aie_stream_in.s:ai_engine_0.DataIn_2  # 对应 clipgraph2

如果修改了模板逻辑但没有同步更新 cfg，会导致连接错位。

🟡 注意事项：HLS 数据类型的位宽对齐

polar_clip 处理的是 32-bit 打包复数：

void polar_clip(hls::stream<ap_axis<32, 0, 0, 0>> &in_sample, ...)
// 低 16-bit: real, 高 16-bit: imag

这与 AIE 侧的 cint16（16-bit 实部 + 16-bit 虚部）天然对齐，但如果未来改为 cfloat（32-bit float x2），整个位宽映射需要重新设计。
