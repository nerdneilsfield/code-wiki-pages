Finder 模块技术深度解析

概述：在噪声中寻找信号的"探针"

想象你正在一个嘈杂的体育场里寻找某个特定的人——你不能逐个检查每个人，而是需要一种系统性的方法：先扫描整个场地找出可能的位置（这是上游 scanner 的工作），然后对每个候选位置进行精确定位，确认那是否真的是你要找的人。这就是 finder 模块在整个 MUSIC（Multiple Signal Classification）算法中的角色。

MUSIC 算法是一种高分辨率波达方向（DOA, Direction of Arrival）估计技术，广泛应用于雷达、声纳和无线通信领域。算法的核心思想是将传感器阵列接收到的信号分解为信号子空间和噪声子空间，然后通过搜索噪声子空间与导向矢量的正交性来估计信号源的方向。在这个流程中，finder 负责峰值精确定位——它接收来自 scanner 的空间谱数据，通过局部搜索找出精确的峰值位置，这些峰值对应着信号源的到达角度。

这个模块的设计体现了典型的流水线并行计算思想：将一个大问题切分成多个小任务，分配给多个 AIE（AI Engine）核并行处理，每个核只负责一小片区域的精细搜索。这种设计不是简单的性能优化，而是由 AIE 架构的物理约束决定的——单个 AIE 核的计算资源和本地内存有限，必须通过空间并行化来实现实时处理。

模块在系统中的位置

finder 是 MUSIC 算法流水线的最后阶段，位于整个信号处理链的末端：

  1. io_adapter —— I/O 适配，负责数据格式转换和流控
  2. qrd —— QR 分解，矩阵正交化处理
  3. svd —— 奇异值分解，提取噪声子空间
  4. doa —— 空间谱计算，生成 MUSIC 谱
  5. scanner —— 空间谱扫描，初步识别峰值区域
  6. finder（本模块） —— 峰值精确定位，确定最终 DOA 估计

这种分层设计遵循渐进精细化原则：scanner 快速扫描整个角度范围找出候选区域，finder 对这些候选区域进行精细分析，避免在全范围内进行高精度搜索带来的计算浪费。

核心组件详解

1. Finder 类模板 —— 精确定位的执行单元

Finder 是一个 C++ 类模板，定义在 finder.h 中，是实际执行峰值查找计算的 AIE kernel：

template<unsigned REGION_ID_0, unsigned REGION_ID_1>
class Finder {
  public:
    void run(adf::input_buffer<cfloat, adf::extents<adf::inherited_extent>> & __restrict in,
             adf::output_buffer<cfloat, adf::extents<adf::inherited_extent>> & __restrict out);
  private:
    alignas(32) aie::vector<cfloat, N> Pm_null[NUM_REGIONS];
};

设计意图与模板参数

模板参数 REGION_ID_0 和 REGION_ID_1 定义了该 kernel 负责处理的角度区域范围。这种设计实现了静态任务分区：

  • 编译时确定每个 kernel 的责任区域，避免运行时调度开销
  • 总共 NUM_FINDER_KERNELS = 16 个 kernel，每个处理 2 个连续区域（NUM_REGIONS_PER_KERNEL = 2）
  • 区域编号从 0 到 31（NUM_REGIONS = 32），覆盖 256 个角度点（NUM_POINTS = 256）

这种静态分区策略是空间并行流水线的典型模式——每个 AIE tile 被赋予固定的几何责任区，数据按顺序流过各个 tile，每个 tile 完成自己的一部分工作后传递给下一个。

内存布局与对齐要求

alignas(32) aie::vector<cfloat, N> Pm_null[NUM_REGIONS];
  • alignas(32) 确保数组在 32 字节边界对齐，这是 AIE 向量加载/存储单元的硬件要求
  • Pm_null 缓存输入的空间谱数据，避免重复访问外部存储器
  • 使用 aie::vector<cfloat, N>（其中 N = COL = 8）表示 SIMD 向量，一次处理 8 个复数样本

2. finder_graph —— 流水线拓扑构建器

finder_graph.h 定义了 AIE graph 的拓扑结构，负责将 16 个 Finder kernel 连接成一条线性流水线：

class finder_graph : public graph {
public:
    template <unsigned START, unsigned END>
    void finder_kernel_create(void) {
      if constexpr (START <= END) {
        finder_kernel[START] = adf::kernel::create_object<Finder<2 * START, 2 * START + 1>>();
        finder_kernel_create<START + 1, END>();
      }
    }
    adf::kernel finder_kernel[NUM_FINDER_KERNELS];
    // ...
};

编译期递归实例化

finder_kernel_create 使用 C++17 编译期递归（if constexpr）来实例化 16 个不同模板参数的 kernel。这种技术：

  • 在编译时生成所有 kernel 实例，避免运行时类型分发
  • 每个 kernel 获得唯一的区域分配（Finder<0,1>, Finder<2,3>, ..., Finder<30,31>）
  • 保持代码紧凑——16 行展开的逻辑用 6 行模板元编程表达

流水线连接模式

// 线性流水线连接
adf::connect<>(sig_i, finder_kernel[0].in[0]);
for (unsigned i = 0; i < NUM_FINDER_KERNELS - 1; ++i)
    adf::connect<>(finder_kernel[i].out[0], finder_kernel[i+1].in[0]);
adf::connect<>(finder_kernel[NUM_FINDER_KERNELS-1].out[0], sig_o);

数据以令牌环方式流动：每个 kernel 消费前一个的输出，产生自己的输出给下一个。这种设计的精妙之处在于：

  • 无中央控制器：没有仲裁逻辑，数据流动由 AIE 硬件流网络自动管理
  • 天然流水线：16 个 kernel 可以同时处理 16 个不同的数据帧
  • 确定性延迟：数据从输入到输出的延迟是可预测的（约 16 个 kernel 的处理周期）

3. dut_graph —— 系统集成层

finder_app.cpp 中的 dut_graph 是完整的可测试设计单元（Design Under Test）：

class dut_graph : public graph {
public:
  finder_graph dut;
  input_plio   sig_i;
  output_plio  sig_o;

  dut_graph(void) {
    sig_i = input_plio::create("PLIO_i", plio_64_bits, "data/sig_i.txt");
    sig_o = output_plio::create("PLIO_o", plio_64_bits, "data/sig_o.txt");
    connect<>(sig_i.out[0], dut.sig_i);
    connect<>(dut.sig_o, sig_o.in[0]);

    // 显式放置：第 0 行，列 12-27
    static constexpr unsigned COL_START = 12;
    for (unsigned i = 0, col = COL_START; i < NUM_FINDER_KERNELS; ++i, ++col) {
      adf::location<adf::kernel>(dut.finder_kernel[i]) = adf::tile(col, ROW_0);
      adf::location<adf::stack>(dut.finder_kernel[i]) = adf::location<adf::kernel>(dut.finder_kernel[i]);
    }
  }
};

物理布局约束

代码中的显式 placement 是关键的设计决策：

  • 固定列位置：kernel 被放置在第 12 到 27 列（共 16 列）
  • 同行放置：所有 kernel 都在第 0 行（ROW_0 = 0）
  • 栈与核同位置：每个 kernel 的栈放在与其计算核相同的 tile，最小化访存延迟

这种布局选择反映了 AIE 架构的特点：

  1. 水平流水线优势：同一行内的 tile 之间有直接的邻居连接，数据横向流动效率最高
  2. 避免垂直路由：跨行的流连接需要通过交换机矩阵，引入额外延迟
  3. 预留其他资源：第 0-11 列留给 pipeline 的其他阶段（svd, scanner 等）

核心算法：Finder::run() 深度解析

输入/输出契约

void Finder<REGION_ID_0, REGION_ID_1>::run(
    adf::input_buffer<cfloat, adf::extents<adf::inherited_extent>> & __restrict in,
    adf::output_buffer<cfloat, adf::extents<adf::inherited_extent>> & __restrict out
);

输入缓冲区结构（总大小：NUM_POINTS + NUM_REGIONS = 256 + 32 = 288 个 cfloat）：

段 内容 大小
Pm_null 缓存 32 个区域的空间谱数据 32 × 8 = 256 个 cfloat
Tag 标记 各区域的峰值候选标记 8 个 cfloat（打包 8 个 float）
直通数据 非本 kernel 处理的数据 剩余部分

输出缓冲区结构：与输入对称，但 tag 可能被修改以记录找到的精确峰值位置。

算法流程

                              ↓

峰值检测逻辑

核心判断条件（finder.cpp 第 64 行）：

if ((testbed[index] < DOA_MIN_THRESHOLD) && 
    (testbed[index-1] > testbed[index]) && 
    (testbed[index] < testbed[index+1])) {
    const unsigned n = i * N + index - 1;
    tag_cfloat[i % N] = cfloat({float(n), 0.0f});
}

这是一个三条件联合检测：

  1. 幅度阈值：testbed[index] < DOA_MIN_THRESHOLD（0.25）—— 确保是噪声子空间的深零点
  2. 左侧下降：testbed[index-1] > testbed[index] —— 左侧斜率为负
  3. 右侧上升：testbed[index] < testbed[index+1] —— 右侧斜率为正

满足这三个条件的点是局部极小值，对应 MUSIC 谱的峰值（因为 MUSIC 谱是噪声投影的倒数，噪声投影的极小值对应信号方向的极大值）。

边界处理策略

// ignore the left most and right most regions
if (i == 0 || i == NUM_REGIONS - 1) continue;

边界区域被排除的原因：

  • 数据完整性：检测需要左右邻域的数据（testbed 构造使用了 Pm_null[i-1] 和 Pm_null[i+1]）
  • 物理合理性：DOA 估计通常有有效角度范围，边界往往对应不可达的物理角度

滑动窗口构造

float testbed[1 + ANGLE_FINDER_REGION_LEN + 1] = {1.0f, 2.0f, ..., 10.0f};
// ...
unsigned index = 0;
cfloat previous = Pm_null[i-1][N-1];
testbed[index++] = previous.real;
for (unsigned j = 0; j < N; ++j) {
    cfloat tmp = Pm_null[i][j];
    testbed[index++] = tmp.real;
}
cfloat tmp = Pm_null[i+1][0];
testbed[index] = tmp.real;

testbed 是一个一维滑动窗口，包含：

  • 左邻域的最后一个点（索引 0）
  • 当前区域的全部 8 个点（索引 1-8）
  • 右邻域的第一个点（索引 9）

这种构造方式实现了跨区域的连续性检测——峰值可能恰好落在两个区域的交界处。

关键设计决策与权衡

1. 静态分区 vs 动态负载均衡

选择的方案：编译期静态分区（每个 kernel 固定处理 2 个区域）

权衡分析：

维度 静态分区（当前） 动态负载均衡（替代方案）
调度开销 零运行时开销 需要任务队列和仲裁逻辑
内存占用 每个 kernel 只需缓存自己的区域 需要全局状态或共享队列
负载均衡 均匀（假设峰值分布均匀） 更好适应非均匀分布
实现复杂度 简单，利用 C++ 模板 复杂，需要运行时系统支持
确定性 完全确定，便于调试和验证 非确定，行为依赖于输入

为什么选择静态分区：MUSIC 算法的应用场景（雷达、通信）通常有严格的实时性要求，确定性行为比平均性能更重要。此外，AIE 的硬件流网络天然适合静态数据流图，动态调度会引入不必要的复杂性。

2. 线性流水线 vs 树形归约

选择的方案：16 个 kernel 排成一条直线，数据顺序流过

为什么不是树形结构：

  • 数据依赖性：每个 kernel 需要看到完整的 Pm_null 缓存来做跨边界检测，树形结构的局部汇聚会破坏这种可见性
  • 延迟敏感性：finder 是 pipeline 的最后阶段，后续还有 doa 模块，线性流水线的延迟（16 周期）是可接受的
  • 面积效率：树形结构需要更多的流连接和缓冲，而 AIE 的流资源是有限的

3. 双缓冲 vs 单缓冲数据流

观察 finder.cpp 的数据访问模式：

auto inIterator  = aie::begin_vector<N>(in);
auto outIterator = aie::begin_vector<N>(out);

这里使用了独立的输入/输出迭代器，暗示了 ping-pong buffer 或双缓冲机制。这种设计的考量：

  • 避免读写冲突：输入和输出使用不同的物理缓冲区，kernel 可以安全地写入输出而不影响正在读取的输入
  • 流水线并发：当当前 kernel 在处理 frame N 时，前一个 kernel 可以写入 frame N+1 的输出，后一个 kernel 可以读取 frame N-1 的输出
  • DMA 友好：双缓冲允许 DMA 在后台传输下一帧数据，同时当前帧正在被处理

4. __restrict 关键字的使用

adf::input_buffer<cfloat, ...> & __restrict in

__restrict 是 C99/C++ 的关键字，告诉编译器该指针是访问其指向数据的唯一方式（没有别名）。在 AIE 编译器中，这允许：

  • 更激进的向量化：编译器可以放心地将内存访问重排为 SIMD 操作
  • 消除冗余加载：知道没有别名后，可以从寄存器复用加载的值
  • 更好的流水线调度：内存访问模式变得完全可预测

依赖关系与数据契约

上游依赖：Scanner 模块

finder 接收来自 scanner 的输出，期望的数据格式：

  • Pm_null：256 个点的空间谱值，组织为 32 个区域 × 8 点/区域
  • Tag：32 个标记位（打包在 8 个 cfloat 中），指示哪些区域包含候选峰值

隐式契约：

  • scanner 保证只在可能存在峰值的区域设置 tag（非零值）
  • 如果 scanner 错误地标记了无峰值区域，finder 会进行空搜索（浪费周期但不会出错）
  • 如果 scanner 漏掉了真实峰值，finder 无法恢复（信息已丢失）

下游依赖：DOA 模块

finder 的输出传递给 doa 模块：

  • 更新的 Tag：包含精确峰值位置的索引（编码在 cfloat.real 中）
  • 直通的 Pm_null：未修改的空间谱数据，供 doa 做进一步处理

输出保证：

  • 对于每个检测到的峰值，finder 在对应的 tag 位置写入 cfloat(peak_index, 0.0f)
  • 未检测到峰值的位置保持原值（通常是 0）
  • 输出缓冲区的结构与输入严格一致，便于下游模块解析

共享配置：music_parameters.h

所有 MUSIC 模块共享 music_parameters.h 中的常量定义：

static constexpr unsigned NUM_POINTS               = 256;
static constexpr unsigned ANGLE_FINDER_REGION_LEN  = 8;
static constexpr unsigned NUM_REGIONS              = 32;  // NUM_POINTS / 8
static constexpr unsigned NUM_FINDER_KERNELS       = 16;  // NUM_REGIONS / 2
static constexpr float    DOA_MIN_THRESHOLD        = 0.25f;

重要约定：这些常量在编译时确定，修改它们需要重新编译所有相关模块。特别需要注意的是：

  • ANGLE_FINDER_REGION_LEN 必须是 COL（8）的因子，因为数据以 8 个 cfloat 的向量形式处理
  • NUM_POINTS 必须能被 ANGLE_FINDER_REGION_LEN 整除，确保区域划分无 remainder
  • DOA_MIN_THRESHOLD 的选择影响检测灵敏度，需要根据信噪比调整

修改参数的影响

如果需要调整算法参数，以下是关键修改点：

增加角度分辨率（更多点数）：

  1. 修改 music_parameters.h 中的 NUM_POINTS
  2. 确保 NUM_POINTS 仍能被 8 整除
  3. 可能需要增加 NUM_FINDER_KERNELS 以保持每个 kernel 的负载合理
  4. 更新 finder_graph.h 中的递归实例化范围

调整检测灵敏度：

  • 修改 DOA_MIN_THRESHOLD（默认 0.25）
  • 较低的值：更容易检测到弱信号，但可能引入假阳性
  • 较高的值：减少假阳性，但可能漏掉弱信号

改变并行度：

  • 当前 16 个 kernel，每个处理 2 个区域
  • 可以增加 kernel 数量（如 32 个，每个处理 1 个区域）以降低延迟
  • 或减少 kernel 数量（如 8 个，每个处理 4 个区域）以节省资源

常见陷阱

  1. 忘记更新所有相关常量

    // 错误：只改了 NUM_POINTS，没改 NUM_REGIONS
static constexpr unsigned NUM_POINTS = 512;  // 原来是 256
// NUM_REGIONS 仍然是 32，导致数据解析错误

  2. 违反向量对齐要求

    // 危险：移除 alignas 可能导致性能下降或运行时错误
// alignas(32) aie::vector<cfloat, N> Pm_null[NUM_REGIONS];
aie::vector<cfloat, N> Pm_null[NUM_REGIONS];  // 未对齐！

  3. 修改 kernel 数量但不更新 placement

    // 错误：增加了 kernel 但没有增加列范围
static constexpr unsigned COL_START = 12;
// 循环只到 16，但可能有 32 个 kernel
for (unsigned i = 0, col = COL_START; i < NUM_FINDER_KERNELS; ++i, ++col)

  4. 误解 tag 的编码方式

    // tag 是打包的：每个 cfloat 包含 4 个 float（实部）
// 直接比较 cfloat 而不是提取实部会导致错误
if (tag_cfloat[i] == 0)  // 错误！应该检查 .real

性能特征

吞吐量估算

基于代码结构和 AIE 架构特性：

  • 每个 kernel 每帧工作量：

    • 读取 32 个向量（Pm_null）：32 周期
    • 处理 2 个区域（最坏情况全搜索）：约 100 周期
    • 写入输出：32 周期
    • 总计：约 200 周期/帧

  • 流水线吞吐量：

    • 16 级流水线，启动间隔（II）约为单个 kernel 的处理时间
    • 估算吞吐：约 200 周期/帧 ≈ 5M 帧/秒 @ 1GHz

  • 延迟：

    • 从输入到输出：16 × 200 = 3200 周期 ≈ 3.2μs @ 1GHz

瓶颈分析

  1. 内存带宽：Pm_null 的加载是主要带宽消耗点（32 向量 × 16 字节/向量 = 512 字节/帧）
  2. 分支预测：峰值检测的条件分支可能影响流水线效率
  3. 标量运算：tag 的解包和重新打包涉及标量操作，可能成为瓶颈

优化建议

如需进一步优化：

  1. 向量化 tag 处理：当前 tag 处理是标量循环，可以尝试向量化
  2. 循环展开：手动展开内层循环以减少分支开销
  3. 数据预取：使用 chess_prepare_for_pipelining 提示（已在代码中使用）
  4. 减少直通数据：如果下游不需要完整的 Pm_null，可以只传递必要部分

总结

finder 模块是 MUSIC 算法流水线中的精确定位引擎，它将粗粒度的候选区域细化为精确的峰值位置。其设计体现了嵌入式信号处理的核心原则：

  • 空间并行：16 个 AIE kernel 分工协作，每个负责一小片角度区域
  • 静态调度：编译期确定任务分配，消除运行时开销
  • 流水线数据流：线性连接实现高吞吐量和确定性延迟
  • 硬件感知：对齐约束、向量操作、显式 placement 都针对 AIE 架构优化

理解这个模块的关键在于把握渐进精细化的思想——scanner 快速筛选，finder 精确定位，两级协作在保证实时性的同时实现高分辨率 DOA 估计。这种分层策略是复杂信号处理系统的通用模式，也是 AIE 编程中平衡资源与性能的典型范例。
