pl_kernels_mm2s_s2mm 模块深度解析

一句话概述

pl_kernels_mm2s_s2mm 是 Versal ACAP 平台上 AIE（AI Engine）与 PL（Programmable Logic）集成的数据搬运基础设施——它提供了一对极简的 HLS 内核 mm2s（Memory-to-Stream）和 s2mm（Stream-to-Memory），负责在 DDR 内存与 AXI Stream 接口之间建立高速数据通道。可以把它们想象成连接两个不同"交通系统"的转换闸口：一边是面向大块数据随机访问的"高速公路"（AXI4-Full 连接到 DDR），另一边是面向流式数据顺序传输的"地铁轨道"（AXI4-Stream 连接到 AIE）。

问题空间：为什么需要这对内核？

异构计算的内存墙问题

Versal ACAP 架构将计算资源分为三个层次：

  • PS（Processing System）：ARM 处理器，运行 Linux 和主机程序
  • PL（Programmable Logic）：可编程逻辑，即传统 FPGA 结构
  • AIE（AI Engine）：专用 AI 加速阵列，拥有高带宽本地存储和 SIMD 能力

这三个世界使用不同的内存模型和通信协议：

  • PS/PL 通过 DDR 共享数据，使用基于地址的随机访问（AXI4-Full）
  • AIE 核心之间以及 AIE 与 PL 之间使用流式数据传输（AXI4-Stream），强调低延迟、确定性时序

核心矛盾：主机程序生成的数据躺在 DDR 中，但 AIE 阵列期望以流的形式消费数据；反之，AIE 的输出是流，但需要写回 DDR 供主机读取。

为什么不能直接用 DMA 控制器？

Xilinx 提供了 hardened DMA（如 XDMA、CDMA），但它们通常：

  1. 配置复杂：需要设置描述符链、中断处理
  2. 灵活性受限：难以与自定义 PL 逻辑或特定的 AIE 数据格式要求配合
  3. 集成开销：在 Vitis 统一软件流程中，HLS 内核比直接实例化 DMA IP 更易于版本控制和参数化

因此，我们需要一种轻量级、可综合、完全受 HLS 工具链管理的数据搬运方案——这就是 mm2s 和 s2mm 存在的意义。

HLS 构建配置解析

在深入代码之前，先理解 Vitis HLS 的构建配置有助于把握模块的整体设计意图。

mm2s.cfg / s2mm.cfg

[hls]
flow_target=vitis              # 目标流程：Vitis 统一软件平台
syn.file=mm2s.cpp              # 综合源文件
syn.cflags=-I.                 # 头文件搜索路径
syn.top=mm2s                   # 顶层函数名（必须与 extern "C" 中的函数名一致）
package.ip.name=mm2s           # 生成的 IP 名称
package.output.syn = true      # 输出综合报告
package.output.format=xo       # 输出格式：Xilinx Object (.xo)
package.output.file=mm2s.xo    # 输出文件名

关键决策分析：

  1. flow_target=vitis vs vivado

    • vitis：生成适用于 Vitis 统一平台的 .xo 文件，可被 v++ -l 链接阶段自动集成
    • vivado：生成传统 RTL IP，需要手动在 Vivado 中实例化
    • 本模块选择前者，因为它是教程演示 Vitis 流程的一部分

  2. package.output.format=xo

    • .xo 是 Vitis 的"对象文件"格式，类似于软件编译中的 .o
    • 它封装了 RTL 网表、接口定义和元数据，使 v++ 链接器能够自动处理时钟、复位、总线连接
    • 对比传统流程：需要手动在 Vivado 中连接 AXI 端口、配置地址映射

  3. 为什么没有 syn.part 指定器件型号？

    • 器件信息由 --platform 参数在命令行传入：v++ -c --platform $PLATFORM_REPO_PATHS/...
    • 这使得同一套 cfg 文件可以在不同平台（VCK190、VEK280 等）上复用

代码深度剖析

mm2s.cpp：内存到流的单向泵

#include <ap_int.h>
#include <hls_stream.h>
#include <ap_axi_sdata.h>

extern "C" {

void mm2s(ap_int<32>* mem, hls::stream<ap_axis<32, 0, 0, 0>>& s, int size) {
#pragma HLS INTERFACE m_axi port=mem offset=slave bundle=gmem
#pragma HLS interface axis port=s
#pragma HLS INTERFACE s_axilite port=mem bundle=control
#pragma HLS INTERFACE s_axilite port=size bundle=control
#pragma HLS interface s_axilite port=return bundle=control

    for(int i = 0; i < size; i++) {
#pragma HLS PIPELINE II=1
        ap_axis<32, 0, 0, 0> x;
        x.data = mem[i];
        s.write(x);
    }
}

}

接口设计的工程考量

1. 三重接口类型的分工

#pragma HLS INTERFACE m_axi port=mem offset=slave bundle=gmem      // 数据通路
#pragma HLS interface axis port=s                                     // 数据通路  
#pragma HLS INTERFACE s_axilite port=mem bundle=control              // 控制通路
#pragma HLS INTERFACE s_axilite port=size bundle=control             // 控制通路
#pragma HLS interface s_axilite port=return bundle=control           // 控制通路

这是一个经典的分离式接口设计：

  • m_axi（AXI4-Full）：用于高带宽数据传输。offset=slave 表示该指针是相对于内核地址空间的从机偏移，bundle=gmem 将所有 gmem 端口的信号捆绑到同一个 AXI 总线接口上，节省资源。
  • axis（AXI4-Stream）：用于流式数据输出。这是与 AIE 阵列通信的标准方式。
  • s_axilite（AXI4-Lite）：用于主机配置。mem、size 和 return 都映射到控制寄存器空间，主机通过 XRT API 写入这些寄存器来启动内核。

为什么用 s_axilite 而不是把控制信号也做成 m_axi？

  • AXI4-Lite 是轻量级寄存器访问协议，适合少量配置数据
  • 它不会占用高性能 AXI4-Full 总线的带宽
  • Vitis 工具链对 s_axilite 有成熟的驱动生成支持

2. 数据类型选择的微妙之处

ap_int<32>* mem          // 32位定点整数指针
ap_axis<32, 0, 0, 0>     // 32位数据，无用户/保持/ID字段
  • ap_int<32> 而非 int：明确告诉 HLS 这是任意精度整数类型，避免 C 语言 int 的歧义（虽然在此场景下等价）
  • ap_axis<32, 0, 0, 0>：ap_axis 是 AXI Stream 的 HLS 封装模板，四个参数分别是数据位宽、用户位宽、保持位宽、ID 位宽。全零表示最简形式——只有数据和有效信号

为什么选择最简 AXIS 格式？

  • AIE 阵列的标准 PLIO（PL I/O）接口期望这种格式
  • 减少不必要的信号线，降低布线拥塞
  • 如果下游需要额外的 sideband 信息（如 packet headers），会在 AIE graph 层处理

3. 流水线策略

for(int i = 0; i < size; i++) {
#pragma HLS PIPELINE II=1
    // ...
}

II=1（Initiation Interval = 1）意味着每个时钟周期启动一次循环迭代。这是该内核能达到的理论最高吞吐率。

关键约束分析：

  • 读端口瓶颈：mem[i] 需要从 DDR 读取。若 DDR 延迟为 L 个周期，HLS 会自动插入流水线阶段隐藏延迟，只要后续迭代不依赖前面的结果即可
  • 写端口瓶颈：s.write(x) 向 AXI Stream FIFO 写入。当下游消费者（AIE）就绪时，写操作在一个周期完成；若下游反压，写会阻塞

为什么没有 #pragma HLS DATAFLOW？

  • 这是一个单阶段内核，没有多个并行执行的子函数
  • DATAFLOW 适用于 producer-consumer 链，此处只有一个简单的循环

s2mm.cpp：流到内存的单向泵

void s2mm(ap_int<32>* mem, hls::stream<ap_axis<32, 0, 0, 0>>& s, int size) {
#pragma HLS INTERFACE m_axi port=mem offset=slave bundle=gmem
#pragma HLS interface axis port=s
#pragma HLS INTERFACE s_axilite port=mem bundle=control
#pragma HLS INTERFACE s_axilite port=size bundle=control
#pragma HLS interface s_axilite port=return bundle=control

    for(int i = 0; i < size; i++) {
#pragma HLS PIPELINE II=1
        ap_axis<32, 0, 0, 0> x = s.read();
        mem[i] = x.data;
    }
}

s2mm 是 mm2s 的镜像对称实现：

  • 输入从 axis 变为 m_axi
  • 输出从 m_axi 变为 axis
  • 循环体内先 read() 后赋值

对称设计的价值：

  • 便于系统级验证——可以 loopback 测试
  • 减少认知负担——理解一个就等于理解两个
  • 便于参数化扩展——如果需要改变数据宽度，两处修改保持一致

[connectivity]

AIE Graph 中的 PLIO 定义

// graph.h
class clipped : public adf::graph {
    adf::input_plio in;   // 对应 DataIn1
    adf::output_plio out; // 对应 DataOut1
    
    clipped() {
        // "DataIn1" 这个名字与 system.cfg 中的连接匹配
        in = adf::input_plio::create("DataIn1", adf::plio_32_bits, "data/input.txt");
        out = adf::output_plio::create("DataOut1", adf::plio_32_bits, "data/output.txt");
        
        // 内部连接：interpolator -> clip -> classifier
        connect(in.out[0], interpolator.in[0]);
        connect(interpolator.out[0], clip.in[0]);
        connect(clip.out[0], classify.in[0]);
        connect(classify.out[0], out.in[0]);
    }
};

关键洞察：mm2s 和 s2mm 对 AIE Graph 来说是完全透明的。Graph 只看到标准的 PLIO 端口，不知道也不关心这些端口是由 HLS 内核、RTL IP 还是 hardened DMA 驱动的。这种抽象使得：

  1. 可替换性：可以将 mm2s 替换为自定义 RTL 而不修改 AIE 代码
  2. 可测试性：可以单独仿真 AIE Graph，用文件 I/O 替代真实的数据搬运
  3. 可移植性：同一套 AIE 内核可以在不同平台（VCK190、VEK280 等）上复用

数据流全景追踪

以一个典型的 AIE 推理任务为例，追踪数据从主机到 AIE 再返回的完整旅程：

Phase 1：主机配置阶段

// 伪代码示意
auto mm2s_kernel = xrt::kernel(device, uuid, "mm2s");
auto s2mm_kernel = xrt::kernel(device, uuid, "s2mm");

// 分配 DDR 缓冲区
auto in_bo = xrt::bo(device, size * sizeof(int32_t), mm2s_kernel.group_id(0));
auto out_bo = xrt::bo(device, size * sizeof(int32_t), s2mm_kernel.group_id(0));

// 填充输入数据
in_bo.write(input_data);
in_bo.sync(XCL_BO_SYNC_BO_TO_DEVICE);

// 启动内核（非阻塞）
auto run_mm2s = mm2s_kernel(in_bo, nullptr, size);  // mem, stream, size
auto run_s2mm = s2mm_kernel(out_bo, nullptr, size); // mem, stream, size

此时：

  • size 值被写入 s_axilite 控制寄存器
  • mem 指针（实际上是缓冲区在设备地址空间的偏移）被写入控制寄存器
  • 内核开始执行，但卡在循环入口等待内部状态机就绪

Phase 2：数据搬运阶段（mm2s）

时钟周期:  0    1    2    3    ...   L    L+1  L+2  ...
          |----|----|----|          |----|----|----|
DDR读请求: REQ0
DDR读数据:      [========DATA0========]  (延迟L周期)
AXIS写:                              WRITE0
DDR读请求:      REQ1
DDR读数据:           [========DATA1========]
AXIS写:                                   WRITE1

由于 #pragma HLS PIPELINE II=1，即使 DDR 读有延迟，HLS 也会自动展开循环并调度指令，使得每周期都能发起新的 DDR 读请求。读回的数据暂存在内部 FIFO，等待写入 AXIS。

Phase 3：AIE 计算阶段

数据流经 AIE graph 的 PLIO 接口进入 AIE 阵列。AIE 核心的计算与 PL 侧的数据搬运时间上重叠——这是 Versal 架构的核心优势之一。

Phase 4：数据回写阶段（s2mm）

时钟周期:  T    T+1  T+2  T+3  ...
          |----|----|----|----|
AXIS读:   READ0
DDR写:         WRITE0
AXIS读:        READ1
DDR写:              WRITE1

同样保持 II=1，形成持续的写事务流。

Phase 5：主机同步阶段

run_mm2s.wait();
run_s2mm.wait();
out_bo.sync(XCL_BO_SYNC_BO_FROM_DEVICE);
out_bo.read(output_data);

设计决策与权衡

1. 显式循环 vs. 突发传输优化

当前实现：

for(int i = 0; i < size; i++) {
    mem[i];  // 每次迭代一次访问
}

可能的优化方向：

// 手动突发传输——预取到本地缓冲区再处理
ap_int<32> local_buf[BURST_LEN];
for (...) {
    #pragma HLS PIPELINE II=1
    // 突发读取 local_buf
    // 处理 local_buf
}

为什么教程代码选择简单实现？

  • 教学清晰度：突发的复杂性会分散对核心概念（接口类型、流水线）的注意力
  • HLS 自动优化：现代 Vitis HLS 对连续的 m_axi 访问会自动合并为突发事务
  • 通用性：简单循环对各种 size 值都适用，而手动突发需要处理边界情况

2. 阻塞读写 vs. 非阻塞 + 超时

当前实现：

s.write(x);    // 若下游满，则阻塞
s.read();      // 若上游空，则阻塞

替代方案：

if (!s.full()) s.write(x);   // 非阻塞，可能丢数据
if (!s.empty()) x = s.read(); // 非阻塞，可能读旧数据

选择阻塞的理由：

  • 背压（Back-pressure）是特性而非缺陷：当下游处理不过来时，自然应该减速上游
  • 数据完整性：非阻塞方案需要复杂的握手协议确保无丢包
  • 死锁可检测性：阻塞导致的死锁可以通过波形分析快速定位；非阻塞的数据丢失是静默的

3. 标量 size 参数 vs. 描述符链

当前实现：单次传输的大小由 s_axilite 寄存器传入。

高端 DMA 的常见做法：使用描述符链表支持分散-聚集（Scatter-Gather）。

取舍分析：

方案 优点 缺点
标量 size 简单、低延迟启动、易调试 每次传输需主机介入
描述符链 主机可一次性提交多个传输 硬件复杂、描述符管理开销

对于 AIE 工作负载，通常是一次性传输大块数据（如整帧图像、整个特征图），标量方案足够且更简单。

4. 硬编码 32-bit 数据宽度 vs. 模板参数化

当前实现：ap_int<32> 和 ap_axis<32, 0, 0, 0> 都是硬编码的。

潜在改进：

template<int DATA_WIDTH>
void mm2s(ap_int<DATA_WIDTH>* mem, ...) { ... }

保持硬编码的原因：

  • AIE 的 PLIO 接口通常配置为 32-bit 或 64-bit，32-bit 是最常见的基础单元
  • 数据宽度变化通常伴随着 AIE graph 的重新设计，不仅仅是 PL 内核的事
  • 简化 cfg 文件配置——模板实例化会增加构建系统的复杂度

新贡献者必读：陷阱与注意事项

1. 内存对齐要求

m_axi 接口对突发传输有对齐要求。虽然 HLS 会自动处理大部分情况，但如果主机分配的缓冲区未对齐到 4KB 页边界，可能导致：

  • 性能下降（无法使用最大突发长度）
  • 在某些平台配置上出现总线错误

最佳实践：

// 使用 XRT 的对齐分配器
auto bo = xrt::bo(device, size, xrt::bo::flags::host_only, group_id);
// 或使用 posix_memalign 在主机侧分配对齐内存

2. Size 参数的隐含契约

size 是以 32-bit 字为单位 的数量，不是字节数。常见错误：

// 错误！传入了字节数
mm2s_kernel(buffer, nullptr, buffer_size_in_bytes);  

// 正确
mm2s_kernel(buffer, nullptr, buffer_size_in_bytes / sizeof(int32_t));

3. Stream 深度与死锁

如果下游 AIE 核心暂时无法消费数据（如正在等待其他输入），mm2s 的 s.write() 会阻塞。此时若 s2mm 也在等待 AIE 输出，而 AIE 又在等待更多输入才能产生输出……就会形成循环依赖死锁。

调试方法：

  • 在仿真波形中观察 TREADY/TVALID 信号
  • 检查 AIE graph 的 fifo_depth 配置是否足够缓冲中间结果
  • 考虑在 PL 侧添加小型 FIFO（hls::stream 的 depth 参数）作为弹性缓冲

4. 多实例并发启动

当系统中存在多个 mm2s/s2mm 实例时（如 normalization_v4_multistream_scaling_flow），注意：

  • 每个实例需要独立的 s_axilite 控制接口（Vitis 自动处理）
  • DDR 带宽可能成为瓶颈——多个 m_axi 端口竞争同一内存控制器
  • 考虑使用 bundle= 将不同实例绑定到不同的 DDR bank

5. 与 RTL IP 的混合使用

本模块属于 versal_integration_clocking_and_rtl_ip 教程的一部分。在同一设计中，你可能需要将 HLS 内核与手写 RTL IP 连接：

  • 确保 AXIS 信号命名规范一致（TDATA, TVALID, TREADY）
  • 注意时钟域——HLS 内核默认使用单一时钟，RTL IP 可能有独立时钟
  • 使用 ap_ctrl_chain 接口模式支持流水线启动（本模块使用简单的 ap_ctrl_hs）

6. 数据格式转换的隐含成本

在本教程的完整系统中，mm2s 输出的是原始 32-bit 数据，但 AIE 侧的 polar_clip 内核期望的是 cint16（复数 16-bit）格式。这种格式转换发生在 AIE graph 的 PLIO 边界：

// PL 侧 (mm2s): 32-bit 整数
ap_axis<32, 0, 0, 0> x;
x.data = mem[i];  // 32-bit

// AIE 侧 (polar_clip): cint16
void polar_clip(input_stream_cint16* in, output_stream_cint16 * out) {
    cint16 sample = readincr(in);  // 自动将 32-bit 拆分为 real/imag
}

关键点：PL 到 AIE 的数据宽度转换是零成本的——硬件自动将 32-bit 流解释为两个 16-bit 复数分量。但开发者必须在心理上明确这种映射关系，否则会在调试时困惑于数据布局。

7. Host 代码中的 Size 计算陷阱

回顾 host.cpp 中的关键代码：

#define SAMPLES 256
int sizeIn = SAMPLES/2;   // 128
int sizeOut = SAMPLES;    // 256

// 输入缓冲区：cint16Input 是 int16_t 数组，每对构成一个复数
auto in_bohdl = xrt::bo(device, sizeIn * sizeof(int16_t) * 2, 0, 0);

// mm2s 调用
auto mm2s_rhdl = mm2s_khdl(in_bohdl, nullptr, sizeIn);

这里存在一个微妙的单位转换链：

  1. sizeIn = 128 表示 128 个复数样本
  2. 每个复数样本 = 2 × int16_t = 4 bytes
  3. 所以总字节数 = 128 × 4 = 512 bytes
  4. 但 mm2s 的 size 参数期望的是 32-bit 字的数量
  5. 512 bytes / 4 bytes per word = 128 —— 恰好等于 sizeIn

这个巧合之所以成立，是因为 cint16 恰好映射到一个 32-bit 字。如果数据格式改变（如使用 cint32），这种"恰好相等"的关系就会被打破，需要显式转换。

总结

pl_kernels_mm2s_s2mm 是一个看似简单但设计精巧的基础设施模块。它的价值不在于代码行数，而在于：以最少的复杂度，可靠地完成异构计算中最基础的任务——数据搬运。

对于新加入团队的工程师，理解这个模块有助于建立以下直觉：

  1. 接口分离原则：数据通路与控制通路永远分开设计
  2. 阻塞即反馈：在流式系统中，背压是协调生产者和消费者的基本机制
  3. 对称简化认知：成对设计的组件（如 mm2s/s2mm）应尽可能保持行为对称
  4. HLS pragmas 是契约：它们不仅是编译提示，更是与其他模块（包括未来的你）的约定

当你需要在真实项目中修改或扩展这对内核时，建议遵循的原则是：保持简单，直到性能数据证明需要复杂化。
