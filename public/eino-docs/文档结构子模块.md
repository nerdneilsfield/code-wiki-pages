# 文档结构子模块

## 1. 模块概述

文档结构子模块提供了一个统一的文档表示和处理方式，它是整个系统中文档数据的核心载体。这个模块解决了在信息检索、知识库构建和文档处理系统中，如何标准化地表示文档内容及其元数据的问题。

在没有这个模块之前，不同的组件可能会使用各自的文档表示方式，导致数据转换复杂、接口不一致等问题。文档结构子模块通过提供一个标准的`Document`结构体，使得文档可以在系统的各个组件之间无缝流动。

## 2. 核心抽象与设计理念

### 2.1 核心抽象

文档结构子模块的核心抽象是`Document`结构体，它可以被想象为一个"信封"：
- **ID**：信封的唯一标识符，用于在系统中追踪和引用这个文档
- **Content**：信封内的主要内容，即文档的实际文本
- **MetaData**：信封上的标签和备注，用于存储各种额外信息

这种设计使得文档既保持了核心内容的简单性，又通过元数据提供了极大的扩展性。

### 2.2 设计理念

1. **简单性与扩展性的平衡**：核心字段（ID、Content）保持简单，同时通过MetaData提供无限扩展可能
2. **流式接口设计**：所有的元数据设置方法都返回`*Document`，支持链式调用
3. **类型安全的元数据访问**：虽然MetaData是`map[string]any`类型，但通过提供专门的getter/setter方法，确保了常用元数据的类型安全
4. **预留关键字段**：通过常量定义预留了一些元数据键，避免命名冲突

## 3. 核心组件详解

### 3.1 Document 结构体

```go
type Document struct {
    ID       string                 `json:"id"`
    Content  string                 `json:"content"`
    MetaData map[string]any         `json:"meta_data"`
}
```

**设计意图**：
- **ID**：唯一标识文档，在检索、存储和追踪时使用
- **Content**：文档的主要内容，通常是纯文本格式
- **MetaData**：灵活的元数据存储，用于存放各种扩展信息

### 3.2 元数据管理方法

文档结构子模块提供了一系列方法来管理常用的元数据：

#### 3.2.1 子索引管理
```go
func (d *Document) WithSubIndexes(indexes []string) *Document
func (d *Document) SubIndexes() []string
```
**用途**：用于搜索引擎，允许文档属于多个子索引，实现更精细的搜索控制。

#### 3.2.2 分数管理
```go
func (d *Document) WithScore(score float64) *Document
func (d *Document) Score() float64
```
**用途**：存储检索相关性分数，在排序和筛选搜索结果时使用。

#### 3.2.3 向量管理
```go
func (d *Document) WithDenseVector(vector []float64) *Document
func (d *Document) DenseVector() []float64
func (d *Document) WithSparseVector(sparse map[int]float64) *Document
func (d *Document) SparseVector() map[int]float64
```
**用途**：存储文档的向量表示，用于语义搜索和相似度计算。密集向量适用于大多数嵌入模型，稀疏向量适用于传统的TF-IDF等表示。

#### 3.2.4 其他元数据
```go
func (d *Document) WithExtraInfo(extraInfo string) *Document
func (d *Document) ExtraInfo() string
func (d *Document) WithDSLInfo(dslInfo map[string]any) *Document
func (d *Document) DSLInfo() map[string]any
```
**用途**：提供额外的灵活存储，ExtraInfo用于简单的文本备注，DSLInfo用于存储查询相关的结构化信息。

## 4. 依赖关系与数据流程

### 4.1 依赖关系

文档结构子模块是一个基础模块，它不依赖于系统中的其他模块，而是被其他模块广泛使用。从模块树中可以看出，它属于`Schema Core Types`，是整个系统的基础数据结构之一。

### 4.2 数据流程

文档在系统中的典型流动路径如下：

1. **文档加载**：通过文档加载器组件读取原始数据，创建`Document`对象
2. **文档处理**：通过文档转换器组件处理文档内容，可能修改Content或添加MetaData
3. **文档索引**：将文档传递给索引器组件，提取向量并存储
4. **文档检索**：从检索器组件获取相关文档，使用Score进行排序
5. **文档使用**：将文档传递给模型或其他组件进行进一步处理

在这个流程中，`Document`对象作为数据载体，保持了数据的一致性和完整性。

## 5. 设计决策与权衡

### 5.1 元数据存储方式选择

**决策**：使用`map[string]any`作为元数据存储

**权衡**：
- ✅ 优点：提供了最大的灵活性，可以存储任意类型的元数据
- ❌ 缺点：失去了编译时类型安全，需要在运行时进行类型断言

**原因**：在系统设计初期，无法预见所有可能的元数据类型，因此选择了灵活性优先的方案。同时，通过提供专门的getter/setter方法，为常用元数据提供了一定程度的类型安全。

### 5.2 方法链式调用设计

**决策**：所有元数据设置方法都返回`*Document`

**权衡**：
- ✅ 优点：支持流畅的链式调用，代码更简洁
- ❌ 缺点：修改了原对象，可能导致意外的副作用

**原因**：在文档处理流程中，通常需要连续设置多个元数据，链式调用可以显著提高代码可读性。同时，文档对象在创建后通常是单向流动的，副作用风险较低。

### 5.3 预留元数据键

**决策**：通过常量定义预留元数据键

**权衡**：
- ✅ 优点：避免命名冲突，提供统一的标准
- ❌ 缺点：限制了用户可用的键名空间

**原因**：系统内部组件需要使用一些特定的元数据字段，通过预留这些键名，可以确保不同组件之间的兼容性。

## 6. 使用指南与最佳实践

### 6.1 基本使用

```go
// 创建文档
doc := &schema.Document{
    ID:      "doc-001",
    Content: "这是文档的主要内容",
}

// 链式设置元数据
doc.WithScore(0.85).
    WithSubIndexes([]string{"tech", "ai"}).
    WithExtraInfo("这是额外信息")

// 获取元数据
score := doc.Score()
indexes := doc.SubIndexes()
```

### 6.2 向量存储

```go
// 存储密集向量（用于语义搜索）
denseVector := []float64{0.1, 0.2, 0.3, 0.4}
doc.WithDenseVector(denseVector)

// 存储稀疏向量（用于传统检索）
sparseVector := map[int]float64{
    100: 0.5,
    200: 0.3,
    300: 0.8,
}
doc.WithSparseVector(sparseVector)
```

### 6.3 最佳实践

1. **始终设置ID**：确保每个文档都有唯一的ID，这对于追踪和引用至关重要
2. **合理使用元数据**：只在必要时使用自定义元数据，优先使用提供的标准方法
3. **注意类型安全**：当直接操作MetaData时，务必进行适当的类型检查
4. **避免修改已传递的文档**：一旦文档被传递给其他组件，避免修改它，以防止意外副作用
5. **使用链式调用**：设置多个元数据时，使用链式调用可以提高代码可读性

## 7. 注意事项与常见问题

### 7.1 元数据类型断言失败

当使用getter方法时，如果元数据的类型不匹配，会返回零值。这可能导致静默失败。

**解决方案**：如果需要严格的类型检查，可以直接访问MetaData并进行类型断言。

### 7.2 并发安全

Document结构体不是并发安全的，如果多个goroutine同时修改同一个Document对象，可能导致数据竞争。

**解决方案**：在并发环境中使用文档时，确保进行适当的同步，或者为每个goroutine创建文档的副本。

### 7.3 向量序列化

当包含向量的文档需要序列化时，注意向量的大小可能导致序列化后的文档很大。

**解决方案**：考虑在不需要向量时移除它们，或者使用更高效的序列化格式。

## 8. 总结

文档结构子模块是一个看似简单但设计精巧的模块，它通过平衡简单性和扩展性，为整个系统提供了统一的文档表示方式。其核心价值在于：

1. **标准化**：提供了统一的文档表示，减少了组件间的转换成本
2. **扩展性**：通过灵活的元数据机制，支持各种复杂的使用场景
3. **易用性**：通过链式调用和便捷方法，提高了开发效率

这个模块是整个系统的基础之一，理解它的设计理念和使用方法对于高效开发和维护系统至关重要。
