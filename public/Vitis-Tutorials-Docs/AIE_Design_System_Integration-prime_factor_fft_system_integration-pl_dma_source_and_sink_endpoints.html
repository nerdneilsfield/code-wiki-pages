PL DMA Source and Sink Endpoints 模块深度解析

一句话概括

pl_dma_source_and_sink_endpoints 是 Versal ACAP 平台上 Prime-Factor FFT (PFA) 1008点FFT 流水线的数据搬运"门户"——它负责将 DDR4 内存中的原始样本数据高效地注入到 AI Engine 计算阵列，并在计算完成后将结果回写到内存。想象它就像一条高速公路的收费站：入口站（DMA Source）把车辆（数据）从普通道路（DDR4）引导上高速（AXI-Stream），出口站（DMA Sink）则让车辆安全驶离高速回到目的地。

问题空间与设计动机

我们面临什么挑战？

在异构计算架构（如 AMD Versal ACAP）中，AI Engine 阵列擅长高吞吐量的数字信号处理，但它无法直接访问片外 DRAM。这就产生了一个根本性的数据搬运问题：

  1. 带宽不匹配：AI Engine 以 1250 MHz 运行，而 PL（Programmable Logic）和 DDR4 控制器通常运行在较低频率（本设计中为 312.5 MHz）
  2. 数据格式转换：FFT 算法需要特定的多相（polyphase）数据排列顺序，但内存中的数据通常是线性存储的
  3. 流水线延迟隐藏：整个 PFA 流水线包含多个处理阶段（Permute → DFT-7 → Transpose → DFT-9 → Transpose → DFT-16 → Permute），每个阶段都有固定的处理延迟

为什么不用简单的 memcpy？

直接使用 CPU 或简单的 DMA 传输会面临几个问题：

  • 协议转换：需要将内存总线协议（AXI4-Full）转换为流式协议（AXI4-Stream）
  • 数据重排：必须在线性存储和多相流式格式之间进行实时转换
  • 背压处理：当下游处理单元繁忙时，需要优雅地处理反压（back-pressure）
  • 延迟补偿：需要插入虚拟周期来匹配下游流水线的固有延迟

解决方案的核心思想

这个模块采用 "双缓冲 + 流式转换" 的架构：

  • Source（入口）：从 DDR4 批量读取数据到片上 BRAM，然后以流式方式按多相顺序输出
  • Sink（出口）：接收流式输入的多相数据，重组为线性顺序后批量写回 DDR4

关键设计参数

参数 值 含义
NFFT 1008 FFT 点数（= 7 × 9 × 16，质因数分解）
NSTREAM 2 并行流通道数
DEPTH 1008 每通道缓冲区深度（支持 8 个变换）
NBITS 128 数据总线宽度（位）
TT_SAMPLE 32-bit 单个样本位宽（复数样本 = I/Q 各 16-bit）

核心组件详解

1. DMA Source (pfa1008_dma_src_wrapper)

功能职责

作为数据入口，负责：

  1. 批量加载：通过 AXI4-Full 接口从 DDR4 读取 128-bit 宽的数据块
  2. 格式转换：将线性存储顺序转换为多相（polyphase）流式顺序
  3. 延迟补偿：插入虚拟零值周期以匹配下游流水线延迟
  4. 循环传输：支持多次重复发送同一数据集（用于性能测试）

内部子模块

load_buffer 函数

  • 输入：mem[NSTREAM*DEPTH] —— DDR4 中的线性数据缓冲区
  • 输出：buff[NSTREAM][DEPTH] —— 片上 BRAM 双缓冲区
  • 核心逻辑：交替写入两个流缓冲区，实现数据的初步分离
// 内存布局示例（8个1008点变换）：
// buff[0]: s0 s1 s2 s3 s8  s9  s10 s11 s16 s17 s18 s19 ...
// buff[1]: s4 s5 s6 s7 s12 s13 s14 s15 s20 s21 s22 s23 ...

transmit 函数

  • 延迟计算：LATENCY = NFFT/2 + 4 = 508 周期
    • 这是下游所有 PL HLS 内核的总处理延迟
    • pfa1008_permute_i: NFFT/8 + 1 = 127 cycles
    • pfa1008_transpose1: NFFT/8 + 2 = 128 cycles
    • pfa1008_transpose2: NFFT/8 + 1 = 127 cycles
    • pfa1008_permute_o: NFFT/8 + 0 = 126 cycles
  • 多相输出：
    • Stream 0 (ss0): 偶数样本 (d6, d4, d2, d0)
    • Stream 1 (ss1): 奇数样本 (d7, d5, d3, d1)

HLS 优化策略

#pragma HLS PIPELINE II=1          // 目标：每个时钟周期输出一个样本
#pragma HLS array_partition dim=1   // 完全分区数组以获得并行访问
#pragma HLS DATAFLOW                // 启用任务级并行：load 和 transmit 重叠执行

2. DMA Sink (pfa1008_dma_snk_wrapper)

功能职责

作为数据出口，负责：

  1. 流捕获：从 AXI4-Stream 接收多相格式的数据
  2. 选择性存储：根据 loop_sel 参数只保存指定迭代的数据（用于调试）
  3. 格式重组：将多相顺序还原为线性存储顺序
  4. 批量回写：通过 AXI4-Full 将结果写回 DDR4

内部子模块

capture_streams 函数

  • 输入：sig_i[NSTREAM] —— 两个 AXI4-Stream 输入
  • 选择逻辑：if (ll == loop_sel) 只保留指定迭代的输出
  • 重组逻辑：
buff[0][dd] = ( val1[1], val0[1], val1[0], val0[0] );  // 低地址部分
buff[1][dd] = ( val1[3], val0[3], val1[2], val0[2] );  // 高地址部分

read_buffer 函数

  • 逆向执行 load_buffer 的操作，将双缓冲区数据交错写回线性内存

关键设计决策与权衡

1. 为什么选择 128-bit 数据宽度？

决策：使用 ap_uint<128> 作为基本数据单元

权衡分析：

  • ✅ 优势：
    • 在 312.5 MHz 下，128-bit 总线提供 40 Gbps 的峰值带宽
    • 每个 128-bit 字包含 4 个 32-bit 复数样本，天然适配 SIMD 处理
    • 与 AI Engine 的 128-bit PLIO 接口宽度匹配
  • ⚠️ 代价：
    • 增加了数据对齐要求（必须 16-byte 对齐）
    • 小批量传输时可能有带宽浪费

替代方案：使用 64-bit 或 32-bit 宽度

  • 会增加所需的时钟频率或流通道数
  • 会增加 HLS 生成的逻辑复杂度

2. 为什么需要显式的延迟补偿？

决策：在 transmit 中插入 LATENCY = NFFT/2 + 4 个虚拟周期

设计原理： 这是一个典型的 "首 token 延迟 vs 吞吐率" 权衡。PFA 流水线中的每个 PL HLS 内核都有固定的处理延迟（主要由乒乓缓冲和流水线填充造成）。如果不进行补偿：

  • 第一个有效样本到达时，下游可能还未准备好接收
  • 会导致数据丢失或流水线气泡

为什么用虚拟零值而不是握手信号？

  • AXI4-Stream 虽然有 TVALID/TREADY 握手，但在 DATAFLOW 模式下，HLS 工具会自动处理大部分反压
  • 显式的虚拟周期简化了时序分析，确保无论下游状态如何，数据都以恒定速率流出

3. 双缓冲区 vs 单缓冲区

决策：使用 buff[NSTREAM][DEPTH] 的双缓冲区结构

权衡分析：

  • ✅ 优势：
    • 允许 load_buffer 和 transmit 在 DATAFLOW 模式下并行执行
    • 自然地将数据分为偶数/奇数流，匹配后续的多相处理需求
  • ⚠️ 代价：
    • 2× 的 BRAM 资源消耗（但这是可接受的，因为 DEPTH=1008 较小）

资源估算：

  • 每个缓冲区：1008 × 128-bit ≈ 16 KB
  • 总计：2 streams × 16 KB = 32 KB BRAM

4. 为什么 Sink 需要 loop_sel 参数？

决策：添加 loop_sel 参数用于选择性捕获

使用场景： 当 loop_cnt > 1 时（例如进行多次 FFT 变换以测量平均性能），我们可能只想保存最后一次或某一次的结果进行验证。loop_sel 允许主机代码指定要捕获的迭代索引。

默认行为：

  • 如果 loop_sel = loop_cnt - 1，则保存最后一次迭代的结果
  • 这种设计避免了不必要的 DDR4 写带宽浪费

新贡献者必读：陷阱与注意事项

1. 数据格式陷阱

问题：输入数据必须以特定的交错格式准备

正确格式：

// 假设有 8 个 1008 点变换，每个样本是 32-bit 复数
// DDR4 缓冲区布局（线性顺序）：
// [transform0_sample0, transform0_sample1, ..., transform0_sample1007,
//  transform1_sample0, transform1_sample1, ..., transform7_sample1007]

常见错误：

  • 忘记样本必须是 16-bit I + 16-bit Q 的打包格式
  • 缓冲区未 16-byte 对齐（128-bit 访问要求）

2. 延迟参数敏感性

问题：LATENCY 常量必须与下游流水线延迟精确匹配

检查清单：

  • 如果修改了 pfa1008_permute_i、transpose1、transpose2 或 pfa1008_permute_o 的实现，必须重新计算 LATENCY
  • 计算公式：LATENCY = permute_i_latency + transpose1_latency + transpose2_latency + permute_o_latency
  • 当前值：(NFFT/8+1) + (NFFT/8+2) + (NFFT/8+1) + (NFFT/8+0) = NFFT/2 + 4

故障症状：

  • 如果 LATENCY 太小：前几个样本可能损坏
  • 如果 LATENCY 太大：只是浪费了几个周期的带宽，不会导致功能错误

3. HLS DATAFLOW 的隐式约束

问题：#pragma HLS DATAFLOW 对函数签名有严格要求

约束条件：

  • 所有数组参数必须是固定大小的（本设计中使用模板参数 DEPTH 和 NSTREAM）
  • 不能有条件地调用 DATAFLOW 区域中的函数
  • 不能使用指针别名（已通过 restrict 隐式处理）

4. AXI4-Lite 寄存器访问时序

问题：loop_cnt 和 loop_sel 通过 AXI4-Lite 配置

注意事项：

  • 必须在启动 DMA 传输前完成寄存器写入
  • 寄存器写入和 DMA 启动之间需要有足够的延迟（通常由 XRT 驱动处理）
  • 不要在数据传输过程中动态修改这些参数

5. 时钟域交叉

问题：PL 运行在 312.5 MHz，AI Engine 运行在 1250 MHz

处理机制：

  • AXI4-Stream 接口自动处理跨时钟域同步
  • 本模块不需要显式的 CDC（Clock Domain Crossing）逻辑
  • 但必须确保 freqhz=312500000 在 system.cfg 中正确设置

6. 仿真 vs 硬件差异

问题：C 仿真和 RTL/硬件行为可能有细微差别

建议：

  • 始终运行 C/RTL Co-simulation 验证时序
  • 注意 csim.setup=false 在 hls.cfg 中禁用了 C 仿真设置（可能是为了加速构建）
  • 实际硬件测试时使用 ILA（Integrated Logic Analyzer）监控 AXI-Stream 接口

总结

pl_dma_source_and_sink_endpoints 模块是整个 Prime-Factor FFT 系统的 "数据门户"。它的设计体现了异构计算中数据搬运的核心原则：

  1. 批量传输优于随机访问：通过 AXI4-Full 突发传输最大化 DDR4 带宽利用率
  2. 流式接口解耦生产消费：AXI4-Stream 允许上下游以不同速率运行
  3. 显式延迟管理：通过虚拟周期确保流水线正确启动
  4. 格式转换与数据移动协同：在搬运数据的同时完成必要的重排，避免额外的处理阶段

对于新加入团队的工程师，理解这个模块的关键在于把握 "内存-流转换" 这一核心抽象，以及它在整个 PFA 流水线中的 边界定位——它既不是纯粹的 DMA 控制器，也不是纯粹的数据格式转换器，而是两者的有机结合，专门服务于 FFT 算法的特定数据访问模式。
