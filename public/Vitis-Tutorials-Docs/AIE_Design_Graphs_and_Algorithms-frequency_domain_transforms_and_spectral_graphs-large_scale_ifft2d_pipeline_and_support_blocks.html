大规模 2D IFFT 流水线与支撑模块 (Large Scale IFFT2D Pipeline and Support Blocks)

一句话总结

这是一个用于 Versal AI Engine 的 64K 点二维逆快速傅里叶变换 (IFFT) 实现，采用分阶段流水线架构——将庞大的 65536 点变换分解为可并行执行的 256 点小 FFT 阶段，通过"前后级分解"策略在 10 个 AI Engine 核心上实现高效数据流处理。

问题空间：为什么需要这个模块？

核心挑战：超大规模 FFT 的"规模墙"

在雷达信号处理、5G Massive MIMO 预编码、以及高分辨率 SAR (合成孔径雷达) 成像中，工程师经常需要处理 64K 点甚至更大规模的 FFT/IFFT。直接在单核上实现这种规模的变换会面临多重困境：

  1. 存储墙 (Memory Wall)：64K 点复数数据需要 1MB 存储，远超单个 AI Engine 核心的本地数据存储 (约 256KB)
  2. 计算墙 (Compute Wall)：\(O(N \log N)\) 复杂度意味着超过 100 万次复数运算，单核无法实时处理
  3. 通信墙 (Communication Wall)：FFT 的蝶形运算需要跨 stage 的全局数据交换，传统缓存架构效率低下

解决方案：分而治之的流水线架构

本模块采用 Cooley-Tukey 分解策略，将 64K 点 IFFT 分解为：

\[ 65536 = 256 \times 256 \]

这产生了一个二维变换结构，可以通过两个 256 点 IFFT 阶段配合转置 (transpose) 操作实现。每个 256 点阶段进一步分解为更小的 4 点或 16 点基运算，形成深度流水线。

心智模型：如何理解这个架构？

类比：现代化的"分拣-配送中心"

想象一个处理海量包裹的大型物流中心：

  1. 前置分拣 (Front Stage)：来自全国各地的包裹首先到达分拣中心，按目的地城市进行粗粒度分类。这对应 IFFT 的"行变换"阶段——对 256 行数据，每行做 256 点 IFFT。

  2. 中转暂存 (Transpose Buffer)：分类后的包裹需要在中转区重新排列——原本按来源组织的包裹现在按目的地组织。这对应"矩阵转置"操作，将行优先存储变为列优先存储。

  3. 末端配送 (Back Stage)：最后按具体街道地址进行细粒度分拣和配送。这对应"列变换"阶段——对转置后的 256 列数据，每列做 256 点 IFFT。

每个阶段由 5 个 AI Engine 核心并行处理 (共 10 核)，就像每个分拣区有 5 条并行的传送带。

核心抽象：Graph 即硬件电路

在 AI Engine 编程模型中，graph 类不是一个简单的数据流图——它直接映射到物理硬件连接。当你写 connect(a, b)，你实际上是在配置 AXI 流连接器和 DMA 通道。

软件声明: connect( front_i[ff].out[0],  dut.front_i[ff] )
        ↓ 编译时展开
硬件配置: PLIO → AXI-Stream Switch → AI Engine Tile (18,0) input port

这种"声明即配置"的范式意味着：

  • 编译时确定性：所有连接在编译时确定，没有运行时动态路由
  • 物理布局感知：location<kernel> = tile(x,y) 直接将代码绑定到物理芯片位置
  • 时序可预测性：数据流延迟主要由传输距离决定，可通过布局优化

端到端数据流详解

以一次完整的 64K 点 IFFT 为例，追踪数据如何流经整个系统：

阶段 1：PL 端 DMA 数据注入 (Host → PL → AIE)

// Host 代码准备 64K 点复数数据 (256x256 矩阵)
// 数据按行优先存储: 256 行, 每行 256 个复数

for (unsigned ff=0; ff < TT_DUT::NFRONT; ff++) {  // 5 个并行通道
    // PLIO 绑定到物理 DMA 通道
    front_i[ff] = input_plio::create(pname_i, plio_64_bits, fname_i);
    // 64-bit PLIO = 1 个时钟周期传输 1 个复数 (32-bit I + 32-bit Q)
}

数据流向：

Host DDR → AXI4-Memory-Mapped → DMA Controller → AXI4-Stream → PLIO → AIE Grid
(256x256)     (64-bit bursts)      (MM2S channel)   (stream)    (port)

每个 front_i[ff] 通道处理 256×(256/5) ≈ 52 行的数据，5 个通道并行覆盖全部 256 行。

阶段 2：Front Stage - 行方向 IFFT (AIE Tiles 18,0-22,0)

// 每个 front 核心执行 ifft256p4: 256 点 IFFT, 基-4 分解
location<kernel>(dut.g_front0.k_kernel) = tile(F_X+0, F_Y+0);  // (18,0)
location<kernel>(dut.g_front1.k_kernel) = tile(F_X+1, F_Y+0);  // (19,0)
...

// IFFT 计算流程 (每个核心)
for each row assigned to this core:
    // 基-4 蝶形运算: 256 = 4^4, 需要 4 stages
    stage0: 64 组 4 点 DFT
    stage1: 16 组 16 点 DFT (twiddle 旋转)
    stage2: 4 组 64 点 DFT (twiddle 旋转)
    stage3: 1 组 256 点 DFT (twiddle 旋转)
    output: 256 点频域结果

计算复杂度：每个 256 点 IFFT 需要约 \(256 \times \log_4(256) \times 2 = 2048\) 次复数运算。

阶段 3：隐式转置 (Transpose)

这是架构的关键所在。Front stage 输出的是按行组织的频域数据，但 Back stage 需要按列处理。这个转置不是显式复制，而是通过 AIE 的数据流重排实现：

Front Stage 输出布局 (Row-Major):
[Row0:  256 点] → 输出到 back_i[0] 缓冲区
[Row1:  256 点] → 输出到 back_i[1] 缓冲区
...
[Row255: 256 点] → 输出到 back_i[4] 缓冲区 (循环)

Back Stage 读取视角 (Column-Major):
[Col0: 256 点] ← 从每行的第 0 个元素聚合
[Col1: 256 点] ← 从每行的第 1 个元素聚合
...

这种"隐式转置"避免了额外的内存拷贝，通过 AIE 的流式架构实现零拷贝数据重排。

阶段 4：Back Stage - 列方向 IFFT (AIE Tiles 18,1-22,1)

// Back 核心执行 ifft256p4_rot: 256 点 IFFT + 旋转因子乘法
location<kernel>(dut.g_back[0].k_kernel) = tile(B_X+0, B_Y+0);  // (18,1)
location<kernel>(dut.g_back[1].k_kernel) = tile(B_X+1, B_Y+1);  // (19,1)
...

// IFFT + 旋转计算 (2D IFFT 需要的 twiddle 旋转)
for each column assigned to this core:
    // Step 1: 256 点 IFFT (同 front)
    ifft256_base4(input_column)
    
    // Step 2: 2D IFFT 旋转因子 (关键差异!)
    for n = 0 to 255:
        output[n] *= exp(j*2*pi*row_idx*col_idx/65536)
    
    output: 256 点时域结果

ifft256p4_rot 相比 ifft256p4 增加了2D IFFT 旋转因子乘法，这是二维变换数学特性的要求。

阶段 5：数据汇出 (AIE → PL → Host)

// Back stage 输出通过 PLIO 回到 PL DMA
for (unsigned bb=0; bb < TT_DUT::NBACK; bb++) {
    back_o[bb] = output_plio::create(pname_o, plio_64_bits, fname_o);
    connect( dut.back_o[bb], back_o[bb].in[0] );
}

// 最终数据流向
AIE Grid → PLIO → AXI4-Stream → DMA Controller → AXI4-MM → Host DDR
(时域结果)                (S2MM channel)         (64-bit bursts) (65536点)

设计决策与权衡

决策 1：固定点 vs 浮点运算

选择：使用定点数 (通常是 cint16 或 cint32) 而非浮点

权衡分析：

维度 定点 (选择) 浮点 (放弃)
计算密度 ✅ AI Engine 单周期可执行 8 个 cint16 MAC ❌ 需要多个周期，吞吐率低
功耗效率 ✅ 单位瓦特处理更多点数 ❌ 浮点单元功耗更高
精度 ⚠️ 需要仔细管理位宽和缩放 ✅ 动态范围大，无需缩放
开发复杂度 ⚠️ 需要量化分析和溢出处理 ✅ 更直观的算法映射

选择原因：对于 5G 和雷达这类信号处理应用，64K 点 IFFT 需要极高的吞吐率。定点运算使 AI Engine 能够达到所需的数据处理速率，而精度损失可以通过适当的位宽选择 (如 cint32 中间结果) 来控制。

决策 2：基-4 vs 基-2 FFT 分解

选择：256 点 IFFT 使用基-4 (radix-4) 分解而非基-2

权衡分析：

基-4 分解：\(256 = 4^4\)，需要 4 个 stage，每个 stage \(256/4 = 64\) 个基-4 蝶形 基-2 分解：\(256 = 2^8\)，需要 8 个 stage，每个 stage \(256/2 = 128\) 个基-2 蝶形

选择原因：

  1. 减少内存访问：基-4 减少了一半的 stage 数，意味着减少了一半的中间数据写回操作
  2. 更高 MAC 效率：AI Engine 的 SIMD 架构可以高效地并行执行 4 路运算
  3. 更好的流水线覆盖：较少的 stage 数意味着更少的同步点，提高了流水线效率

决策 3：显式 vs 隐式内核放置

选择：使用显式 location<kernel> 将内核绑定到特定 tile

location<kernel>(dut.g_front0.k_kernel) = tile(F_X+0, F_Y+0);  // (18,0)
// ... 明确的 10 个 tile 位置

权衡分析：

维度 显式放置 (选择) 隐式自动布局 (放弃)
确定性 ✅ 完全可预测的数据流延迟 ⚠️ 每次编译可能不同
性能调优 ✅ 可优化核间距离以减少延迟 ❌ 无法手动微调
可移植性 ⚠️ 绑定到特定芯片布局 ✅ 可在不同 Versal 设备上运行
调试 ✅ 已知位置便于波形分析 ❌ 需要反向查找实际位置

选择原因：这是一个高性能信号处理流水线，每一周期的延迟都影响吞吐率。显式放置确保：

  1. Front stage 的 5 个核心在 tile 行 18，列 0-4 (物理上相邻，低延迟互联)
  2. Back stage 的 5 个核心在 tile 行 18，列 0-4，但行 1 (与 front 垂直相邻)
  3. 数据从 front 到 back 的转置通过片上存储高效完成

关键设计决策详解

决策：双阶段流水线的缓冲区管理

在 2D IFFT 中，第一维变换后必须转置数据才能进行第二维变换。本设计采用了一种隐式双缓冲策略：

传统做法 (显式转置):
1. Front stage 输出到 Buffer A (行优先)
2. CPU 或 DMA 转置 Buffer A → Buffer B (列优先)
3. Back stage 从 Buffer B 读取
[问题: 额外的内存拷贝和延迟]

本设计 (隐式流式转置):
1. Front stage 输出直接流式传输
2. AI Engine 的流缓冲和路由自然形成数据重排
3. Back stage 以列优先方式消费
[优势: 零拷贝，流式处理，全流水线]

决策：5 通道并行的负载均衡

选择 5 个并行通道 (NFRONT=NBACK=5) 是基于 256 点 IFFT 的处理粒度和 AI Engine 的数据通路宽度：

总数据量: 256 行 × 256 点/行 = 65536 点
每通道: 65536 / 5 = 13107 点 (实际分配考虑对齐)

5 是最佳选择，因为:
- 2 或 4 通道: 单通道负载过高，无法实时处理
- 8 通道: 超出 256 行的整除便利性，增加同步开销
- 5 通道: 在 Versal AI Engine 阵列 (典型 50+ 核心) 中资源占比合理

新贡献者须知：陷阱与边界情况

1. 内核放置的物理约束

location<kernel> 不是装饰性的——它直接决定硬件布局。修改时必须考虑：

// 当前布局: Front 在 tile 行 0，Back 在 tile 行 1，同列对齐
location<kernel>(dut.g_front0.k_kernel) = tile(18,0);
location<kernel>(dut.g_back[0].k_kernel) = tile(18,1);  // 同列，相邻行

// 危险做法: 将 back 核心分散到不同列
location<kernel>(dut.g_back[0].k_kernel) = tile(18,1);
location<kernel>(dut.g_back[1].k_kernel) = tile(25,1);  // 远离其他核心!
// 结果: 跨列通信延迟增加，可能破坏流水线时序

2. PLIO 位宽与数据格式的隐含契约

sig_i = input_plio::create(pname_i, plio_64_bits, fname_i);

plio_64_bits 建立了严格的内存到流映射：

  • 物理层：64 位 PLIO 端口 = 每时钟周期传输 64 位
  • 数据层：64 位 = 32 位 I (实部) + 32 位 Q (虚部) = 1 个复数样本
  • 文件层：输入文本文件每行一个十六进制数，表示一个 64 位复数

常见错误：

// 错误: 假设 128 位可以传输 2 个复数
plio_128_bits  // 实际上不存在这种标准配置

// 错误: 修改数据格式但保持 plio_64_bits
// 如果改为 cint16 (16+16=32 位)，plio_64_bits 现在传输 2 个样本
// 这需要同步修改 kernel 代码和连接配置

3. AIE_SIM_ONLY 宏的双生世界

#ifdef AIE_SIM_ONLY
    front_i[ff] = input_plio::create(pname_i, plio_64_bits, fname_i);
#else
    front_i[ff] = input_plio::create(pname_i, plio_64_bits);
#endif

这个宏区分了两个完全不同的执行环境：

特性 AIE_SIM_ONLY (仿真) 硬件运行
数据源 从文本文件读取 从 DMA 流接收
输出目标 写入文本文件 发送到 DMA 流
调试 可检查中间文件 需要 ILA/Vivado 调试
运行速度 慢 (软件仿真) 实时

关键陷阱：在仿真和硬件之间切换时，必须确保 data/ 目录存在且包含正确格式的输入文件。硬件运行时这些文件不会被使用，但编译时仍然需要 #ifdef 分支的语法完整性。

4. 流水线启动的隐藏时序

int main(void) {
  aie_dut.init();    // 配置所有 DMA 和 AIE 核心
  aie_dut.run( TT_DUT::NSAMP/TT_DUT::NFRONT );  // 运行特定迭代数
  aie_dut.end();     // 刷新流水线，确保所有数据完成
  return 0;
}

这里的 run(count) 不是简单的"重复 count 次"，而是涉及复杂的流水线填充和排空：

时钟周期: 0    1    2    ...  L    ...  L+count-1  ...
          ↓    ↓    ↓         ↓         ↓          ↓
Front:   [F0] [F1] [F2] ... [FL] ... [数据流] ... [排空]
Back:         [B0] [B1] ... [BL] ... [数据流] ... [排空]
Output:            [O0] ...     ... [结果]   ... [完成]

其中 L = 流水线深度 (约等于 stage 数)

关键洞察：run(count) 中的 count 必须精确计算为 "样本数 / 并行通道数"，因为每个通道处理的是交错的数据流。错误的 count 会导致：

  • count 太小：流水线未完成，输出数据不完整
  • count 太大：重复处理或缓冲区溢出

5. runtime<ratio> 的调度含义

runtime<ratio>(dut.k_kernel) = 0.9;

这行代码声明了该内核占用其所在 AI Engine tile 的 90% 计算资源。剩余 10% 留给：

  • DMA 数据传输
  • 与其他核心的同步
  • 中断处理

危险值：设置为 1.0 (100%) 会导致 tile 无法响应 DMA 请求，造成死锁。设置为过低 (< 0.5) 则浪费计算资源。0.9 是经过验证的安全平衡点。

性能特征与调优指引

理论吞吐率计算

参数:
- 总点数: N = 65536 (64K)
- 时钟频率: F = 1 GHz (典型 AIE 频率)
- 每个 256 点 IFFT 周期数: C ≈ 256 × log4(256) × k ≈ 2048 周期 (含开销)
- 并行通道: P = 5

计算:
- 每批次总 IFFT 次数: (256 行 + 256 列) = 512 次 256 点 IFFT
- 每个通道处理: 512 / 5 ≈ 103 次 IFFT
- 每批次周期数: 103 × 2048 ≈ 210,944 周期
- 每批次时间: 210,944 / 1G = 0.21 ms
- 吞吐率: 64K 点 / 0.21 ms ≈ 311 MSPS (百万样本/秒)

实际性能调优检查清单

  • [ ] 确认所有 PLIO 连接使用 64-bit 宽度 - 更低位宽会限制吞吐率
  • [ ] 验证 DMA 描述符配置 - 确保 burst 长度匹配 AXI4-Stream 最佳实践 (通常 16-64 beats)
  • [ ] 检查 tile 放置的邻接性 - Front 和 Back stage 的对应通道应在同一列 (如 front0 在 (18,0)，back0 在 (18,1))
  • [ ] 运行时比例平衡 - 所有核心的 runtime<ratio> 应一致 (0.9)，避免某个核心成为瓶颈
  • [ ] 输入数据格式验证 - 文本文件中的十六进制数应与 cint32 或 cint16 的位宽匹配

扩展与演化路径

如何修改为 256K 点 IFFT？

当前: \(65536 = 256 \times 256\) 目标: \(262144 = 512 \times 512\)

需要修改：

  1. 核心算法：从 ifft256p4 改为 ifft512p8 (512 点，基-8 或基-2 混合)
  2. 并行度：从 5 通道增加到 8 或 16 通道以维持吞吐率
  3. 内存布局：512 行 × 512 列需要 4 倍存储，可能需要引入外部存储 (DDR) 缓冲
  4. 放置策略：需要更多的 AI Engine tiles，可能需要扩展到多行 (目前只用 2 行)

如何集成到 OFDM 接收机？

典型的 5G NR 接收机链路：

天线数据 → CP 移除 → 64K IFFT (本模块) → 子载波去映射 → 均衡 → 解调 → 译码

集成要点：

  1. 时序同步：确保 CP 移除后的数据边界与 IFFT 的帧边界对齐
  2. 缩放管理：定点 IFFT 的输出需要合适的缩放因子，避免后续处理溢出
  3. 循环前缀处理：本模块输出的是纯 IFFT 结果，如需添加 CP，需在下游处理

总结：核心设计哲学

本模块体现了 AI Engine 编程的三个核心原则：

  1. 数据流即架构 —— 不是先写算法再考虑映射，而是从数据移动模式出发设计。64K 点 IFFT 的天然二维结构直接映射到 Front/Back 两级流水线。

  2. 显式优于隐式 —— 内核位置、PLIO 绑定、DMA 配置都在代码中明确声明。这种"声明式硬件编程"使性能可预测、可复现。

  3. 分而治之的规模化 —— 通过 5 通道并行将问题分解到 10 个核心，每个核心处理可管理的 256 点 IFFT。这是可扩展性的关键——需要更大变换？增加更多通道和阶段。

理解这三点，你就能不仅使用这个模块，还能演化它，创造适应新需求的变体。
