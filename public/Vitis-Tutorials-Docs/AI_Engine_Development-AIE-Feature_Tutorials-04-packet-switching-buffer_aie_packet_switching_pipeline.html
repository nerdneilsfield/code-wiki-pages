Buffer AIE Packet Switching Pipeline 深度解析

一句话总结

本模块展示了一种基于包交换的多路数据复用架构——通过 PL 侧 HLS 包处理器将多个独立数据流打包注入 AIE 引擎，由 AIE 内部的 pktsplit/pktmerge 组件完成数据包路由与解复用，最终将处理后的多路数据通过包接收器分发回 PL 侧。这相当于在片上实现了一个微型邮政系统：多个"发件人"将信件投入邮筒，邮递员（AIE）根据信封上的地址分拣投递，最终由"收件人"按地址取信。

1. 问题空间与设计动机

1.1 为什么需要包交换？

在典型的 AIE 加速系统中，数据从 PL（可编程逻辑）侧流入 AIE 阵列进行处理，然后将结果输出回 PL。传统的静态连接模式存在以下局限：

局限 说明
连接数受限 每个 AIE 核心只有有限的输入/输出端口，无法直接连接大量数据源
灵活性差 连接在编译时固定，运行时无法动态调整数据路由
带宽碎片化 为每个数据源分配独立通道，可能导致带宽利用率低下

包交换（Packet Switching） 借鉴了网络通信的思想：将数据打包，每个包携带路由信息（包 ID），通过网络中的交换节点动态路由到目的地。这种方式：

  • 共享物理通道：多路数据时分复用同一物理链路
  • 动态路由：根据包 ID 决定数据流向
  • 可扩展性：新增数据流只需分配新 ID，无需新增物理连接

1.2 本模块要解决的问题

本模块是一个教学示例，展示如何在 Versal ACAP 架构中实现多路数据流的包交换处理。具体目标包括：

  1. 演示 PL-AIE 数据包封装与解封装：在 PL 侧使用 HLS 实现包发送器和接收器
  2. 展示 AIE 内部包路由：使用 pktsplit<4> 和 pktmerge<4> 在 AIE 阵列内完成包路由
  3. 验证端到端数据流：4 个独立的 MM2S 数据流经过打包→AIE 处理→解包→分发到 4 个 S2MM 缓冲区

核心价值：提供了一个可运行的参考设计，开发者可以基于它构建更复杂的包交换系统。

2. 核心抽象与思维模型

2.1 类比：邮政系统

理解本模块的最佳类比是邮政系统：

邮政系统组件 本模块对应 职责
发件人 MM2S + 包发送器 从内存读取数据，装入信封（添加头部），投入邮筒
信封上的地址 包 ID（header 中的 ID 字段） 决定信件应该投递到哪个目的地
邮递员分拣中心 AIE 的 pktsplit 根据信封地址将信件分拣到不同的投递路线
收件人处理 AIE 核心（core1~core4） 拆开信封，处理信件内容（+1, +2, +3, +4）
重新装信封 AIE 的 pktmerge 将处理后的信件装入新信封，集中发回
邮局分发 包接收器 根据地址将信件分发给各个收件人
收件人 S2MM 将收到的信件内容存入自己的邮箱（内存）

2.2 核心抽象层

本模块包含三个核心抽象层次：

层 1：原始数据流（Data Stream）

  • 表现形式：hls::stream<ap_axiu<32,0,0,0>>
  • 特点：连续的数据流，无边界信息（除了 TLAST）
  • 问题：无法区分数据来源/去向

层 2：数据包（Packet）

  • 表现形式：头部（32-bit）+ 数据载荷（8 x 32-bit）
  • 头部格式：
    [4:0]   - Packet ID (0-31)
[11:5]  - Reserved
[14:12] - Packet Type
[15]    - Reserved
[20:16] - Source Row (-1 for unknown)
[27:21] - Source Column (-1 for unknown)
[30:28] - Reserved
[31]    - Parity bit (XOR of [30:0])
  • 特点：自描述的数据单元，包含路由信息

层 3：逻辑通道（Virtual Channel）

  • 表现形式：通过 Packet ID 区分的独立数据流
  • 特点：多路数据时分复用同一物理链路
  • 优势：动态路由，灵活扩展

2.3 数据包格式

本模块使用固定的数据包格式：

+--------------------------------------------------+
|  Word 0: Header (32-bit)                         |
|  +---------------------------------------------+ |
|  | [4:0]   | Packet ID (0-3)                   | |
|  | [11:5]  | Reserved (0)                      | |
|  | [14:12] | Packet Type (0)                   | |
|  | [15]    | Reserved (0)                      | |
|  | [20:16] | Source Row (0x1F = -1)            | |
|  | [27:21] | Source Col (0x1F = -1)            | |
|  | [30:28] | Reserved (0)                      | |
|  | [31]    | Parity (XOR of [30:0])          | |
|  +---------------------------------------------+ |
+--------------------------------------------------+
|  Word 1-8: Data Payload (8 x 32-bit)              |
|  +---------------------------------------------+ |
|  | Word 1 | Data[0]                             | |
|  | Word 2 | Data[1]                             | |
|  | ...    | ...                                 | |
|  | Word 8 | Data[7] (TLAST=1)                   | |
|  +---------------------------------------------+ |
+--------------------------------------------------+

关键约定：

  • 每个包固定 9 个 word（1 header + 8 data）
  • 最后一个 data word 的 TLAST=1，标记包结束
  • Packet ID 必须在 0-3 范围内（对应 4 个 AIE 核心）

4. 组件深度解析

4.1 PL 侧数据搬运器 (MM2S/S2MM)

职责：在 DDR 内存与 AXI Stream 之间搬运数据。

MM2S (Memory-Mapped to Stream):

void mm2s(ap_uint<32>* mem, hls::stream<ap_axiu<32, 0, 0, 0>>& s, int size)
  • 使用 m_axi 接口从 DDR 读取数据
  • 使用 axis 接口输出 AXI Stream
  • #pragma HLS PIPELINE II=1 保证每个周期输出一个数据

S2MM (Stream to Memory-Mapped):

void s2mm(ap_uint<32>* mem, hls::stream<ap_axiu<32, 0, 0, 0>>& s, int size)
  • 与 MM2S 方向相反，从 Stream 读取，写入 DDR

为什么需要这些组件：

  • Host 只能直接访问 DDR 内存，无法直接操作 AXI Stream
  • MM2S/S2MM 作为"数据泵"，将数据从 Host 可见的内存空间泵入/泵出 AIE 处理流水线

4.2 包发送器 (hls_packet_sender)

职责：将多个输入流的数据封装成包，注入 AIE。

核心逻辑（简化）：

for (每个迭代):
    for (i = 0 to 3):  // 4 个输入流
        // 1. 生成头部
        header = generateHeader(pktType, packet_ids[i]);
        out.write(header);
        
        // 2. 写入 8 个数据 word
        for (j = 0 to 7):
            data = read_from_stream(i);  // 从 s0, s1, s2, s3 读取
            if (j == 7) data.last = 1;     // 最后一个 word 标记 TLAST
            out.write(data);

关键设计决策：

  1. 固定包长度 (PACKET_LEN=8)：

    • 简化发送器和接收器的逻辑
    • 与 AIE 核心的 buffer 大小匹配 (extents<8>)
    • 权衡：灵活性降低，但实现简单

  2. 头部生成策略：

    • 头部中的 ID 字段决定 AIE 中的路由目标
    • 使用 packet_ids_c.h 中定义的宏（如 Datain0_0, Datain0_1 等）
    • 注意：这些 ID 必须与 AIE 图中 pktsplit 的配置匹配

  3. 轮询调度：

    • 发送器轮流从 4 个输入流读取数据
    • 保证每个流的带宽公平分配
    • 限制：如果某个流没有数据，发送器会阻塞（read() 是阻塞操作）

4.3 AIE 图与包路由

AIE 图定义（graph.h）：

class mygraph : public adf::graph {
private:
    adf::kernel core[4];           // 4 个 AIE 核心
    adf::pktsplit<4> sp;           // 包分离器（1 入 4 出）
    adf::pktmerge<4> mg;           // 包合并器（4 入 1 出）
public:
    adf::input_plio in;            // PL 输入接口
    adf::output_plio out;          // PL 输出接口
    
    mygraph() {
        // 创建 4 个 AIE 核心，每个对应一个处理函数
        core[0] = adf::kernel::create(aie_core1);  // +1
        core[1] = adf::kernel::create(aie_core2);  // +2
        core[2] = adf::kernel::create(aie_core3);  // +3
        core[3] = adf::kernel::create(aie_core4);  // +4
        
        // 创建 PLIO 接口
        in = adf::input_plio::create("Datain0", plio_32_bits, "data/input.txt");
        out = adf::output_plio::create("Dataout0", plio_32_bits, "data/output.txt");
        
        // 创建包分离器和合并器
        sp = adf::pktsplit<4>::create();
        mg = adf::pktmerge<4>::create();
        
        // 连接拓扑：
        // pktsplit[0..3] -> core[0..3] -> pktmerge[0..3]
        for(int i=0; i<4; i++) {
            adf::runtime<ratio>(core[i]) = 0.9;  // 90% 时间片分配给核心
            adf::connect<>(sp.out[i], core[i].in[0]);
            adf::connect<>(core[i].out[0], mg.in[i]);
        }
        
        // 包流连接（使用 pktstream 类型）
        adf::connect<adf::pktstream>(in.out[0], sp.in[0]);
        adf::connect<adf::pktstream>(mg.out[0], out.in[0]);
    }
};

关键概念解释：

  1. pktsplit<4> - 包分离器：

    • 1 个输入，4 个输出
    • 根据输入包的头部 ID 字段，将包路由到对应的输出端口
    • 相当于"分拣员"，看信封地址后放到对应的格子里

  2. pktmerge<4> - 包合并器：

    • 4 个输入，1 个输出
    • 将来自多个输入端的包合并到单一输出
    • 相当于"汇集点"，多个来源的信件汇聚到一条邮路

  3. pktstream 类型：

    • 特殊的 AXI-Stream 类型，支持包边界信息传递
    • 与 plio_32_bits 配合，实现 PL-AIE 间的包流传输

AIE 核心处理逻辑（以 core1 为例）：

void aie_core1(input_buffer<int32,extents<8>> &in,
               output_buffer<int32,extents<8>> &out) {
    auto inIter = aie::begin(in);
    auto outIter = aie::begin(out);
    for(int i=0; i<8; i++) {
        *outIter++ = *inIter++ + 1;  // 每个元素加 1
    }
}
  • 每个核心处理 8 个 int32 数据
  • 4 个核心分别执行 +1, +2, +3, +4 操作
  • 这种设计使得验证时可以明确区分每个数据流是否被正确路由

4.4 包接收器 (hls_packet_receiver)

职责：解析 AIE 输出的包流，根据头部 ID 将数据分发到对应的输出流。

核心逻辑（简化）：

void hls_packet_receiver(hls::stream<ap_axiu<32,0,0,0>> &in,
                        hls::stream<ap_axiu<32,0,0,0>> &out0,
                        hls::stream<ap_axiu<32,0,0,0>> &out1,
                        hls::stream<ap_axiu<32,0,0,0>> &out2,
                        hls::stream<ap_axiu<32,0,0,0>> &out3,
                        const unsigned int total_num_packet) {
    for(unsigned int iter=0; iter<total_num_packet; iter++) {
        // 1. 读取头部
        ap_axiu<32,0,0,0> tmp = in.read();
        unsigned int ID = getPacketId(tmp.data);  // 提取 ID 字段
        unsigned int channel = packet_ids[ID];     // 映射到输出通道
        
        // 2. 读取 8 个数据 word，根据 channel 分发
        for(int j=0; j<PACKET_LEN; j++) {
            tmp = in.read();
            switch(channel) {
                case 0: out0.write(tmp); break;
                case 1: out1.write(tmp); break;
                case 2: out2.write(tmp); break;
                case 3: out3.write(tmp); break;
            }
        }
    }
}

关键实现细节：

  1. 头部解析：

    • 使用 getPacketId() 从头部提取 ID 字段（低 5 位）
    • 通过 packet_ids[ID] 映射到输出通道（0-3）
    • 关键约束：ID 必须在 0-3 范围内，否则映射失败

  2. 包边界处理：

    • 第一个 word 总是头部（无 TLAST）
    • 随后的 8 个 word 是数据，第 8 个 word 的 TLAST=1
    • 接收器不检查 TLAST，只按固定长度读取

  3. 输出分发：

    • 使用 switch 语句根据 channel 选择输出流
    • 每个输出流连接到对应的 S2MM 内核

5. 设计权衡与决策

5.1 为什么选择包交换而非固定连接？

方案 优点 缺点 适用场景
固定连接 低延迟、确定性时序、实现简单 连接数受限、灵活性差 数据流固定、连接数少的场景
包交换 共享物理链路、动态路由、可扩展 需要包头开销、延迟略高、实现复杂 多路数据、动态路由、连接数多的场景

本模块选择包交换的理由：

  1. 教学目的：展示 Versal 架构中 AIE 包交换能力的典型用法
  2. 连接数限制：4 个输入流共享 1 个 PLIO 连接到 AIE，无法使用固定连接
  3. 可扩展性演示：通过修改 Packet ID 即可改变数据路由，无需修改硬件连接

5.2 头部开销与带宽权衡

头部开销计算：

  • 每个包：1 header word + 8 data words = 9 words
  • 头部开销比例：1/9 ≈ 11.1%

权衡分析：

方案 头部大小 开销比例 优点 缺点
当前方案 (1 word) 4 bytes 11.1% 实现简单、开销较低 可扩展字段有限
扩展头部 (2 words) 8 bytes 22.2% 支持更多元数据（时间戳、序列号等） 开销增加一倍
变长头部 可变 可变 灵活性最高 实现复杂，需要解析逻辑

当前设计选择 1 word 头部的理由：

  • 教学示例，保持简单
  • 11% 开销在多数应用场景可接受
  • 5-bit ID 支持最多 32 个虚拟通道，对于本示例足够

5.3 AIE 内部 pktsplit/pktmerge vs. 外部逻辑

关键问题：为什么使用 AIE 内部的 pktsplit/pktmerge，而不是在 PL 侧实现相同功能？

方案 优点 缺点
AIE 内部 (当前) 利用 AIE 专用硬件，效率高；减少 PL 资源使用；展示 AIE 包交换能力 灵活性受限（只能使用 AIE 支持的 split/merge 模式）
PL 侧实现 灵活性高，可实现任意路由逻辑；易于调试和修改 消耗 PL 资源（LUT/FF/BRAM）；延迟可能更高
混合方案 结合两者优势 实现复杂，需要仔细划分功能

本模块选择 AIE 内部方案的理由：

  1. 演示 AIE 特性：这是 AIE Feature Tutorial 的核心目标——展示 AIE 的包交换能力
  2. 资源优化：pktsplit/pktmerge 使用 AIE 专用硬件，不消耗 PL 的 LUT/FF 资源
  3. 性能优势：AIE 内部的包路由是硬件加速的，延迟和吞吐量优于 PL 实现

6. 关键陷阱与调试建议

6.1 包 ID 不匹配

症状：数据被错误路由到非预期的 AIE 核心，或包接收器输出到错误的 S2MM 通道。

根本原因：

  • 包发送器使用的 ID 与 AIE 图中 pktsplit 期望的 ID 不匹配
  • packet_ids_c.h 中的宏定义与 AIE 图配置不一致

调试建议：

  1. 检查 packet_ids_c.h： 确保以下宏定义与 AIE 图配置一致：

    #define Datain0_0  0  // 对应 pktsplit 的 out[0]
#define Datain0_1  1  // 对应 pktsplit 的 out[1]
#define Datain0_2  2  // 对应 pktsplit 的 out[2]
#define Datain0_3  3  // 对应 pktsplit 的 out[3]

  2. 验证 AIE 图配置： 在 graph.h 中，确认以下连接关系：

    adf::connect<> (sp.out[0], core[0].in[0]);  // ID 0 -> core1
adf::connect<> (sp.out[1], core[1].in[0]);  // ID 1 -> core2
adf::connect<> (sp.out[2], core[2].in[0]);  // ID 2 -> core3
adf::connect<> (sp.out[3], core[3].in[0]);  // ID 3 -> core4

  3. 使用仿真验证： 在硬件运行前，使用 AIE 仿真器验证数据路由是否正确。

6.2 TLAST 信号处理

症状：AIE 核心或包接收器无法正确识别包边界，导致数据解析错误。

根本原因：

  • 包发送器未正确设置最后一个数据 word 的 TLAST 信号
  • AIE 核心或包接收器依赖 TLAST 识别包边界

调试建议：

  1. 验证发送器 TLAST 设置： 在 hls_packet_sender.cpp 中，确保以下逻辑正确：

    for(int j=0; j<PACKET_LEN; j++) {
    tmp = read_from_input_stream();
    if(j == PACKET_LEN-1) {
        tmp.last = 1;  // 最后一个 word 设置 TLAST=1
    } else {
        tmp.last = 0;
    }
    out.write(tmp);
}

  2. 检查 AIE 核心 buffer 大小： 确保 AIE 核心的输入/输出 buffer 大小与包数据量匹配：

    // kernels.h
void aie_core1(input_buffer<int32,extents<8>> &in,
               output_buffer<int32,extents<8>> &out);
// extents<8> 表示 8 个 int32 = 8 word，与 PACKET_LEN=8 匹配

  3. 使用波形调试： 在硬件运行时，使用 ILA（Integrated Logic Analyzer）抓取 AXI Stream 波形，验证 TLAST 信号时序。

6.3 AIE 核心计算与预期不符

症状：Host 验证时发现输出数据与预期不符（如期望 +1 但实际 +2）。

根本原因：

  • 数据被错误路由到非预期的 AIE 核心
  • AIE 核心代码与预期处理逻辑不匹配

调试建议：

  1. 核对核心处理逻辑： 确认每个核心的处理函数与预期一致：

    // aie_core1.cpp: +1
*outIter++ = *inIter++ + 1;

// aie_core2.cpp: +2
*outIter++ = *inIter++ + 2;

// aie_core3.cpp: +3
*outIter++ = *inIter++ + 3;

// aie_core4.cpp: +4
*outIter++ = *inIter++ + 4;

  2. 验证 Host 检查逻辑： 在 host.cpp 中，确认验证逻辑与 AIE 核心处理对应：

    if(*(host_out1+i) != *(host_in1+i) + 1) match = 1;  // core1: +1
if(*(host_out2+i) != *(host_in2+i) + 2) match = 1;  // core2: +2
if(*(host_out3+i) != *(host_in3+i) + 3) match = 1;  // core3: +3
if(*(host_out4+i) != *(host_in4+i) + 4) match = 1;  // core4: +4

  3. 追踪单个数据包： 如果可能，在输入数据中插入特定的"标记值"（如 0xDEADBEEF），然后在输出中追踪该值经过了哪个核心，验证路由是否正确。
