FIR Filter AIE vs HLS Datamover Kernels 模块详解

概述

本模块是 Versal ACAP AI Engine/HLS FIR Filter Tutorial 的核心组成部分，提供了一个完整的系统级基准测试框架，用于比较和评估两种不同硬件加速实现方式的性能：AI Engine (AIE) 实现与 HLS + DSP58 实现。

想象一下这个场景：你正在设计一个高性能数字信号处理系统，需要在 Versal 器件上实现 FIR 滤波器链。面对 AI Engine（专为矢量计算优化的处理器阵列）和可编程逻辑中的 DSP58 引擎（传统 FPGA 信号处理单元）两种选择，你如何做出明智的决策？本模块正是为了回答这个问题而存在——它提供了一个"苹果对苹果"的比较平台，让工程师能够在完全相同的测试条件下评估两种架构的性能、功耗和资源占用。

核心目标

  1. 公平比较：通过统一的数据搬运内核（datamover）接口，消除 I/O 差异对性能评估的影响
  2. 灵活配置：支持可变数量的滤波器（1-10 个）和抽头数（15/64/129/240），适应不同复杂度场景
  3. 真实测量：提供硬件仿真（hw_emu）和实际硬件运行两种模式，获取真实的吞吐量、延迟和功耗数据
  4. 教学示范：展示 Vitis 统一软件平台的完整开发流程，包括内核编译、系统链接和主机控制

关键设计思想

1. 对称 harness 设计（Symmetric Harness）

本模块最核心的设计理念是一致性 harness。无论是 AIE 还是 HLS 实现，都使用完全相同的 datamover 内核作为数据输入源和结果校验器。这种设计的价值在于：

  • 隔离变量：性能差异只反映计算内核本身的特性，而非 I/O 子系统的差异
  • 简化对比：工程师可以专注于分析 AIE 与 DSP58 的架构差异，而不必纠结于不同的数据搬运机制
  • 代码复用：同一套主机控制代码（datamover_class）适用于两种实现

2. 自包含测试模式（Self-Contained Test Mode）

datamover 内核采用了一种特殊的测试策略：不依赖外部 DDR 内存，所有测试数据都在 PL 内部生成和验证：

  • 输入生成：mm2s 函数在第一个时钟周期输出脉冲信号（0x3FFF3FFF，即实部和虚部均为 16383），其余周期输出零
  • 结果校验：s2mm 函数将 FIR 输出与预计算的 golden 数据进行逐样本比较，累计错误计数
  • 零延迟反馈：由于不涉及 NoC/DDR，可以测量到 FIR 内核的理论最大吞吐量，不受内存带宽瓶颈限制

这种模式就像一个"封闭实验室"——你可以精确测量 FIR 内核本身的处理能力，而不会被外部存储器的延迟和带宽所干扰。

3. 参数化设计空间

整个系统通过一组宏定义实现高度可配置：

参数 含义 典型值
N_FIR_FILTERS 级联滤波器数量 1, 2, ..., 10
N_FIR_TAPS 每个滤波器的抽头数 15, 64, 129, 240
FIR_WINDOW_SIZE 处理窗口大小 256, 512, 8192
SAMP_PERIOD 采样周期（HLS） 1, 2, 4

这种参数化设计允许快速探索设计空间，例如：

  • 固定抽头数，增加滤波器数量 → 评估系统扩展性
  • 固定滤波器数量，增加抽头数 → 评估单核处理能力上限
  • 调整 SAMP_PERIOD → 在吞吐量和资源占用之间权衡

子模块结构

本模块包含两个主要变体，分别对应 AIE 和 HLS 实现路径：

AIE Design

AI Engine 实现路径，包含：

  • AIE Graph: FirGraph 类定义，使用 DSPLib 的 fir_sr_sym_graph
  • PL Datamover: 与 HLS 版本共享相同接口的数据搬运内核
  • Host Application: 基于 XRT AIE API 的主机控制程序
  • System Config: 连接 AIE 和 PL 的 system.cfg 配置文件

HLS Design

HLS + DSP58 实现路径，包含：

  • FIR HLS Kernel: 使用 hls::FIR 模板类的滤波器实现
  • PL Datamover: 与 AIE 版本功能相同的数据搬运内核
  • Host Application: 基于标准 XRT API 的主机控制程序
  • Directives: HLS 综合优化指令（hls_pre.tcl, post_sys_link.tcl）

关键设计决策与权衡

决策 1：为什么使用自包含测试而非 DDR 往返？

选择的方案：datamover 内核在 PL 内部生成测试数据并直接校验结果，不经过 DDR 内存。

替代方案：传统的 DMA 模式——从 DDR 读取输入数据，处理完成后写回 DDR，由主机进行结果校验。

权衡分析：

维度 自包含模式 DDR 往返模式
测量精度 ✅ 纯 FIR 吞吐量，无外部干扰 ❌ 包含 NoC/DDR 延迟
实现复杂度 ✅ 无需缓冲区管理 ❌ 需要分配/同步 DDR 缓冲区
真实性 ⚠️ 理想条件，非典型应用场景 ✅ 更接近实际部署
调试难度 ✅ 错误定位更直接 ❌ 需区分计算错误与数据传输错误

为何这样选择：本教程的首要目标是比较 AIE 与 DSP58 的计算能力，而非评估整个系统 I/O 性能。自包含模式提供了最纯净的计算基准。当需要评估端到端性能时，可以在保持计算内核不变的情况下，添加 DDR 数据路径。

决策 2：为什么 AIE 和 HLS 使用相同的 datamover？

选择的方案：两种实现复用几乎完全相同的 datamover.cpp 和 datamover.h 代码。

替代方案：为每种实现定制优化的数据搬运逻辑。

权衡分析：

维度 统一 datamover 定制优化
公平性 ✅ 确保 I/O 行为一致 ❌ 难以保证对比的公正性
代码维护 ✅ 单一代码库 ❌ 重复代码，容易分叉
性能极限 ⚠️ 可能不是各自最优 ✅ 可针对每种架构调优

为何这样选择：在性能对比场景中，控制变量比追求每个个体的局部最优更重要。统一的 datamover 确保了任何观测到的性能差异都源于计算内核（AIE vs HLS FIR），而非数据供给方式的不同。

决策 3：为什么选择特定的抽头数和滤波器数量组合？

选择的方案：预定义了 15/64/129/240 四种抽头数和最多 10 级级联的配置。

设计考量：

  1. 15 taps：最小配置，测试轻量级滤波器的基础开销
  2. 64 taps：中等配置，平衡计算强度和并行度
  3. 129 taps：接近 AIE 单核处理能力边界，展示级联必要性
  4. 240 taps：大滤波器场景，强制多核协作

这些数值的选择反映了 Versal AIE 架构的特性：

  • AIE 内核具有 8 个 MAC（乘累加）单元，每周期可执行 8 次操作
  • 对称 FIR 可利用系数对称性减少一半乘法运算
  • 超过一定抽头数后，单核无法在 1 GHz 时钟下维持单样本/周期的吞吐量，必须通过 N_AIES_PER_FIR 或级联分解工作负载

新贡献者注意事项

1. 黄金数据（Golden Data）的生成与验证

datamover.h 中包含了大量预计算的 golden_16b 数组，这些是 FIR 滤波器对脉冲输入的理论响应。如果你修改了：

  • 滤波器系数（FIR_COEF_VALUES）
  • 抽头数（N_FIR_TAPS）
  • 级联深度（N_FIR_FILTERS）

必须重新生成 golden 数据！项目提供了 Python 脚本 create_fir_coefs.py 来自动化这一过程。

2. AIE 与 HLS 的系数定义差异

注意以下微妙但关键的差异：

AIE 版本（system_settings_comm.h）：

#define FIR_COEF_VALUES  { 255, 511, 767, ... }  // 仅前半部分（对称性由 DSPLib 处理）

HLS 版本（system_settings_comm.h）：

#define FIR_COEF_VALUES  { 255, 511, 767, 1023, 1279, 1535, 1791, 2047, 1791, 1535, ... }  // 完整对称系数

这是因为：

  • AIE DSPLib 的 fir_sr_sym_graph 自动利用系数对称性，只需要前半部分
  • HLS hls::FIR 需要完整的系数数组（尽管设置了 coeff_structure = symmetric）

混合使用会导致错误的滤波器响应！

3. 窗口大小与采样周期的匹配

在 HLS 实现中，WINDOW_LENGTH 必须与 AIE 的 FIR_WINDOW_SIZE 协调：

  • AIE 使用 plio_128_bits，每次传输 4 个 cint16 样本（64 位 × 4 = 128 位）
  • HLS 的 complex_split 每次处理 4 个样本（128 位 AXI-Stream 宽度）
  • 确保 WINDOW_LENGTH 能被 4 整除，否则会出现未处理样本或越界访问

4. 硬件仿真 vs 实际硬件的行为差异

  • hw_emu：模拟精确的硬件时序，适合功能验证和初步性能估计，但运行速度慢（分钟到小时级别）
  • hardware：真实硬件运行，获取准确的功耗和最大吞吐量数据

某些时序敏感的问题（如背压处理、FIFO 深度不足导致的死锁）可能在 hw_emu 中无法完全复现。

5. 错误阈值的设定

主机代码中的通过/失败判定：

return ((errCnt > 1500) ? EXIT_FAILURE : EXIT_SUCCESS);

这个阈值（1500）相对宽松，因为：

  • 浮点/定点舍入差异可能导致微小误差
  • 多级级联会累积量化误差
  • 不同实现（AIE vs HLS）的舍入模式可能略有不同

如果你的修改导致错误计数显著增加，首先检查：

  1. 系数定义是否正确
  2. 数据类型转换是否溢出（cint16 范围是 -32768 到 32767）
  3. 缩放因子（FIR_DOWNSHIFT）是否匹配

与其他模块的关系

本模块位于 AIE_Design_System_Integration 分类下，是 Versal 系统集成教程家族的一员。相关模块包括：

  • lenet_system_dma_kernel_instance：展示 LeNet CNN 的 DMA 集成模式，使用内存映射接口而非流接口
  • gemm_aie_vs_dsp_plio_and_memory_kernels：矩阵乘法的 AIE vs HLS 对比，使用 PLIO 和内存两种接口
  • fft2d_aie_vs_hls_scaling：2D FFT 的规模化实现，展示多实例配置

本模块的独特价值在于其极简主义设计——剥离了不必要的复杂性，专注于单一问题的清晰对比。当你理解了这里的设计原则，可以更容易地理解更复杂的系统（如通道化器或多实例 FFT）是如何构建的。

总结

fir_filter_aie_vs_hls_datamover_kernels 模块是一个精心设计的基准测试平台，其核心贡献在于：

  1. 公平的比较框架：通过对称 harness 设计，确保 AIE 和 HLS 实现面临相同的 I/O 条件
  2. 纯净的测量环境：自包含测试模式消除了 DDR 带宽瓶颈，聚焦于计算内核本身
  3. 灵活的配置空间：支持广泛的参数扫描，适应不同应用场景的探索需求
  4. 教学价值：展示了 Vitis 平台的完整开发流程，从内核编码到系统集成

对于刚加入团队的工程师，建议按以下顺序深入：

  1. 阅读 AIE Design 和 HLS Design 子模块文档
  2. 运行默认配置（1 filter, 15 taps）的 hw_emu，观察波形和数据流
  3. 修改 N_FIR_FILTERS 和 N_FIR_TAPS，观察资源占用和吞吐量的变化趋势
  4. 尝试添加 DDR 数据路径，将本模块扩展为更完整的应用原型
