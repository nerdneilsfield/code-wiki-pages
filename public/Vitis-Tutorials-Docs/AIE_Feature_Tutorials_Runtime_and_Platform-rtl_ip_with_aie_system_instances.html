RTL IP with AIE System Instances 模块详解

概述

RTL IP with AIE System Instances 是一个教学示例模块，展示了如何在 AMD Versal ACAP 平台上将自定义 RTL IP（寄存器传输级知识产权核）与 AI Engine (AIE) 计算图集成。这个模块的核心价值在于演示了异构计算架构中不同计算单元之间的无缝协作——即如何将传统的硬件描述语言（HDL）编写的 RTL 模块，与基于 C++/ADF 的高级综合 HLS 内核以及 AI Engine 的计算内核协同工作。

想象一个现代化的工厂流水线：AI Engine 就像是高度优化的专用工作站，擅长执行数字信号处理等并行计算任务；RTL IP 则是定制的精密机械臂，执行特定的硬件加速功能；而 HLS 内核则充当物料搬运系统，负责在内存和计算单元之间高效传输数据。本模块展示的就是如何将这些不同"工种"整合到一条流畅的生产线上。

问题空间与设计动机

为什么要混合 RTL、HLS 和 AIE？

在 Versal ACAP 这样的异构计算平台上，开发者面临一个关键决策：应该使用哪种编程模型来实现特定功能？

实现方式 优势 局限性
纯 AI Engine 极高的并行度、易于用 C++ 编写 DSP 算法 某些控制逻辑或特殊运算难以高效表达
纯 HLS 快速开发、高级抽象、自动优化 对某些精细时序控制不够灵活
纯 RTL 完全控制硬件行为、最高性能、可复用既有 IP 开发周期长、验证复杂、难以修改

实际系统中，往往需要结合三者的优势：

  • AI Engine 处理数据密集型 DSP 计算（如 FIR 滤波、FFT）
  • RTL IP 复用已有的硬件设计或实现特殊的硬件优化（如本例中的极坐标裁剪）
  • HLS 内核 处理数据搬移和接口转换（MM2S/S2MM）

本模块解决的具体问题

  1. RTL IP 封装：如何将现有的 RTL 代码（VHDL/Verilog）封装成 Vitis 工具链可识别的 XO 内核格式
  2. 跨域连接：如何在 ADF 计算图中声明与 RTL 内核的接口（PLIO）
  3. 系统集成：如何通过 system.cfg 配置将所有组件连接成一个完整的系统
  4. 主机控制：如何从主机程序同时控制 HLS 内核和 AIE 计算图的执行

组件职责详解

1. MM2S（Memory Map to Stream）HLS 内核

文件：pl_kernels/mm2s.cpp

这是数据管道的入口点。它从 DDR 内存读取数据，通过 AXI-Stream 接口输出到 AI Engine。

void mm2s(ap_int<32>* mem, hls::stream<ap_axis<32, 0, 0, 0>>& s, int size)

关键设计点：

  • 使用 m_axi 接口访问 DDR（#pragma HLS INTERFACE m_axi port=mem）
  • 使用 axis 接口输出数据流（#pragma HLS interface axis port=s）
  • 使用 s_axilite 接口接收主机控制参数（size）
  • 循环流水线化（#pragma HLS PIPELINE II=1），每个时钟周期输出一个样本

为什么这样设计？ 这种分离的接口设计遵循了 Xilinx 推荐的"控制-数据分离"模式：数据路径使用高带宽的 AXI-Stream，控制路径使用轻量级的 AXI-Lite。

2. Interpolator（插值器）AIE 内核

文件：aie/kernels/interpolators/hb27_2i.cc

这是一个 27 抽头半带 FIR 滤波器，执行 2 倍上采样插值。它是典型的 AI Engine 计算密集型任务。

关键特性：

  • 输入：128 个 cint16 样本（窗口缓冲区）
  • 输出：256 个 cint16 样本（2x 插值）
  • 使用 AIE API（aie_api/aie.hpp）进行向量化计算
  • 利用对称系数特性减少乘法运算

数据类型转换：此内核将窗口接口（Window）转换为流接口（Stream），为后续连接到 RTL 内核做准备。

3. Polar Clip（极坐标裁剪）RTL 内核

文件：pl_kernels/hdl/vhdl/polar_clip.vhd

这是本模块的核心教学点——一个用 VHDL 实现的 RTL IP。它计算复数输入向量的模，如果超过阈值则裁剪输出。

关键接口：

  • in_sample_TDATA[31:0]：输入复数样本（实部16位 + 虚部16位）
  • out_sample_TDATA[31:0]：输出裁剪后的复数样本
  • ap_clk：100MHz 时钟（由 system.cfg 配置）

为什么用 RTL 实现？ 这个模块可能是：

  • 遗留的经过验证的 IP 核
  • 需要精确的时序控制
  • 对面积/功耗有特殊优化要求

4. Classifier（分类器）AIE 内核

文件：aie/kernels/classifiers/classify.cc

根据复数输入所在的象限输出分类结果（0-3）。虽然注释说"没人会这样构建分类器"，但它展示了 AIE 如何处理来自 RTL 的数据。

关键设计点：

  • 输入：流接口 input_stream_cint16
  • 输出：窗口接口 output_buffer<int32>
  • 支持仿真（__chess__ 宏）和实际硬件两种模式

5. S2MM（Stream to Memory Map）HLS 内核

文件：pl_kernels/s2mm.cpp

数据管道的出口点，将 AIE 输出的流数据写回 DDR。

void s2mm(ap_int<32>* mem, hls::stream<ap_axis<32, 0, 0, 0>>& s, int size)

端到端数据流追踪

让我们追踪一个数据样本的完整旅程：

1. 主机阶段
   host.exe → xrt::bo 分配输入/输出缓冲区
   → 加载 xclbin → 映射内核

2. 输入准备
   cint16Input[] (data.h) → in_bohdl.sync(TO_DEVICE)
   → DDR 物理内存

3. 数据注入 (PL)
   mm2s_khdl(in_bohdl, nullptr, sizeIn)
   → mm2s 从 DDR 读取 128 个 cint16 样本
   → 通过 AXI-Stream 发送到 AIE

4. 插值处理 (AIE)
   interpolator.in[0] 接收 128 样本窗口
   → fir_27t_sym_hb_2i 执行 27 抽头 FIR
   → 输出 256 样本（2x 上采样）
   → clip_in PLIO 输出到 PL

5. 极坐标裁剪 (PL)
   polar_clip.in_sample 接收流数据
   → 计算幅度并裁剪
   → 通过 out_sample 返回 AIE

6. 分类处理 (AIE)
   classifier 接收裁剪后的样本
   → 判断象限输出 0-3
   → DataOut1 PLIO 输出到 PL

7. 数据收集 (PL)
   s2mm 接收分类结果
   → 写入 DDR 输出缓冲区

8. 结果验证 (Host)
   out_bohdl.sync(FROM_DEVICE)
   → 与 golden[] 数组比较
   → 输出 "Test passed" 或错误计数

关键设计决策与权衡

1. 接口选择：Window vs Stream

决策：Interpolator 使用 Window 输入、Stream 输出；Classifier 使用 Stream 输入、Window 输出。

权衡分析：

  • Window 接口：适合块处理，可利用 AIE 的本地内存做双缓冲，但需要额外的同步开销
  • Stream 接口：适合连续流处理，低延迟，但需要在 PL 侧处理反压（back-pressure）

为什么选择这种混合模式？ 这反映了真实系统中的常见模式：数据以突发（burst）形式到达（Window），处理后以连续流形式传递（Stream），最后又汇聚成块输出（Window）。

2. PLIO 位宽配置

代码（graph.h）：

in = adf::input_plio::create("DataIn1", adf::plio_32_bits, "data/input.txt");

决策：使用 32-bit PLIO 接口。

理由：

  • cint16（复数16位）= 实部16位 + 虚部16位 = 32位
  • 完美匹配数据类型，无需位宽转换
  • 100MHz 时钟下，32-bit 接口提供 3.2 Gbps 带宽，足以满足本设计需求

3. RTL 内核的时钟频率

配置（system.cfg）：

[clock]
freqHz=100000000:polar_clip.ap_clk

决策：将 polar_clip 固定在 100MHz。

权衡：

  • 优点：简化时序收敛，降低功耗
  • 代价：相比更高频率，吞吐率受限
  • 替代方案：可以使用时钟分频器或异步 FIFO 来桥接不同时钟域

4. 控制协议选择

配置（polar_clip_rtl_kernel.tcl）：

package_xo -kernel_name $kernelName \
    -ctrl_protocol ap_ctrl_none \
    ...

决策：使用 ap_ctrl_none（无显式控制协议）。

含义：RTL 内核一旦启动就持续处理数据，不需要主机通过 AXI-Lite 发送启动/停止命令。这与 HLS 内核默认的 ap_ctrl_hs（握手协议）形成对比。

适用场景：适合纯数据流处理，没有内部状态需要重置的场景。

5. 数据对齐与内存分配

代码（sw/host.h）：

struct aligned_allocator {
    T* allocate(std::size_t num) {
        void* ptr = nullptr;
        if (posix_memalign(&ptr, 4096, num*sizeof(T)))
            throw std::bad_alloc();
        return reinterpret_cast<T*>(ptr);
    }
};

决策：使用 4KB 页对齐的内存分配。

原因：

  • DMA 引擎通常要求缓冲区页对齐以获得最佳性能
  • XRT（Xilinx Runtime）底层使用 Linux 内核驱动，页对齐可减少 TLB miss

子模块说明

本模块包含三个主要子模块，分别对应不同类型的计算单元：

PL Kernels 子模块

包含 HLS 编写的 MM2S 和 S2MM 数据搬移内核。这些内核充当 DDR 内存与 AI Engine 之间的数据桥梁，使用标准的 AXI4 协议族（AXI4-Full 用于内存访问，AXI4-Stream 用于数据流传输）。

AIE Kernels 子模块

包含 AI Engine 计算内核的实现，包括 27 抽头 FIR 插值器和象限分类器。这些内核展示了如何使用 AIE API 进行高性能 DSP 计算，以及如何处理 Window 和 Stream 两种接口类型。

System Integration 子模块

包含系统级配置文件和 RTL IP 封装流程。核心文件 system.cfg 定义了所有内核实例及其连接关系，polar_clip_rtl_kernel.tcl 展示了如何将 VHDL 代码封装为 Vitis 可用的 XO 内核。

新贡献者注意事项

常见陷阱

  1. PLIO 名称不匹配

    // graph.h
clip_in = adf::output_plio::create("clip_in", ...);

    # system.cfg
sc=ai_engine_0.clip_in:polar_clip.in_sample

    PLIO 对象名（clip_in）必须与 system.cfg 中的接口名完全一致，否则链接阶段会报错。

  2. RTL 时钟配置遗漏 如果忘记在 system.cfg 中指定 freqHz，Vitis 编译器可能无法正确推断时钟域，导致时序违规或功能错误。

  3. 数据类型大小假设

    auto in_bohdl = xrt::bo(device, sizeIn * sizeof(int16_t) * 2, 0, 0);

    注意 cint16 占用 4 字节（实部2字节 + 虚部2字节），不是 2 字节。这里的 * 2 是因为 int16_t 是 2 字节，而复数是 2 个 int16。

  4. 仿真与硬件的行为差异

    #ifdef __chess__
  int32 from_stream = get_ss(0);  // 仿真模式
#else
  cint16 sample = readincr(input); // 硬件模式
#endif

    AIE 内核代码中经常使用条件编译区分仿真和硬件。修改时要确保两种模式都测试。

调试技巧

  1. 使用 Vivado 仿真波形

    ./launch_hw_emu.sh -g  # 启动 GUI 仿真

    可以观察 AXI-Stream 上的实际数据传输，验证数据格式是否正确。

  2. 检查 kernel.xml RTL IP 封装后会生成 kernel_${kernelName}_auto.xml，其中描述了 RTL 内核的端口和协议。如果连接失败，首先检查此文件。

  3. 启用 HLS 调试信息

    [hls]
syn.debug.enable=1

    这使得在 Vivado 中可以单步调试 HLS 内核的 C/C++ 源代码。

扩展建议

如果要基于此模块开发自己的设计：

  1. 替换 polar_clip：修改 polar_clip_rtl_kernel.tcl 指向你的 RTL 源码，保持 AXI-Stream 接口兼容
  2. 调整数据宽度：如果改变样本位宽，需要同步修改：
    • graph.h 中的 plio_XX_bits
    • RTL 内核的 TDATA 宽度
    • 主机代码中的缓冲区大小计算
  3. 增加流水线深度：如果插入更多 RTL 或 HLS 处理阶段，注意在 system.cfg 中添加相应的 sc= 连接行

总结

RTL IP with AIE System Instances 模块是一个精心设计的教学示例，它展示了 Versal ACAP 平台最强大的特性之一：异构计算的灵活性。通过将 RTL IP（精确硬件控制）、HLS 内核（快速数据搬移）和 AI Engine（高性能 DSP）结合在一起，开发者可以在同一个设计中利用最适合每种任务的编程模型。

理解这个模块的关键在于把握接口契约：AXI-Stream 连接各组件，AXI-Lite 用于控制，而 system.cfg 则是整个系统的"接线图"。掌握这些概念后，你就可以开始构建自己的异构加速系统了。
