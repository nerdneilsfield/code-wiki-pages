FIR AIE vs HLS 链式与内核契约模块

概述

本模块是 AMD Versal 自适应计算平台中 FIR 滤波器设计对比教程 的核心组成部分，旨在系统性地比较和评估两种截然不同的硬件加速实现路径：AI Engine (AIE) 阵列编程模型 与 HLS (高层次综合) 基于 DSP58 的 RTL 级实现。

想象你正在设计一个高性能数字信号处理流水线——就像在一个工厂中安排生产线。你可以选择两种生产方式：一种是雇佣大量专业化工人（AIE 核心）协同完成复杂任务，另一种是使用一台高度优化的自动化机器（HLS 生成的硬件）。本模块就是这场"人力协作 vs 机器自动化"对决的技术试验场。

为什么需要这个模块？

在现代异构计算架构中，开发者面临一个关键决策：何时使用 AI Engine 的可编程并行性，何时使用 HLS 的专用硬件优化？ 这个模块通过实现功能等价的 FIR 滤波器链，提供了：

  1. 公平的性能基准测试框架 —— 确保两种实现在相同数据流、相同接口契约下进行对比
  2. 设计空间探索方法论 —— 展示如何根据滤波器阶数、通道数量、采样率等参数权衡选择
  3. 系统集成最佳实践 —— 演示 PL (可编程逻辑) 与 AIE/RTL 内核的统一控制接口

关键观察：统一的控制抽象

尽管底层硬件截然不同（AIE 阵列 vs DSP58 Slice），两个路径的上层控制代码呈现出惊人的对称性：

// AIE 路径 - 图级抽象
fir_chain_class fir_graph;
fir_graph.init(dhdl, top);   // 打开图句柄
fir_graph.run();             // 启动图执行（无限循环模式）
...
fir_graph.close();           // 释放图资源

// HLS 路径 - 内核级抽象  
fir_class fir_krnl;
fir_krnl.init(dhdl, top, iterCnt);  // 打开内核句柄
fir_krnl.run();                      // 启动内核执行
...
fir_krnl.close();                    // 释放内核资源

这种对称性并非偶然，而是刻意的设计选择：它允许主机代码在几乎不修改的情况下切换底层实现，为公平的性能对比创造了条件。

核心设计决策与权衡

决策 1：XRT Native API vs OpenCL 抽象层

选择：直接使用 experimental/xrt_aie.h 和 experimental/xrt_kernel.h 中的原生 XRT API，而非更高层的 OpenCL 封装。

权衡分析：

  • ✅ 优势：更细粒度的控制（如 xrtGraphReset 的显式调用）、更低的运行时开销、更接近硬件的语义表达
  • ❌ 代价：代码冗长（需手动管理 xrtGraphHandle、xrtRunHandle 等句柄）、平台可移植性降低

为何如此选择：这是一个面向教学的参考设计，首要目标是清晰展示 Versal 平台的原生编程模型，而非追求跨平台兼容性。对于生产代码，可能需要考虑添加一个薄封装层。

决策 2：RAII 封装 vs 裸句柄操作

选择：将 XRT 资源句柄封装在 C++ 类中，利用构造函数/析构函数模式（虽然此处显式调用 init/close，但语义上遵循 RAII）。

权衡分析：

  • ✅ 优势：异常安全（栈展开时会自动调用 close）、代码可读性提升、减少资源泄漏风险
  • ❌ 代价：额外的虚函数表开销（此处无虚函数）、类的定义和维护成本

关键代码模式：

class fir_chain_class {
    xrtGraphHandle fir_g_hdl;  // 所有权明确归属于此类实例
public:
    int init(...) {  // 获取资源
        fir_g_hdl = xrtGraphOpen(...);
        if (fir_g_hdl == NULL) throw ...;
    }
    void close() {   // 释放资源
        xrtGraphClose(fir_g_hdl);
    }
};

决策 3：编译时参数化 vs 运行时配置

选择：HLS 路径大量使用模板特化和宏定义（N_FIR_TAPS、N_FIR_FILTERS、SAMP_PERIOD），AIE 路径依赖外部图定义文件。

权衡分析：

  • ✅ 优势：编译期优化机会最大化（HLS 可针对特定系数生成最优硬件）、零运行时配置开销
  • ❌ 代价：每次参数变更需重新综合、二进制文件膨胀（不同配置需独立 xclbin）

为何如此选择：FIR 滤波器的系数和结构在设计阶段通常已确定，属于静态配置域。将其提升到编译期符合"将不变量固化到硬件"的 FPGA 设计哲学。

决策 4：共享 datamover 抽象 vs 路径专用数据通路

选择：两种实现复用完全相同的 datamover_class，包括 DMA 缓冲区大小 (INP_SIZE)、错误检查机制 (golden_check) 和 XRT 调用序列。

深层考量：

  • 这是实验控制变量的关键——确保任何性能差异都源自 FIR 实现本身，而非数据 I/O 路径
  • 强制两种实现遵循相同的流式数据契约（AXI-Stream 128-bit 位宽、ap_axiu<128,0,0,0> 格式）

关键实现细节与潜在陷阱

1. 内存所有权与生命周期

XRT 句柄的生命周期管理：

  • xrtDeviceHandle dhdl：由 main 函数拥有，传递给各子系统借用，最后由 xrtDeviceClose 释放
  • xrtGraphHandle / xrtKernelHandle：由各自的 wrapper class 独占拥有，禁止拷贝（应声明为 = delete 的拷贝构造，虽然代码中未显式声明）
  • xrtRunHandle：与对应的 Kernel/Graph Handle 绑定，必须先关闭 Run Handle 再关闭 Kernel Handle

潜在风险：

// 危险顺序：若先关闭 kernel handle，run handle 将变为悬空
xrtKernelClose(datamover_khdl);  // ❌ 错误
xrtRunClose(datamover_rhdl);     // 此时可能访问已释放资源

// 正确顺序
xrtRunClose(datamover_rhdl);     // ✅ 先关闭运行实例
xrtKernelClose(datamover_khdl);  // 再关闭内核

2. 错误处理策略的不一致性

观察到的技术债务：

  • fir_chain_class::init 返回 int 错误码并抛出异常（双重机制）
  • fir_chain_class::run 仅返回 int 错误码
  • datamover_class::init 返回 void，失败时直接打印日志

建议：新贡献者应优先使用异常处理路径（std::runtime_error），这与现代 C++ 实践一致，且能自动处理栈展开。

3. 隐式的全局状态依赖

// 这些宏定义来自外部头文件，实际影响程序行为
#define INP_SIZE 2048           // AIE 路径
#define INP_SIZE 8192           // HLS 路径（注意：同名宏不同值！）
#define N_FIR_FILTERS ...       // system_settings_comm.h
#define N_FIR_TAPS ...          // system_settings_comm.h

陷阱警告：两个路径的 INP_SIZE 定义不同（AIE: 2048, HLS: 8192），这并非 bug，而是反映了底层实现的处理能力差异。但在阅读代码时极易产生困惑。

4. HLS FIR 的采样周期自适应逻辑

#if N_FIR_TAPS > 64
   #define SAMP_PERIOD 4  // 降速以保证可行性
#else
   #define SAMP_PERIOD 1  // 全速运行
#endif

工程洞察：当抽头数超过 64 时，HLS 无法在一个时钟周期内完成所有乘累加操作，必须插入流水线气泡。这是一种时序收敛驱动的性能降级策略。

5. AIE 图的无限循环模式

errCode = xrtGraphRun(fir_g_hdl, -1);  // -1 表示无限循环

与 HLS 的关键差异：

  • HLS 内核是迭代驱动的（iterCnt 参数明确指定迭代次数）
  • AIE 图是数据流驱动的，-1 让其持续运行直到被显式停止或复位

同步机制：主机通过等待 datamover 完成来间接确认 AIE 图已处理完所需数据，这是一种隐式同步设计。

扩展建议

若需添加新的滤波器配置（如 32-tap）：

  1. 更新 system_settings_comm.h 中的 N_FIR_TAPS 定义
  2. HLS 路径：在 fir_hls.h 中添加对应的 #if N_FIR_TAPS == 32 分支，确定合适的 SAMP_PERIOD
  3. AIE 路径：更新图定义文件（不在本模块代码中，需参考 AIE 工具链文档）
  4. 重新生成 golden 参考数据，更新 golden_check 的预期错误阈值
