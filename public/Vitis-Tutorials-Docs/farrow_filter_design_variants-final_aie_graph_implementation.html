final_aie_graph_implementation 模块技术深度解析

概述：这个模块解决了什么问题？

想象你正在设计一个音频处理系统，需要实时改变信号的采样率——比如将 48kHz 的音频流转换为 44.1kHz。传统的做法是使用多相滤波器组配合插值和抽取，但当分数倍率变化时（比如 48/44.1 ≈ 1.088），这种方法会变得异常复杂。Farrow 滤波器正是为解决这类任意分数倍率重采样问题而生的一种优雅架构。

本模块 final_aie_graph_implementation 是 Farrow 滤波器在 AMD AIE-ML（AI Engine with Machine Learning enhancements）上的最终优化实现。它代表了从初始移植版本经过两轮迭代优化后的成熟方案，核心目标是：在保持算法精度的同时，最大化利用 AIE-ML 的 SIMD 向量运算能力和专用 MAC（乘累加）单元，实现高吞吐量的实时分数延迟滤波。

与软件实现的 Farrow 滤波器不同，这里的挑战在于：AIE-ML 是高度并行的 VLIW 架构，拥有固定的 512-bit 向量宽度和特定的内存访问模式。 naive 的移植会导致严重的资源浪费和流水线停顿。本模块的设计精髓在于将 Farrow 的多项式插值结构映射到 AIE-ML 的滑动窗口乘法指令集上，通过精心安排的系数布局和双核流水线分工，实现了接近理论峰值的计算效率。

心智模型：如何理解这个架构？

类比：精密钟表工坊的流水线

想象一个制作精密钟表的工坊：

  • 第一道工序（Kernel 1）：工匠 A 负责将原材料（输入信号）切割成标准规格的零件（滤波后的多项式基带信号 y0, y1, y2, y3）。他使用一套固定模板（FIR 系数）批量加工，每次产出 4 种不同规格的零件。

  • 第二道工序（Kernel 2）：工匠 B 接收这些零件，根据客户定制的订单要求（分数延迟参数 μ），将零件组装成最终的钟表（插值输出）。他需要临时工作台（中间缓冲区 z）来存放半成品，因为组装过程需要分阶段完成。

  • 工坊布局（Graph）：两个工匠的工作台紧密相连，零件通过传送带直接传递，无需仓库存储。整个工坊有统一的进出口（PLIO）对接外部物流系统。

这个类比揭示了 Farrow 滤波器的核心数学结构：

\[ y(\mu) = y_0 + \mu \cdot (y_1 + \mu \cdot (y_2 + \mu \cdot y_3)) \]

其中 \(y_0, y_1, y_2, y_3\) 是四个并行 FIR 滤波器的输出，\(\mu\) 是时变的分数延迟参数。Kernel 1 计算所有 \(y_i\)，Kernel 2 执行嵌套的 Horner 求值。

分层架构说明

1. dut_graph —— 测试平台包装层

位于 farrow_app.cpp 中的 dut_graph 是一个测试平台适配层，而非核心算法实现。它的设计体现了 AIE 开发中常见的模拟/硬件双模式编译模式：

#ifdef AIE_SIM_ONLY
    // 仿真模式：从文本文件读取输入，输出写入文本文件
    sig_i[0] = input_plio::create("PLIO_i_0", plio_64_bits, "data/sig_i.txt");
#else
    // 硬件模式：纯接口定义，数据来自 PL（Programmable Logic）
    sig_i[0] = input_plio::create("PLIO_i_0", plio_64_bits);
#endif

这种条件编译策略允许同一套图描述在两种场景下工作：

  • AIE Simulator：独立验证算法正确性，无需完整的 Versal 硬件平台
  • Vitis/Vivado 集成：作为子图嵌入更大的系统，与 PL 侧的 DMA/数据搬移器协同

2. farrow_graph —— 核心计算图

farrow_graph（定义于 farrow_graph.h）是真正的算法载体。它继承自 adf::graph，这是 AMD AIE 工具链提供的领域特定语言（DSL）基类。

关键配置决策：

runtime<ratio>(farrow_kernel1_ins) = 0.9;
runtime<ratio>(farrow_kernel2_ins) = 0.9;

runtime<ratio> 设置内核占用其所在 AIE 核的时间比例。0.9 表示该内核预计占用 90% 的计算周期，留下 10% 的余量给调度开销。这是性能调优的关键参数——设置过低会浪费算力，设置过高可能导致时序违例。

location<stack>(farrow_kernel1_ins) = location<kernel>(farrow_kernel1_ins);

这行代码将栈空间分配到内核所在的同一个 AIE Tile，避免跨 Tile 内存访问带来的延迟。在 AIE 架构中，数据局部性就是性能。

3. Kernel 1：多项式基带生成

farrow_kernel1 实现了 Farrow 结构的核心创新——将分数延迟滤波分解为固定系数的并行 FIR 滤波器组。

数学原理：

传统分数延迟滤波器需要为每个延迟值 \(\mu\) 重新计算或查找一组滤波器系数。Farrow 结构的洞察是：可以将理想频率响应展开为 \(\mu\) 的多项式：

\[ H(e^{j\omega}, \mu) \approx \sum_{k=0}^{K} C_k(e^{j\omega}) \cdot \mu^k \]

其中 \(C_k\) 是固定的多项式系数滤波器。本实现使用 \(K=3\)（三次多项式），因此需要 4 个并行 FIR 滤波器（对应 \(\mu^0, \mu^1, \mu^2, \mu^3\) 的系数）。

向量化策略：

// 16 抽头系数被紧凑打包为一个向量
alignas(32) static constexpr int16 f_taps[16] = {
    // flip(f3_taps.int(end-3:end)), flip(f2_taps...), flip(f1_taps...), flip(f0_taps...)
    206,-1264,6606,-14835,906,-3543,10352,-7628,
    -51,316,-1652,20093,-226,886,-2588,10099
};

注意注释中的关键信息：系数数组是经过精心排列的，每 4 个元素对应一个子滤波器的最后 4 个抽头，且顺序是翻转的（flip）。这种排列是为了匹配 AIE-ML 的 sliding_mul_sym_xy_ops 指令的硬件预期格式。

滑动窗口乘法：

acc_f3 = aie::sliding_mul_sym_xy_ops<8,8,1,1,int16,cint16>::mul_antisym(
    f_coeffs, 0, v_buff, 9);

这行代码是性能的核心。sliding_mul_sym_xy_ops 是 AIE-ML 特有的对称/反对称滑动窗口乘法指令：

  • <8,8,1,1>：8 输入 × 8 系数，步长 1， lanes 数 1
  • mul_antisym：利用系数的反对称性减少乘法次数
  • 偏移参数 0, 4, 8, 12 分别选取 f_taps 中对应子滤波器的起始位置

状态管理：

v_buff.insert(1, aie::load_v<8>(f_state));  // 加载历史状态
// ... 处理 ...
*(v8cint16*)f_state = v_buff.extract<8>(1);  // 保存新状态

FIR 滤波是有状态的（需要前一次的样本）。Kernel 1 使用 f_state[8] 保存 8 个历史样本，通过 v_buff 的 insert/extract 操作实现零拷贝的状态传递。

4. Kernel 2：Horner 求值

farrow_kernel2 接收 Kernel 1 产生的 4 路基带信号和外部输入的延迟参数 \(\mu\)，执行多项式求值：

\[ output = y_0 + \mu \cdot (y_1 + \mu \cdot (y_2 + \mu \cdot y_3)) \]

三阶段流水线：

代码中三个几乎相同的循环块对应 Horner 求值的三次迭代：

// Stage 1: 计算 z = y2 + μ·y3
acc_x = aie::mac(aie::from_vector<TT_ACC>(*p_y2++, DNSHIFT), *p_y3++, del);
*p_z2++ = acc_x.to_vector<TT_SIG>(DNSHIFT);

// Stage 2: 计算 z' = y1 + μ·z
acc_y = aie::mac(aie::from_vector<TT_ACC>(*p_y1++, DNSHIFT), *p_z2++, del);
*p_z1++ = acc_y.to_vector<TT_SIG>(DNSHIFT);

// Stage 3: 输出 = y0 + μ·z'
acc_z = aie::mac(aie::from_vector<TT_ACC>(*p_y0++, DNSHIFT), *p_z1++, del);
*p_sig_o++ = acc_z.to_vector<TT_SIG>(DNSHIFT);

为什么需要中间缓冲区 z[BUFFER_SIZE]？

这是一个关键的架构决策。Horner 求值本质上是递归的——下一阶段的输入依赖上一阶段的输出。由于 AIE-ML 的缓冲区模型要求明确的输入/输出契约，无法在一个内核调用内完成三阶段流水。因此需要一个乒乓缓冲区 z 来存储中间结果。

注意 z 的大小是 BUFFER_SIZE（1024），这意味着它可以在单次内核调用中容纳所有中间数据，避免了多次调用的上下文切换开销。

设计权衡与决策分析

1. 双核分工 vs. 单核融合

现状：Kernel 1 和 Kernel 2 分布在两个独立的 AIE 核上，通过流连接通信。

替代方案：将两者融合到一个内核中，消除核间通信开销。

权衡考量：

  • 选择分离的原因：

    • Kernel 1 是纯 FIR，计算密集但无递归依赖；Kernel 2 是多阶段 MAC，有数据依赖。两者的最优流水线深度不同。
    • 分离允许独立的 runtime<ratio> 调优——如果合并，必须取两者需求的较大值。
    • AIE-ML 的本地内存有限（32KB 数据 + 16KB 程序），合并可能导致内存压力。

  • 代价：核间流连接引入约 10-20 周期的延迟，但对于 BUFFER_SIZE=1024 的批处理，这部分开销被充分摊销。

2. 定点点数与动态范围

static constexpr unsigned DNSHIFT = 14;
typedef cint16 TT_SIG;   // 16-bit 实部 + 16-bit 虚部
typedef cacc48 TT_ACC;   // 48-bit 累加器

设计选择：使用 16-bit 定点而非浮点。

理由：

  • AIE-ML 的峰值性能（TOPS）主要针对定点运算标定，浮点吞吐量约为定点的 1/4~1/8。
  • Farrow 滤波器的系数动态范围可控，通过合理的缩放（DNSHIFT=14）可以避免溢出。
  • cacc48 累加器提供足够的精度余量，防止多抽头累加的精度损失。

风险：输入信号幅度过大时可能发生饱和。构造函数中设置了饱和模式：

aie::set_saturation(aie::saturation_mode::saturate);

这比环绕（wrap-around）更安全，但仍需上游保证信号电平。

3. 缓冲区大小与延迟权衡

BUFFER_SIZE = 1024 意味着：

  • 最小延迟：1024 个样本周期（约 21μs @ 48kHz 采样率）
  • 批处理优势：摊销了内核启动开销，允许更深的软件流水线

对于实时音频应用，1024 样本是可接受的（专业音频通常容忍 1-10ms 延迟）。如果需要更低延迟，可以减小 BUFFER_SIZE，但会降低吞吐效率。

4. 系数存储布局优化

注意到 f_taps 的声明：

alignas(32) static constexpr int16 f_taps[16] = {...};

alignas(32) 确保系数数组对齐到 256-bit（32 byte）边界，这是 AIE-ML 向量加载指令的最优对齐方式。未对齐访问会导致额外的周期惩罚。

系数值的排列顺序（flip + interleave）是为了匹配 sliding_mul_sym_xy_ops 的硬件预期。这是算法专家与硬件架构协同设计的典型例子——如果系数按自然顺序排列，同样的数学结果需要更多指令周期完成。

C/C++ 实现细节深度分析

内存所有权模型

本模块采用静态分配 + 栈上状态的混合内存策略，这是 AIE 编程的典型模式：

1. 系数数组（只读静态存储）

alignas(32) static constexpr int16 f_taps[16] = {...};
  • 分配者：编译器在 .rodata 段分配
  • 所有者：类模板（隐式共享），每个实例只读访问
  • 生命周期：程序全程有效
  • 对齐要求：32 字节对齐以满足 AIE-ML 向量加载指令的硬件约束

2. 滤波器状态（实例私有）

alignas(32) TT_SIG f_state[STATE_LEN];  // Kernel 1
alignas(32) TT_SIG z[BUFFER_SIZE];      // Kernel 2
  • 分配者：AIE 工具链在 init() 时为每个内核实例分配本地数据内存（DMEM）
  • 所有者：内核对象实例（farrow_kernel1 / farrow_kernel2）
  • 借用关系：通过 v_buff.insert/extract 操作临时借用，无堆分配开销
  • RAII 策略：遵循 Rule of Zero —— 无动态分配，无需自定义析构

3. 输入/输出缓冲区（调用者提供）

void run(input_buffer<TT_SIG,...>& __restrict sig_i, ...)
  • 分配者：ADF 运行时（基于 BUFFER_SIZE 配置）
  • 所有者：ADF 运行时管理，内核仅获得引用
  • 契约：__restrict 承诺无别名，违反将导致向量化错误

对象生命周期与值语义

Rule of Zero 适用性分析：

两个内核类（farrow_kernel1、farrow_kernel2）均遵循 Rule of Zero：

  • 无原始指针成员（状态数组是固定大小的值类型）
  • 无动态分配
  • 编译器生成的拷贝/移动/析构行为完全正确

这意味着：

farrow_kernel1 k1;           // OK
farrow_kernel1 k2 = k1;      // OK - 浅拷贝状态数组（独立副本）
farrow_kernel1 k3 = std::move(k1);  // OK - 移动语义等价于拷贝

注意：虽然语法上允许拷贝，但在实际 AIE 部署中，内核对象由 kernel::create_object<>() 工厂方法构造，通常以单例形式存在。

错误处理策略

本模块采用前置条件断言 + 饱和保护的双重策略：

1. 编译时契约（静态断言）

static_assert(BUFFER_SIZE % 16 == 0, "Buffer size must be multiple of 16");
// 隐含在 chess_loop_range 和循环计数中的假设

这些约束由架构保证，而非运行时检查。

2. 数值溢出保护

farrow_kernel1::farrow_kernel1(void) {
    aie::set_rounding(aie::rounding_mode::symmetric_inf);
    aie::set_saturation(aie::saturation_mode::saturate);
}
  • 异常安全等级：无异常（AIE 运行时禁用 C++ 异常）
  • 错误传播：数值溢出被静默钳位到最大/最小值，无错误码返回
  • 前置条件：调用者必须保证输入信号幅度经过适当缩放，避免频繁饱和

3. 调试支持 仿真模式下可通过对比 data/sig_o.txt 与 MATLAB 参考输出发现数值问题。

Const-Correctness 与可变性模型

类级别分析：

成员 可变性 说明
f_taps static constexpr 编译期常量，物理存储于只读存储器
DNSHIFT, BUFFER_SIZE static constexpr 模板参数式常量，影响代码生成
f_state 实例可变 每次 run() 调用后更新，保持 FIR 连续性
z 实例可变 Kernel 2 的乒乓缓冲区，调用间不保留（重新初始化）

方法级别分析：

// Kernel 1
void run(input_buffer<...>& __restrict sig_i,   // 输入：逻辑上 const，但 API 非 const&
         output_buffer<...>& __restrict y3,     // 输出：必然可变
         ...);

// 注意：input_buffer 不是 const&，这是 ADF API 的设计选择
// 实际语义：内核不会修改输入缓冲区内容，但运行时可能更新元数据

关键观察：run() 方法本身不是 const，因为它需要修改内部状态（f_state）。这准确反映了 FIR 滤波器的有状态本质。

API 契约与前置条件

调用者必须保证：

  1. 缓冲区对齐：输入/输出缓冲区必须 32 字节对齐（由 ADF 运行时保证）
  2. 大小匹配：所有缓冲区的大小必须等于 BUFFER_SIZE * sizeof(TT_SIG)
  3. 延迟参数格式：del_i 中的值必须是 Q15 格式的定点数（范围 [-1, 1)），已左移 14 位
  4. 无重叠缓冲区：__restrict 要求输入/输出缓冲区指向不相交的物理内存区域
  5. 连续调用：若输入流中断，必须在边界处重新初始化图（end() + init()）以重置 f_state

未定义行为风险：

  • 传入未对齐的缓冲区 → 总线错误
  • 缓冲区大小不匹配 → 内存越界
  • 延迟参数未缩放 → 错误的插值结果（静默失败）
  • 输入/输出缓冲区重叠 → 向量化结果损坏

并发与线程安全

单线程设计声明：本模块严格单线程，无并发安全考虑。

执行模型：

  • 每个内核映射到一个专用的 AIE 核
  • 内核间通过 FIFO 流连接通信，无共享内存竞争
  • 同一内核不会被重入调用（ADF 运行时分批调度）

多实例部署：

// 合法：多个独立实例
farrow_graph g1, g2, g3;

每个实例拥有独立的内核对象和状态数组，互不干扰。

性能架构

热点路径分析：

代码区域 执行频率 优化策略
sliding_mul_sym_xy_ops 每样本 4 次调用 利用对称/反对称性减少 50% 乘法次数
aie::mac (Horner 求值) 每样本 3 次调用 48-bit 累加器避免中间溢出
循环主体 BUFFER_SIZE/16 次迭代 chess_prepare_for_pipelining 生成软件流水线

数据布局决策：

// Kernel 1: 向量寄存器布局（256-bit = 16 × cint16）
// v_buff[0:7]  = 新输入样本
// v_buff[8:15] = 历史状态（来自 f_state）
v_buff.insert(1, aie::load_v<8>(f_state));  // 加载历史到高位
v_buff.insert(0, *p_sig_i++);              // 插入新样本到低位

这种布局允许单次 sliding_mul 操作同时访问新旧样本，无需额外的数据重组指令。

分支预测与循环展开：

for (unsigned rr=0; rr < BUFFER_SIZE/16; rr++)
    chess_loop_range(1,0)        // 告知编译器：至少 1 次迭代，无上限
    chess_prepare_for_pipelining // 启用软件流水线
  • chess_loop_range(1,0) 中的 0 表示"无上限"，允许编译器针对大缓冲区优化
  • 循环体内部手动展开（两次迭代），匹配 AIE-ML 的双发射能力

算法复杂度权衡：

FIR 滤波的理论复杂度是 \(O(N \cdot M)\)（\(N\) 样本数，\(M\) 抽头数）。本实现通过以下方式降低常数因子：

  • 向量化：一次处理 8 个样本（SIMD 因子 = 8）
  • 对称性利用：mul_sym / mul_antisym 减少约 50% 乘法
  • 有效吞吐量：每个 AIE-ML 核可达约 1 GMAC/s（取决于时钟频率）

修改滤波器特性

如需适配不同的分数延迟精度或通带要求：

  1. 重新设计系数：修改 farrow_kernel1.h 中的 f_taps 数组。注意保持 16 元素长度和特定的排列顺序。

  2. 调整定点缩放：如果新系数的动态范围不同，可能需要修改 DNSHIFT。增大 DNSHIFT 提高精度但降低有效位宽，减小则相反。

  3. 验证数值稳定性：使用 check_sim_output.m MATLAB 脚本对比参考实现输出。

瓶颈场景 优化策略

陷阱与注意事项

1. 严格的类型契约

farrow_kernel2 的延迟输入类型是 int32（TT_DEL），但实际只使用低 16-bit 转换为 int16：

del = aie::vector_cast<int16>(*p_del_i++);

隐含假设：调用者保证延迟参数已经过适当的定点缩放，且高位无有效信息。传入未缩放的浮点转换值将导致错误结果。

2. 缓冲区别名限制

Kernel 2 使用了 __restrict 关键字标记指针：

void run(input_buffer<TT_DEL,...>& __restrict del_i, ...)

这向编译器承诺输入/输出缓冲区互不重叠。违反此假设（例如将同一物理缓冲区同时作为输入和输出）将导致未定义的向量化行为。

3. 状态持久性

Kernel 1 的 f_state 是成员变量，在 run() 调用之间保持：

private:
    alignas(32) TT_SIG f_state[STATE_LEN];

这意味着：

  • 连续性要求：如果输入流不连续（存在间隙），必须在间隙处重置状态或保证恰好填满 BUFFER_SIZE 边界。
  • 多实例隔离：每个 farrow_graph 实例有独立的 Kernel 对象，状态不会交叉污染。

4. 仿真与硬件的行为差异

AIE_SIM_ONLY 宏控制的代码路径仅在仿真时使用。确保：

  • 仿真通过后，在硬件编译前移除或更新数据文件路径
  • 硬件模式下，PLIO 的连接需要在 Vitis 链接配置中显式声明（通常在 system.cfg 或 Vivado IPI 中）

5. chess 编译器提示

chess_loop_range(1,0)
chess_prepare_for_pipelining

这些是 AMD chess 编译器的专用 pragma：

  • chess_loop_range(min,max)：告知编译器循环迭代次数范围，辅助软件流水线调度
  • chess_prepare_for_pipelining：指示编译器尝试生成流水线化的循环主体

警告：如果实际迭代次数超出声明的范围，生成的二进制可能行为异常。修改 BUFFER_SIZE 时必须同步调整这些提示。

总结

final_aie_graph_implementation 是一个经过深度优化的 AIE-ML 实现范例，展示了如何将经典的 DSP 算法（Farrow 滤波器）映射到异构 AI 加速器架构。其核心设计智慧在于：

  1. 算法-硬件协同设计：系数排列、向量宽度、滑动窗口指令的选择都紧密围绕 AIE-ML 的微架构特性
  2. 显式流水线划分：双核分工平衡了计算负载与数据依赖性
  3. 零开销抽象：通过 adf::graph DSL 在保持高层表达力的同时，生成接近手写汇编的效率

对于新加入团队的开发者，建议按照以下顺序深入理解：

  1. 阅读 farrow_kernel1.h 理解 FIR 的向量化策略
  2. 跟踪 farrow_kernel2.cpp 的三阶段 Horner 求值数据流
  3. 研究 farrow_graph.h 的连接拓扑和资源配置
  4. 最后审视 farrow_app.cpp 的测试平台封装模式

这一学习路径遵循"由内向外"的原则，先掌握计算核心，再理解系统集成。
