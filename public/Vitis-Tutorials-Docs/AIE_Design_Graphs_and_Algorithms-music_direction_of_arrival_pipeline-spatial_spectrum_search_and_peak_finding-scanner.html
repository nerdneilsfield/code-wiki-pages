Scanner 模块深度解析

一句话概括

Scanner 是 MUSIC（Multiple Signal Classification）算法流水线中的阈值检测与标记阶段，它负责在 DOA（Direction of Arrival）阶段生成的空间谱中扫描潜在峰值区域，为后续的 Finder 精确定位提供候选标记。你可以把它想象成一个**"雷达筛子"**——先用粗粒度阈值快速筛选出可能藏有信号峰值的区域，打上标记后再交给更精细的探测器去确认。

问题空间：为什么需要 Scanner？

MUSIC 算法的最终目标是估计信号到达角度（DOA）。整个流水线大致如下：

输入信号 → QRD（QR分解）→ SVD（奇异值分解）→ DOA（空间谱计算）→ Scanner（阈值检测）→ Finder（峰值精确定位）→ 输出角度估计

在 DOA 阶段，系统生成了一个长度为 NUM_POINTS = 256 的空间谱数组 Pm_null，表示不同角度方向上的信号功率。这个谱的特点是：

  • 大部分区域的值接近噪声水平
  • 真正的信号源会在特定角度产生尖锐的峰值
  • 需要一种高效的方式识别哪些区域值得进一步分析

一个朴素的做法是让 Finder 直接扫描全部 256 个点，但这会带来两个问题：

  1. 计算效率低：Finder 需要进行复杂的局部极大值检测和插值计算，对每个点都执行这样的操作是浪费的
  2. 并行度受限：单个 AIE 核的处理能力有限，难以在时序约束内完成全谱扫描

Scanner 的设计洞察是：先用一个简单的阈值测试快速排除明显不是峰值的区域，只把"可疑区域"标记出来传给 Finder。这类似于图像处理中的"感兴趣区域（ROI）检测"——先找出可能包含目标的区域，再对这些区域做精细处理。

核心抽象与心智模型

类比：机场安检的分级筛查

想象机场安检流程：

  1. 第一级（Scanner）：所有人通过金属探测门，只有报警的人才会被标记出来
  2. 第二级（Finder）：被标记的人接受人工手检，精确定位身上携带的物品

Scanner 就是 MUSIC 流水线中的"金属探测门"——它不关心峰值的具体位置，只回答一个问题："这个区域是否值得 Finder 关注？"

核心数据结构

概念 含义 参数来源
NUM_POINTS = 256 空间谱总点数（角度分辨率） music_parameters.h
ANGLE_FINDER_REGION_LEN = 8 每个区域的长度（8个连续点） music_parameters.h
NUM_REGIONS = 32 总区域数（256 / 8） 计算得出
NUM_SCANNER_KERNELS = 2 Scanner 使用的 AIE 核数量 music_parameters.h
DOA_MIN_THRESHOLD = 0.25f 阈值，低于此值视为噪声 music_parameters.h

空间谱被划分为 32 个区域，每个区域 8 个点。两个 Scanner 核各自处理 16 个区域（第一个核处理区域 0-15，第二个核处理区域 16-31）。

输出标记（Tag）的语义

Scanner 的输出是一个长度为 NUM_REGIONS 的二进制标记数组：

  • tag[i] = 1：第 i 个区域中存在至少一个点超过阈值 DOA_MIN_THRESHOLD
  • tag[i] = 0：第 i 个区域所有点都低于阈值

这些标记以 cfloat 格式存储（实部为 0 或 1，虚部为 0），便于在 AIE 数据流中传输。

组件深度解析

1. Scanner 类模板 (scanner.h, scanner.cpp)

设计意图

Scanner 是一个模板类，使用非类型模板参数 START 和 END 来指定处理的区域范围。这种设计允许：

  • 编译期确定工作范围：不同的实例处理不同的区域，无运行时分支开销
  • 代码复用：同一个类模板实例化为多个独立内核
  • 显式负载划分：第一个核处理 [0, NUM_REGIONS/2)，第二个核处理 [NUM_REGIONS/2, NUM_REGIONS)

内存布局与对齐

class Scanner {
private:
    alignas(32) aie::vector<cfloat, N>  noiseSS;           // 占位符，未实际使用
    alignas(32) aie::vector<cfloat, N>  Pm_null_cfloat[NUM_REGIONS];  // 缓存空间谱
};
  • alignas(32)：确保数据按 32 字节对齐，满足 AIE 向量加载指令的对齐要求
  • aie::vector<cfloat, N>：使用 AIE API 的向量类型，N = 8 表示每次处理 8 个复数
  • 栈上分配：所有数据都在内核栈上，无动态内存分配

run() 方法详解

template<unsigned START, unsigned END>
void Scanner<START,END>::run(
    adf::input_buffer< cfloat, adf::extents<adf::inherited_extent>> & __restrict in,
    adf::output_buffer<cfloat, adf::extents<adf::inherited_extent>> & __restrict out
);

参数说明：

  • in / out：输入/输出缓冲区，使用 __restrict 修饰告诉编译器这两个指针不会别名，支持激进优化
  • adf::inherited_extent：缓冲区大小由 graph 定义中的 dimensions() 指定，而非内核代码

执行流程：

// Step 1: 如果是第一个 kernel，跳过 noiseSS（来自 DOA 的噪声子空间，Scanner 不使用）
if constexpr (START == 0) {
    for (unsigned i = 0; i < N; ++i) {
        noiseSS = *inIterator++;  // 读取但不使用
    }
}

// Step 2: 读取 Pm_null 空间谱并直通输出
for (unsigned i = 0; i < NUM_REGIONS; ++i) {
    Pm_null_cfloat[i] = *inIterator++;  // 缓存到本地
    *outIterator++ = Pm_null_cfloat[i]; // 直通给 Finder
}

// Step 3: 如果是第二个 kernel，读取第一个 kernel 生成的 tags
if constexpr (START != 0) {
    *outIterator++ = *inIterator++;  // 传递前 8 个 tags
    *outIterator++ = *inIterator++;  // 传递后 8 个 tags
}

// Step 4: 核心逻辑 - 阈值检测
const aie::vector<float,N> THRESHOLD_VECTOR = aie::broadcast<float,N>(DOA_MIN_THRESHOLD);
for (unsigned i = START; i < END; ++i) {
    // 提取实部转换为 float 向量
    for (unsigned j = 0; j < N; ++j) {
        Pm_null_float[j] = Pm_null_cfloat[i][j].real;
    }
    // 向量化比较：取 min(值, 阈值)
    const aie::vector<float,N> compare_result = aie::min(Pm_null_float, THRESHOLD_VECTOR);
    // 如果结果等于阈值，说明所有值都 ≥ 阈值（min 返回的是较小值）
    tag[i] = !(aie::equal(compare_result, THRESHOLD_VECTOR));
}

// Step 5: 输出 tags（打包为 cfloat 向量）
for (unsigned i = 0, index = START; i < 2; ++i) {
    for (unsigned j = 0; j < N; ++j) {
        tag_cfloat[j] = cfloat({float(tag[index++]), 0.0f});
    }
    *outIterator++ = tag_cfloat;
}

关键技巧 - 向量化阈值检测：

这里的逻辑利用了 AIE 向量指令的特性来实现高效的阈值判断。在 MUSIC 算法中，Pm_null 表示 null spectrum 的值，值越小表示越可能是信号峰值（因为 null spectrum 对应分母，值小意味着伪谱值大）。因此 tag=1 表示该区域存在小于阈值的点。

代码中的技巧是：

  • aie::min(vector, threshold) 对每个元素取最小值
  • 如果某元素 < threshold，则 min 返回该元素本身；否则返回 threshold
  • aie::equal(result, threshold) 检查每个位置是否等于阈值
  • 如果 result 中有任何位置不等于阈值，说明对应位置的原始值 < threshold
  • 因此 !(equal(...)) 为 true 时表示存在小于阈值的点

2. scanner_graph 类 (scanner_graph.h)

职责

scanner_graph 是 ADF（Adaptive DataFlow）图容器，负责：

  1. 实例化 Scanner 内核：创建两个 Scanner 实例，分别处理前半段和后半段区域
  2. 配置内核属性：设置运行时间比例、源代码文件
  3. 建立数据流连接：串联两个内核形成流水线
  4. 声明缓冲区维度：告知编译器每个端口的缓冲区大小

内核实例化

scanner_graph(void) {
    // 第一个 kernel：处理区域 [0, NUM_REGIONS/2)
    scanner_kernel[0] = adf::kernel::create_object<Scanner<0, NUM_REGIONS / 2>>();
    
    // 第二个 kernel：处理区域 [NUM_REGIONS/2, NUM_REGIONS)
    scanner_kernel[1] = adf::kernel::create_object<Scanner<NUM_REGIONS / 2, NUM_REGIONS>>();
    
    // 配置运行时比例（90% 的时间可以运行）
    for (unsigned i = 0; i < NUM_SCANNER_KERNELS; ++i) {
        adf::runtime<ratio>(scanner_kernel[i]) = 0.9;
        adf::source(scanner_kernel[i]) = "scanner.cpp";
    }
    
    // 连接成链：input → kernel0 → kernel1 → output
    adf::connect<>(sig_i, scanner_kernel[0].in[0]);
    adf::connect<>(scanner_kernel[0].out[0], scanner_kernel[1].in[0]);
    adf::connect<>(scanner_kernel[1].out[0], sig_o);
    
    // 声明缓冲区维度（单位：cfloat 元素个数）
    adf::dimensions(scanner_kernel[0].in[0])  = {N * N + NUM_POINTS};        // 320 = 64 + 256
    adf::dimensions(scanner_kernel[0].out[0]) = {NUM_POINTS + NUM_REGIONS/2}; // 264 = 256 + 8
    adf::dimensions(scanner_kernel[1].in[0])  = {NUM_POINTS + NUM_REGIONS/2}; // 264
    adf::dimensions(scanner_kernel[1].out[0]) = {NUM_POINTS + NUM_REGIONS};   // 288 = 256 + 32
}

维度计算说明：

  • N = 8，所以 N * N = 64（noise subspace 的大小）
  • NUM_POINTS = 256（空间谱点数）
  • NUM_REGIONS = 32，所以 NUM_REGIONS/2 = 16

Kernel 0 的输入包含 noise subspace（64 个 cfloat）和空间谱（256 个 cfloat），共 320 个。Kernel 0 的输出包含空间谱（256 个）和它计算的 tags（16 个，但打包成 2 个 vector，即 16 个 cfloat），共 272？实际上代码写的是 264，这是因为 tags 是以 vector 形式（每 8 个 tag 打包成一个 aie::vector<cfloat, 8>）输出的，所以 16 个 tag 占 16 个 cfloat。

等等，重新计算：NUM_REGIONS/2 = 16，这是 tag 的数量，每个 tag 是一个 cfloat，所以是 16 个 cfloat。加上 256 个 Pm_null，总共 272。但代码写的是 264... 这里可能有其他考虑。

实际上仔细看代码，scanner_kernel[0].out[0] 的维度是 {NUM_POINTS + NUM_REGIONS / 2} = 256 + 16 = 272，但注释写的是 264。这可能是注释错误，或者是我的理解有误。无论如何，实际运行时 ADF 会根据 dimensions 声明来分配缓冲区。

3. dut_graph 类 (scanner_app.cpp)

这是 Scanner 模块的独立测试封装，用于单独验证 Scanner 功能而不依赖完整的 MUSIC 流水线。

class dut_graph : public graph {
public:
    scanner_graph dut;
    input_plio    sig_i;   // PLIO: Programmable Logic I/O
    output_plio   sig_o;

    dut_graph(void) {
        // 创建 PLIO 端口，64-bit 宽度，连接到数据文件
        sig_i = input_plio::create("PLIO_i", plio_64_bits, "data/sig_i.txt");
        sig_o = output_plio::create("PLIO_o", plio_64_bits, "data/sig_o.txt");
        
        // 连接 PLIO 到 scanner_graph
        connect<>(sig_i.out[0], dut.sig_i);
        connect<>(dut.sig_o, sig_o.in[0]);
        
        // 物理布局约束：将 kernel 映射到特定 AIE tile
        static constexpr unsigned COL_START = 12;
        for (unsigned i = 0, col = COL_START; i < NUM_SCANNER_KERNELS; ++i, ++col) {
            adf::location<adf::kernel>(dut.scanner_kernel[i]) = adf::tile(col, ROW_0);
            adf::location<adf::stack >(dut.scanner_kernel[i]) = adf::location<adf::kernel>(dut.scanner_kernel[i]);
        }
    }
};

关键概念：

  • input_plio / output_plio：ADF 的可编程逻辑 IO 端口，用于 AIE 阵列与外部 PL（Programmable Logic）之间的数据传输
  • plio_64_bits：数据宽度为 64 位（即 2 个 cfloat，因为 cfloat = 32-bit float × 2）
  • 物理布局约束：adf::location 明确指定内核运行在哪个 AIE tile 上，这对时序和资源规划至关重要

设计决策与权衡

1. 为什么选择双核流水线？

决策：使用 2 个 AIE 核串行处理，每个核处理 16 个区域。

替代方案：

  • 单核处理全部 32 个区域
  • 更多核并行处理（如 4 核各处理 8 个区域）

权衡分析：

方案 延迟 吞吐量 资源占用 复杂度
单核 高 低 最低 最低
双核（当前） 中 中 中 中
四核 低 高 高 高

选择理由：

  1. 任务特性适合：Scanner 的计算量不大（主要是简单的阈值比较），双核已能满足时序
  2. 数据依赖：Kernel 1 需要 Kernel 0 生成的 tags（虽然代码中 Kernel 1 自己也会计算全部 tags，但仍需传递前一半），天然适合流水线
  3. 资源平衡：MUSIC 整体流水线中，Scanner 不是瓶颈，无需过度并行

2. 为什么使用模板参数指定区域范围？

决策：使用 template<unsigned START, unsigned END> 而非运行时参数。

好处：

  • 编译期确定循环边界，编译器可充分展开和优化
  • 避免运行时分支判断（if (START == 0) 等是 constexpr）
  • 生成独立的内核二进制，便于独立调优

代价：

  • 代码膨胀：每个实例生成独立的内核代码
  • 灵活性降低：无法运行时动态调整负载分配

适用性：在 AIE 编程中，内核实例通常是静态配置的，这种权衡是可接受的。

3. 为什么传递完整的 Pm_null 而不是只传标记结果？

观察：Scanner 将输入的 Pm_null 完整透传给输出，Finder 会再次使用这些数据。

为什么不只传 tags？

  1. Finder 需要上下文：Finder 不仅要看到标记，还需要原始谱值来进行精确的峰值定位和插值
  2. 流水线一致性：保持数据流格式统一，简化 buffer 管理
  3. AIE 内存带宽充足：相比计算，数据传输不是瓶颈

4. 阈值检测的向量化策略

关键代码：

const aie::vector<float,N> THRESHOLD_VECTOR = aie::broadcast<float,N>(DOA_MIN_THRESHOLD);
aie::vector<float,N> Pm_null_float;
for (unsigned j = 0; j < N; ++j) {
    const cfloat tmp = Pm_null_cfloat[i][j];
    Pm_null_float[j] = tmp.real;
}
const aie::vector<float,N> compare_result = aie::min(Pm_null_float, THRESHOLD_VECTOR);
tag[i] = !(aie::equal(compare_result, THRESHOLD_VECTOR));

为什么这样写？

  • 提取实部：Pm_null 是复数，但阈值比较只看幅度（这里简化为实部）
  • aie::broadcast：创建常量向量，利用 SIMD 并行比较
  • min + equal 技巧：避免显式的逐元素比较和归约，充分利用 AIE 的向量 ALU

注意：这里的逻辑是"是否存在小于阈值的点"，因为 MUSIC 的 null spectrum 越小表示越可能是峰值。

边缘情况与注意事项

1. 阈值选择的敏感性

DOA_MIN_THRESHOLD = 0.25f 是一个经验值：

  • 太高：会漏掉弱信号峰值
  • 太低：会产生过多 false positive，增加 Finder 负担

这个值需要根据实际信噪比环境调优。

2. 边界区域处理

注意 Finder 会忽略第一个和最后一个区域，因为它们无法构成完整的"左邻-当前-右邻"三元组用于峰值检测。这意味着：

  • 如果真实峰值恰好在谱的边缘（角度 -90° 或 +90°），可能会被漏检
  • 这是 MUSIC 算法的固有限制，通常通过扩展扫描范围来缓解

3. 数据对齐要求

alignas(32) 是必须的：

  • AIE 的向量加载指令（如 vload）要求 32 字节对齐
  • 未对齐访问会导致未定义行为或性能急剧下降
  • 在 AIE 编程中，始终对向量数据进行 32 字节对齐

4. chess_prepare_for_pipelining 和 chess_flatten_loop

这些是 Xilinx AIE 编译器的 pragma：

  • chess_prepare_for_pipelining：提示编译器尝试软件流水线优化
  • chess_flatten_loop：提示编译器展开循环以减少循环开销

它们影响生成的汇编代码质量，但对功能正确性无影响。

5. 多核协作的数据同步

Kernel 0 和 Kernel 1 之间通过 ADF 的 streaming 接口连接：

adf::connect<>(scanner_kernel[0].out[0], scanner_kernel[1].in[0]);

ADF 运行时保证：

  • 数据按 FIFO 顺序传递
  • 阻塞语义：如果 Kernel 1 的输入缓冲区未满而 Kernel 0 未完成写入，Kernel 1 会等待

开发者无需手动同步，但需要注意缓冲区尺寸匹配。
