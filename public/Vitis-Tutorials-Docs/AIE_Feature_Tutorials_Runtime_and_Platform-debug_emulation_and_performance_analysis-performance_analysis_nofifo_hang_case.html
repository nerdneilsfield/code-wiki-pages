performance_analysis_nofifo_hang_case 模块深度解析

一句话概括

这是一个故意设计为有缺陷的 AIE-PL 系统案例，用于演示当 AIE 核间通信缺少 FIFO 缓冲时，系统如何在硬件仿真中发生死锁（hang）。它像一面"反面的镜子"——通过展示错误的设计模式，教会开发者理解流式数据通路中背压（back-pressure）和缓冲深度的关键作用。

问题空间：为什么这个模块存在？

背景：AIE 编程中的常见陷阱

在 Versal ACAP 架构中，AI Engine（AIE）阵列与可编程逻辑（PL）之间的数据流动遵循流式语义。AIE 核之间通过两种机制通信：

  1. Stream 通道：类似 FIFO 的流式接口，支持阻塞读写
  2. Buffer 连接：基于共享内存的异步数据传输

一个常见的性能优化手段是在 Stream 连接上配置 fifo_depth 来添加缓冲。但许多初学者会忽略这一点，或者为了节省 BRAM 资源而刻意省略 FIFO。

本模块的核心目的

performance_analysis_nofifo_hang_case 是一个教学用例，专门展示以下场景：

当两个 AIE 核通过 Stream 通道直接相连（无 FIFO 缓冲），且它们的生产/消费速率不匹配时，系统会发生死锁。

这个案例属于 13-aie-performance-analysis 教程的一部分，与另外两个案例形成对比：

  • performance_analysis_dmafifo_optimized_case：使用 DMA FIFO 优化的正确实现
  • performance_analysis_ssfifo_case：使用单速率 FIFO 的案例

心智模型：想象一条没有缓冲区的生产线

想象一条汽车装配线：

  • 工位 A（aie_dest1）：负责安装引擎，每 10 分钟完成一辆
  • 工位 B（aie_dest2）：负责安装轮胎，每 8 分钟完成一辆
  • 传送带（net1）：连接两个工位

如果传送带没有任何缓冲区（即只能容纳一辆车），会发生什么？

  • 当工位 A 完成一辆车，必须立即推给工位 B
  • 但如果工位 B 还在处理上一辆车，工位 A 就必须等待
  • 同时，工位 A 的上游（PL 侧的 mm2s）还在不断送车过来
  • 最终，整个生产线因为相互等待而完全停滞——这就是死锁

这就是本案例的设计：故意移除 FIFO 缓冲，让生产者和消费者直接耦合，演示速率不匹配导致的死锁。

关键设计特征

组件 角色 关键特性
mm2s PL 数据源 从 DDR 读取数据，通过 AXI4-Stream 发送给 AIE
s2mm PL 数据汇 从 AIE 接收数据，写回 DDR
aie_dest1 AIE 计算核 0 输入 buffer → 输出 stream + buffer（分叉）
aie_dest2 AIE 计算核 1 输入 stream + buffer → 输出 stream（汇合）
net1 核间 Stream 连接 故意不设置 FIFO 深度，导致潜在死锁
net2 核间 Buffer 连接 并行数据路径，使用 buffer 语义

数据流详解

1. 初始化阶段（Host）

// host.cpp
auto device = xrt::device(0);
auto uuid = device.load_xclbin(xclbinFilename);

// 创建 kernel 句柄
auto s2mm = xrt::kernel(device, uuid, "s2mm");
auto mm2s = xrt::kernel(device, uuid, "mm2s");

// 分配 buffer
auto out_bo = xrt::bo(device, output_size_in_bytes, s2mm.group_id(0));
auto in_bo = xrt::bo(device, output_size_in_bytes, mm2s.group_id(0));

Host 代码使用 XRT（Xilinx Runtime） API 加载 xclbin，创建 kernel 对象，并分配设备内存缓冲区。

2. 启动性能分析探针

xrt::aie::profiling handle(device), handle2(device);
handle.start(xrt::aie::profiling::profiling_option::io_stream_running_event_count, 
             "gr.dataout", "", 0);
handle2.start(xrt::aie::profiling::profiling_option::io_stream_running_event_count, 
              "gr.in", "", 0);

这里设置了 AIE 事件计数器，监控 gr.dataout（输出 PLIO）和 gr.in（输入 PLIO）上的数据流动。这是诊断死锁的关键工具——如果事件计数停止增长，说明数据流已停滞。

3. 启动数据通路

auto s2mm_run = s2mm(out_bo, nullptr, OUTPUT_SIZE);  // 先启动 sink
auto mm2s_run = mm2s(in_bo, nullptr, OUTPUT_SIZE);   // 再启动 source
auto ghdl = xrt::graph(device, uuid, "gr");
ghdl.run(4);  // 运行 4 次迭代
ghdl.end(5000);  // 最多等待 5000 AIE 周期

注意启动顺序：先启动 s2mm（消费者），再启动 mm2s（生产者）。这是良好的实践，确保下游准备好接收数据。

4. AIE 图内部数据流

// graph.h - 数据连接拓扑
connect< >net0(in.out[0], k[0].in[0]);           // PLIO → aie_dest1 (buffer)
connect< stream >net1(k[0].out[0], k[1].in[0]);  // aie_dest1 → aie_dest2 (stream, NO FIFO!)
connect< >net2(k[0].out[1], k[1].in[1]);         // aie_dest1 → aie_dest2 (buffer)
connect< stream >net3(k[1].out[0], dataout.in[0]); // aie_dest2 → PLIO (stream)
// fifo_depth(net1)=32;  // ← 这行被注释掉了！

关键观察：fifo_depth(net1)=32; 被注释掉了。这意味着 net1 是一个无缓冲的 stream 连接。

5. AIE 核的计算逻辑

aie_dest1（数据分叉）：

void aie_dest1(input_buffer<int32,extents<32>> &in, 
               output_stream<int32> *out, output_buffer<int32,extents<32>> &outm) {
    for(int i=0; i<8; i++) {
        tmp = *inIter++;
        writeincr(out, tmp);    // 写入 stream (net1)
        *outmIter++ = tmp;       // 写入 buffer (net2)
    }
}

aie_dest2（数据汇合）：

void aie_dest2(input_stream<int32> *in, input_buffer<int32,extents<32>> &inm, 
               output_stream<int32> *outm) {
    for(int i=0; i<8; i++) {
        tmp = readincr_v<4>(in);   // 从 stream 读取 (net1)
        tmp2 = *inmIter++;          // 从 buffer 读取 (net2)
        writeincr(outm, tmp + tmp2); // 输出到 PL
    }
}

死锁机制分析

为什么会死锁？

想象以下执行时序：

  1. aie_dest1 开始执行，从输入 buffer 读取 4 个整数（vector<4>）
  2. 它尝试通过 writeincr(out, tmp) 写入 net1
  3. 如果 aie_dest2 还没有启动或暂时落后，net1 的 stream 会阻塞
  4. 由于没有 FIFO 缓冲，writeincr 必须等待 aie_dest2 的 readincr
  5. 同时，aie_dest1 还持有 net2 buffer 的写入权
  6. 如果 aie_dest2 因为某种原因（如等待 net2 的数据）也阻塞了...
  7. 双方互相等待，形成循环依赖 → 死锁

关键代码证据

在 graph.h 中，第 38 行的注释明确说明了设计意图：

//fifo_depth(net1)=32;

这一行被注释掉，意味着：

  • net1 使用默认的 stream 连接（无显式 FIFO）
  • 生产者（aie_dest1）和消费者（aie_dest2）直接耦合
  • 任何速率不匹配都会导致阻塞传播

Host 代码的预期行为

有趣的是，host.cpp 中的测试逻辑暗示了预期的失败：

std::cout << "TEST " << (match ? "PASSED expected by timeout" : "FAILED") << "\n";
return (match ? EXIT_SUCCESS : EXIT_FAILURE);

注释 "PASSED expected by timeout" 表明：这个测试用例预期会因为超时而"通过"——这正是死锁的表现。

设计权衡与决策

1. 有 FIFO vs 无 FIFO

方案 优点 缺点 本案例选择
无 FIFO（当前） 节省 BRAM 资源；延迟最低 极易死锁；对时序要求苛刻 ✅ 故意选择（教学目的）
小 FIFO（如 16） 平衡资源与鲁棒性 可能仍不足应对突发流量 ❌ 未采用
大 FIFO（如 128） 高容错性；解耦 producer/consumer 消耗更多 BRAM；延迟增加 ❌ 未采用

教学意图：通过展示"错误"的设计，让开发者理解 FIFO 的必要性。

2. Buffer + Stream 混合拓扑

本案例使用了复杂的连接模式：

  • net0：buffer 连接（PLIO → aie_dest1）
  • net1：stream 连接，无 FIFO（aie_dest1 → aie_dest2）⚠️
  • net2：buffer 连接（aie_dest1 → aie_dest2）
  • net3：stream 连接（aie_dest2 → PLIO）

这种设计故意制造不对称：aie_dest1 通过两条路径向 aie_dest2 发送数据，一条是 stream（同步），一条是 buffer（异步）。这种复杂性增加了死锁的可能性。

3. 运行时比例配置

runtime<ratio>(k[0]) = 0.8;
runtime<ratio>(k[1]) = 0.8;

两个核都配置了 80% 的运行时比例，意味着它们各自占用一个 AIE tile 的 80% 计算资源。这在理论上应该足够，但由于缺乏 FIFO 缓冲，调度器的微小波动就可能触发死锁。

新贡献者必读：陷阱与注意事项

🚨 关键警告

  1. 不要将此设计复制到生产代码中

    • 这是一个故意损坏的教学示例
    • 实际项目中应始终为 stream 连接配置适当的 fifo_depth

  2. 理解 "hang" 的表现

    • 硬件仿真（hw_emu）会卡住不动
    • 事件计数器（profiling handles）停止增长
    • 可能需要强制终止仿真进程

  3. 调试技巧

    // 在 host.cpp 中检查事件计数
long long data_out_count = handle.read();
long long data_in_count = handle2.read();
std::cout << "Output data received:" << data_out_count << std::endl;
std::cout << "Input data sent:" << data_in_count << std::endl;

    如果输出显示计数远低于预期（如 2048），说明发生了数据丢失或死锁。

🔧 修复建议

要修复此设计，只需取消注释 graph.h 中的 FIFO 配置：

// 修改前
//fifo_depth(net1)=32;

// 修改后
fifo_depth(net1)=32;  // 为 stream 连接添加 32 深度的 FIFO

或者参考相邻的优化案例：

  • performance_analysis_dmafifo_optimized_case
  • performance_analysis_ssfifo_case

总结

performance_analysis_nofifo_hang_case 是一个以反面教材形式存在的教学模块。它的价值不在于代码本身，而在于它所演示的设计陷阱：

在 AIE 流式数据通路中，省略 FIFO 缓冲会导致生产者-消费者死锁。

对于新加入团队的工程师，理解这个案例有助于：

  1. 识别 hw_emu 中"hang"的根本原因
  2. 掌握使用 XRT profiling API 诊断数据流问题的方法
  3. 建立正确的直觉：何时需要 FIFO，以及配置多大深度

记住：当你看到 fifo_depth() 被注释掉时，就应该警惕潜在的死锁风险。
