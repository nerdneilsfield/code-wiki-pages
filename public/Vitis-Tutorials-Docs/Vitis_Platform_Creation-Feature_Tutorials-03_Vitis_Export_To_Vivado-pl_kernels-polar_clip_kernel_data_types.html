polar_clip_kernel_data_types 模块深度解析

一句话概括

polar_clip 是一个用于 Versal ACAP 平台的 HLS PL Kernel，它在 AIE（AI Engine）和 PL（Programmable Logic）之间充当一个智能信号门控器——接收来自 AIE 的复数数据流，计算每个样本的幅度，将超过阈值的信号进行裁剪后返回给 AIE。你可以把它想象成一个音频系统中的"限幅器"：当输入信号太强时，它不是简单地削波，而是计算出超出阈值的部分并只输出这部分差值。

问题空间与设计动机

为什么需要这个模块？

在无线通信或雷达信号处理系统中，经常需要对信号进行峰值因子降低（CFR, Crest Factor Reduction）。CFR 的核心思想是检测信号幅度是否超过某个阈值，如果超过，则生成一个补偿信号来削减峰值。

polar_clip 正是实现这一功能的关键组件。它位于 AIE 数据处理流水线的中间位置：

  1. 上游：AIE 的 classifier kernel 对信号进行初步分类/处理
  2. 本模块：接收分类后的复数信号，计算其极坐标表示（幅度 + 相位），输出超过阈值的部分
  3. 下游：AIE 的 interpolator kernel 将裁剪后的信号进行插值处理

为什么不用软件实现？

CFR 是一个对延迟敏感的操作，需要在数据流经过时实时处理。HLS 实现的 PL Kernel 可以提供：

  • 确定性延迟：每个样本的处理周期是固定的（II=1）
  • 高吞吐量：流水线化设计可以每时钟周期处理一个样本
  • 与 AIE 的无缝集成：通过 AXI4-Stream 接口直接与 AIE 的 PLIO 连接

为什么选择 CORDIC 算法？

计算复数的幅度和相位通常需要平方根和反正切运算，这在硬件中代价高昂。CORDIC（COordinate Rotation DIgital Computer）算法通过一系列移位和加法操作逼近这些三角函数，避免了复杂的乘法器和查找表。

核心抽象与心智模型

数据类型抽象：complex_types<ELEMENT>

template <typename ELEMENT> struct complex_types {
  ELEMENT real;
  ELEMENT imag;
};

这是一个简单的泛型复数结构体，允许使用不同的底层数值类型。在本模块中，它被实例化为：

typedef complex_types<ap_int<16>> ap_cint16;  // 16位有符号整数复数

设计意图：使用模板可以在仿真时使用标准 C++ 类型（如 int），在综合时使用 HLS 特定的任意精度类型（ap_int<16>），实现同一份代码的双重用途。

数据流模型：AXI4-Stream

void polar_clip(
    hls::stream<ap_axis<32, 0, 0, 0>>& in_sample,
    hls::stream<ap_axis<32, 0, 0, 0>>& out_sample
)

想象数据流就像一条传送带：

  • in_sample 是从上游 AIE 送来的包裹队列
  • out_sample 是处理后送往下游 AIE 的包裹队列
  • 每个 "包裹"（ap_axis<32, 0, 0, 0>）包含 32 位数据，没有额外的 sideband 信号（USER=0, ID=0, DEST=0）

极坐标转换的心智模型

对于复数 \(z = I + jQ\)：

  • 幅度：\(|z| = \sqrt{I^2 + Q^2}\)
  • 相位：\(\theta = \text{atan2}(Q, I)\)

但直接计算平方根和反正切在硬件中很慢。CORDIC 算法的直觉是：通过一系列预定义的旋转，逐步将向量对齐到 x 轴，此时 x 坐标就是幅度。

组件深度解析

1. complex_types<ELEMENT> 模板结构体

template <typename ELEMENT> struct complex_types {
  ELEMENT real;
  ELEMENT imag;
};

设计意图：

  • 类型擦除与特化的平衡：模板参数允许在仿真和综合时使用不同的数值类型
  • 内存布局保证：C++ 标准保证 POD 类型的成员按声明顺序布局，这对 HLS 综合很重要
  • 无构造函数/析构函数：遵循 Rule of Zero，确保可以被安全地 memcpy 和位操作

实际使用：

typedef complex_types<ap_int<16>> ap_cint16;
// 等价于 struct { ap_int<16> real; ap_int<16> imag; };
// 总位宽：32 位，正好匹配 AXIS 数据宽度

2. cos_sin_mag 函数 —— CORDIC 核心

这是整个模块最复杂的部分，实现了定点 CORDIC 算法。

关键常量解析

const ap_int32 P_SCALE = 512;        // 2^9，用于归一化
const ap_int32 SHIFT_BITS = 9;       // 右移 9 位 = 除以 512
const ap_int32 P_fixed = 311;        // 近似于 512/√2 ≈ 362，但这里是经验值
const ap_int32 nsteps = 6;           // CORDIC 迭代次数

为什么是 6 次迭代？

  • 每次迭代将角度误差减少约一半
  • 6 次迭代后，角度精度约为 \(2^{-6} \approx 1.5°\)，对于 CFR 应用足够
  • 更多迭代会增加延迟和资源，收益递减

LUT（查找表）的设计

const ap_int32 rotation_cos_lut[64] = {...};
const ap_int32 rotation_sin_lut[64] = {...};
  • 64 个条目：对应 6 次迭代的所有可能旋转模式（\(2^6 = 64\)）
  • 预计算值：在编译期计算好的 cos/sin 值，避免运行时计算
  • 定点表示：这些值已经是定点格式，可以直接参与运算

象限处理逻辑

sign_I = (x_I >> 16) & 0x1;  // 提取符号位
sign_Q = (x_Q >> 16) & 0x1;

if (sign_I > 0) abs_I = ~(x_I) + 1;  // 补码取绝对值
else            abs_I = x_I;

CORDIC 算法只在第一象限工作，因此需要将输入映射到第一象限，记录原始象限信息用于后续恢复。

迭代核心

for(ap_int32 i = 1; i < nsteps; i++) {
    if (Y_step >= 0) {
        rotation_index += (1 << i);
        next_X_step = X_step + (Y_step >> i);
        next_Y_step = Y_step - (X_step >> i);
    } else {
        next_X_step = X_step - (Y_step >> i);
        next_Y_step = Y_step + (X_step >> i);
    }
    X_step = next_X_step;
    Y_step = next_Y_step;
}

这是 CORDIC 的经典形式：

  • 根据 Y 坐标的符号决定旋转方向
  • 每次旋转角度为 \(\arctan(2^{-i})\)
  • 使用移位代替乘法，节省资源

幅度计算

ap_int32 mag = X_step * P_fixed;
*magout = mag >> SHIFT_BITS;

CORDIC 完成后，X_step 近似等于 \(|I| + |Q|\)（L1 范数），需要通过乘以 \(P_{fixed}\) 并移位来校正为 L2 范数（欧几里得距离）。

3. polar_clip 顶层函数

接口定义

void polar_clip(
    hls::stream<ap_axis<32, 0, 0, 0>>& in_sample,
    hls::stream<ap_axis<32, 0, 0, 0>>& out_sample
)

pragma 指令解析：

#pragma HLS INTERFACE ap_ctrl_none port=return    // 无控制接口，自由运行
#pragma HLS INTERFACE axis port=out_sample        // AXI4-Stream 接口
#pragma HLS INTERFACE axis port=in_sample         // AXI4-Stream 接口

ap_ctrl_none 意味着这个 kernel 一旦启动就会持续运行，不需要主机通过 ap_start/ap_done 信号控制。这对于流式数据处理很常见。

数据解包

ap_axis<32, 0, 0, 0> in_x = in_sample.read();
short value_real = short(in_x.data & 0xFFFF);           // 低 16 位
short value_imag = short((in_x.data >> 16) & 0xFFFF);   // 高 16 位

注意：这里假设数据的 little-endian 布局——实部在低 16 位，虚部在高 16 位。这与 AIE 端的 cint16 类型布局一致。

阈值判断与裁剪

#define CFR_THRESHOLD (11626)

if (mag_sq > CFR_THRESHOLD * CFR_THRESHOLD) {
    res_real = (ap_int32)cs_fixed_real * (magout - CFR_THRESHOLD);
    res_imag = (ap_int32)cs_fixed_imag * (magout - CFR_THRESHOLD);
    ovalue.real = (short)(res_real & 0xFFFF);
    ovalue.imag = (short)(res_imag & 0xFFFF);
} else {
    ovalue.real = 0;
    ovalue.imag = 0;
}

关键设计决策：

  • 比较的是 mag_sq（幅度平方）而不是 mag，避免开方运算
  • 阈值 11626 是经验值，对应大约 -3dBFS 的满量程电平
  • 输出的是 (mag - THRESHOLD) 乘以单位向量，即超出阈值的部分
  • 如果未超阈值，输出零向量（不是原信号！）

数据打包

out_x.data = (((short)ovalue.real & 0x0000FFFF) << 0) | 
             (((short)ovalue.imag & 0x0000FFFF) << 16);
out_sample.write(out_x);

设计权衡与决策分析

1. 定点 vs 浮点

选择：纯定点实现（ap_int<16>, ap_int<32>）

理由：

  • FPGA DSP slice 对定点运算优化更好
  • 与 AIE 的数据类型保持一致（AIE 主要使用 int16/int32）
  • 避免浮点带来的面积和延迟开销

代价：

  • 需要手动管理小数点位置（scaling）
  • 动态范围受限，需要仔细选择位宽

2. CORDIC vs 直接计算

选择：CORDIC 算法

理由：

  • 只需要移位、加法和一个小 LUT
  • 面积效率高，适合并行实现
  • 延迟可预测（固定 6 个周期）

替代方案：

  • 使用 HLS 的 sqrt() 和 atan2() —— 会消耗大量 DSP 和 BRAM
  • 使用更大的 LUT 直接查表 —— 精度与面积的权衡

3. 输出零 vs 输出原信号

选择：未超阈值时输出 (0, 0)

理由：

  • 这是 CFR 的典型行为：只输出需要削减的峰值分量
  • 下游的 interpolator 会将此与原始信号结合

风险：如果误解这一点，会认为信号被错误地清零了。

4. 单样本处理 vs 块处理

选择：单样本流式处理（II=1）

理由：

  • 最小化延迟
  • 与 AIE 的流式数据模型匹配

代价：

  • 无法利用数据重用（如 FFT 的蝶形运算）
  • 每个样本都需重新计算 CORDIC

5. ap_ctrl_none vs 标准控制接口

选择：ap_ctrl_none

理由：

  • 这是一个"永远在线"的数据处理管道
  • 不需要主机干预即可持续处理流数据

代价：

  • 无法通过标准 XRT API 启动/停止单个 kernel
  • 需要通过 AIE graph 的控制来间接控制数据流

新贡献者必读：陷阱与注意事项

🔴 严重：数据格式不匹配

问题：如果 AIE 端和 PL 端的复数打包格式不一致，会导致实部/虚部互换或符号错误。

当前约定：

AXIS 数据 [31:16] = 虚部（Imaginary）
AXIS 数据 [15:0]  = 实部（Real）

验证方法：检查 aie/kernels/polar_clip.cpp 和本文件的位操作是否一致。

🔴 严重：阈值常量的含义

#define CFR_THRESHOLD (11626)

这个值是定点表示的。对于 16 位有符号整数，满量程是 32767。11626 约等于 \(32767 / \sqrt{8}\)，对应 -9dBFS 左右。

不要直接修改这个值而不理解定点缩放关系。

🟡 警告：CORDIC 的精度限制

6 次迭代的 CORDIC 有约 1-2 度的角度误差。如果你的应用需要更高精度：

  • 增加 nsteps（需要重新生成 LUT）
  • 或者考虑使用 DSP-based 的直接计算

🟡 警告：溢出风险

value_real_sq = value_real * value_real;  // 16-bit * 16-bit = 32-bit

这里依赖 ap_int<32> 来容纳 32 位结果。如果 value_real 是标准 short，会发生 16 位溢出！

确保所有变量都是 ap_int 类型。

🟢 提示：仿真 vs 综合的行为差异

代码中有一些被注释掉的 printf 语句：

// printf("-[KERNEL]: Inside the kernel input  real and imag respectively  %d %d \n",...)

在 HLS 仿真时可以取消注释用于调试，但在综合前必须注释掉——printf 在硬件中没有意义。

🟢 提示：LUT 的生成

如果你修改了 nsteps，现有的 64 条目 LUT 将不再适用。你需要重新计算 LUT 值：

import math
N = 6  # nsteps
for i in range(2**N):
    angle = sum(((i >> j) & 1) * math.atan(2**(-j-1)) for j in range(N))
    cos_val = int(math.cos(angle) * 32768)  # 根据你的缩放调整
    sin_val = int(math.sin(angle) * 32768)
    print(f"{cos_val},{sin_val}")

扩展与修改指南

如何修改阈值？

编辑 #define CFR_THRESHOLD 行。注意保持与 AIE 端的一致性。

如何支持不同的位宽？

  1. 修改 typedef complex_types<ap_int<16>> ap_cint16; 中的位宽
  2. 调整 AXIS 数据宽度（ap_axis<32, ...> → ap_axis<64, ...>）
  3. 重新计算 CORDIC 的 scaling 常数

如何添加更多的 sideband 信号？

修改 ap_axis<32, 0, 0, 0> 的参数：

  • 第 2 个参数：USER 位宽
  • 第 3 个参数：ID 位宽
  • 第 4 个参数：DEST 位宽

例如，ap_axis<32, 4, 0, 0> 添加 4 位 USER 信号用于传输元数据。
