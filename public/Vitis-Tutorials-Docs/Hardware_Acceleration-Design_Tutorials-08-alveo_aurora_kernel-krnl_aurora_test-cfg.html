Alveo Aurora Kernel Stream Configuration

概述：这不是代码，是硬件的"接线图"

当你打开 krnl_aurora_test.cfg 这个文件时，第一眼可能会困惑——这里没有 if 语句，没有循环，没有算法逻辑。相反，你看到的是一个充满神秘符号的配置文件：nk=、stream_connect=、connect=。这不是软件代码，而是Xilinx Vitis 平台的硬件连接配置——本质上，它是 Alveo 加速卡的"接线图"。

这个模块解决的核心问题是：如何在 Alveo 卡上的 Aurora 高速串行收发器与测试数据生成/捕获逻辑之间建立正确的数据通路？ Aurora 是 Xilinx 的高速串行通信协议，利用 FPGA 的 GT（Gigabit Transceiver）收发器实现每秒数十 Gbps 的数据传输。这个配置文件定义了三个关键内核（kernel）的实例化方式、它们之间的 AXI 流连接，以及 Aurora 内核与物理 QSFP 接口的时钟和信号绑定。

理解这个文件的关键在于认识到它的声明式本质：你不是在描述"怎么做"，而是在描述"是什么"——这些内核存在，这些连接存在，这些物理引脚绑定存在。Vitis 编译器会读取这份"接线图"，生成对应的硬件比特流。

一句话概括

这是一个 Alveo 加速卡 Aurora 高速串行通信测试的系统连接蓝图——它像剧场的灯光控制台，把逻辑的数据源/检测器内核（演员）通过精确的"线缆"连接到物理 QSFP 光口（舞台设备），定义了数据流如何在 FPGA 可编程逻辑与片外高速收发器之间流动。

1. 问题域：为什么需要这个配置？

1.1 背景：Aurora 是什么？

Aurora 是 Xilinx 的高速串行通信协议，基于 GT（Gigabit Transceiver）收发器，通常运行在 10Gbps、25Gbps 甚至更高。它是 Alveo 数据中心加速卡与外部设备（网络交换机、传感器阵列、其他 FPGA）通信的"高速公路"。

1.2 核心挑战：软硬件协同的鸿沟

在 Alveo 上实现 Aurora 通信，开发者面临三个层次的复杂性：

┌─────────────────────────────────────────────────────────────────┐
│  Layer 3: 软件应用层                                            │
│  - 主机程序 (C++/Python)                                        │
│  - 调度数据传输、配置寄存器                                       │
├─────────────────────────────────────────────────────────────────┤
│  Layer 2: 内核逻辑层 (Vitis Kernels)                            │
│  - strm_issue: 生成测试数据流                                   │
│  - krnl_aurora: Aurora 协议处理 (帧封装/解封装)                 │
│  - strm_dump: 接收并校验数据                                    │
├─────────────────────────────────────────────────────────────────┤
│  Layer 1: 物理接口层 (GT/QSFP)                                  │
│  - GT 收发器 (MGT)                                              │
│  - QSFP 光模块                                                  │
│  - 参考时钟 (Refclk)                                            │
└─────────────────────────────────────────────────────────────────┘

关键问题：Layer 2 中的内核实例（如 strm_issue_0、krnl_aurora_0）是逻辑实体，而 Layer 1 的 GT 端口和时钟是物理实体。如何让 Vitis 编译器知道：

  1. strm_issue_0 的数据输出要连接到 krnl_aurora_0 的发送端口？
  2. krnl_aurora_0 要使用哪个 GT 收发器和参考时钟？
  3. 不同 Alveo 卡（U200、U250、U50 等）的引脚映射不同，如何处理？

1.3 本模块的答案

krnl_aurora_test.cfg 就是解决上述问题的系统连接配置——它是 Vitis 链接阶段的输入，告诉 v++ --link 命令：

  • 实例化哪些内核（nk=）
  • 内核之间的 AXI Stream 如何连接（stream_connect=）
  • 内核如何连接到物理引脚（connect=）

2.2 组件角色详解

组件 类型 角色描述
strm_issue_0 数据生成内核 产生测试数据流，作为 Aurora 发送通道的数据源。类似"测试模式生成器"。
krnl_aurora_0 协议处理内核 Aurora 协议的封装/解封装。将 AXI Stream 数据转换为 GT 收发器所需的串行流，处理帧边界、流量控制、错误检测。这是连接逻辑内核与物理收发器的桥梁。
strm_dump_0 数据检测内核 接收从 Aurora 接收通道回来的数据，进行校验（如 CRC 比对、序列号检查），验证传输完整性。

2.3 数据流路径详解

发送路径（TX Path）：

strm_issue_0.data_output 
    ↓ [AXI4-Stream, via stream_connect]
krnl_aurora_0.tx_axis
    ↓ [帧封装：加 SOF/EOF、CRC]
Aurora TX Lane
    ↓ [8b/10b 或 64b/66b 编码]
GT Transceiver
    ↓ [串行化]
QSFP Port → 光纤 → 对端设备

接收路径（RX Path）：

对端设备 → 光纤 → QSFP Port
    ↓ [反串行化]
GT Transceiver
    ↓ [解码]
Aurora RX Lane
    ↓ [帧解封装：验 CRC、去 SOF/EOF]
krnl_aurora_0.rx_axis
    ↓ [AXI4-Stream, via stream_connect]
strm_dump_0.data_input
    ↓ [数据校验]
验证结果 → Host

3. 配置深度解析

3.1 内核实例化（nk=）

nk=krnl_aurora:1:krnl_aurora_0
nk=strm_issue:1:strm_issue_0
nk=strm_dump:1:strm_dump_0

语法：nk=<kernel_name>:<num_instances>:<instance_name0>:<instance_name1>:...

设计意图：

  • 每个 nk 行将一个 RTL/Vitis 内核声明为系统中可用的组件
  • 冒号分隔的三个字段分别是：内核类型名（必须匹配 .xo 文件名）、实例数量、实例化后的命名前缀
  • 这里每个内核都只有 1 个实例，命名采用 <kernel>_<index> 的惯例

3.2 AXI Stream 连接（stream_connect=）

stream_connect=krnl_aurora_0.rx_axis:strm_dump_0.data_input
stream_connect=strm_issue_0.data_output:krnl_aurora_0.tx_axis

语法：stream_connect=<source_instance>.<port>:<dest_instance>.<port>

数据流向分析：

连接 方向 含义
strm_issue_0.data_output → krnl_aurora_0.tx_axis 发送 测试数据从生成器流入 Aurora 发送通道
krnl_aurora_0.rx_axis → strm_dump_0.data_input 接收 Aurora 接收到的数据流出到检测器进行验证

关键洞察：这两行配置建立了**回环测试（Loopback Test）**的拓扑——数据从 strm_issue 发出，经 Aurora 发送，通常在外部光纤环回（或对接另一设备）后，从 Aurora 接收端回到 strm_dump 进行校验。

3.3 物理引脚连接（connect=）

# uncomment following lines for xilinx_u200_gen3x16_xdma_2_202110_1
connect=io_clk_qsfp_refclkb_00:krnl_aurora_0/gt_refclk
connect=krnl_aurora_0/gt_port:io_gt_qsfp_00
connect=krnl_aurora_0/init_clk:ii_level0_wire/ulp_m_aclk_freerun_ref_00

语法：connect=<source_pin>:<dest_pin> 或 connect=<kernel>/<pin>:<board_interface>

引脚功能解析：

连接 类型 作用
io_clk_qsfp_refclkb_00 → gt_refclk 参考时钟 QSFP 模块的差分参考时钟（通常 156.25MHz 或 161.1328125MHz），是 GT 收发器 CDR（时钟数据恢复）的基准
gt_port → io_gt_qsfp_00 高速收发器 GT 收发器的物理引脚映射到 QSFP 连接器（4 对差分信号：TX+/-, RX+/-），实现串行数据的发送/接收
init_clk → ulp_m_aclk_freerun_ref_00 初始化时钟 Aurora IP 初始化/配置逻辑的独立时钟，确保在 GT 参考时钟稳定前，Aurora 的状态机仍能工作

关键设计决策：注释中显示多卡支持的策略——通过注释/取消注释不同代码块来适配 U200、U250、U50、U55c、U280 等不同 Alveo 卡。这体现了**"编译期板卡选择"**的权衡：

  • ✅ 优势：每个配置针对特定板卡优化，时钟频率、引脚约束精确匹配
  • ❌ 代价：维护多份配置，切换板卡需要重新链接（re-link）

4. 设计权衡与决策

4.1 抽象层级选择：为什么用 .cfg 而非 C++？

备选方案 权衡分析
方案A（本模块采用）：Vitis 配置语法（.cfg） ✅ 声明式、简洁、工具链原生支持
✅ 链接期捕获连接错误（如端口类型不匹配）
❌ 学习曲线（特定语法）、调试信息有限
方案B：C++ 运行时动态配置 ✅ 灵活性高，可运行时重配置
❌ 复杂性高，需管理内存和资源生命周期
❌ 错误可能延迟到运行时
方案C：JSON/YAML 配置 ✅ 通用格式，易编辑
❌ 需额外解析层，Xilinx 工具链不原生支持

决策理由：Vitis 工具链（v++ --link）原生解析 .cfg 格式，能在链接阶段验证连接合法性（如检查 AXI Stream 的位宽匹配），这比运行时错误更易调试。

4.2 拓扑选择：回环测试 vs 点对点通信

本配置建立了经典的回环测试拓扑：

┌─────────────┐     TX     ┌──────────┐     物理环回     ┌──────────┐     RX     ┌─────────────┐
│ strm_issue  │ ─────────→ │ Aurora   │ ────────────────→ │ Aurora   │ ─────────→ │ strm_dump   │
│  (数据源)    │            │ TX Lane  │    (光纤短接或     │ RX Lane  │            │  (数据校验器) │
└─────────────┘            └──────────┘     对端设备)      └──────────┘            └─────────────┘

为什么选回环？

  • ✅ 自包含验证：单张卡即可测试，无需外部设备
  • ✅ 故障隔离：若测试失败，问题必在卡内（内核、Aurora IP、物理层）
  • ✅ 性能基准：测量单链路最大吞吐量和延迟

替代拓扑：若取消回环，将 Aurora TX 连接到另一张卡的 Aurora RX，可构建多卡 Scale-out 网络，但这需要同步时钟和更复杂的错误处理。

4.3 时钟域设计：三时钟策略

配置中连接了三个不同的时钟源，这是高速串行设计的经典模式：

┌─────────────────────────────────────────────────────────────────────────────┐
│                           时钟域架构                                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   ┌──────────────┐      ┌──────────────┐      ┌──────────────┐             │
│   │  init_clk    │      │  gt_refclk   │      │  dclk/user   │             │
│   │  (初始化时钟) │      │  (GT参考时钟) │      │  (数据时钟)   │             │
│   │  ~100-300MHz │      │  156.25MHz   │      │  与线速率相关 │             │
│   └──────┬───────┘      └──────┬───────┘      └──────┬───────┘             │
│          │                     │                     │                       │
│          ▼                     ▼                     ▼                       │
│   ┌──────────────┐      ┌──────────────┐      ┌──────────────┐             │
│   │ Aurora State │      │ GT CDR/PLL   │      │ Data Path    │             │
│   │ Machine      │      │ (时钟恢复)    │      │ (FIFO/AXIS)  │             │
│   │ (复位/初始化) │      │              │      │              │             │
│   └──────────────┘      └──────────────┘      └──────────────┘             │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘

三时钟的作用：

时钟 连接目标 频率 作用
init_clk ulp_m_aclk_freerun_ref_00 ~100-300MHz 独立初始化时钟。确保即使在 gt_refclk 不稳定或缺失时，Aurora 的状态机仍能运行，完成复位序列和链路初始化。这是可靠性关键。
gt_refclk io_clk_qsfp_refclkb_00 156.25MHz (典型) GT 参考时钟。这是 GT 收发器 CDR 电路的基准。CDR 从串行数据流中恢复时钟，其精度直接依赖于参考时钟的稳定性和抖动性能。
dclk 隐含在 AXI Stream 中 与线速率/协议相关 数据时钟。Aurora IP 内部从恢复的时钟生成数据域时钟，与 AXI Stream 的握手信号同步。这是用户逻辑（strm_issue、strm_dump）实际工作的时钟域。

关键设计洞察：这种三时钟架构实现了故障隔离和独立启动。

  • init_clk 可由板载自由运行振荡器提供，不依赖于外部 QSFP 模块是否插入或链路伙伴是否上电
  • gt_refclk 通常来自 QSFP 模块的时钟输出（如果模块支持）或板载时钟分配器
  • 这种分离允许 Aurora 在参考时钟不稳定时仍能被复位和配置，待时钟稳定后再进入正常操作

5.2 数据契约与接口规范

AXI4-Stream 连接契约：

stream_connect 指令建立的连接必须满足以下硬件契约：

信号 方向 宽度 描述
TDATA 源→宿 通常为 32/64/128/256/512 位 有效数据负载
TVALID 源→宿 1 位 数据有效指示
TREADY 宿→源 1 位 下游就绪接收
TLAST 源→宿 1 位 数据包边界指示
TKEEP 源→宿 TDATA/8 位 字节有效掩码

契约要求：

  1. 位宽匹配：源和宿的 TDATA 宽度必须相同（由 krnl_aurora 的接口定义决定）
  2. 时钟域一致：连接的双方必须在同一时钟域，或插入异步 FIFO 进行跨时钟域处理
  3. 反压传播：TREADY 信号必须正确传播，防止数据丢失

物理引脚连接契约：

connect 指令建立了内核逻辑端口到板级物理接口的映射：

内核端口 物理接口 信号标准 约束
gt_refclk io_clk_qsfp_refclkb_00 差分 LVDS/LVPECL 必须匹配 GT 原语的时钟要求（如 156.25MHz ±100ppm）
gt_port io_gt_qsfp_00 GTY/GTM 收发器引脚 特定于器件封装，不可随意更改
init_clk ulp_m_aclk_freerun_ref_00 单端/差分时钟 通常为 100-300MHz，与数据速率无关

7. 设计权衡与关键决策

7.1 配置策略：编译期 vs 运行期

观察：所有连接关系（stream_connect、connect）在 .cfg 文件中静态定义，由 Vitis 链接器处理。

权衡分析：

维度 编译期配置（本模块） 运行期配置（假想）
灵活性 低：需重新编译链接生成 xclbin 高：可动态调整连接
性能 高：连接硬连线，无运行期开销 低：需交换机/路由逻辑
验证难度 低：链接器静态检查 高：运行期才可能暴露错误
资源占用 最小：精确实例化所需资源 额外：需通用互联结构

决策合理性：对于 Aurora 测试这种拓扑固定、性能敏感的场景，编译期配置是最优选择。这类似于 ASIC 设计中的硬连线 vs 片上网络（NoC）的权衡——当拓扑确定时，硬连线提供最低延迟和最高效率。

7.2 多板卡支持策略：条件编译 vs 动态探测

观察：配置文件中包含多个注释块，分别对应 U200、U250、U50、U55c、U280 等不同 Alveo 卡。

设计决策：采用人工条件编译（注释切换）而非自动板卡探测。

权衡分析：

策略 优势 劣势
人工条件编译（当前） - 显式控制，无魔法
- 精确匹配每个板卡的时钟架构（U200/U250 使用 level0_wire，U250 使用 level1_wire）
- 易于代码审查		 - 切换板卡需手动编辑
- 维护多份注释代码
- 无编译期错误检查（如取消注释了 U200 又取消 U250 的块，链接会失败但提示可能晦涩）
动态板卡探测（假想） - 单一配置适配所有板卡
- 用户体验更好		 - 需运行时读取板卡 EEPROM 或 DSA 元数据
- 时钟引脚命名规则需标准化（实际各卡差异大）
- 无法处理本质不同的时钟架构（如 U250 的 level1_wire vs U200 的 level0_wire）

关键洞察：Alveo 卡之间的差异不仅是"参数不同"，而是架构不同——不同的时钟树结构、不同的 GT Quad 位置、不同的 level shifter 层级。这种差异无法通过简单参数化抽象，必须显式处理。

类比：这类似于嵌入式开发中不同 MCU 的启动代码——STM32F1 和 STM32F4 虽有相似性，但时钟树、Flash 等待状态、电源稳压器的差异，使得它们需要不同的启动文件，而非单一通用代码。

7.3 初始化时钟与 GT 参考时钟的分离

关键设计：init_clk（初始化时钟）与 gt_refclk（GT 参考时钟）是两个独立的物理时钟源。

设计原理：

┌────────────────────────────────────────────────────────────────────────┐
│                    为什么需要独立的 init_clk？                            │
├────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   场景：系统刚上电，QSFP 模块未插入或链路伙伴未就绪                      │
│                                                                         │
│   若 init_clk 依赖于 gt_refclk：                                        │
│   ┌─────────────┐    无 QSFP      ┌─────────────┐                      │
│   │  Board      │───→ 无 refclk ──→│ Aurora IP   │  无法初始化！        │
│   │  Power On   │    (或不稳定)    │  Stuck      │                      │
│   └─────────────┘                   └─────────────┘                      │
│                                                                         │
│   有独立 init_clk 时：                                                   │
│   ┌─────────────┐    板载振荡器   ┌─────────────┐    等待 refclk        │
│   │  Board      │───→ init_clk ──→│ Aurora IP   │←─── 稳定后启动 GT     │
│   │  Power On   │    (始终可用)   │  正常工作   │                      │
│   └─────────────┘                   └─────────────┘                      │
│                                                                         │
│   关键收益：                                                            │
│   ✓ Aurora 可在 GT 参考时钟不稳定时完成初始化序列                       │
│   ✓ 支持"热插拔" QSFP 模块——模块插入后软件触发重新初始化                │
│   ✓ 避免时钟域交叉导致的亚稳态问题                                      │
│                                                                         │
└────────────────────────────────────────────────────────────────────────┘

工程类比：这就像现代汽车的 ECU（发动机控制单元）设计——即使发动机未启动（GT 时钟未就绪），ECU 仍需电源（init_clk）来管理仪表盘、诊断系统和启动序列。

9. 边界情况与陷阱

9.1 常见配置错误

错误模式 症状 根因分析 修复方法
多板卡配置同时启用 链接失败，报错 "multiple drivers for clock" 或引脚冲突 不同板卡的 connect 行都取消注释，导致同一物理引脚被多个信号驱动 确保仅目标板卡的配置块启用，其他用 # 注释
AXI Stream 位宽不匹配 编译通过但运行时数据错乱或死锁 strm_issue 输出位宽与 krnl_aurora 输入位宽不同，或 TLAST/TKEEP 处理不一致 检查各内核的 .xo 接口定义，确保 TDATA 位宽一致
时钟未连接 Aurora IP 报告初始化失败或链路无法建立 init_clk 或 gt_refclk 未正确连接到板级时钟 检查时钟引脚名称与 Platform 文档匹配；确认时钟频率在 Aurora IP 支持范围内
循环依赖连接 链接器报错 "circular dependency" 或无限循环 意外创建了环形拓扑（如 A→B→C→A） 本配置是 DAG（有向无环图），但扩展时需避免环

9.2 运行时陷阱

GT 参考时钟不稳定：

  • 现象：Aurora 链路间歇性失步，性能波动
  • 诊断：检查 QSFP 模块是否插入到位、光纤是否完好、对端设备是否上电
  • 缓解：确保 init_clk 始终稳定，利用 Aurora 的自动重初始化机制

AXI Stream 反压传播：

  • 现象：吞吐量低于理论值，或完全停滞
  • 诊断：检查 TREADY 信号是否被正确传播。若 strm_dump 处理速度低于数据到达速度，反压会传播到 Aurora，进而可能触发流量控制
  • 缓解：确保接收端（strm_dump）有足够缓冲或处理能力；在 Aurora 配置中启用流量控制（Flow Control）

跨时钟域（CDC）问题：

  • 现象：数据偶发错误，难以复现
  • 诊断：检查是否有异步时钟域交叉未正确处理。本配置中，若 strm_issue、krnl_aurora、strm_dump 使用不同频率的时钟，需在边界插入异步 FIFO
  • 缓解：确保 Vitis 链接器插入了适当的 CDC 逻辑，或在内核 RTL 中处理 CDC

9.3 扩展与修改指南

添加新的 Aurora 链路：

若要测试多条 Aurora 链路（如使用 QSFP 的多个通道），需：

# 1. 实例化更多内核
nk=krnl_aurora:2:krnl_aurora_0:krnl_aurora_1
nk=strm_issue:2:strm_issue_0:strm_issue_1
nk=strm_dump:2:strm_dump_0:strm_dump_1

# 2. 建立新的流连接
stream_connect=strm_issue_1.data_output:krnl_aurora_1.tx_axis
stream_connect=krnl_aurora_1.rx_axis:strm_dump_1.data_input

# 3. 连接新的物理引脚（以 U200 的第二个 QSFP 为例）
connect=io_clk_qsfp_refclkb_01:krnl_aurora_1/gt_refclk
connect=krnl_aurora_1/gt_port:io_gt_qsfp_01
connect=krnl_aurora_1/init_clk:ii_level0_wire/ulp_m_aclk_freerun_ref_00

关键注意事项：

  • 确保新使用的 QSFP 端口在目标 Alveo 卡上物理存在
  • 检查 Platform 文档中时钟引脚的可用性（如 refclkb_01 是否存在）
  • 多个 Aurora 实例可共享 init_clk，但每个实例需要独立的 gt_refclk（若使用不同 GT Quad）

10. 总结：核心设计洞察

10.1 架构隐喻

想象这个系统是一个现代化港口：

  • strm_issue 是港口的装货码头，源源不断地产生集装箱（数据包）
  • krnl_aurora 是海关与运输公司，负责将集装箱封装到标准货柜车（Aurora 帧），并运送到跨海大桥（GT 收发器）
  • QSFP 端口是跨海大桥的入口，连接着外部高速公路网络
  • strm_dump 是卸货码头，接收从跨海大桥运来的集装箱，检查是否完好无损

krnl_aurora_test.cfg 这个配置文件，就是港口的总体规划蓝图——它定义了码头、海关、跨海大桥的相对位置，以及它们之间的道路连接。这个规划必须在施工（FPGA 比特流生成）前确定，因为一旦建成，修改道路（重新配置 FPGA 互连）需要重新施工（重新生成 xclbin）。

10.2 关键设计决策回顾

决策 选择 理由
配置方式 静态 .cfg 文件 编译期验证、零运行期开销、工具链原生支持
多板卡支持 条件注释块 架构差异无法简单参数化，显式优于隐式
时钟架构 三时钟分离 (init/gt/data) 故障隔离、独立启动、支持热插拔
测试拓扑 回环测试 自包含验证、故障隔离、单卡可测
连接类型 AXI4-Stream 行业标准、反压支持、HLS/RTL 兼容

10.3 新贡献者行动指南

如果你是刚加入团队的新工程师，需要修改或扩展这个配置：

第一步：理解物理约束

  • 查阅目标 Alveo 卡的 Platform 文档，确认：
    • QSFP 端口编号（qsfp_00 vs qsfp_01）
    • 参考时钟引脚名称（refclka vs refclkb）
    • 时钟频率要求（156.25MHz vs 161.1328125MHz）
  • 检查 XDC 约束文件（若有），了解引脚的物理位置

第二步：验证内核接口

  • 确认 strm_issue、krnl_aurora、strm_dump 的 .xo 接口定义：
    • TDATA 位宽（通常为 64 或 128 位）
    • 是否包含 TLAST、TKEEP、TUSER 等信号
  • 确保连接的端口位宽匹配（Vitis 链接器会检查，但提前确认可避免迭代）

第三步：渐进式修改

  • 不要一次性修改多个连接：每次只改一行配置，链接验证通过后再继续
  • 使用 --save-temps：保留 Vitis 中间文件，便于调试连接问题
  • 检查链接报告：v++ 生成的 _link_summary 文件包含实际实现的拓扑图

第四步：硬件验证

  • 烧录 xclbin 后，首先验证 Aurora 链路状态（通常通过寄存器读取 channel_up 和 lane_up 信号）
  • 使用 xbutil 或自定义工具检查 GT 眼图（eye scan），确认信号完整性
  • 逐步增加数据速率，观察误码率（BER）变化

常见陷阱清单：

  • [ ] 忘记切换板卡配置时，注释掉旧的 connect 行（导致多驱动错误）
  • [ ] 混淆 refclka 和 refclkb（不同 QSFP 端口使用不同参考时钟）
  • [ ] 忽视 init_clk 的频率要求（过高或过低都会导致 Aurora IP 初始化失败）
  • [ ] 未检查 QSFP 模块的速率等级（使用 10G 模块测试 25G Aurora 会失败）

  • 下游消费：本配置的输出（krnl_aurora_test.xclbin）被主机应用程序通过 XRT（Xilinx Runtime）加载到 Alveo 卡上执行。主机应用通常使用 Hardware_Acceleration-Design_Tutorials-08-alveo_aurora_kernel-host 中描述的 API 与内核交互。

  • 平台依赖：本配置紧密耦合于特定 Alveo 卡的平台定义（Platform），如 xilinx_u200_gen3x16_xdma_2_202110_1。这些平台定义了 io_clk_qsfp_*、io_gt_qsfp_* 等板级接口的可用性与物理位置。理解 Hardware_Acceleration-Design_Tutorials-08-alveo_aurora_kernel-platform 对于正确映射物理引脚至关重要。

  • 同系列教程：本模块属于 Hardware_Acceleration-Design_Tutorials 系列中的第 8 个教程。该系列的前序教程（如 Hardware_Acceleration-Design_Tutorials-01-convolution）介绍了 Vitis 流程的基础知识，而本教程专注于高速串行通信这一高级主题。建议新用户先完成基础教程再深入本模块。

  • 相关 IP 文档：Aurora 8B/10B 或 Aurora 64B/66B IP 的官方文档（PG046、PG074）是理解 krnl_aurora 内部工作原理的关键参考。本配置文件中涉及的 gt_refclk、init_clk 等信号的时序要求，需以这些 IP 文档为准。

  • GT 收发器指南：UltraScale/UltraScale+ GTY/GTM 收发器的用户指南（UG578、UG576）解释了 io_gt_qsfp_* 等引脚的物理层特性、眼图要求、预加重/均衡设置。当 Aurora 链路无法建立时，这些文档是调试信号完整性问题的必备参考。
