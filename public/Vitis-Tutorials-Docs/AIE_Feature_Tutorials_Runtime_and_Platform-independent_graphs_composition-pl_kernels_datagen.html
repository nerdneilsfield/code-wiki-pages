pl_kernels_datagen 模块技术深度解析

一句话概括

pl_kernels_datagen 是一个合成数据生成器，它像一个"数字水龙头"——在硬件测试环境中按需产生可预测的整数序列，为 AI Engine 分区提供可控的测试输入。它是独立图组合流程中的关键组件，让多个团队能够在不依赖真实数据源的情况下并行验证各自的 AIE 分区。

问题空间：为什么需要这个模块？

场景困境：多团队并行开发 AIE 系统

想象一个大型 Versal 项目，三个团队分别负责不同的信号处理阶段：

  • 团队 A：负责前端滤波（pr0_gmio 分区）
  • 团队 B：负责 RTP 重配置逻辑（pr1_rtp 分区）
  • 团队 C：负责性能优化核心（pr2_perf 分区）

每个团队需要测试自己的 AIE 图，但存在根本性矛盾：

  1. 上游数据不可用：团队 B 和 C 的团队还在开发中，无法提供稳定的数据源
  2. 测试可重复性：使用真实传感器数据会导致测试结果不可复现
  3. 边界条件验证：需要特定的数据模式（如全零、递增序列、最大值等）来验证鲁棒性

朴素方案的缺陷

方案 问题
等待其他团队完成 串行化开发，违背"独立分区"的设计初衷
从 DDR 读取预存数据 增加内存带宽压力，且需要额外的 DMA 配置
随机数生成 难以调试，错误无法复现

设计洞察

需要一个轻量级、确定性、零外部依赖的数据源，能够：

  • 直接在 PL（Programmable Logic）中生成数据，无需 DDR 访问
  • 通过 AXI4-Stream 直接注入 AIE 阵列
  • 支持参数化配置（数据量、起始值），适应不同测试场景

心智模型：把 datagen 想象成什么？

类比：编程式"数字信号发生器"

就像电子实验室中的函数发生器可以输出正弦波、方波等标准波形，datagen 是一个纯数字的序列发生器。它的核心特性：

  • 确定性输出：给定相同的 size 和 init_val，总是产生相同的序列
  • 流式接口：数据像水流一样持续流出，而非批量突发
  • 零状态记忆：每次调用都是独立的，不保留历史上下文

代码深度解析

文件：datagen.cpp

/*
Copyright (C) 2023, Advanced Micro Devices, Inc. All rights reserved.
SPDX-License-Identifier: MIT
*/
#include <stdlib.h>
#include <iostream>
#include <hls_stream.h>
#include <ap_axi_sdata.h>

extern "C" void datagen(hls::stream<ap_axis<32,0,0,0>> & out, int size, int init_val=0) {
#pragma HLS INTERFACE axis port=out
#pragma HLS INTERFACE s_axilite port=return bundle=control
#pragma HLS INTERFACE ap_ctrl_hs port=return
#pragma HLS interface s_axilite port=size bundle=control
#pragma HLS interface ap_none port=size
#pragma HLS interface s_axilite port=init_val bundle=control
#pragma HLS interface ap_none port=init_val
        for(int i=0;i<size;i++){
                ap_axis<32,0,0,0> tmp;
                tmp.data=i+init_val;
                tmp.keep=-1;
                tmp.last=0;
                out.write(tmp);
        }
}

函数签名解析

void datagen(
    hls::stream<ap_axis<32,0,0,0>> & out,  // 输出流端口
    int size,                               // 生成数据个数
    int init_val = 0                        // 序列起始值（默认0）
)

模板参数 ap_axis<32,0,0,0> 的含义：

  • 32：数据位宽 32-bit（对应 int 类型）
  • 0：无用户定义信号（TUSER 宽度为 0）
  • 0：无 ID 信号（TID 宽度为 0）
  • 0：无目的地信号（TDEST 宽度为 0）

这是一个精简的 AXI4-Stream 协议实现，仅保留必需的信号线：

  • TDATA[31:0]：实际数据
  • TKEEP[3:0]：字节有效指示（固定为全1，即 -1）
  • TLAST：包结束指示（固定为0，表示连续流）
  • TVALID/TREADY：由 hls::stream 自动管理

HLS 接口指令详解

指令 作用 设计意图
#pragma HLS INTERFACE axis port=out 将 out 映射为 AXI4-Stream 接口 与 AIE 的 PLIO 直接对接
#pragma HLS INTERFACE s_axilite port=return bundle=control 块控制接口（启动/完成/空闲） 主机通过 XRT 控制核的执行
#pragma HLS INTERFACE ap_ctrl_hs port=return 握手式控制协议 确保主机知道核何时完成
#pragma HLS interface s_axilite port=size bundle=control size 参数通过 AXI4-Lite 配置 运行时动态指定数据量
#pragma HLS interface ap_none port=size size 无时钟同步要求 简化时序，假设配置后稳定
#pragma HLS interface s_axilite port=init_val bundle=control init_val 通过 AXI4-Lite 配置 支持多种测试模式
#pragma HLS interface ap_none port=init_val init_val 无时钟同步要求 同 size，简化设计

关键洞察：s_axilite + ap_none 的组合是一种常见的 HLS 设计模式：

  • s_axilite 负责软件可见性——让主机能够读写该寄存器
  • ap_none 声明硬件无时序约束——告诉综合工具该信号在核启动后保持稳定，无需打拍同步

核心算法逻辑

for(int i = 0; i < size; i++) {
    ap_axis<32,0,0,0> tmp;
    tmp.data = i + init_val;    // 生成递增序列
    tmp.keep = -1;               // 所有字节有效（0xFFFFFFFF）
    tmp.last = 0;                // 非包尾（连续流模式）
    out.write(tmp);              // 阻塞写入流
}

生成的数据模式：

  • 当 init_val=0, size=5 时，输出：[0, 1, 2, 3, 4]
  • 当 init_val=100, size=3 时，输出：[100, 101, 102]

这种简单可预测的模式是刻意为之：

  1. 易于验证：主机知道期望输出，可直接比对
  2. 调试友好：波形中出现 0x00000005 就知道是第6个数据
  3. 溢出检测：如果看到不连续的跳跃，说明有丢数或乱序

设计决策与权衡

决策 1：为何不用 #pragma HLS PIPELINE？

观察：循环体没有 PIPELINE 指令，这意味着默认行为是顺序执行。

权衡分析：

选项 II（启动间隔） 资源消耗 适用场景
无 PIPELINE（当前） N/A（非流水线） 最小 低速数据生成，简化时序
PIPELINE II=1 1 周期 中等 高速连续数据流

选择理由：

  1. 需求匹配：作为测试数据源，通常不需要极高吞吐量（400 MHz 下单核已足够）
  2. 面积优先：独立分区教程强调模块化，小面积核更易布局
  3. 时序安全：非流水线设计对跨时钟域（400 MHz PL → 250 MHz AIE）更友好

潜在风险：如果下游 AIE 分区需要饱和吞吐，当前的顺序写入可能成为瓶颈。可通过添加 PIPELINE II=1 优化。

决策 2：tmp.last = 0 的含义

TLAST 信号的作用：在 AXI4-Stream 中标记数据包的边界。

设计选择：

  • last = 0：表示这是一个无限流，没有显式的包边界
  • 替代方案 last = (i == size-1)：在最后一个数据标记包尾

为何选择连续流模式：

  1. 协议简化：下游 AIE 核无需处理包边界逻辑
  2. 灵活连接：可与需要连续流的 AIE 核直接对接
  3. 数量控制：通过 size 参数精确控制总数据量，而非依赖 TLAST

注意事项：如果下游核依赖 TLAST 触发内部状态机，此设计需要修改。

决策 3：为何使用 ap_axis 而非 ap_axiu？

对比 mm2s.cpp 中的用法：

  • mm2s 使用 ap_axiu<128, 0, 0, 0>（unsigned 数据）
  • datagen 使用 ap_axis<32, 0, 0, 0>（signed 数据）

选择依据：

  • datagen 生成的是有符号整数（int 类型），使用 ap_axis 语义更准确
  • 虽然硬件层面两者等价（都是比特向量），但代码可读性更好

决策 4：默认参数 init_val=0 的工程意义

便利性设计：

  • 最常见的测试场景是从 0 开始计数
  • 默认参数允许主机代码简化为 datagen(out, 1024) 而非 datagen(out, 1024, 0)

可扩展性：

  • 支持非零起始值，便于生成偏移序列（如模拟 ADC 采样值的直流偏置）
  • 负数起始值可用于测试有符号数处理逻辑

构建系统集成

配置文件：tmp.cfg

[hls]
freqhz=400000000          # PL 核目标频率：400 MHz
syn.top=datagen           # 顶层函数名
syn.file=datagen.cpp      # 源文件
flow_target=vitis         # Vitis 统一流程
package.output.format=xo  # 输出 Xilinx Object 格式
package.output.file=datagen.xo

频率选择 rationale：

  • 400 MHz 是 Versal PL 的常见工作频率
  • 高于 AIE 的 250 MHz，确保数据供给不会成为瓶颈

Makefile 构建流程

%.xo: %.cpp
    sed 's/s2mm/\((basename \)<)/' config.cfg > tmp.cfg
    v++ -c --mode hls --platform ${PLATFORM} --config tmp.cfg

关键技巧：使用 sed 动态替换配置模板，避免为每个核维护独立的 .cfg 文件。

系统连接：system.cfg

nk=datagen:1:datagen
stream_connect=datagen.out:ai_engine_0.pr1_Datain0

命名规则：

  • nk=datagen:1:datagen：创建 1 个实例，名为 datagen
  • datagen.out：引用核的 out 端口
  • ai_engine_0.pr1_Datain0：引用 AIE 分区 pr1 的图端口 Datain0

分区前缀 pr1_：这是独立分区流程的关键约定——所有端口名都带分区前缀，防止链接时的命名冲突。

使用指南与示例

典型使用模式

模式 1：基础功能验证

// Host 代码片段
auto datagen_kernel = xrt::kernel(device, uuid, "datagen");
auto run = datagen_kernel(1024, 0);  // 生成 1024 个整数，从 0 开始
run.wait();

验证方法：检查 AIE 输出是否等于输入（直通测试）或符合预期算法结果。

模式 2：多批次不同起始值

for (int batch = 0; batch < 10; batch++) {
    auto run = datagen_kernel(100, batch * 100);  // 批次间无重叠
    run.wait();
    // 验证输出范围 [batch*100, batch*100+99]
}

用途：验证 AIE 核的状态重置是否正确，确保批次间无残留数据污染。

模式 3：边界条件测试

// 测试空传输
auto run1 = datagen_kernel(0, 0);
run1.wait();  // 应无任何数据输出

// 测试单数据
auto run2 = datagen_kernel(1, INT_MAX);
run2.wait();  // 验证最大正值处理

// 测试负起始值
auto run3 = datagen_kernel(10, -5);
run3.wait();  // 验证有符号数处理

配置参数建议

参数 推荐值 说明
size 64 ~ 65536 太小无法摊平启动开销，太大占用仿真时间
init_val 0, 0x80000000, 0x7FFFFFFF 覆盖零、最小负值、最大正值三种典型情况

边缘情况与陷阱

陷阱 1：size 为负数或未初始化

风险：size 是有符号 int，如果传入负数，循环条件 i < size 初始即为假，核立即退出，无数据输出。

症状：下游 AIE 核挂起等待数据，系统死锁。

防御：Host 代码应验证 size > 0：

if (size <= 0) {
    throw std::invalid_argument("size must be positive");
}

陷阱 2：跨时钟域数据丢失

场景：datagen 运行在 400 MHz，下游 AIE 运行在 250 MHz，且 AIE 内部有反压（back-pressure）。

机制：hls::stream::write() 是阻塞操作——当 FIFO 满时会 stall。由于 HLS 未加 PIPELINE，每次写入需要多个周期，天然提供了反压容忍。

监控点：观察 out 端口的 TREADY 信号，如果长期为低，说明下游消费能力不足。

陷阱 3：init_val 溢出

数学表达：实际输出值为 (i + init_val) mod \(2^{32}\)

当 i + init_val > INT_MAX 时，会发生有符号整数溢出（UB in C++，但在 HLS 中定义为回绕）。

示例：

datagen(out, 10, 2147483640);  // init_val = INT_MAX - 7
// 输出: [2147483640, ..., 2147483647, -2147483648, ..., -2147483641]

应对：如果测试需要严格单调递增且无回绕，确保 init_val + size - 1 <= INT_MAX。

陷阱 4：与其他数据源的混用

在同一个分区测试中，如果交替使用 datagen（合成数据）和 mm2s（真实 DDR 数据），注意：

  • datagen 输出 32-bit 数据
  • mm2s 输出 128-bit 数据

不匹配后果：如果 AIE 图的 PLIO 配置为 128-bit，直接连接 datagen 会导致数据错位。

解决方案：确保 AIE 图的 plin/plout 位宽与数据源匹配，或在中间插入位宽转换核。

扩展与定制

扩展点 1：改变数据模式

当前实现生成线性递增序列。如需其他模式（如伪随机、三角波、常数值），修改循环体：

// 伪随机（LFSR）
static unsigned lfsr = init_val;
lfsr = (lfsr >> 1) ^ (-(lfsr & 1u) & 0xD0000001u);
tmp.data = lfsr;

// 三角波
tmp.data = (i < size/2) ? i : (size - i);

// 常数值（压力测试）
tmp.data = init_val;  // 所有数据相同，测试 AIE 的常量折叠

扩展点 2：添加 TLAST 支持

如需与依赖包边界的 AIE 核对接：

tmp.last = (i == size - 1) ? 1 : 0;

扩展点 3：多通道输出

复制端口和数据生成逻辑，支持并行多路输出：

void datagen_dual(
    hls::stream<ap_axis<32,0,0,0>> & out0,
    hls::stream<ap_axis<32,0,0,0>> & out1,
    int size, int init_val0, int init_val1
) {
    for(int i = 0; i < size; i++) {
        // 生成两个独立序列
        ap_axis<32,0,0,0> tmp0, tmp1;
        tmp0.data = i + init_val0;
        tmp1.data = i + init_val1;
        // ... 设置 keep/last
        out0.write(tmp0);
        out1.write(tmp1);
    }
}

总结

pl_kernels_datagen 是一个小而精的测试基础设施核。它的设计哲学是：

用最简单的确定性行为，解决最复杂的系统集成验证问题。

对于新加入团队的开发者，理解这个模块的关键在于把握三点：

  1. 它是假的，但是有用的：合成数据不能替代真实场景测试，但能让并行开发成为可能
  2. 它是慢的，但是够用的：非流水线设计牺牲了极限性能，换取了时序安全和面积效率
  3. 它是封闭的，但是灵活的：零外部依赖使其可在任意上下文中复用，参数化设计又保留了测试多样性

在多团队、多分区的 AIE 项目中，像 datagen 这样的"数字水龙头"是不可或缺的调试利器——它让问题的边界变得清晰：如果输出错了，问题一定在下游；如果输出对了，就可以自信地移交集成。
