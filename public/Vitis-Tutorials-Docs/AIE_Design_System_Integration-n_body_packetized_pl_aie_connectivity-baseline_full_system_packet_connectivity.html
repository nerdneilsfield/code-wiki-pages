baseline_full_system_packet_connectivity 模块深度解析

一句话概述

本模块是 N-Body 仿真器完整系统设计的数据高速公路调度中心，它通过 Vitis 连接配置文件（conn.cfg）将 PL（可编程逻辑）端的 DMA 数据搬运核、包分发/收集核与 AIE（AI 引擎）阵列的 100 个计算单元精确地编织在一起，实现了"一对多广播输入、多对一汇聚输出"的大规模并行数据流架构。

想象一个大型机场的行李分拣系统：DMA 源端像是值机柜台，将旅客行李（数据）送上传送带；packet_sender 像是智能分拣机器人，根据目的地标签（packet ID）将行李分发到 100 条不同的登机通道（AIE 输入端口）；AIE 阵列完成计算后，packet_receiver 像到达大厅的行李转盘，从 100 个出口收集行李并按航班重新归类；最后 DMA 汇端将行李送到提取区。本模块就是这套系统的"交通规划图"。

问题空间：为什么需要这种架构？

N-Body 问题的计算特性

N-Body 仿真是计算物理学中的经典问题：模拟 \(N\) 个粒子在相互引力作用下的运动轨迹。每个粒子的加速度取决于与其他所有粒子的相互作用：

\[\\mathbf{a}_i = \\sum_{j \\neq i} \\frac{G m_j (\\mathbf{r}_j - \\mathbf{r}_i)}{|\\mathbf{r}_j - \\mathbf{r}_i|^3}\]

这种计算的复杂度为 \(O(N^2)\)，当 \(N=100\) 时，每轮迭代需要计算约 10,000 对相互作用。为了加速，设计采用了：

  • 空间并行：100 个 AIE 计算单元（CU），每个处理一个粒子的受力计算
  • 时间流水线：4 个数据包（packet）组织成突发传输（burst），隐藏延迟
  • 数据复用：每个粒子位置数据需要广播给所有 CU 进行交互计算

为什么不能用简单的点对点连接？

如果采用朴素的直连方案：

  • 每个 CU 需要独立的 DMA 通道 → 100 个 DMA 读通道，资源爆炸
  • AIE 阵列的物理端口有限 → 无法直接连接 100 个逻辑流
  • 数据传输效率低下 → 小粒度随机访问，无法利用突发传输带宽

核心洞察：需要一种"数据包交换网络"架构——PL 端负责数据的路由和打包，AIE 端专注于计算，通过 packet switching 实现逻辑上的全连接而无需物理上的全连接。

组件深度解析

1. conn.cfg —— 系统连接配置

这是整个系统的"接线图"，使用 Vitis 连接配置语法定义了核实例化和流连接。

核实例化 (nk)

nk=mm2s_mp:1:mm2s_mp_0          # 1 个 DMA 内存到流核
nk=packet_sender:1:packet_sender_0  # 1 个包发送核
nk=packet_receiver:1:packet_receiver_0  # 1 个包接收核
nk=s2mm_mp:1:s2mm_mp_0          # 1 个 DMA 流到内存核

设计决策：为什么选择 4 个独立核而不是合并？

  • 模块化：每个核可以独立优化、综合和替换
  • 资源隔离：DMA 和计算逻辑分离，便于时序收敛
  • 可测试性：每个核可以单独进行 HLS C-simulation

输入数据流（Ingress Path）

stream_connect = mm2s_mp_0.s0:packet_sender_0.rx
# DMA → packet_sender（待分发的粒子数据）

stream_connect = mm2s_mp_0.s1:ai_engine_0.in_j
# DMA → AIE 共享端口（所有粒子的位置数据，广播）

关键洞察：in_j 是一个特殊的广播端口，包含所有粒子的位置信息，被所有 CU 共享读取。这避免了在 PL 端复制 100 份相同数据。

分发网络（Distribution Network）

stream_connect = packet_sender_0.tx0:ai_engine_0.in_i0
stream_connect = packet_sender_0.tx1:ai_engine_0.in_i1
...
stream_connect = packet_sender_0.tx99:ai_engine_0.in_i99

100 路扇出：packet_sender 将输入流拆分到 100 个独立通道，每个对应一个 AIE CU 的 in_i* 端口。这种设计允许每个 CU 以独立速率消费数据，天然支持 back-pressure。

收集网络（Collection Network）

stream_connect = ai_engine_0.out_i0:packet_receiver_0.rx0
...
stream_connect = ai_engine_0.out_i99:packet_receiver_0.rx99

100 路扇入：AIE 计算完成后，结果从 100 个输出端口流出，进入 packet_receiver 进行汇聚。

输出聚合（Eggregation）

stream_connect = packet_receiver_0.tx0:s2mm_mp_0.s0
stream_connect = packet_receiver_0.tx1:s2mm_mp_0.s1
stream_connect = packet_receiver_0.tx2:s2mm_mp_0.s2
stream_connect = packet_receiver_0.tx3:s2mm_mp_0.s3

4→1 归约：100 路输入被整理为 4 路输出，匹配 DMA sink 的 4 个 stream 接口。注意这里不是简单的一对一映射，而是基于 packet ID 的动态路由。

2. packet_sender —— 包分发核

位于 baseline_pl_packet_kernels 模块中，这是一个 HLS 实现的 PL 核，负责将 DMA 传来的原始数据打包并分发到 100 个 AIE 输入端口。

核心功能

void packet_sender(hls::stream<axis_pkt>& rx,           // 输入：来自 DMA
                   hls::stream<axis_pkt>& tx0, ...tx99  // 输出：到 AIE
                  )

数据包格式：

  • Header（第 1 个字）：包含 packet ID（5-bit）、packet type（3-bit）、奇偶校验位
  • Payload（224 个字）：实际的粒子数据
  • TLAST：标记包的结束

工作流程

for(int cu=0; cu<NUM_CU; cu++) {              // 遍历 100 个计算单元
    for (int h = 0; h < PACKET_NUM; h++) {    // 每个 CU 4 个包
        unsigned int ID = packet_ids[h];      // 获取包 ID
        ap_uint<32> header = generateHeader(PKTTYPE, ID);
        tx[cu].write(header);                 // 发送包头
        
        for(int i = 0; i < PACKET_LEN; i++) { // 224 个 payload 字
            #pragma HLS PIPELINE II=1         // 单周期吞吐
            x = rx.read();
            x.last = (i == PACKET_LEN-1);     // 最后一个字标记 TLAST
            tx[cu].write(x);
        }
    }
}

关键设计点：

  • 双重循环结构：外层按 CU 组织，内层按 packet 组织，确保数据顺序符合 AIE 期望
  • Pipeline II=1：保证每个时钟周期输出一个字，最大化带宽
  • 动态路由：通过 switch(cu) 语句实现 100 路分发，HLS 会将其综合为多路选择器网络

Header 生成逻辑

ap_uint<32> generateHeader(unsigned int pktType, unsigned int ID) {
    ap_uint<32> header = 0;
    header(4,0)   = ID;           // Packet ID: 0-3
    header(11,5)  = 0;            // Reserved
    header(14,12) = pktType;      // Type: 0
    header[15]    = 0;            // Reserved
    header(20,16) = -1;           // Source row (broadcast)
    header(27,21) = -1;           // Source col (broadcast)
    header(30,28) = 0;            // Reserved
    header[31]    = header(30,0).xor_reduce() ? 0 : 1;  // Parity bit
    return header;
}

Packet ID 来源：packet_ids 数组从 packet_ids_c.h 引入，这是由 AIE 编译器自动生成的头文件，确保 PL 和 AIE 对 packet routing 的理解一致。

3. packet_receiver —— 包收集核

同样位于 baseline_pl_packet_kernels 模块，负责从 100 个 AIE 输出端口收集数据，并按 packet ID 重组为 4 路输出。

核心功能

void packet_receiver(hls::stream<axis_pkt>& rx0, ...rx99,  // 输入：来自 AIE
                     hls::stream<axis_pkt>& tx0, ...tx3    // 输出：到 DMA
                    )

工作流程

for (int cu = 0; cu < 100; cu++) {
    for (int h = 0; h < PACKET_NUM; h++) {
        // 1. 读取包头，解析 packet ID
        tmp = rx[cu].read();
        unsigned int ID = getPacketId(tmp.data);
        unsigned int channel = packet_ids[ID];  // 映射到输出通道 0-3
        
        // 2. 读取 payload，按 ID 路由到对应输出
        for(int i = 0; i < PACKET_LEN; i++) {
            #pragma HLS PIPELINE II=1
            x = rx[cu].read();
            tx[channel].write(x);  // 动态路由到 tx0-tx3
        }
    }
}

与 sender 的关键区别：

  • Sender 是 1→100 广播：同一个输入流被拆分到 100 个输出
  • Receiver 是 100→4 汇聚：100 个输入流按 packet ID 分类到 4 个输出

数据流追踪：端到端视角

让我们追踪一组粒子数据从主机内存出发，经过完整计算，最终回到主机的全过程：

Phase 1: 主机准备数据

// host 代码示意
for (int cu = 0; cu < 100; cu++) {
    for (int p = 0; p < 4; p++) {
        for (int i = 0; i < 224; i++) {
            input_buffer[cu*4*224 + p*224 + i] = particle_data[cu][p][i];
        }
    }
}
// 启动 DMA 传输
mm2s_mp_0.start(input_buffer, BURST_SIZE);

数据在 DDR 中按 CU-major order 排列：先排满 CU0 的 4 个 packet，再 CU1，以此类推。

Phase 2: DMA 读取与分发

Phase 3: AIE 计算

每个 AIE CU 执行以下计算：

// AIE kernel 伪代码
void nbody_kernel(input_stream in_i,      // 本粒子数据
                  input_stream in_j,      // 所有粒子位置（共享）
                  output_stream out_i) {  // 计算结果
    particle me = read(in_i);
    for (int step = 0; step < 4; step++) {
        for (int other = 0; other < 100; other++) {
            particle them = read(in_j, other);
            force += compute_gravity(me, them);
        }
        me.velocity += force * dt;
        me.position += me.velocity * dt;
    }
    write(out_i, me);
}

设计权衡与决策分析

权衡 1：100 路并行 vs. 时分复用

问题：为什么要 100 个物理端口，而不是用更少的端口时分复用？

选择的方案：100 路并行（fan-out/fan-in）

理由：

  • 吞吐量需求：N-Body 计算是 memory-bound，需要最大化数据并行度
  • AIE 架构：AIE 阵列天然支持大量并行 stream 接口
  • 简化同步：每路流独立，无需复杂的时分调度逻辑

代价：

  • PL 端需要大量的 switch-case 逻辑（每个核 100 个 case）
  • 布线资源消耗增加
  • 时序收敛挑战（长组合逻辑链）

权衡 2：固定路由 vs. 动态路由

问题：数据包的路由是在编译时确定还是运行时确定？

选择的方案：混合模式

  • Sender：固定路由（CU index 决定目标端口）
  • Receiver：半动态路由（根据 packet ID 查表决定输出通道）

理由：

  • Sender 侧数据顺序是确定的（按 CU 顺序），固定路由更简单高效
  • Receiver 侧需要按 packet ID 重组，查表路由提供灵活性

权衡 3：Packetization 开销

问题：每个 packet 增加 1 个 header 字（224→225），约 0.45% 开销，是否值得？

选择的方案：保留 header

理由：

  • 路由必需：AIE packet switching 网络需要 header 进行路由决策
  • 同步点：header 作为 transaction boundary，便于调试和错误检测
  • 可扩展性：未来可扩展更多元数据（timestamp、sequence number 等）

权衡 4：PL 端数据处理策略

观察到的代码模式：

switch(cu) {
    case 0: tx0.write(tmp); break;
    case 1: tx1.write(tmp); break;
    ...
    case 99: tx99.write(tmp); break;
}

为什么不使用数组索引？

// 更简洁的写法，但 HLS 综合结果可能不同
tx[cu].write(tmp);

原因：

  • HLS 对数组索引的综合可能是顺序访问或需要额外的仲裁逻辑
  • 显式的 switch-case 确保 HLS 生成并行的多路选择器
  • 每个 case 分支可以独立优化（如不同的 FIFO depth）

隐式契约

  1. 数据顺序：Host 必须按 CU-major order 填充输入 buffer，否则 packet 内容会错乱
  2. Packet ID 一致性：packet_ids_c.h 必须在 PL 和 AIE 之间保持一致，由 AIE 编译器生成
  3. Buffer 对齐：DMA 缓冲区通常需要 4KB 对齐以满足 AXI 性能要求

新贡献者注意事项

常见陷阱

1. Packet ID 不匹配

症状：仿真通过，硬件运行结果错误

原因：修改了 AIE graph 的 connectivity，导致 packet_ids_c.h 重新生成，但 PL 代码未重新编译

解决：确保 make clean 后全量重建，或手动同步 packet ID 定义

2. Stream 深度不足

症状：仿真挂死或数据丢失

原因：默认的 hls::stream depth 可能不足以缓冲 burst 数据

解决：在 HLS 中为关键 stream 指定足够深度：

#pragma HLS stream variable=tx0 depth=256

3. TLAST 信号处理

症状：DMA 传输不完整或重复

原因：packet_sender 中 x.last 只在 packet 最后一个字设置，但某些 DMA 配置需要 frame-level TLAST

检查点：确认 PACKET_LEN 与实际数据量匹配，且 TLAST 出现在正确的位置

4. 并发访问竞争

症状：非确定性错误，偶尔出现数据错乱

原因：in_j 是广播端口，多个 CU 同时读取需要 AIE 内部正确仲裁

验证：检查 AIE graph 中 in_j 的连接方式是否为 broadcast 而非 point-to-point

扩展指南

如果需要修改计算单元数量（如从 100 改为 50）：

  1. 修改 packet_sender.hpp 和 packet_receiver.hpp 中的 NUM_CU
  2. 更新 conn.cfg 中的 stream_connect 行数
  3. 重新生成 AIE graph，更新 packet_ids_c.h
  4. 调整 Host 代码中的 buffer 分配逻辑
