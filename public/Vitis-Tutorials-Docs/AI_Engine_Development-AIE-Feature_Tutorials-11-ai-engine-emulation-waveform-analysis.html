Emulation Waveform Analysis Pipeline 技术深度解析

概述：这个模块解决了什么问题？

想象一下，你正在调试一个复杂的异构系统——里面有 CPU、FPGA 可编程逻辑（PL）、AI 引擎阵列，还有连接它们的片上网络（NoC）。当数据流出现问题时，你可能会问："数据到底走到哪里去了？" "是 PL 没发出数据，还是 AI 引擎没收到？" "RTP 参数更新到底生效了没有？"

emulation_waveform_analysis_pipeline 模块正是为了解决这些调试难题而设计的。它提供了一个完整的硬件仿真波形分析环境，让开发者能够在没有实际硬件的情况下，通过详细的波形视图来观察和理解 Versal 自适应 SoC 中各个组件之间的数据交互。

这个模块的核心价值在于：

  1. 可见性：在硬件仿真环境中提供近乎真实的信号级可视能力
  2. 可调试性：通过 Transaction Level Modeling (TLM) 捕获高层事务信息
  3. 教学性：作为教程设计，展示了如何搭建一个可观测的 AIE-PL 协同系统

心智模型：把它想象成什么？

把这个系统想象成一个交响乐团的总谱分析室：

  • random_noise（PL 噪声源） = 第一小提琴声部，持续产生音符（数据）
  • FIR24 滤波器（AIE 内核） = 指挥家，实时调整演奏风格（滤波系数）
  • s2mm（PL 数据收集器） = 录音设备，把最终演奏保存下来
  • 波形查看器 = 总谱分析师的工作台，可以同时看到所有乐手的演奏细节

关键洞察：这不是一个简单的数据处理流水线，而是一个为调试和教学而设计的"可观测系统"。每一个组件都被精心选择，以展示特定的调试场景：

  • PL → AIE 的数据流传输
  • 异步 RTP（Run-Time Parameter）参数更新
  • AIE → DDR 内存的数据回写

数据流详解

整个系统的数据流动遵循一条清晰的单向路径：

  1. 数据源（PL）：random_noise 内核生成复数随机噪声样本，通过 AXI-Stream 接口输出
  2. 数据处理（AIE）：fir24_sym 内核对输入数据进行 24 抽头对称 FIR 滤波
  3. 数据汇聚（PL）：s2mm 内核将 AIE 输出的流数据写入 DDR 内存
  4. 主机验证（PS）：host.exe 读取 DDR 中的结果并与黄金参考数据比对

与此同时，还存在一条控制流：

  • 主机通过 RTP（Run-Time Parameter）接口动态更新 FIR 滤波器的系数
  • 支持读回机制验证参数是否成功写入

核心组件剖析

1. random_noise —— 可控噪声源

这是一个 HLS 实现的 PL 内核，其核心职责是为系统提供可重复的测试数据。

extern "C" void random_noise(hls::stream<std::complex<short> > & out, int size) {
#pragma HLS INTERFACE axis port=out
#pragma HLS INTERFACE s_axilite port=return bundle=control
#pragma HLS INTERFACE ap_ctrl_hs port=return
#pragma HLS interface s_axilite port=size bundle=control
#pragma HLS interface ap_none port=size
    for(int i=0;i<size;i++){
        std::complex<short> sample;
        sample.real((my_rand() % 256) - 127);
        sample.imag((my_rand() % 256) - 127);
        out.write(sample);
    }
}

设计要点：

  • 使用 hls::stream 实现 AXI-Stream 接口，确保与 AIE 的无缝连接
  • s_axilite 接口用于标量控制（size 参数），允许主机动态配置输出样本数
  • 自定义 LCG（线性同余发生器）伪随机数生成器，保证跨平台结果一致性
  • 输出范围控制在 [-127, 128]，适合定点 DSP 处理

2. fir24_sym —— 自适应 FIR 滤波器

这是系统的计算核心，运行在 AI 引擎阵列上。它是一个 24 抽头对称 FIR 滤波器，支持运行时系数更新。

void fir24_sym(
    input_buffer<cint16,extents<64>,margin<24>> &iwin,
    output_buffer<cint16,extents<64>> &owin,
    const int32 (&coeffs)[12],
    int32 (&coeffs_readback)[12]
)

关键设计决策：

对称性利用：由于滤波器是对称的，只需要传递 12 个系数而非 24 个，减少 RTP 带宽需求。

向量化处理：代码大量使用 aie::vector 和 aie::sliding_mul_sym，充分利用 AIE 的 SIMD 能力：

  • 每次循环处理 16 个样本（4 次向量操作 × 4 元素/向量）
  • 使用 sliding_mul_sym 指令高效实现滑动窗口相关运算

双缓冲策略：xbuff 和 ybuff 交替使用，隐藏数据加载延迟：

// xbuff 存储当前处理窗口的前半部分
// ybuff 存储后半部分，同时预取下一批数据
xbuff.insert(0, ybuff.extract<4>(1));  // 数据复用
ybuff.insert(0, isamples);             // 新数据注入

3. s2mm —— 流数据到内存的桥梁

这个 PL 内核负责将 AIE 输出的流数据写入 DDR 内存，是数据流的终点。

void s2mm(ap_int<32>* mem, hls::stream<ap_axis<32, 0, 0, 0>>& s, int size) {
#pragma HLS INTERFACE m_axi port=mem offset=slave bundle=gmem
#pragma HLS interface axis port=s
#pragma HLS INTERFACE s_axilite port=mem bundle=control
#pragma HLS INTERFACE s_axilite port=size bundle=control
#pragma HLS interface s_axilite port=return bundle=control
    for(int i = 0; i < size; i++) {
#pragma HLS PIPELINE II=1
        ap_axis<32, 0, 0, 0> x = s.read();
        mem[i] = x.data;
    }
}

性能特征：

  • II=1 的流水线设计，每个时钟周期处理一个样本
  • m_axi 接口支持突发传输，最大化 DDR 带宽利用率
  • 使用 ap_axis 类型明确声明 AXI-Stream 协议，包含 TDATA/TVALID/TREADY 信号

4. system.cfg —— 系统集成蓝图

配置文件定义了内核实例化和连接关系：

[connectivity]
nk=s2mm:1:s2mm_1
nk=random_noise:1:random_noise_1
stream_connect=ai_engine_0.Dataout0:s2mm_1.s
stream_connect=random_noise_1.out:ai_engine_0.Datain0
[advanced]
param=hw_emu.enableProfiling=false
param=compiler.addOutputTypes=hw_export

关键配置解读：

  • nk（num kernels）：声明每个内核的实例数量及实例名
  • stream_connect：建立 AXI-Stream 连接，格式为 source:destination
  • hw_emu.enableProfiling=false：在硬件仿真中禁用性能分析（避免干扰波形纯净度）

TCL 脚本：波形分析的利器

本模块提供了一组精心设计的 TCL 脚本，用于在 Vivado XSIM 中快速定位感兴趣的信号：

add_waveforms.tcl —— 全局视角

添加所有关键信号组：CIPS（处理器）、NOISE/S2MM（PL 内核）、DDR4（内存）、AIENGINE（AIE 阵列）。

pl_to_aie.tcl —— PL 到 AIE 的数据通路

聚焦于 random_noise → ai_engine_0 的数据流，用于调试输入侧问题。

aie_to_ddr.tcl —— AIE 到 DDR 的数据通路

聚焦于 ai_engine_0 → s2mm → noc_ddr4 的数据流，用于调试输出侧问题。

rtp_signals.tcl —— RTP 参数更新通路

专门用于观察 Run-Time Parameter 的读写事务，这是 AIE 动态重配置的关键机制。

bootup_signals.tcl —— 启动序列验证

验证 PMC（Platform Management Controller）配置流程，是排查启动问题的首选工具。

设计权衡与决策分析

1. 为什么选择 FIR 滤波器作为示例？

替代方案考虑：

  • 矩阵乘法：计算密集但缺乏动态参数更新的教学价值
  • FFT：过于复杂，会分散对调试技术的注意力
  • 简单加法器：太简单，无法展示 AIE 的向量处理能力

FIR 的优势：

  • 计算模式典型（乘累加），易于理解
  • 天然支持系数更新，完美展示 RTP 机制
  • 结果可验证（与黄金参考数据比对）

2. 为什么采用 PL 生成数据而非 DDR 读取？

权衡分析：

  • PL 生成：简化系统，减少 NoC/DDR 流量干扰，专注于 AIE-PL 接口调试
  • DDR 读取：更接近真实应用场景，但会增加波形复杂度

决策理由：这是一个调试教学系统，不是生产系统。简化数据路径有助于学习者聚焦核心概念。

3. 为什么使用同步 RTP 更新？

在 host.cpp 中可以看到：

ghdl.update("gr.fir24.in[1]", narrow_filter);
ghdl.run(16);
ghdl.wait();  // 等待完成后再进行下一步操作

这种同步模式确保了：

  • 参数更新与数据处理的确定性顺序
  • 便于波形分析时关联特定时间点的行为
  • 简化了调试过程（避免竞态条件的不确定性）

代价是牺牲了一部分灵活性，但对于教学目的而言是可接受的。

新贡献者必读：陷阱与注意事项

1. 波形查看的时机至关重要

关键规则：必须在仿真启动前添加所有需要的信号！

# 正确做法：在 run 之前执行
source add_waveforms.tcl
run all

# 错误做法：暂停后添加新信号不会有历史数据
run 100us
add_wave ...  # 新信号从这里开始才有数据

2. TLM 信号的解读

波形中看到的宽色块不代表单个事务——那是微秒级的时间尺度压缩效果。必须放大到纳秒级才能看到实际的事务序列。

3. RTP 读回的时序陷阱

// 这段代码有微妙的时序含义
ghdl.update(...);      // 发起异步更新
ghdl.run(16);
ghdl.read(...);        // 立即读回——可能读到旧值！
ghdl.wait();
ghdl.read(...);        // wait 后读回——保证是新值

第一个 read() 可能在 AIE 内核实际应用新系数之前就执行了，因此返回的是旧值。这是预期行为，不是 bug。

4. 随机数种子的确定性

random_noise 使用固定的种子初始化（next = 1），这意味着每次仿真产生的数据序列是相同的。如果需要不同的测试数据，需要修改种子值或实现外部种子注入机制。

5. 内存对齐要求

s2mm 内核假设 DDR 地址是 4 字节对齐的（ap_int<32> 访问）。如果主机分配的缓冲区未对齐，可能导致总线错误或性能下降。

  • debug_walkthrough_pl_data_movers：更基础的 PL 数据搬运调试教程

总结

emulation_waveform_analysis_pipeline 是一个精心设计的教学型调试系统。它的价值不在于处理多么复杂的算法，而在于展示了一个完整的异构系统是如何被构建、配置和调试的。

对于新加入团队的工程师，建议按照以下顺序学习：

  1. 先阅读 README.md 了解整体流程
  2. 研究 system.cfg 理解系统连接关系
  3. 逐个分析 PL 内核（random_noise、s2mm）的 HLS 代码
  4. 深入理解 AIE 内核的向量化技巧
  5. 实际运行仿真，配合 TCL 脚本观察波形
  6. 尝试修改滤波器系数或数据长度，观察系统行为变化

通过这种方式，你将不仅学会如何使用这个特定的系统，更能掌握调试 Versal 异构系统的通用方法论。
