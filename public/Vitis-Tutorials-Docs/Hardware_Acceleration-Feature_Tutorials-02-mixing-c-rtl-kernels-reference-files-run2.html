run2_mixed_c_rtl_kernel_integration_configuration 模块深度解析

一句话概括

这个模块是 Vitis 硬件加速开发流程中的系统链接配置层，它解决了一个核心问题：如何将两种完全不同设计范式的内核（HLS C++ 内核与 RTL 内核）无缝集成到同一个 FPGA 比特流中，并让主机代码以完全一致的 API 调用它们。想象你是一位乐团指挥，左手边是古典乐手（HLS C++ 内核，遵循标准化的编程模型），右手边是即兴爵士乐手（RTL 内核，拥有完全自定义的硬件接口），而 run2.cfg 就是那份让两者和谐演奏的总谱。

问题空间：为什么需要这个模块？

在异构计算系统中，FPGA 加速的核心挑战从来不是"写一个能工作的内核"，而是如何让多个内核协同工作。具体来说：

1. 设计范式的鸿沟

维度 HLS C++ 内核 (krnl_vadd) RTL 内核 (rtl_kernel_wizard_0)
设计入口 C/C++ 高级综合 Verilog/VHDL 手工设计
接口抽象 编译器自动生成 AXI 接口 设计者完全控制端口定义
内存模型 m_axi/s_axilite 自动推断 需显式声明 AXI4/AXI-Lite/AXIS
打包方式 v++ -c 直接生成 .xo RTL Kernel Wizard 或手动打包
仿真支持 原生支持软件仿真 需提供 C++ 行为模型才能软件仿真

如果没有统一的链接配置，这两种内核就像说不同语言的两个人——无法协作。

2. 数据流的连续性需求

本教程的核心场景是一个流水线：C++ 内核先执行向量加法，将结果写入 DDR；RTL 内核紧接着读取该结果，对每个元素执行 +1 操作。关键约束是：

Host → [DDR] → krnl_vadd (C++) → [DDR] → rtl_kernel_wizard_0 (RTL) → [DDR] → Host
              ↑_____________________________________________________↓
                              零拷贝数据传递

两个内核必须共享同一个物理缓冲区 (boOut)，且主机代码不应在中间介入数据搬运。

3. 为什么简单的方案行不通？

朴素方案：分别编译两个内核，然后在主机代码里顺序调用。

问题：

  • Vitis 链接器 (v++ -l) 需要知道所有内核的存在才能分配 SLR、规划 NoC/DDR 连接、解决资源冲突
  • 没有统一的 xclbin，XRT 无法同时识别两个内核的元数据
  • 内存端口的 bank 分配可能冲突（例如两个内核都声称独占 gmem0）

核心组件职责

1. run2.cfg —— 系统链接的"总谱"

这是整个模块的灵魂所在。它是一个 INI 格式的配置文件，被 v++ -l（链接阶段）消费：

[connectivity]
nk=krnl_vadd:1:krnl_vadd_1
nk=rtl_kernel_wizard_0:1:rtl_kernel_wizard_0_1

nk 指令的语义：nk=<kernel_name>:<num_instances>:<instance_name_1>:...:<instance_name_N>

  • krnl_vadd:1:krnl_vadd_1：从 krnl_vadd.xo 中提取内核定义，实例化 1 个 CU（Compute Unit），命名为 krnl_vadd_1
  • rtl_kernel_wizard_0:1:rtl_kernel_wizard_0_1：同理，为 RTL 内核创建一个名为 rtl_kernel_wizard_0_1 的实例

为什么需要显式命名？ 因为主机代码通过 xrt::kernel(device, uuid, "rtl_kernel_wizard_0") 查找内核时，实际匹配的是内核类型名，而非实例名。但在性能分析、调试和底层 XRT 调用中，实例名用于区分同一类型的多个 CU。

2. xrt-host_step2.cpp —— 统一访问抽象

这是体现 Vitis "一次编写，到处运行" 哲学的关键代码：

// 获取 C++ 内核对象
auto krnl = xrt::kernel(device, uuid, "krnl_vadd");
// 获取 RTL 内核对象 —— 完全相同的 API！
auto krnlRTL = xrt::kernel(device, uuid, "rtl_kernel_wizard_0");

// 分配缓冲区（注意：使用 krnl 的 group_id，但 buffer 可被 krnlRTL 复用）
auto boOut = xrt::bo(device, vector_size_bytes, krnl.group_id(2));

// 顺序执行，数据在 DDR 中零拷贝传递
auto run1 = krnl(boIn1, boIn2, boOut, DATA_SIZE);
run1.wait();
auto run2 = krnlRTL(boOut);  // RTL 内核直接读取 C++ 内核的输出
run2.wait();

关键洞察：krnl.group_id(2) 返回的是全局内存 bank 的标识符。虽然这里用的是 C++ 内核的第 3 个参数（输出端口 c）的 bank 信息，但由于两个内核在链接时被配置为访问相同的 DDR bank，这个 buffer object 对两者都是有效的。

3. Makefile 的条件分支逻辑

ifeq (\((LAB),\)(filter $(LAB),run1))
   EXECUTABLE := host_1
   LDCLFLAGS := -t \((TARGET) --platform \)(DEVICE) --config run1.cfg
else
   EXECUTABLE := host_2
   LDCLFLAGS := -t \((TARGET) --platform \)(DEVICE) --config run2.cfg
endif

以及链接规则的关键差异：

\((XCLBIN): \)(XO) 
ifeq (\((LAB),\)(filter $(LAB),run1))
        v++ \((LDCLFLAGS) -l -o'\)@' $(+)          # 仅链接 C++ 内核
else
        v++ \((LDCLFLAGS) -l -o'\)@' \((+) \)(RTL_KRNL)  # 链接 C++ + RTL 内核
endif

这展示了 Vitis 构建系统的增量扩展模式：从单内核 (run1) 到多内核混合 (run2)，只需修改配置文件和链接输入列表，无需改变工具链或主机 API。

数据流深度追踪

让我们跟随一个数据元素走完整个处理流程：

Step 1: 主机初始化与内存分配

size_t vector_size_bytes = sizeof(int) * DATA_SIZE;  // 4096 * 4 = 16KB
auto boIn1 = xrt::bo(device, vector_size_bytes, krnl.group_id(0));
auto boIn2 = xrt::bo(device, vector_size_bytes, krnl.group_id(1));
auto boOut = xrt::bo(device, vector_size_bytes, krnl.group_id(2));

幕后发生了什么？

  1. XRT 向 Linux 内核驱动申请物理连续的 DMA 缓冲区（或通过 IOMMU 映射的 scatter-gather 列表）
  2. 根据 krnl.group_id(N) 返回的内存组 ID，确定该 buffer 应绑定到哪个 DDR/HBM bank
  3. 返回的 xrt::bo 对象持有文件描述符和虚拟地址映射

Step 2: 主机数据填充与下行同步

auto bo0_map = boIn1.map<int*>();  // mmap() 获得用户空间指针
for (int i = 0; i < DATA_SIZE; ++i) {
    bo0_map[i] = i;
    bo1_map[i] = i;
    bufReference[i] = bo0_map[i] + bo1_map[i] + 1;  // 预期结果：i + i + 1
}
boIn1.sync(XCL_BO_SYNC_BO_TO_DEVICE);
boIn2.sync(XCL_BO_SYNC_BO_TO_DEVICE);

注意 bufReference 的计算：+ 1 是因为 RTL 内核会对每个元素加 1。这是验证端到端正确性的黄金标准。

Step 3: C++ 内核执行（HLS 生成硬件）

auto run1 = krnl(boIn1, boIn2, boOut, DATA_SIZE);
run1.wait();

krnl_vadd 的内部流水线（来自 krnl_vadd.cpp）：

// 分块处理，每块 BUFFER_SIZE=256 个元素
for (int i = 0 ; i < n_elements ; i += BUFFER_SIZE) {
    // 1. 从 DDR burst 读取输入 A 和 B 到本地 BRAM
    readA: for (int j = 0 ; j < size ; j++) {
        #pragma HLS pipeline ii = 1 rewind
        arrayA[j] = a[i+j];
        arrayB[j] = b[i+j];
    }
    
    // 2. 执行加法并 burst 写回 DDR
    vadd_wrteC: for (int j = 0 ; j < size ; j++){
        #pragma HLS pipeline ii = 1 rewind
        c[i+j] = arrayA[j] + arrayB[j];
    }
}

HLS 优化要点：

  • max_read_burst_length = 256 / max_write_burst_length = 256：最大化 AXI burst 长度，摊销地址通道开销
  • ii = 1：目标启动间隔为 1 周期，即每个周期处理一个元素
  • rewind：启用循环展开后的流水线连续执行，消除循环边界气泡

Step 4: RTL 内核执行（手工设计硬件）

auto run2 = krnlRTL(boOut); 
run2.wait();

RTL 内核 rtl_kernel_wizard_0 是一个预打包的 .xo 文件，其行为是：

  • 接收一个指向 DDR buffer 的 AXI4 主设备接口
  • 遍历 buffer，对每个 32-bit 整数执行 +1 操作
  • 写回原地址

关键观察：主机代码传入的是同一个 boOut，它在 DDR 中的物理地址对 RTL 内核可见。这就是零拷贝数据传递——没有额外的 memcpy，没有主机介入。

Step 5: 结果回传与验证

boOut.sync(XCL_BO_SYNC_BO_FROM_DEVICE);
if (std::memcmp(bo2_map, bufReference, vector_size_bytes))
    throw std::runtime_error("Value read back does not match reference");

设计决策与权衡

决策 1：显式内核实例命名 vs. 自动生成

选择：run2.cfg 中使用 nk=krnl_vadd:1:krnl_vadd_1 显式指定实例名

替代方案：依赖 v++ 自动生成实例名（如 krnl_vadd_0）

权衡分析：

  • 显式命名：增加配置复杂度，但确保 Makefile、主机代码、性能分析报告中的名称一致；便于调试和多实例场景下的精确控制
  • 自动生成：配置更简单，但任何工具链版本变更导致的命名规则变化都会破坏下游脚本

为何当前选择最优：这是一个教学示例，显式命名让学习者清楚看到"内核类型"与"硬件实例"的区别。

决策 2：共享 DDR Bank vs. 流式连接（Stream）

选择：两个内核通过 DDR 共享数据（boOut 作为中间缓冲区）

替代方案：使用 hls::stream 或 AXI4-Stream 直接连接两个内核，绕过 DDR

权衡分析：

  • DDR 共享：延迟更高（需两次 DDR 访问），但耦合松散，内核可独立开发、测试和复用；符合本教程演示"混合语言内核"的核心目标
  • 流式直连：延迟极低，但需要两个内核同时在线协商流控（ready/valid 握手），且 RTL 内核必须暴露 AXI-S 接口而非 AXI-Full

为何当前选择最优：RTL 内核 rtl_kernel_wizard_0 设计为内存映射接口（AXI4 Full），改造为流式会改变其基本架构。本教程的目标是展示最小侵入性的集成方式。

决策 3：单一配置文件 vs. 分层配置

选择：所有链接配置集中在 run2.cfg

替代方案：拆分为 kernels.cfg（内核列表）、connectivity.cfg（连接关系）、memory.cfg（bank 分配）

权衡分析：

  • 单一文件：简单直观，适合小型项目；但随着内核数量增长会变得臃肿
  • 分层配置：模块化更好，支持条件包含（#include），适合大型 SoC 设计

扩展建议：当您的项目超过 5-6 个内核时，考虑使用 v++ --config 的多文件支持：

v++ -l --config common.cfg --config kernel_list.cfg --config memory_map.cfg ...

决策 4：HLS 内核的 BUFFER_SIZE=256 选择

选择：使用 256 元素的本地缓冲区分块处理

替代方案：一次性处理全部 4096 元素；或更小的分块（如 64）

权衡分析：

  • 256 元素：占用 1KB BRAM（256 × 4 bytes），在大多数 Xilinx 器件上是高效的；burst 长度 256 接近 AXI4 最大 burst 长度，最大化带宽利用率
  • 4096 元素：需要 16KB BRAM，对于大向量可能超出小器件容量；且更大的 buffer 不会显著提升性能，因为 DDR burst 效率已饱和
  • 64 元素：BRAM 占用更少，但更多循环迭代次数意味着更多控制开销和潜在流水线气泡

深层原理：这是典型的** Roofline 模型**考量——当计算强度（每字节数据的运算量）很低时（向量加法只有 1 FLOP/4 bytes = 0.25），性能由内存带宽决定。此时优化目标是最大化 DRAM burst 效率，而非减少本地 buffer。

新贡献者必读：陷阱与最佳实践

陷阱 1：Bank 分配不匹配

症状：运行时出现数据损坏，或 xrt::bo 构造失败

根本原因：两个内核默认可能被分配到不同的 DDR bank。如果 krnl_vadd 写入 bank 0，而 RTL 内核从 bank 1 读取，它们看到的是不同的物理内存。

排查方法：检查 v++ -l 生成的 _link/vivado.log 或 .xclbin.info 文件，确认两个内核的 AXI 主接口连接到相同的 sp（slave port）。

修复：在 run2.cfg 中显式指定 bank：

[connectivity]
sp=krnl_vadd_1.c:bank0
sp=rtl_kernel_wizard_0_1.m_axi_gmem:bank0

陷阱 2：RTL 内核的软件仿真缺失

症状：make all TARGET=sw_emu LAB=run2 编译成功，但运行时崩溃或结果错误

根本原因：RTL 内核的 .xo 文件通常只包含硬件网表。要在软件仿真中运行，必须附带一个 C++ 行为模型（通常由 RTL Kernel Wizard 生成）。

解决方案：确保 RTL 内核打包时包含了软件仿真模型。检查 rtl_kernel_wizard_0.xo 的内容：

unzip -l rtl_kernel_wizard_0.xo | grep -E "\.(cpp|c)$"

应有类似 swEmu/rtl_kernel_wizard_0.cpp 的文件存在。

陷阱 3：group_id 的隐式契约

危险代码模式：

// 假设 krnl_vadd 和 rtl_kernel_wizard_0 的 group_id(0) 指向同一 bank
auto boShared = xrt::bo(device, size, krnl.group_id(0));
// ... 传递给 krnlRTL ...

风险：虽然本例中恰好成立（两个内核的第一个参数都绑定到默认 bank），但这是实现细节依赖。正确的做法是显式查询或使用配置文件强制 bank 对齐。

陷阱 4：nk 指令的实例数与名称数不匹配

错误示例：

nk=krnl_vadd:2:krnl_vadd_1  # 声明 2 个实例，但只给 1 个名称

这将导致链接错误。正确的多实例声明：

nk=krnl_vadd:2:krnl_vadd_1:krnl_vadd_2

最佳实践 1：版本控制配置文件

run2.cfg 是构建系统的关键输入，应与源代码同等对待：

git add run2.cfg
# 不要忽略 .cfg 文件！

最佳实践 2：使用 profile 段进行性能基线建立

[profile]
data=all:all:all

这会启用所有内核、所有端口、所有内存事务的性能计数器。在硬件仿真和实际硬件运行后，使用 vitis -a xrt.run_summary 分析 Timeline Trace，验证：

  • 两个内核是否真正并行执行（或至少流水线重叠）
  • DDR 带宽利用率是否接近理论峰值
  • 是否存在不必要的 stalls

最佳实践 3：渐进式验证

不要直接跳到 run2。遵循教程的设计意图：

  1. Step 1 (run1)：验证单 C++ 内核工作正常，建立性能基线
  2. Step 2 (run2)：引入 RTL 内核，对比 Timeline Trace 的变化
  3. Step 3 (自定义)：修改数据大小、尝试不同的 bank 分配策略
