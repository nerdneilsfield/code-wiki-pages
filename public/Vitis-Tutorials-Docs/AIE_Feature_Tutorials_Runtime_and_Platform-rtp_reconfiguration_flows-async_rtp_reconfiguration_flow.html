异步标量 RTP 重配置流程 (Async RTP Reconfiguration Flow)

一句话概括

这个模块展示了如何在 AI Engine 图执行期间动态地、非阻塞地修改内核参数——就像在高速行驶的列车上更换引擎的燃料配比，而不需要停车。这是 Versal ACAP 平台的核心能力之一，让硬件加速具备了软件般的灵活性。

问题空间：为什么需要运行时参数重配置？

在传统的 FPGA/ASIC 设计中，算法参数（如滤波器系数、增益值、阈值等）通常是编译时固定的。这意味着：

  1. 静态配置的局限：如果你想改变一个 FIR 滤波器的截止频率，必须重新综合、重新布局布线、重新生成比特流——这个过程可能需要数小时。
  2. 多场景适配困难：同一个加速器需要在不同工作模式下运行（例如通信系统中的不同调制方式），静态设计意味着你需要为每种模式准备一个独立的比特流。
  3. 自适应算法的障碍：现代信号处理经常需要根据输入数据动态调整参数（如 AGC 自动增益控制），这在静态设计中几乎不可能实现。

解决方案的演进

方案 优点 缺点
完全静态配置 资源最少、时序最确定 无灵活性
通过 BRAM/URAM 查表 可切换预存配置 需要额外的存储和切换逻辑
RTP (Runtime Parameter) 真正的动态更新、细粒度控制 需要额外的控制路径和同步考虑

RTP 的核心价值：它提供了一条从处理器（PS）到 AI Engine 内核的控制通道，让你可以在纳秒级时间尺度上修改内核行为，而主数据通路保持全速运行。

心智模型：把 RTP 想象成什么？

类比：赛车比赛中的维修区无线电

想象 F1 赛车比赛：

  • 数据流 = 赛车在赛道上的高速行驶（AI Engine 内核处理数据）
  • RTP 端口 = 车队通过无线电给车手发送战术指令（修改增益、策略等）
  • 同步 RTP = 必须在特定圈数进站时才能接收新指令（阻塞式）
  • 异步 RTP = 车手边开车边听无线电，随时可以使用最新指令（非阻塞式）

这个模块展示的是异步 RTP——内核在每个迭代都会读取最新的 RTP 值，但不需要与 RTP 更新操作同步。

核心抽象

┌─────────────────────────────────────────────────────────────┐
│                      主机程序 (Host)                          │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────────┐  │
│  │ datagen     │───→│  AI Engine  │───→│ s2mm            │  │
│  │ (数据生成)   │    │  vect_add   │    │ (数据写出)       │  │
│  └─────────────┘    └─────────────┘    └─────────────────┘  │
│                            ↑                                │
│                            │ async RTP update                │
│                       ┌────┴────┐                           │
│                       │ value=10 │ (运行时动态更新)           │
│                       └─────────┘                           │
└─────────────────────────────────────────────────────────────┘

关键理解点：

  1. 数据流和控制流分离：数据从 datagen → AI Engine → s2mm 流动，而 RTP 更新走独立的控制路径
  2. 异步语义：ghdl.update() 调用不会阻塞等待内核完成当前迭代，而是立即返回
  3. 最终一致性：内核会在下一次读取 RTP 端口时看到新值，但不保证精确的时序对齐

组件职责

组件 类型 职责
datagen HLS Kernel 生成测试数据序列，支持初始值偏移 (init_val)
vect_add AIE Kernel 执行向量加法，将输入数据加上 RTP 传入的 value

数据流与控制流追踪

完整执行时序

时间轴 ──────────────────────────────────────────────────────────→

Host 侧:
  ├─→ xrt::device(0)              [打开设备]
  ├─→ load_xclbin()               [加载比特流]
  ├─→ xrt::kernel(s2mm/datagen)   [创建 kernel 句柄]
  ├─→ xrt::bo()                   [分配输出缓冲区]
  │
  ├─→ s2mm_run = s2mm(out_bo, ...)   [启动 s2mm (非阻塞)]
  ├─→ datagen_run = datagen(...)     [启动 datagen (非阻塞)]
  │
  ├─→ ghdl = xrt::graph(device,...)   [获取 graph 句柄]
  ├─→ ghdl.run(ITERATION)             [启动 AIE graph，运行 4 次迭代]
  ├─→ ghdl.update("gr.k.in[1]", 10)   [★ 异步更新 RTP 值为 10]
  ├─→ ghdl.end()                      [结束 graph 执行]
  │
  ├─→ s2mm_run.wait()                 [等待 DMA 完成]
  └─→ out_bo.sync(FROM_DEVICE)        [同步数据回主机]

AIE 侧 (每个迭代):
  ├─→ 从 in[0] 读取 256 个 int32 数据
  ├─→ 从 in[1] (RTP) 读取 value (=10)
  ├─→ vresult = vdata + broadcast(value)
  └─→ 写入 out[0]

关键观察

  1. 启动顺序的重要性：s2mm 和 datagen 必须先启动，因为它们会立即开始消费/生产数据。如果 AIE graph 先启动而没有数据源，会导致死锁。

  2. 异步更新的时机：ghdl.update() 在 ghdl.run() 之后调用，这是异步 RTP 的典型用法——图已经在运行，我们在途中注入新的参数值。

  3. 单一更新 vs 多次迭代：本例中只调用了一次 update，但 graph 运行了 4 次迭代。这意味着后 3 次迭代都复用了相同的 RTP 值 (10)。

代码深度解析

AIE Graph 定义 (aie/graph.h)

class rtpgraph: public adf::graph
{
private:
  adf::kernel k;

public:
  adf::output_plio out;
  adf::input_plio in; 
  adf::port<adf::direction::in> value;  // ★ RTP 端口声明

  rtpgraph()
  {
    k = adf::kernel::create(vect_add<256>);
    out = adf::output_plio::create("Dataout0", adf::plio_32_bits, "data/output0.txt");
    in = adf::input_plio::create("Datain0", adf::plio_32_bits, "data/input0.csv");
    
    // ★ 关键：使用 async() 包装 RTP 连接
    adf::connect<adf::parameter>(value, adf::async(k.in[1]));

    adf::connect< >(k.out[0], out.in[0]);
    adf::connect< >(in.out[0], k.in[0]);
    adf::source(k) = "vect_add.cc";
    adf::runtime<adf::ratio>(k)= 0.9;
  };
};

设计要点：

  • adf::async(k.in[1]) 明确指定了异步 RTP 语义。如果不加 async()，默认是同步 RTP，要求每次迭代都必须有对应的 update() 调用。
  • k.in[1] 对应 vect_add 函数的第三个参数 int32 value（前两个是输入/输出缓冲区）。

AIE Kernel 实现 (aie/vect_add.cc)

template<int NUM>
void vect_add(input_buffer<int32,extents<NUM>>& __restrict in,
              output_buffer<int32,extents<NUM>>& __restrict out,
              int32 value)  // ★ RTP 参数
{
    auto inIter=aie::begin_vector<ELEM>(in);
    auto outIter=aie::begin_vector<ELEM>(out);
    aie::vector<int32,ELEM> vv=aie::broadcast<int32>(value);  // 广播 RTP 值
    
    for(int i=0;i<NUM/ELEM;i++)
    chess_prepare_for_pipelining
    {
        aie::vector<int32,ELEM> vdata=*inIter++;
        aie::vector<int32,ELEM> vresult=aie::add(vdata,vv);  // 向量加法
        *outIter++=vresult;
    }
}

性能特征：

  • 使用 aie::vector 和 aie::add 利用 AIE 的 SIMD 能力（16 路并行）
  • chess_prepare_for_pipelining 指示编译器进行软件流水线优化
  • __restrict 关键字确保指针不别名，允许更激进的优化

Host 控制程序 (sw/host.cpp)

// 核心执行流程
auto s2mm_run = s2mm(out_bo, nullptr, ELEM_per_iter*ITERATION);
auto datagen_run = datagen(nullptr, ELEM_per_iter*ITERATION, 0);

auto ghdl = xrt::graph(device, uuid, "gr");
ghdl.run(ITERATION);
ghdl.update("gr.k.in[1]", 10);  // ★ 异步 RTP 更新
ghdl.end();

s2mm_run.wait();
out_bo.sync(XCL_BO_SYNC_BO_FROM_DEVICE);

XRT API 关键点：

  • xrt::graph::run(n) 启动 graph 执行 n 次迭代
  • xrt::graph::update(port_path, value) 异步更新 RTP 端口
  • 端口路径 "gr.k.in[1]" 遵循命名约定：graph_name.kernel_name.port_index

PL Kernel: s2mm (pl_kernels/s2mm.cpp)

void s2mm(ap_int<32>* mem, hls::stream<ap_axis<32, 0, 0, 0>>& s, int size) {
    #pragma HLS INTERFACE m_axi port=mem offset=slave bundle=gmem
    #pragma HLS interface axis port=s
    #pragma HLS INTERFACE s_axilite port=mem bundle=control
    #pragma HLS INTERFACE s_axilite port=size bundle=control
    #pragma HLS interface s_axilite port=return bundle=control

    for(int i = 0; i < size; i++) {
        #pragma HLS PIPELINE II=1
        ap_axis<32, 0, 0, 0> x = s.read();
        mem[i] = x.data;
    }
}

接口设计：

  • m_axi 接口用于访问 DDR 内存（主数据通路）
  • axis 接口用于接收 AIE 输出的 stream 数据
  • s_axilite 接口用于接收来自 host 的控制参数（mem 地址、size）

PL Kernel: datagen (pl_kernels/datagen.cpp)

void datagen(hls::stream<ap_axis<32,0,0,0>> & out, int size, int init_val=0) {
    #pragma HLS INTERFACE axis port=out
    #pragma HLS INTERFACE s_axilite port=return bundle=control
    #pragma HLS INTERFACE ap_ctrl_hs port=return
    #pragma HLS interface s_axilite port=size bundle=control
    #pragma HLS interface ap_none port=size
    #pragma HLS interface s_axilite port=init_val bundle=control
    #pragma HLS interface ap_none port=init_val
    
    for(int i=0; i<size; i++){
        ap_axis<32,0,0,0> tmp;
        tmp.data = i + init_val;
        tmp.keep = -1;
        tmp.last = 0;
        out.write(tmp);
    }
}

设计权衡与决策分析

1. 同步 vs 异步 RTP

特性 同步 RTP 异步 RTP (本模块)
更新时机 每次迭代开始前必须完成 随时可更新，内核按需读取
阻塞性 Host 阻塞等待握手完成 立即返回，无阻塞
适用场景 严格时序控制、确定性要求 高性能、低延迟控制
复杂度 简单（一对一映射） 需要处理竞争条件和时序不确定性
吞吐量 受限于控制握手开销 数据通路全速运行

为什么选择异步？ 这个教程选择异步 RTP 是为了展示最大灵活性。在实际应用中，如果你需要确保每个迭代使用特定的参数值（例如每帧使用不同的滤波器系数），应该使用同步 RTP。

2. 系统配置选择

system.cfg 连接关系：

[connectivity]
nk=s2mm:1:s2mm
nk=datagen:1:datagen
stream_connect=ai_engine_0.Dataout0:s2mm.s
stream_connect=datagen.out:ai_engine_0.Datain0

注意这里没有显式的 RTP 连接配置——RTP 端口由 XRT 运行时通过专用的控制路径管理，不需要在 connectivity 中声明。这体现了 Versal 架构中数据平面和控制平面的分离。

3. 验证策略

新贡献者必读：陷阱与最佳实践

常见错误

  1. 忘记至少一次 RTP 更新

    • 异步 RTP 如果没有初始值设置，内核会在第一次读取 RTP 时阻塞
    • 解决方案：要么在 host 中调用 update()，要么在 graph 中使用 initial_value(k.in[1])=10;

  2. 错误的端口路径

    • "gr.k.in[1]" 中的索引必须匹配 graph 定义中的连接顺序
    • 调试技巧：使用 aiesimulator 时可以打印端口信息验证

  3. 启动顺序导致的死锁

    • 如果先启动 AIE graph 而后启动 PL kernels，可能导致 AIE 等待永远不会到达的数据
    • 推荐顺序：分配内存 → 启动 PL kernels → 启动 AIE graph → 更新 RTP → 等待完成

  4. 混淆同步和异步语义

    • 如果在 graph 中使用 async() 但在 host 中按同步方式思考（期待精确的一次更新对应一次迭代），会导致困惑
    • 明确你的需求：是要严格的时序控制（用 sync）还是要最大性能（用 async）

扩展建议

  1. 多 RTP 端口：一个内核可以有多个 RTP 参数，分别连接到不同的 adf::port
  2. 数组 RTP：参考同目录下的 async_array_rtp 示例，了解如何传递数组参数
  3. RTP 读回：参考 async_array_rtp_read 示例，了解如何从内核读取状态

总结

这个模块虽然功能简单（只是一个带偏移量的向量加法），但它展示了 Versal ACAP 最核心的价值主张之一：硬件加速的软件化控制。通过 RTP 机制，你可以在不中断数据流的情况下动态调整算法参数，这为自适应信号处理、软件定义无线电、实时控制系统等应用场景打开了大门。
