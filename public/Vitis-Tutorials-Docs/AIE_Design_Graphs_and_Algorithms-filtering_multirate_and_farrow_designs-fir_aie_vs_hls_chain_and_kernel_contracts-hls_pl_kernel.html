HLS PL 内核层 (fir_hls.h)

概述

本文件定义了基于 HLS (高层次综合) 的 FIR 滤波器内核的接口与配置参数。它是连接算法描述（C++）与硬件实现（RTL）的关键桥梁，展示了如何利用 Vitis HLS 的 hls::FIR 库将数字信号处理算法高效地映射到 Versal 平台的 DSP58 计算单元。

想象你是一位建筑设计师——fir_hls.h 就是你的设计蓝图。你不是直接堆砌砖块（编写 RTL），而是通过标准化的建筑规范（HLS pragma 和模板参数）来描述你想要的结构，让自动化工具生成最优的施工方案。

核心组件：fir_params 结构体

设计意图

fir_params 继承自 hls::ip_fir::params_t，是 HLS FIR IP 的编译时配置接口。通过模板特化，它将滤波器的所有静态属性（系数数量、位宽、采样周期等）编码到类型系统中，使 HLS 编译器能够针对特定配置生成最优硬件。

结构体定义详解

struct fir_params : hls::ip_fir::params_t {
    static const unsigned num_coeffs    = N_FIR_TAPS;        // 抽头数（滤波器阶数+1）
    static const double   coeff_vec[N_FIR_TAPS];             // 系数数组（外部定义）
    static const unsigned coeff_width   = 16;                // 系数量化位宽
    static const unsigned input_width   = 16;                // 输入样本位宽
    static const unsigned output_width  = 16;                // 输出样本位宽
    static const unsigned output_rounding_mode = hls::ip_fir::truncate_lsbs;  // 截断舍入
    static const unsigned input_length  = WINDOW_LENGTH;     // 输入窗口大小
    static const unsigned output_length = WINDOW_LENGTH;     // 输出窗口大小
    static const unsigned sample_period = SAMP_PERIOD;       // 采样周期（时钟周期/样本）
    static const unsigned coeff_structure = hls::ip_fir::symmetric;  // 对称系数结构优化
};

关键参数解析

数据通路宽度 (*_width)

所有数据路径统一使用 16-bit，这是定点数 DSP 优化的黄金平衡点：

  • 足够表示典型的音频/通信信号动态范围 (~90 dB SNR)
  • 完美匹配 DSP58 的 18x18 乘法器（无资源浪费）
  • 避免 32-bit 带来的双倍资源消耗和时序压力

采样周期自适应逻辑 (SAMP_PERIOD)

#if N_FIR_TAPS > 64
   #define SAMP_PERIOD 4  // 降速以保证可行性
#else
   #define SAMP_PERIOD 1  // 全速运行
#endif

#if N_FIR_TAPS == 129
   #define SAMP_PERIOD 2
#endif

#if N_FIR_TAPS == 240
   #define SAMP_PERIOD 4
#endif

工程洞察：

这是一个典型的时序收敛驱动的性能权衡。DSP58 在一个时钟周期内能完成的乘累加操作是有限的：

  • 当抽头数 ≤ 64：单周期完成所有 MAC 运算，SAMP_PERIOD = 1（最大吞吐）
  • 当抽头数 65~128：需要 2 周期流水线，SAMP_PERIOD = 2
  • 当抽头数 > 128：需要 4 周期流水线，SAMP_PERIOD = 4

这种自适应确保了设计在更高抽头数下仍能通过时序约束，代价是成比例降低的采样率。

对称系数结构优化

static const unsigned coeff_structure = hls::ip_fir::symmetric;

对于线性相位 FIR 滤波器，系数满足 \(h[n] = h[N-1-n]\) 的对称性。声明此属性允许 HLS 利用对称性将乘法器数量减半，显著提升资源效率。

窗口长度条件编译

#if N_FIR_TAPS == 15
    static const unsigned input_length  = WINDOW_LENGTH_HALF;  // 4096
    static const unsigned output_length = WINDOW_LENGTH_HALF;
#else
    static const unsigned input_length  = WINDOW_LENGTH;       // 8192
    static const unsigned output_length = WINDOW_LENGTH;
#endif

15-tap 滤波器使用半窗口长度，可能是因为：

  1. 短滤波器的处理延迟低，可以使用更小的批处理粒度
  2. 测试平台为短滤波器配置了不同的数据量期望

注意：这与 AIE 路径的 INP_SIZE = 2048 形成对比，体现了两种实现的数据流差异。

FIR 实例数组声明

#if N_FIR_TAPS == 15
static hls::FIR<fir_params> fir_real[N_FIR_FILTERS*2];
static hls::FIR<fir_params> fir_imag[N_FIR_FILTERS*2];
#else
static hls::FIR<fir_params> fir_real[N_FIR_FILTERS];
static hls::FIR<fir_params> fir_imag[N_FIR_FILTERS];
#endif

实部/虚部分离架构

复数 FIR 滤波被分解为独立的实部和虚部处理通道，这是 DSP 硬件的常见模式：

\[y_{real}[n] = \sum_{k=0}^{N-1} h[k] \cdot x_{real}[n-k] - \sum_{k=0}^{N-1} h[k] \cdot x_{imag}[n-k]\]
\[y_{imag}[n] = \sum_{k=0}^{N-1} h[k] \cdot x_{real}[n-k] + \sum_{k=0}^{N-1} h[k] \cdot x_{imag}[n-k]\]

15-tap 特殊处理的推测

15-tap 情况下滤波器实例数翻倍（N_FIR_FILTERS*2），可能的解释：

  1. 双通道并行：每个逻辑滤波器对应两个物理实例，分别处理奇偶样本
  2. I/Q 分离的进一步细分：可能涉及额外的解交织逻辑
  3. 遗留代码：早期实验配置的残留

内核函数接口

void fir_hls(
    hls::stream<ap_axiu<128, 0, 0, 0>> &strmInp,
    hls::stream<ap_axiu<128, 0, 0, 0>> &strmOut,
    uint32_t iterCnt
);

接口契约分析

参数 方向 类型 语义
strmInp Input hls::stream<ap_axiu<128,0,0,0>> AXI-Stream 输入，128-bit 数据宽度，无 sideband 信号
strmOut Output hls::stream<ap_axiu<128,0,0,0>> AXI-Stream 输出，同上
iterCnt Input uint32_t 迭代次数，控制总处理数据量

AXI-Stream 协议细节

ap_axiu<128, 0, 0, 0> 表示：

  • 数据位宽：128 bit（可承载 8 个 16-bit 样本）
  • 用户自定义位宽：0（无 TID/TDEST 等路由信息）
  • 保留位宽：0
  • 最后指示位宽：0（不使用 TLAST 进行帧边界标记）

隐含的同步机制：由于不使用 TLAST，数据流的边界由 iterCnt * WINDOW_LENGTH 的总量隐式确定。这要求发送方和接收方对数据量有预先约定的共识。

数据类型定义

#define WINDOW_LENGTH 8192
#define WINDOW_LENGTH_HALF 4096
#define FIR_VALUE_WIDTH 16
#define FIR_SAMPLES WINDOW_LENGTH

typedef ap_fixed<FIR_VALUE_WIDTH, FIR_VALUE_WIDTH> Data_t;
typedef Data_t DataWindow_t[WINDOW_LENGTH];

ap_fixed 的语义

ap_fixed<16, 16> 表示 16-bit 定点数，其中：

  • 整数部分：16 bit（包含符号位）
  • 小数部分：0 bit

实际上这就是普通的 16-bit 有符号整数（Q0.15 或 Q1.14 格式，取决于具体解释）。使用 ap_fixed 而非标准 int16_t 是为了：

  1. 保持与 HLS 数学库的兼容性
  2. 为未来扩展（如引入小数位）预留灵活性
  3. 明确表达"这是信号处理数据"的意图

设计权衡与最佳实践

编译时 vs 运行时配置

本模块采用激进的编译时配置策略：所有关键参数（抽头数、窗口大小、采样周期）都是宏定义或模板常量。这带来了：

优势 代价
最大化编译器优化机会 每次配置变更需重新综合
零运行时配置开销 二进制文件膨胀
类型安全的配置验证 无法动态适应不同工作负载

适用场景判断：

  • ✅ 适合：产品化的固定功能加速器（如无线基站的标准化滤波器）
  • ❌ 不适合：需要运行时切换多种滤波器配置的多功能设备

资源-性能权衡矩阵

抽头数 SAMP_PERIOD 理论采样率 相对资源占用 适用场景
≤ 64 1 最高 基准 高带宽实时处理
65~128 2 中 ~2x DSP 平衡型应用
129~240 4 较低 ~4x DSP 高精度/复杂滤波

潜在陷阱与调试建议

1. 系数数组的外部定义依赖

static const double coeff_vec[N_FIR_TAPS];  // 声明在此，定义在外部

风险：如果链接时找不到对应的定义，将导致未解析符号错误。系数通常在单独的 .cpp 文件中由 MATLAB/Python 脚本生成。

2. 窗口大小与 DMA 缓冲区的不匹配

HLS 内核期望的 WINDOW_LENGTH 必须与主机侧 datamover 配置的 INP_SIZE 严格对齐。任何不匹配都会导致：

  • 数据欠读：内核等待更多数据而挂死
  • 数据过读：读取无效内存，产生错误结果

3. 采样周期的隐性约束

修改 N_FIR_TAPS 后，必须重新评估并可能更新 SAMP_PERIOD。忘记这一步可能导致：

  • 时序违例（综合失败）
  • 运行时数据损坏（若强行绕过 DRC）

4. AXI-Stream 的死锁风险

由于接口使用阻塞式 hls::stream，以下情况会导致死锁：

  • 上游停止发送数据但内核仍在读取
  • 下游消费速度低于内核生产速度（缓冲区溢出）

确保数据通路的流量匹配是系统集成者的责任。

与 AIE 实现的对比视角

维度 HLS (本模块) AIE (对应实现)
编程模型 基于 C++/pragma 的显式硬件描述 基于数据流图的隐式并行调度
优化粒度 时钟周期级（sample_period） 缓冲深度级（buffer depth）
资源映射 直接映射到 DSP58/LUT/FF 映射到 AIE 核心阵列 + 互连网络
重配置能力 需重新编译 xclbin 支持 RTP（运行时参数）重配置
适用信号带宽 中高带宽（受限于 PL 时钟） 超高带宽（利用 AIE 并行性）

理解这些差异有助于在实际项目中做出明智的技术选型决策。
