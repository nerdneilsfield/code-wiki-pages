PL Kernels 模块技术深度解析

概述

PL Kernels 模块是 AIE-ML 性能分析教程中的可编程逻辑（Programmable Logic）数据搬运内核集合，其核心使命是作为 AI Engine 与外部世界之间的"数据高速公路" —— 以尽可能高的吞吐量为 AIE 计算阵列提供测试数据，并接收处理结果，同时不成为系统性能的瓶颈。

想象一个高性能计算工厂：AIE 内核是精密的加工机床，而 PL Kernels 则是连接仓库与车间的自动化输送带。这些输送带必须足够快，确保机床始终有原料可用；同时又要足够简单，避免因自身复杂度引入额外的延迟或故障点。

问题空间与设计动机

为什么需要专门的 PL 数据搬运内核？

在 Versal ACAP 架构中，AI Engine 阵列专注于高密度、低功耗的矢量计算，但它无法直接与 DDR 内存或外部接口通信。数据必须通过 PL（基于 FPGA 的可编程逻辑区域）进行中转。这就带来了一个关键挑战：

如何设计 PL 端的数据搬运逻辑，使其能够饱和 AIE 的输入/输出带宽，而不引入背压（back-pressure）导致 AIE 流水线停滞？

传统的 DMA 控制器虽然可以完成数据传输，但在以下场景下显得力不从心：

  1. 流式数据对齐：AIE 使用 AXI4-Stream 接口，需要精确的握手信号管理
  2. 吞吐量匹配：需要持续、无间隙地发送/接收数据以匹配 AIE 的计算速率
  3. 测试与验证：在性能优化迭代中，需要可控的数据源来隔离 AIE 端的性能瓶颈

本模块的设计选择

本模块采用 HLS（高层次综合） 方式实现数据搬运内核，而非手写 RTL。这一选择背后的权衡：

方案 优势 劣势 本模块的选择
HLS C++ 开发快速、易于修改、可移植性好 对底层时序控制较粗 ✅ 采用
RTL (Verilog/VHDL) 时序精确、面积最优 开发周期长、调试复杂 ❌ 未采用

考虑到这是一个教学示例，重点在于展示 AIE-ML 的性能优化方法，而非极致的 PL 优化，HLS 提供了最佳的开发效率与可读性平衡。

核心抽象与心智模型

"数据源-汇"（Source-Sink）模型

本模块围绕两个核心角色构建：

┌─────────────┐      AXI4-Stream      ┌─────────────┐
│   datagen   │ ────────────────────> │  AIE Graph  │
│  (数据源)    │    128-bit data       │  (计算阵列)  │
└─────────────┘                       └─────────────┘
                                           │
                                           │ AXI4-Stream
                                           ▼
                                      ┌─────────────┐
                                      │    s2ss     │
                                      │   (数据汇)   │
                                      └─────────────┘
  • datagen（Data Generator）：数据生成器，作为 AIE 的输入数据源。它按配置产生固定模式的 bfloat16 数据（值 0-7），通过 AXI4-Stream 接口持续输出。
  • s2ss（Stream to Stream Sink）：流式数据接收器，作为 AIE 的输出数据汇。它从 AXI4-Stream 接口读取数据并丢弃（用于吞吐量测试）。

数据格式约定

datagen 生成的每个 128-bit 数据包包含 8 个 bfloat16 数值：

┌─────────────────────────────────────────────────────────────┐
│ 127:112 │ 111:96 │ 95:80 │ 79:64 │ 63:48 │ 47:32 │ 31:16 │ 15:0 │
├─────────┼────────┼───────┼───────┼───────┼───────┼───────┼──────┤
│  0x40e0 │ 0x40c0 │0x40a0 │ 0x4080│ 0x4040│ 0x4000│ 0x3f80│ 0x0000│
│  bf16=7 │ bf16=6 │bf16=5 │ bf16=4│ bf16=3│ bf16=2│ bf16=1│ bf16=0│
└─────────┴────────┴───────┴───────┴───────┴───────┴───────┴──────┘

这种固定的数据模式便于：

  1. 功能验证：AIE 输出可与预期值比对
  2. 性能测试：消除了从 DDR 读取数据的延迟不确定性
  3. 死锁调试：确定性的数据流便于分析问题根因

核心组件详解

1. datagen - 数据生成器

文件: datagen.cpp

功能：产生固定模式的 bfloat16 测试数据流。

HLS 接口配置：

#pragma HLS INTERFACE axis port=out           // AXI4-Stream 输出
#pragma HLS INTERFACE s_axilite port=return bundle=control  // 控制寄存器
#pragma HLS INTERFACE ap_ctrl_hs port=return  // 握手控制
#pragma HLS interface s_axilite port=size bundle=control    // 传输大小配置
#pragma HLS interface ap_none port=size       // size 为只读标量

内部实现要点：

  • 循环产生 size 个 128-bit 数据包
  • 每个数据包包含 8 个预定义的 bfloat16 值（0.0 到 7.0）
  • keep=-1 表示所有字节有效
  • last=0 表示非包尾（由 AIE 端处理帧边界）

关键设计决策：

为什么不从 DDR 读取真实数据？

在性能分析场景中，我们希望消除 DDR 访问延迟的不确定性，将焦点集中在 AIE 内部的计算效率和数据搬移上。datagen 提供了确定性的、可持续的最大速率数据源。

2. s2ss - 流式数据接收器

文件: s2ss.cpp

功能：从 AXI4-Stream 接口读取数据并丢弃，用于测试 AIE 输出吞吐量。

HLS 接口配置：

#pragma HLS interface axis port=s             // AXI4-Stream 输入
#pragma HLS INTERFACE s_axilite port=size bundle=control    // 传输大小配置
#pragma HLS interface s_axilite port=return bundle=control  // 控制寄存器

流水线优化：

#pragma HLS PIPELINE II=1

这一 pragma 要求 HLS 工具以 Initiation Interval = 1 的速率执行循环，即每个时钟周期读取一个 128-bit 数据。在 400 MHz 的 PL 时钟下，理论峰值吞吐量为：

\[ \text{Throughput}_{max} = 128 \text{ bits} \times 400 \times 10^6 \text{ Hz} = 51.2 \text{ Gbps} = 6.4 \text{ GB/s} \]

这远高于 AIE-ML 的典型输出速率，确保不会成为瓶颈。

关键设计决策：

为什么是 "sink" 而不是写回 DDR？

在纯性能测试场景中，我们只需要测量 AIE 能够产生数据的最大速率，而不关心数据内容。将数据直接丢弃（blackhole）避免了 DDR 写入带宽可能成为限制因素的情况。

3. config.cfg - HLS 综合配置

功能：定义 HLS 综合的顶层函数、目标频率和输出格式。

[hls]
freqhz=400000000          # PL 目标频率: 400 MHz
syn.top=s2mm              # 顶层函数名（会被 Makefile 动态替换）
syn.file=s2mm.cpp         # 源文件（会被 Makefile 动态替换）
flow_target=vitis         # Vitis 流程
package.output.format=xo  # 输出 Xilinx Object 格式
package.output.file=s2mm.xo

注意：实际使用时，Makefile 会通过 sed 命令将 s2mm 替换为实际的 kernel 名称（datagen 或 s2ss）。

4. Makefile - 构建编排

关键规则：

%.xo: %.cpp
    sed 's/s2mm/\((basename \)<)/' config.cfg > tmp.cfg
    v++ -c --mode hls --platform ${PLATFORM} --config tmp.cfg

构建流程：

  1. 对每个 .cpp 文件，生成对应的 .xo（Xilinx Object）文件
  2. 使用 sed 动态修改 config.cfg 中的顶层函数名
  3. 调用 v++ 进行 HLS 综合和打包

设计权衡与决策分析

1. 单通道 vs 多通道（Version 1 → Version 4）

维度 Version 1 Version 4
复杂度 简单，易于调试 复杂，需处理多流同步
吞吐量 受限于单通道带宽 3 倍理论峰值提升
资源占用 较少 LUT/FF 3 倍 PL 资源
适用场景 功能验证、死锁分析 性能扩展、生产部署

演进逻辑：

  • Version 1 首先解决"能不能工作"的问题，通过单通道验证 AIE 图的正确性
  • 发现单通道带宽无法满足 AIE 计算能力后，Version 4 引入 3 路并行，实现线性扩展

2. 固定模式数据 vs 随机/真实数据

选择固定模式数据的理由：

  1. 确定性调试：已知输入意味着预期输出可预先计算，便于自动验证
  2. 死锁定位：当系统挂起时，可以确定问题不在数据依赖上
  3. 性能基线：消除了数据生成的不确定性，测量的是纯粹的系统能力

代价：无法发现数据相关的时序问题（如某些值导致的异常延迟）。

3. AXI4-Stream vs AXI4-Full（Memory Mapped）

本模块全部使用 AXI4-Stream，原因：

  • AIE 阵列原生支持 Stream 接口，无需地址转换开销
  • Stream 接口天然支持背靠背（back-to-back）传输，无总线仲裁延迟
  • 符合数据流应用的本质特征（连续、有序、无随机访问）

新贡献者须知

常见陷阱

  1. 频率域交叉（Clock Domain Crossing）

    • PL 运行在 400 MHz，AIE 运行在 300 MHz
    • AXI4-Stream 接口会自动处理异步时钟域转换，但需注意 FIFO 深度设置

  2. TLAST 信号处理

    • datagen 设置 tmp.last=0，表示由 AIE 端负责帧边界标记
    • 如果 AIE 期望 TLAST 来触发内部状态机，可能需要修改 datagen

  3. Keep 信号全置位

    • tmp.keep=-1 表示所有 16 字节都有效
    • 部分系统设计可能利用 keep 进行字节级掩码，此处不适用

调试技巧

  1. 使用 ChipScope 抓包

    [debug]
aie.chipscope=Dataout0
aie.chipscope=Datain0

  2. 检查 HLS 综合报告

    • 确认 s2ss 的 II=1 是否达成
    • 检查 datagen 的循环延迟是否符合预期

  3. 仿真验证

    • 先运行 x86sim 验证功能正确性
    • 再运行 aiesim 检查时序和性能
