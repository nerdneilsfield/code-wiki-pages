ifft256p4 模块深度解析

一句话概括

ifft256p4 是一个专为 AMD Versal AI Engine 设计的 256点基-4 IFFT（逆快速傅里叶变换）计算内核，它采用四阶段 DIT（Decimation-In-Time）流水线结构，通过乒乓缓冲策略在 AIE 向量处理器上高效执行复数频域到时域的转换。想象它像一个精密的音频解码器——将频谱数据"翻译"回时域波形，但处理的是 256 个复数样本的高吞吐量数据流。

问题空间与设计动机

为什么需要这个模块？

在大型 2D-IFFT 系统（如 ifft64k 所需的 65536 点变换）中，直接计算是不可行的：

  1. 计算复杂度：\(O(N^2)\) 的直接 DFT 对于 64K 点需要超过 40 亿次复数运算
  2. 内存限制：AIE 内核的本地内存有限，无法容纳完整的数据集
  3. 并行度需求：需要利用 AIE 阵列的 SIMD 向量处理能力

核心洞察：通过将 2D-IFFT 分解为两个 256 点 1D-IFFT 阶段（行列分解），并在中间插入旋转因子乘法（twiddle rotation），可以将 \(O(N^2)\) 问题转化为可管理的 \(O(N \log N)\) 流水线。

为什么是 256 点？

\[256 = 4^4\]

这个选择并非随意：

  • 基-4 友好：256 是 4 的整数幂，完美适配 AIE API 的 fft_dit_r4_stage 原语
  • 向量对齐：256 点可以被 AIE 的 128-bit 向量宽度（4 x 32-bit 复数）整除
  • 内存效率：256 x 8 bytes = 2KB，恰好适合 AIE 内核的本地数据存储器

核心组件深度解析

1. ifft256p4 内核类

位于 ifft256p4.h 和 ifft256p4.cpp，这是模块的核心计算引擎。

类型系统与常量定义

typedef cint32 TT_DATA;      // 数据类型：32-bit 实部 + 32-bit 虚部
typedef cint16 TT_TWID;      // 旋转因子：16-bit 实部 + 16-bit 虚部（Q15 定点）
static constexpr unsigned NFFT = 256;      // FFT 点数
static constexpr unsigned EXTRA = 4;       // 额外填充样本（满足 AIE DMA 对齐要求）
static constexpr unsigned NSAMP = NFFT + EXTRA;  // 总缓冲区大小 = 260
static constexpr bool INVERSE = true;      // IFFT 模式（vs FFT）
static constexpr unsigned SHIFT_TW = 15;   // 旋转因子右移位数（Q15 归一化）
static constexpr unsigned SHIFT_DT = 15;   // 数据右移位数（防止溢出）

设计决策：使用 cint16 存储旋转因子而非 cint32，是因为：

  1. 旋转因子的动态范围已知且有限（单位圆上的点）
  2. 16-bit 存储节省 50% 的常量内存
  3. AIE 向量乘法单元支持混合精度（32-bit x 16-bit → 32-bit）

构造函数与初始化

ifft256p4::ifft256p4(TT_DATA (&tbuff_i)[NFFT]) : tbuff(tbuff_i)
{
    aie::set_rounding(aie::rounding_mode::conv_even);     // 偶数舍入
    aie::set_saturation(aie::saturation_mode::saturate);  // 饱和算术
}

关键细节：

  • tbuff 通过引用绑定到外部分配的数组，遵循 RAII 外部所有权 模式
  • 舍入模式设为 conv_even（收敛偶数舍入），避免累积偏置误差
  • 饱和模式防止定点溢出导致的数值灾难

run() 方法：四阶段流水线

void ifft256p4::run(input_buffer<TT_DATA,extents<NSAMP>>& __restrict sig_i,
                    output_buffer<TT_DATA,extents<NSAMP>>& __restrict sig_o)
{
    auto itw = aie::begin_vector<4>(sig_o);
    TT_DATA* ibuff = sig_i.data();
    TT_DATA* obuff = sig_o.data();

    // 四阶段基-4 DIT FFT
    aie::fft_dit_r4_stage<64>(ibuff, tw4a_1, tw4a_0, tw4a_2, NFFT, SHIFT_TW, SHIFT_DT, INVERSE, tbuff);
    aie::fft_dit_r4_stage<16>(tbuff, tw4b_1, tw4b_0, tw4b_2, NFFT, SHIFT_TW, SHIFT_DT, INVERSE, ibuff);
    aie::fft_dit_r4_stage< 4>(ibuff, tw4c_1, tw4c_2, tw4c_2, NFFT, SHIFT_TW, SHIFT_DT, INVERSE, tbuff);
    aie::fft_dit_r4_stage< 1>(tbuff, tw4d_1, tw4d_0, tw4d_2, NFFT, SHIFT_TW, SHIFT_DT, INVERSE, obuff);

    // 写入额外 4 个零样本
    itw += NFFT/4;
    *itw++ = aie::zeros<TT_DATA,4>();
}

算法解释：

这是一个 基-4 DIT（Decimation-In-Time） FFT，将 256 点分解为 4 级，每级处理 4 点蝶形运算：

阶段 Stride 输入缓冲区 输出缓冲区 旋转因子表
1 64 ibuff (sig_i) tbuff tw4a_0/1/2
2 16 tbuff ibuff (sig_i 重用) tw4b_0/1/2
3 4 ibuff tbuff tw4c_0/1/2
4 1 tbuff obuff (sig_o) tw4d_0/1/2

乒乓缓冲策略：

  • 阶段 1、3 使用 tbuff 作为输出
  • 阶段 2 写回 ibuff（重用输入缓冲区）
  • 这种 in-place / out-of-place 混合 策略最小化内存占用

模板参数 <stride> 的含义：

  • 控制蝶形运算的元素间距
  • 从 64 递减到 1，反映 DIT 算法的位反转特性

2. ifft256p4_graph ADF 图容器

class ifft256p4_graph : public graph {
public:
    kernel k_kernel;
    port<input> sig_i;
    port<output> sig_o;

    ifft256p4_graph(void)
    {
        // 创建内核实例，传递临时缓冲区参数
        k_kernel = kernel::create_object<ifft256p4>(
            std::vector<ifft256p4::TT_DATA>(ifft256p4::NFFT)
        );
        source(k_kernel) = "ifft256p4.cpp";
        runtime<ratio>(k_kernel) = 0.9;  // 90% 时间片分配给此内核
        
        connect<>(sig_i, k_kernel.in[0]);
        connect<>(k_kernel.out[0], sig_o);
    }
};

关键设计点：

  1. std::vector 参数传递：向内核构造函数传递临时缓冲区，create_object 负责在 AIE 本地内存中分配
  2. runtime<ratio>(k_kernel) = 0.9：内核占用 90% 的 AIE 周期，保留 10% 给数据移动和其他开销
  3. 单输入单输出：简洁的流式接口，符合 AIE 数据流编程模型

class dut_graph : public graph {
public:
    TT_DUT dut;
    input_plio sig_i;
    output_plio sig_o;
    
    dut_graph(void)
    {
        // PLIO 配置：64-bit 宽度，关联文本文件

        sig_o = output_plio::create("PLIO_o_0", plio_64_bits, "data/sig_o_0.txt");

  • 仿真时从文本文件读取激励
  • 验证时捕获输出与参考值比对

数据格式契约

输入 (sig_i)：

  • 类型：cint32 数组，260 元素（256 数据 + 4 填充）
  • 布局：自然顺序（非位反转）
  • 来源：通常是前一级转置操作的输出

输出 (sig_o)：

  • 类型：cint32 数组，260 元素
  • 布局：自然顺序
  • 最后 4 元素强制置零（aie::zeros<TT_DATA,4>()）

设计权衡与决策分析

1. 定点 vs 浮点

选择：Q15 定点旋转因子 + Q31 定点数据

理由：

  • ✅ AIE 向量单元对定点运算优化更好（更高吞吐量）
  • ✅ 功耗显著低于浮点
  • ⚠️ 需要仔细管理缩放（SHIFT_TW、SHIFT_DT）防止溢出

替代方案：使用 cfloat 类型——会简化代码但降低 2-4x 吞吐量

2. 乒乓缓冲 vs 单缓冲

选择：双缓冲（tbuff + 重用 ibuff）

理由：

  • ✅ 允许阶段间并行（前一阶段写后一阶段读）
  • ✅ 避免数据拷贝开销
  • ⚠️ 增加 2KB 内存占用

观察到的模式：奇数阶段输出到 tbuff，偶数阶段输出到 ibuff

3. 编译时常量 vs 运行时可配置

选择：所有参数为 static constexpr

理由：

  • ✅ 编译器可以展开循环、优化地址计算
  • ✅ 旋转因子表可以放在常量内存区
  • ⚠️ 每个 FFT 大小需要独立的内核实例（代码膨胀）

对比：ifft256p4_rot 使用模板参数 NSTREAM 和 PHASE，在编译时生成不同变体

4. 90% 运行时比率

runtime<ratio>(dut.k_kernel) = 0.9;

含义：内核每 10 个周期执行 9 个周期，留出 10% 给：

  • DMA 数据传输
  • 其他共享此 AIE tile 的内核
  • 中断处理

风险：如果实际计算需要 >90% 周期，会导致流水线停滞

新贡献者注意事项

🔴 关键假设与隐含契约

  1. 缓冲区对齐：alignas(32) 确保 SIMD 加载不会跨越缓存行边界
  2. EXTRA=4 的用途：满足 AIE DMA 的 16-byte 对齐要求，同时也是向量迭代的便利终止条件
  3. 旋转因子预计算：ifft256p4_twiddle.h 中的值由 MATLAB/Octave 脚本生成，不要手动编辑

🟡 常见陷阱

问题 症状 解决方案
修改 NFFT 后忘记更新旋转因子表 输出完全错误 重新运行 gen_vectors.m 生成新表
移除 __restrict 关键字 性能下降 20-50% 保留 __restrict 允许编译器优化别名分析
更改 SHIFT_TW/DT 而不验证 静默数值溢出或精度损失 使用 regression.m 验证信噪比
忘记调用 registerKernelClass 链接错误：未定义的符号 确保 .h 文件中包含静态注册块

🟢 扩展点

如需修改此模块：

  1. 改变 FFT 大小：需要重新计算所有旋转因子表，并调整四级 stride 参数
  2. 改为 FFT（正变换）：将 INVERSE = false，并可能需要调整输出缩放
  3. 增加吞吐量：考虑实例化多个 ifft256p4_graph 并行处理（如 ifft64k_graph 所做）
