DMA FIFO 优化案例：性能分析教程

一句话概括

这是一个Versal ACAP AIE-PL 数据流性能调优的标杆案例，演示了如何通过合理配置 AIE 内核间的 stream 连接深度（FIFO depth）来解决流水线背压（back-pressure）问题，从而最大化系统吞吐率。想象它像是一条高速公路上的缓冲车道——当后续路段拥堵时，足够长的缓冲区能让上游交通继续流动，而不是完全停滞。

问题空间与设计动机

为什么需要这个模块？

在 Versal ACAP 架构中，AI Engine (AIE) 阵列与可编程逻辑 (PL) 之间的数据搬移是系统性能的瓶颈所在。AIE 内核以固定时钟频率（通常 ~1GHz）运行，而 PL 侧的数据搬运器（DMA）可能以不同频率运行。当数据生产者和消费者速率不匹配时，会产生背压（back-pressure）：

  • 场景 A：如果下游消费者处理速度慢于上游生产者，数据会在连接处堆积
  • 场景 B：如果没有足够的缓冲，上游会被迫停顿，导致整体吞吐率下降
  • 场景 C：极端情况下，死锁可能发生，系统完全挂起

本模块属于 13-aie-performance-analysis 教程系列中的成功案例，展示了通过显式设置 fifo_depth 来消除背压的正确做法。

对比：三种性能案例的设计意图

案例 状态 核心问题 教学目的
testcase_dmafifo_opt (本模块) ✅ 正确 FIFO 深度充足 展示正确的性能优化方法
testcase_nofifo_hang ❌ 错误 未设置 FIFO，导致死锁 演示缺少缓冲的后果
testcase_ssfifo ⚠️ 次优 FIFO 深度不足 展示部分优化的局限性

这三个案例共同构成一个完整的教学叙事：从失败中学习，理解原理，最终掌握正确的调优方法。

核心抽象与心智模型

类比：工厂装配线的缓冲库存

想象一条汽车装配线：

  • 工位 A（AIE 内核 1）：安装发动机，每 10 分钟完成一辆
  • 工位 B（AIE 内核 2）：安装轮胎，每 12 分钟完成一辆
  • 传送带（stream 连接）：两工位之间的运输通道

如果传送带只能容纳一辆车（无缓冲/FIFO=0），当工位 B 还在忙时，工位 A 必须停工等待——这就是背压。解决方法是增加传送带的容量（FIFO depth），让工位 A 可以继续生产，把成品暂存在传送带上。

在本设计中：

关键代码路径详解

1. PL 数据搬运器 (mm2s.cpp / s2mm.cpp)

这两个 HLS 内核是最简单的"数据泵"：

// mm2s.cpp - 内存到流
void mm2s(ap_uint<128>* mem, hls::stream<ap_axiu<128, 0, 0, 0>>& s, int size) {
    #pragma HLS INTERFACE m_axi port=mem offset=slave bundle=gmem
    #pragma HLS interface axis port=s
    #pragma HLS INTERFACE s_axilite port=mem bundle=control
    #pragma HLS INTERFACE s_axilite port=size bundle=control
    #pragma HLS interface s_axilite port=return bundle=control
    
    for(int i = 0; i < size; i++) {
        #pragma HLS PIPELINE II=1  // 关键：每个周期输出一个 128-bit
        ap_axiu<128, 0, 0, 0> x;
        x.data = mem[i];
        x.keep = -1;  // 所有字节有效
        x.last = 0;   // 非包尾（连续流）
        s.write(x);
    }
}

设计要点：

  • II=1（Initiation Interval = 1）意味着每个时钟周期可以启动一次新的迭代
  • 在 400MHz 下，理论峰值带宽 = 128 bits × 400M = 51.2 Gbps = 6.4 GB/s
  • s_axilite 接口用于主机配置 mem 指针和 size，不占用数据传输路径

2. AIE 内核 1：aie_dest1 —— 数据分流器

void aie_dest1(input_buffer<int32,extents<1024>> & __restrict in, 
               output_stream<int32> * __restrict out, 
               output_buffer<int32,extents<1024>> & __restrict outm) {
    auto inIter = aie::begin_vector<4>(in);    // 输入迭代器，每次 4 个 int32
    auto outmIter = aie::begin_vector<4>(outm); // buffer 输出迭代器
    aie::vector<int32,4> tmp;
    
    for(int i = 0; i < 128; i++)  // 128 × 4 = 512? 等等，实际是 256 × 4 = 1024
    chess_prepare_for_pipelining   // 提示编译器准备流水线优化
    {
        tmp = *inIter++;           // 读取 4 个 int32
        writeincr(out, tmp);       // 写入 stream（去 aie_dest2 的路径 0）
        *outmIter++ = tmp;         // 写入 buffer（去 aie_dest2 的路径 1）
        
        tmp = *inIter++;           // 再读 4 个
        writeincr(out, tmp);       // 再次分流
        *outmIter++ = tmp;
    }
}

关键洞察：

  • chess_prepare_for_pipelining 是 AMD AIE 编译器特定的 pragma，提示该循环应被充分流水线化
  • 数据被复制到两个输出：一个走 stream（net1），一个走 buffer（net2）
  • 这种设计故意制造了数据分叉，要求下游 aie_dest2 同时消费两条路径的数据

3. AIE 内核 2：aie_dest2 —— 数据合并与计算

void aie_dest2(input_stream<int32> * __restrict in, 
               input_buffer<int32,extents<1024>> & __restrict inm, 
               output_stream<int32> * __restrict outm) {
    auto inmIter = aie::begin_vector<4>(inm);
    aie::vector<int32,4> tmp, tmp2;
    
    for(int i = 0; i < 128; i++)
    chess_prepare_for_pipelining
    {
        tmp = readincr_v<4>(in);   // 从 stream 读取（来自 aie_dest1 路径 0）
        tmp2 = *inmIter++;          // 从 buffer 读取（来自 aie_dest1 路径 1）
        writeincr(outm, aie::add(tmp, tmp2));  // 向量相加后输出
        
        tmp = readincr_v<4>(in);
        tmp2 = *inmIter++;
        writeincr(outm, tmp + tmp2);  // 等价写法
    }
}

计算语义：

  • 功能上，aie_dest2 将同一份数据的两个副本相加：(data × 1) + (data × 1) = data × 2
  • 这就是为什么 host 验证代码检查 host_in[i] * 2 == host_out[i]
  • 使用 __restrict 关键字告诉编译器指针不别名，允许激进优化

4. Graph 拓扑与 FIFO 配置

// graph.h
class adaptive_graph : public graph {
public:
    input_plio in;
    output_plio dataout;
    kernel k[2];

    adaptive_graph() {
        k[0] = kernel::create(aie_dest1);
        source(k[0]) = "aie_dest1.cc";
        k[1] = kernel::create(aie_dest2);
        source(k[1]) = "aie_dest2.cc";

        in = input_plio::create("Datain0", plio_128_bits, "data/input.txt");
        dataout = output_plio::create("Dataout0", plio_128_bits, "data/output.txt");

        runtime<ratio>(k[0]) = 0.8;  // 占用 80% AIE 计算资源
        runtime<ratio>(k[1]) = 0.8;

        connect<> net0(in.out[0], k[0].in[0]);           // 默认缓冲
        connect<stream> net1(k[0].out[0], k[1].in[0]);   // stream 连接
        connect<> net2(k[0].out[1], k[1].in[1]);         // 默认缓冲
        connect<stream> net3(k[1].out[0], dataout.in[0]); // stream 输出
        
        fifo_depth(net1) = 1024;  // ⭐ 关键优化：为 net1 设置 1024 深度 FIFO
    }
};

连接类型说明：

连接 类型 是否设置 FIFO 说明
net0 connect<> (默认) 否 PLIO 到内核，使用默认缓冲
net1 connect<stream> 是，1024 内核间 stream，显式深度
net2 connect<> (默认) 否 buffer 类型连接
net3 connect<stream> 否 内核到 PLIO，单向输出无需深缓冲

6. HLS 编译配置 (pl_kernels/config.cfg)

[hls]

syn.top=s2mm
syn.file=s2mm.cpp

设计决策说明：

  • freqhz=400000000：设置 HLS 综合目标频率为 400 MHz，高于系统级的 312.5 MHz。这是为了给工具更大的时序裕量，确保在系统级约束下仍能稳定运行
  • flow_target=vitis：指定使用 Vitis 统一软件平台进行编译，而非独立的 Vivado HLS
  • package.output.format=xo：输出格式为 Xilinx Object (XO)，这是 Vitis 链接阶段的标准输入格式
  • syn.top=s2mm：指定顶层函数为 s2mm，对应 s2mm.cpp 中的 extern "C" 函数

关键设计决策与权衡

决策 1：为什么只为 net1 设置 fifo_depth？

观察：四个连接中只有 net1 显式设置了 fifo_depth=1024。

推理：

  • net0（PLIO→K1）：PLIO 作为数据源，其速率由外部 DMA 控制，且 aie_dest1 总是准备好接收（它是纯消费者）
  • net1（K1→K2）：这是关键路径。aie_dest1 产生数据的速度可能快于 aie_dest2 消费，因为 K2 要等待两条路径的数据都就绪才能计算
  • net2（K1→K2）：虽然是 buffer 类型，但 buffer 语义本身包含存储，不像 stream 那样需要显式 FIFO
  • net3（K2→PLIO）：K2 是生产者，PLIO 是消费者，下游的 s2mm 会持续拉取数据

权衡：设置过大的 FIFO 会消耗宝贵的 AIE tile 片上 SRAM 资源。1024 个 int32 = 4KB，这是一个合理的折中。

决策 2：为什么要设计两条数据路径（stream + buffer）？

这看起来像是人为复杂化，但有其教学目的：

  1. 模拟真实工作负载：许多信号处理算法需要同时访问"当前样本"（stream）和"历史窗口"（buffer）
  2. 制造同步点：aie_dest2 必须协调两个输入源的到达，这自然产生了潜在的速率不匹配
  3. 展示 FIFO 的效果：如果只有一条路径，背压问题不会显现；两条路径创造了需要缓冲的场景

决策 3：runtime<ratio> = 0.8 的含义

runtime<ratio>(k[0]) = 0.8;
runtime<ratio>(k[1]) = 0.8;

这告诉 AIE 编译器：每个内核预计占用其所在 AIE tile 的 80% 计算资源。剩余 20% 可用于：

  • 与其他内核共享 tile（如果资源允许）
  • 留给编译器进行指令调度优化
  • 保证即使最坏情况下也能满足时序

为什么不设为 1.0？

  • 1.0 意味着独占 tile，可能导致资源浪费
  • 留有余地可以让编译器更灵活地进行流水线调度

决策 4：HLS 内核的 PIPELINE II=1 vs AIE 的 chess_prepare_for_pipelining

特性 HLS (PL) AIE
Pragma #pragma HLS PIPELINE II=1 chess_prepare_for_pipelining
目标 每个时钟周期启动一次迭代 充分利用 VLIW 槽位，最大化 SIMD
约束 受限于资源依赖和内存端口 受限于数据移动和寄存器压力
时钟域 400 MHz (本设计) ~1 GHz

两者都追求高吞吐，但实现机制不同。HLS 关注硬件流水线阶段，AIE 关注超长指令字 (VLIW) 调度和向量单元利用率。

性能测量方法论

仿真模式 (graph.cpp)

#if defined(__AIESIM__) || defined(__X86SIM__)
int main(int argc, char **argv) {
    gr.init();
    int iterations = 20;
    int bytes_per_iteration = 4096;
    int total_bytes = bytes_per_iteration * iterations;
    
    // 使用 AIE 事件性能计数器
    event::handle handle = event::start_profiling(
        gr.dataout, 
        event::io_stream_start_to_bytes_transferred_cycles, 
        total_bytes
    );
    
    gr.run(iterations);
    gr.wait();
    long long cycle_count = event::read_profiling(handle);
    
    // 吞吐率计算：总字节数 / (周期数 × 每周期纳秒)
    double throughput = (double)total_bytes / (cycle_count * 1e-3); 
    printf("Throughput of the graph: %f MB/s\n", throughput);
}
#endif

硬件执行模式 (host.cpp)

// XRT 性能分析 API
xrt::aie::profiling handle(device);
handle.start(
    xrt::aie::profiling::profiling_option::io_stream_start_to_bytes_transferred_cycles,
    "gr.dataout", 
    "", 
    output_size_in_bytes
);

ghdl.run(iterations);
ghdl.end();
long long cycle_count = handle.read();

// 注意这里的 0.8 因子！
double throughput = (double)output_size_in_bytes / (cycle_count * 0.8 * 1e-3);

关键差异：硬件模式下乘以 0.8，因为生产板上的 AIE 时钟周期为 0.8 纳秒（即 1.25 GHz），而仿真模式使用不同的计时基准。

新贡献者须知：陷阱与注意事项

🚨 常见错误 1：忘记设置 fifo_depth

// 错误写法（会导致死锁或性能极差）
connect<stream> net1(k[0].out[0], k[1].in[0]);
// 没有 fifo_depth() 调用！

症状：系统在硬件上挂起，或吞吐率远低于理论值。

诊断：使用 Vitis Analyzer 查看波形，观察 TREADY/TVALID 信号是否频繁拉低。

🚨 常见错误 2：混淆 stream 和 buffer 语义

特性 connect<stream> connect<> (默认/buffer)
数据粒度 逐个元素（或向量） 整块 buffer
阻塞行为 write/read 可能阻塞 异步，通过同步令牌协调
需要 FIFO 通常需要显式 fifo_depth 不需要
适用场景 连续数据流 块状数据处理

🚨 常见错误 3：忽略 __restrict 的重要性

在 AIE 内核中，__restrict 不是可选优化——它是指令级并行的前提。如果省略，编译器必须假设指针可能别名，会生成保守的串行代码。

🚨 常见错误 4：buffer 大小不匹配

// kernel.h
void aie_dest1(input_buffer<int32, extents<1024>> & __restrict in, ...)

调用者必须确保传入的 buffer 确实包含 1024 个 int32。如果实际数据更少，会发生越界读取；如果更多，多余数据被忽略。

🚨 常见错误 5：性能计数器句柄耗尽

event::handle handle = event::start_profiling(...);
if(handle == event::invalid_handle) {
    printf("ERROR:Invalid handle. Only two performance counter in a AIE-PL interface tile\n");
    return 1;
}

每个 AIE-PL 接口 tile 只有两个性能计数器。如果尝试同时监控多个端口，会返回无效句柄。

总结

performance_analysis_dmafifo_optimized_case 是一个精心设计的教学案例，它用最小的代码量展示了 Versal ACAP 性能调优的核心原则：

  1. 理解数据流：明确知道数据从哪里来、到哪里去、经过哪些转换
  2. 识别瓶颈：找到生产者和消费者之间可能存在速率不匹配的点
  3. 适度缓冲：在关键连接点设置合适的 FIFO 深度，既解决背压又不浪费资源
  4. 测量验证：使用硬件性能计数器量化优化效果，而不是凭感觉

对于新加入团队的工程师，建议按以下顺序学习：

  1. 先阅读本模块（成功案例）
  2. 再研究 testcase_nofifo_hang（失败案例）
  3. 对比两者的 system.cfg 和 graph.h 差异
  4. 修改 FIFO 深度参数，观察对吞吐率的影响
  5. 将学到的模式应用到实际项目的数据流设计中
