alveo_aurora_kernel_stream_config 模块深度解析

概述：高速串行链路验证的"测试夹具"

想象你正在调试一条时速 1000 公里的超级高速公路，但你无法直接看到路面上行驶的车辆。你能做的就是：在入口放一个自动发车机器人，在出口放一个检测相机，然后让车辆在这条路上往返跑。通过比对发出和收到的车辆编号，你就能知道这条路有没有丢车、有没有损坏。

这就是 alveo_aurora_kernel_stream_config 模块所做的事情——它是 Xilinx Alveo 加速卡上 Aurora 高速串行通信协议的硬件级环回测试配置。Aurora 是一种基于 GTX/GTH/GTY 收发器的轻量级链路层协议，常用于 10Gbps 甚至 25Gbps+ 的芯片间通信。

这个模块不实现算法逻辑，而是定义系统拓扑：它把三个 HLS 内核（strm_issue、krnl_aurora、strm_dump）通过 AXI4-Stream 接口拼接成一个完整的测试回路，同时处理不同 Alveo 卡（U200、U250、U55C 等）的引脚差异和时钟域配置。

想象你正在设计一个数据中心的高速互联方案——需要将数据以每秒数十亿比特的速度从一个 Alveo 加速卡传输到另一张卡。传统的 PCIe 或以太网方案要么带宽受限，要么延迟过高。Aurora 64B/66B 是一种轻量级串行通信协议，它直接利用 FPGA 的 GT（Gigabit Transceiver）收发器，在单根光纤上实现高达 100 Gbps 的全双工通信。

但这个看似简单的目标面临三个核心挑战：

  1. 时钟域的"巴别塔"：Aurora IP 运行在 161 MHz 的用户时钟域，而 Vitis 平台内核运行在 300 MHz，主机接口则是另一个异步域。没有精心设计的 CDC（Clock Domain Crossing），数据会在时钟边界处损坏。

  2. GT 收发器的"黑盒"特性：QSFP 光口、参考时钟、初始化时钟——这些底层硬件信号需要精确连接到 Alveo 卡的特定引脚，任何错位都会导致链路无法建立。

  3. 流式数据的"接力赛"：从主机内存 → AXI4 总线 → AXI Stream → Aurora TX → 光纤 → Aurora RX → AXI Stream → AXI4 → 主机内存，每一步都需要无缝衔接，否则会出现反压（back-pressure）或数据丢失。

本模块（alveo_aurora_kernel_stream_config）正是解决这些问题的系统集成配置层。它不是单一的代码文件，而是一套完整的硬件设计流程：通过 krnl_aurora_test.cfg 配置文件，将 RTL 内核（krnl_aurora）、HLS 数据搬运内核（strm_issue/strm_dump）与 Alveo 平台的物理接口（GT 收发器、参考时钟）精确连接在一起。

心智模型：把系统设计想象成什么？

把这个系统想象成一个跨国高速物流网络：

  • krnl_aurora 是海关口岸（RTL 内核）：负责将货物（数据）装上/卸下国际班列（Aurora 链路）。它有复杂的内部设施（Aurora IP、FIFO、控制寄存器），但对外只暴露标准化的装卸接口（AXI Stream）。

  • strm_issue / strm_dump 是内陆集散中心（HLS 内核）：负责将货物从本地仓库（主机内存）运到口岸，或将口岸到达的货物运回仓库。它们执行简单的"搬运"职能，但必须在正确的时机启动，否则口岸会空闲或堵塞。

  • krnl_aurora_test.cfg 是铁路调度图：它不处理任何数据，而是定义了所有设施之间的连接关系——哪条轨道通向哪个口岸、哪个仓库对接哪条轨道、口岸使用哪个国际通道（GT 端口）。

  • Host 程序 是物流指挥中心：它生成测试货物、监控口岸状态、协调集散中心的启停，并最终验证货物完整性。

关键洞察：配置文件的每一行都在回答一个问题——数据从哪里来？到哪里去？经过哪些中间节点？使用时钟域 A 还是 B？这种显式连接（explicit connectivity）是 Vitis 硬件链接阶段的核心抽象。

核心组件角色

1. strm_issue —— 发送端数据泵

这是一个极简的 HLS 内核，功能可以用一句话概括：将 AXI4 内存映射访问转换为 AXI Stream 连续流。

void strm_issue(hls::stream<ap_axiu<PTR_WIDTH, 0, 0, 0>>& data_output,
                ap_uint<PTR_WIDTH> *data_input,
                unsigned int byte_size)
{
    for (int i = 0; i < (byte_size / PTR_BYTE_WIDTH); i++) {
        #pragma HLS PIPELINE II = 1
        ap_axiu<PTR_WIDTH, 0, 0, 0> temp; 
        temp.data = data_input[i];
        data_output.write(temp);
    }
}

设计要点解析：

  • #pragma HLS PIPELINE II=1：这是性能的关键。II（Initiation Interval）为 1 意味着每个时钟周期都能发起一次新的内存读取和流写入。在 300 MHz 下，这提供约 9.6 GB/s 的理论带宽（256 bits × 300 MHz ÷ 8），足以喂饱 40 Gbps 的 Aurora 链路。

  • ap_axiu<PTR_WIDTH, 0, 0, 0>：使用零宽度的 sideband 信号（keep/strb/last/user），因为 Aurora 链路不需要这些元数据——它传输的是原始字节流。

  • 无流控依赖：该内核假设下游（krnl_aurora）总能及时消费数据。如果 Aurora 链路出现反压，data_output.write() 会阻塞，但由于是阻塞写，不会丢失数据——只是吞吐量下降。

2. krnl_aurora —— Aurora 链路的封装壳

这是整个系统的核心 RTL 模块，其内部结构体现了分层解耦的设计哲学：

// 顶层端口：平台接口 + GT 物理接口
module krnl_aurora (
    input           ap_clk,          // 300MHz 平台时钟
    input           ap_rst_n,
    // AXIS 平台侧接口
    output [255:0]  rx_axis_tdata,   // 接收数据到平台
    input  [255:0]  tx_axis_tdata,   // 发送数据来自平台
    // GT 物理接口
    input  [3:0]    gt_rxp_in, gt_rxn_in,  // 差分接收对
    output [3:0]    gt_txp_out, gt_txn_out, // 差分发送对
    input           gt_refclk_p, gt_refclk_n, // GT 参考时钟
    input           init_clk       // Aurora 初始化时钟
);

内部子模块分工：

子模块 功能 时钟域 关键作用
aurora_64b66b_0 Xilinx Aurora IP 核 user_clk (161 MHz) 完成 64B/66B 编码、链路训练、误码检测
axis_data_fifo_0_tx/rx 异步 AXI Stream FIFO 跨 ap_clk ↔ user_clk 解决时钟域交叉，提供弹性缓冲
krnl_aurora_control_s_axi AXI-Lite 从接口 ap_clk 暴露链路状态寄存器给主机
xpm_cdc_* XPM 原语 CDC 多域 同步状态信号，避免亚稳态

为什么需要两个 FIFO？

TX FIFO（发送）和 RX FIFO（接收）都承担时钟域隔离的职责，但它们的工作模式不同：

  • TX FIFO：ap_clk（写）→ user_clk（读）。主机以 300 MHz 突发写入，Aurora 以 161 MHz 匀速读出。FIFO 深度吸收了速率差异。

  • RX FIFO：user_clk（写）→ ap_clk（读）。Aurora 接收是匀速的，但主机可能间歇性读取。FIFO 防止接收溢出。

复位时序的精妙之处：

// 使用 27 位计数器产生 >1 秒的复位脉冲（假设 100MHz init_clk）
always @ (posedge init_clk or negedge ap_rst_n_i) begin
    if (!ap_rst_n_i) begin
        reset_pb_i <= 1'b1;
        pma_init_i <= 1'b1;
        rst_cnt    <= 27'b0;
    end else begin
        if (rst_cnt != 27'h7ff_ffff) rst_cnt <= rst_cnt + 1'b1;
        if (rst_cnt == 27'h7ff_ff00) pma_init_i <= 1'b0;  // 先释放 PMA
        if (rst_cnt == 27'h7ff_ffff) reset_pb_i <= 1'b0;  // 后释放逻辑复位
    end
end

这里隐藏着 Aurora IP 的上电时序要求：PMA（Physical Medium Attachment）必须先于逻辑层初始化。pma_init 提前 256 个时钟周期释放，确保 GT 收发器稳定后再启动协议层。

3. strm_dump —— 接收端数据收集器

与 strm_issue 对称，将 AXI Stream 写回 AXI4 内存：

void strm_dump(hls::stream<ap_axiu<PTR_WIDTH, 0, 0, 0>>& data_input,
               ap_uint<PTR_WIDTH> *data_output,
               unsigned int byte_size)
{
    for (int i = 0; i < (byte_size / PTR_BYTE_WIDTH); i++) {
        #pragma HLS PIPELINE II = 1
        ap_axiu<PTR_WIDTH, 0, 0, 0> temp = data_input.read();
        data_output[i] = temp.data;
    }
}

关键细节：data_input.read() 是阻塞读。如果上游没有数据，内核会停滞等待。这就是为什么 Host 代码要先启动 strm_dump，再启动 strm_issue——确保接收端"张开口袋"后再开始发送。

配置文件的连接语义

krnl_aurora_test.cfg 是 Vitis 链接阶段的蓝图。理解它的语法就是理解硬件如何组装。

内核实例化

nk=krnl_aurora:1:krnl_aurora_0
nk=strm_issue:1:strm_issue_0
nk=strm_dump:1:strm_dump_0
  • nk=<kernel_type>:<num_instances>:<instance_name_prefix>
  • 这里声明了三个内核各一个实例，分别命名为 *_0

AXI Stream 连接

stream_connect=krnl_aurora_0.rx_axis:strm_dump_0.data_input
stream_connect=strm_issue_0.data_output:krnl_aurora_0.tx_axis

这定义了数据流的有向边：

  • strm_issue_0.data_output → krnl_aurora_0.tx_axis（发送路径）
  • krnl_aurora_0.rx_axis → strm_dump_0.data_input（接收路径）

注意命名约定：rx_axis 是从 Aurora 视角的"接收"，实际上是输出到下游；tx_axis 是 Aurora 的"发送"，实际上是接收自上游。

GT 和时钟的物理连接

connect=io_clk_qsfp_refclkb_00:krnl_aurora_0/gt_refclk
connect=krnl_aurora_0/gt_port:io_gt_qsfp_00
connect=krnl_aurora_0/init_clk:ii_level0_wire/ulp_m_aclk_freerun_ref_00

这是平台特定的绑定：

  • io_clk_qsfp_refclkb_00：Alveo U200 的 QSFP0 参考时钟接口（161.1328125 MHz）
  • io_gt_qsfp_00：QSFP0 的 GT 收发器差分对
  • ii_level0_wire/ulp_m_aclk_freerun_ref_00：平台提供的 100 MHz 自由运行时钟

为什么需要针对不同板卡注释/取消注释？

不同 Alveo 卡的时钟和 GT 命名不同：

  • U200/U250：使用 refclkb 和 level0_wire
  • U280/U50：使用 refclka 或其他变体
  • U55C：QSFP0 的命名也有差异

这种硬编码的平台依赖是当前设计的主要痛点——移植到新卡需要手动编辑配置文件。

Host 程序的 orchestration 逻辑

Host 代码不仅是"调用内核"，更是时序敏感的协调器。

初始化序列

// 1. 加载 xclbin
xrt::device device = xrt::device(DEVICE_ID);
xrt::uuid xclbin_uuid = device.load_xclbin(xclbin_file);

// 2. 创建内核对象（注意 krnl_aurora 用 xrt::ip，其他用 xrt::kernel）
xrt::ip krnl_aurora = xrt::ip(device, xclbin_uuid, "krnl_aurora");
xrt::kernel krnl_strm_issue = xrt::kernel(device, xclbin_uuid, "strm_issue");
xrt::kernel krnl_strm_dump = xrt::kernel(device, xclbin_uuid, "strm_dump");

关键区别：krnl_aurora 是 RTL IP 核，不是可启动的内核，因此使用 xrt::ip 接口进行寄存器读写；strm_issue/dump 是可启动的 HLS 内核，使用 xrt::kernel。

Aurora 就绪轮询

// 等待 Aurora 链路建立，3 秒超时
while (1) {
    uint32_t reg_read_data = krnl_aurora.read_register(0x10);
    if (reg_read_data == 0x11ff) {  // 14'b1_0001_1111_1111
        // channel_up=1, soft_err=0, hard_err=0, mmcm_locked=0, pll_lock=1
        // line_up[3:0]=1111, gt_powergood[3:0]=1111
        break;
    }
    if (timeout) return EXIT_FAILURE;
}

这里的魔数 0x11ff 是位域掩码，对应 aurora_status 寄存器的理想状态：

  • Bit 12 (channel_up)：链路层就绪
  • Bit 11-10 (soft_err/hard_err)：无错误
  • Bit 8 (gt_pll_lock)：GT PLL 锁定
  • Bit 7-4 (line_up)：4 条 lane 全部对齐
  • Bit 3-0 (gt_powergood)：4 个 GT 收发器电源正常

内核启动顺序

// 必须先启动接收端！
run_strm_dump.start();
std::this_thread::sleep_for(std::chrono::milliseconds(1));  // 给它时间准备

// 再启动发送端
run_strm_issue.start();
run_strm_issue.wait();
run_strm_dump.wait();

为什么顺序至关重要？

如果先启动 strm_issue，数据会涌入 Aurora TX，但 RX 端的 strm_dump 尚未准备好接收。虽然 FIFO 可以提供一定缓冲，但 Aurora 是全双工但同步的——如果接收端不消费，链路级的流量控制最终会影响发送端。更关键的是，loopback 测试模式下，接收不到的数据会堆积在 RX FIFO 直至溢出。

设计权衡与决策分析

1. RTL vs HLS 的分工选择

决策：Aurora 核心用 RTL 封装，数据搬运用 HLS。

权衡分析：

方案 优势 劣势 本设计选择
纯 RTL 完全控制时序和资源 开发慢、难维护 Aurora 核心部分采用
纯 HLS 快速迭代、易移植 对 GT 等硬核控制不足 数据搬运部分采用
混合（实际） 各取所长 接口一致性挑战 ✅ 最终方案

深层原因：Aurora IP 本身就是黑盒 RTL，且需要精确的时钟、复位和 GT 连接，这些是 HLS 难以表达的。而数据搬运是标准的"内存→流→内存"模式，HLS 的 hls::stream 和 m_axi 接口正好擅长此道。

2. 异步 FIFO 的深度选择

配置中使用的 axis_data_fifo 默认深度未显式设置（通常为 512 或 1024）。为什么不更大？

  • 资源成本：每增加一倍深度，BRAM 用量线性增长。Alveo 卡的 BRAM 虽多，但多个 Aurora 实例会快速耗尽。
  • 延迟考量：更大的 FIFO 意味着更高的最坏情况延迟。对于实时性敏感的应用，这是不可接受的。
  • 足够性分析：40 Gbps 链路在 161 MHz 下的有效数据率为 5 GB/s（256 bits × 161 MHz ÷ 8）。512 深度的 FIFO 可容纳约 16 KB，足够吸收毫秒级的时钟漂移。

3. 无 CRC 和流控的简化

Aurora IP 配置中关闭了 CRC 和流控（Flow Control）：

CONFIG.CRC {false}
CONFIG.FlowControl {None}

风险与收益：

  • 收益：降低逻辑复杂度、减少延迟、提高有效带宽（无协议开销）
  • 风险：位错误无法检测、接收端过载时无背压机制

适用场景判断：本设计用于受控环境的 loopback 测试——短距离光纤、低误码率、固定块大小传输。在这种场景下，CRC 可由上层协议补充，流控可通过 Host 层面的速率控制替代。

4. 平台相关性的硬编码

配置文件中对不同 Alveo 卡的时钟和 GT 连接使用了条件注释而非自动检测。

技术债务：

  • 每次移植到新卡需要手动编辑
  • 容易出错（忘记注释/取消注释）
  • 不支持运行时动态适配

改进方向：使用 Tcl 脚本在构建时根据 PLATFORM 变量自动生成正确的连接配置，或通过 Vitis 的平台抽象层查询可用接口。

新贡献者必读：陷阱与最佳实践

常见错误

1. 时钟连接错误

# 错误：将 init_clk 连接到错误的时钟源
connect=krnl_aurora_0/init_clk:ii_level1_wire/...  # U200 应该用 level0

症状：Aurora 状态寄存器显示 mmcm_not_locked_out=1，链路无法建立。

排查：检查目标平台的 platforminfo --verbose 输出，确认时钟名称。

2. 内核启动顺序颠倒

// 错误：先启动 issue
run_strm_issue.start();
run_strm_dump.start();  // 后启动 dump

症状：数据校验失败，或程序挂起（如果 RX FIFO 溢出导致 Aurora 链路级流控）。

3. 忽略 Aurora 就绪状态

// 错误：不检查状态直接启动内核
run_strm_issue.start();

症状：随机数据错误，因为链路尚未训练完成就开始传输。

扩展指南

添加第二个 Aurora 实例

  1. 复制 nk=krnl_aurora:1:krnl_aurora_0 为 nk=krnl_aurora:1:krnl_aurora_1
  2. 添加对应的 stream_connect 行
  3. 关键：确保使用不同的 GT Quad（如 io_gt_qsfp_01）和参考时钟
  4. 更新 Host 代码，创建第二个 xrt::ip 对象并轮询其状态

修改数据宽度

当前设计使用 256-bit（4 lanes × 64-bit/lane）。若要改为 128-bit（2 lanes）：

  1. 修改 Aurora IP 配置：CONFIG.C_AURORA_LANES {2}
  2. 更新 HLS 内核中的 PTR_WIDTH 为 128
  3. 重新计算吞吐量预期：128 bits × 161 MHz ÷ 8 = 2.58 GB/s

启用 CRC

在 gen_aurora_ip.tcl 中添加：

CONFIG.CRC {true}

然后修改 Host 代码，在传输完成后检查 Aurora 状态寄存器的 CRC 错误位。

总结

alveo_aurora_kernel_stream_config 模块展示了如何在 Vitis 流程中集成底层 GT 收发器功能。其核心设计智慧在于：

  1. 分层抽象：用 HLS 处理标准数据流，用 RTL 封装专用硬件，用配置文件声明连接关系
  2. 时钟域隔离：在每一处异步边界（161 MHz ↔ 300 MHz）使用 FIFO 和 CDC 原语
  3. 显式状态机：Host 代码通过轮询状态寄存器，将硬件的异步就绪事件转化为软件的同步控制流

理解这个模块，就理解了 Vitis 平台上自定义 RTL 内核与标准 HLS 内核协同工作的基本范式。
