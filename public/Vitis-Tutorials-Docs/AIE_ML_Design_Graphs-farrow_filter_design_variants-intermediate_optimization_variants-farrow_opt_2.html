farrow_opt_2 模块技术深度解析

一句话概括

farrow_opt_2 是 Farrow 滤波器的第二阶段优化实现，它通过将多项式插值计算拆分为多个并行 FIR 子滤波器 + 一个 Horner 结构合并核，在 AIE-ML 引擎上实现了可变分数延迟滤波的高效流水线处理。你可以把它想象成一个"精密加工车间"：四个并行的"磨床"（FIR 子滤波器）同时处理原始信号，然后一个"装配工"（Horner 合并核）根据实时变化的延迟参数，将这些半成品组装成最终的精确输出。

问题空间：为什么需要这个模块？

背景：Farrow 滤波器是什么？

在数字信号处理中，我们经常面临一个难题：如何对信号进行任意精度的分数延迟？ 传统做法是设计一组固定延迟的滤波器，但这在延迟需要动态变化的场景（如通信系统的时序同步、音频采样率转换）中非常低效。

Farrow 滤波器的核心洞察是：将理想分数延迟滤波器的系数表示为延迟参数的多项式函数。这样，对于任意延迟值 \(\mu\)，滤波器系数可以通过多项式求值动态生成，而无需重新设计或存储大量滤波器组。

计算挑战

Farrow 滤波器的直接实现面临两个关键瓶颈：

  1. 计算复杂度：每个输出样本需要进行多次 FIR 卷积和多项式求值
  2. 数据依赖：多项式求值的 Horner 结构存在顺序依赖，难以并行化

优化的演进路径

从模块树可以看出，farrow_opt_2 位于优化路径的中间位置：

farrow_baseline_graph → farrow_optimization_stage_1_graph → farrow_opt_2 → farrow_final_implementation_graph

farrow_opt_2 的核心设计决策是：将多项式系数的计算从运行时转移到编译时，通过预计算的 FIR 子滤波器实现并行化。具体来说，它将 3 阶多项式分解为 4 个并行 FIR 滤波器（对应多项式的 4 个系数），然后用 Horner 结构在最后一个核中完成合并。

组件角色详解

1. dut_graph —— 测试封装层

这是整个设计的顶层入口，负责连接外部世界（PLIO）与内部计算图（farrow_graph）。

关键设计点：

  • 使用条件编译区分仿真模式（AIE_SIM_ONLY）和实际硬件部署
  • 仿真模式下从文本文件读取输入/写入输出，便于验证
  • 硬件模式下仅创建裸 PLIO 端口，由外部系统连接
#ifdef AIE_SIM_ONLY
    sig_i[0] = input_plio::create("PLIO_i_0", plio_64_bits, "data/sig_i.txt");
#else
    sig_i[0] = input_plio::create("PLIO_i_0", plio_64_bits);
#endif

这种设计体现了可测试性优先的理念：相同的内核代码可以在纯软件仿真中快速迭代验证，无需每次都部署到硬件。

2. farrow_graph —— 计算拓扑定义

这是模块的核心，定义了 5 个内核实例及其连接关系。

关键设计决策：

决策 说明 权衡
4+1 内核拆分 4 个 FIR 核 + 1 个合并核 增加内核间通信开销，换取并行度和模块化
相同内核类复用 K1-K4 都使用 farrow_kernel1 代码复用，通过构造函数参数区分不同系数
90% 运行时比例 runtime<ratio> = 0.9 保留 10% 余量防止时序违规
栈位置绑定 location<stack> 与内核位置相同 减少内存访问延迟

系数设计：

std::vector<int16> taps1 = {206,-1264,6606,-14835,14835,-6606,1264,-206};   // y3
std::vector<int16> taps2 = {906,-3543,10352,-7628,-7628,10352,-3543,906};   // y2 (对称)
std::vector<int16> taps3 = {-51,316,-1652,20093,-20093,1652,-316,51};       // y1 (反对称)
std::vector<int16> taps4 = {-226,886,-2588,10099,10099,-2588,886,-226};     // y0 (对称)

这些系数不是任意的——它们对应于 Farrow 结构中 Lagrange 或样条插值的多项式基系数。注释中的 "sym" 和 "anti-sym" 标记暗示了潜在的进一步优化空间（利用对称性减少乘法次数），但在此优化阶段尚未充分利用。

3. farrow_kernel1 —— FIR 子滤波器

这是一个通用的 8 抽头 FIR 滤波器内核，被实例化 4 次分别处理不同的多项式系数分支。

内存模型：

private:
    alignas(32) TT_SIG f_state[STATE_LEN];      // 8 个状态的循环缓冲区
    alignas(32) int16 (&f_taps)[8];              // 对外部传入系数的引用
  • 状态数组：保存前 7 个输入样本，实现滑动窗口 FIR
  • 对齐要求：alignas(32) 确保 SIMD 加载指令可以高效工作
  • 引用语义：f_taps 是对构造时传入数组的引用，无拷贝开销

核心算法：

// 使用 AIE API 的滑动窗口乘法
acc_f3 = aie::sliding_mul_ops<16, 8, 1, 1, 1, int16, cint16>::mul(
    f_coeffs, 0, v_buff, 25
);

这里使用了 AIE 特有的 sliding_mul_ops——它可以在单个周期内完成 16 个输出的部分积计算，并通过滑动窗口机制高效复用输入数据。参数含义：

  • 16：每次处理的输出点数
  • 8：抽头数
  • 1, 1, 1：各种步长和重复参数

编译器提示：

chess_loop_range(BUFFER_SIZE/32, BUFFER_SIZE/32)
chess_prepare_for_pipelining

这些是 Xilinx AIE 编译器特有的 pragma，用于指导循环展开和流水线优化。chess_loop_range 告诉编译器循环的确切迭代次数，使其能够生成最优的调度方案。

4. farrow_kernel2 —— Horner 合并核

这是整个结构的"大脑"，负责根据延迟参数 \(\mu\) 将 4 个 FIR 输出合并为最终结果。

Horner 结构回顾： 对于 3 阶多项式 \(P(\mu) = y_0 + y_1\mu + y_2\mu^2 + y_3\mu^3\)，Horner 形式为：

\[P(\mu) = y_0 + \mu(y_1 + \mu(y_2 + \mu \cdot y_3))\]

这种形式将乘法次数从 6 次减少到 3 次，并具有良好的数值稳定性。

实现细节：

// 第一遍：计算 z = y2 + μ·y3
acc_x = aie::mac(aie::from_vector<TT_ACC>(*p_y2++, DNSHIFT), *p_y3++, del);
*p_z2++ = acc_x.to_vector<TT_SIG>(DNSHIFT);

// 第二遍：计算 z' = y1 + μ·z
acc_y = aie::mac(aie::from_vector<TT_ACC>(*p_y1++, DNSHIFT), *p_z2++, del);
*p_z1++ = acc_y.to_vector<TT_SIG>(DNSHIFT);

// 第三遍：计算输出 = y0 + μ·z'
acc_z = aie::mac(aie::from_vector<TT_ACC>(*p_y0++, DNSHIFT), *p_z1++, del);
*p_sig_o++ = acc_z.to_vector<TT_SIG>(DNSHIFT);

关键观察：

  1. 三轮遍历设计：由于 Horner 结构的顺序依赖性（每一层依赖上一层的输出），无法在一个循环内完成所有计算。代码采用三轮独立遍历，每轮处理一层 Horner 嵌套。

  2. 中间缓冲区 z：大小为 BUFFER_SIZE（1024），用于存储第一轮到第二轮、第二轮到第三轮之间的中间结果。

  3. 延迟参数的复用：注意 p_del_i 在三轮中被重置三次（p_del_i = aie::begin_vector<4>(del_i)），这意味着同一个延迟参数被读取三次。这是为了简化地址生成逻辑，代价是略微增加的内存带宽。

  4. 量化处理：DNSHIFT = 14 表示定点数右移 14 位，这是定点 DSP 的典型做法，用于在保持精度的同时避免溢出。

数据流追踪：一个样本的旅程

让我们追踪一个复数样本从输入到输出的完整路径：

1. dut_graph::sig_i[0] (PLIO)
   ↓ stream
2. farrow_graph::sig_i[0] (port<input>)
   ↓ connect<> (广播到 4 个内核)
3. farrow_kernel{1,2,3,4}_ins.in[0]
   ↓ 内部 FIR 处理 (8 抽头卷积)
4. farrow_kernel{1,2,3,4}_ins.out[0]
   ↓ connect<> (分别标记为 y3, y2, y1, y0)
5. farrow_kernel5_ins.in[1..4]
   ↓ Horner 合并 (三轮 MAC 运算)
6. farrow_kernel5_ins.out[0]
   ↓ connect<>
7. farrow_graph::sig_o[0]
   ↓ stream
8. dut_graph::sig_o[0] (PLIO)

延迟参数的路径：

dut_graph::del_i[0] → farrow_graph::del_i[0] → farrow_kernel5_ins.in[0]

注意延迟参数只进入 kernel5，不经过 FIR 子滤波器。这符合 Farrow 结构的数学定义：延迟参数仅在最后的插值步骤使用。

设计决策与权衡分析

1. 并行度 vs 通信开销

决策：将单核实现拆分为 5 个内核（4 FIR + 1 合并）

收益：

  • 4 个 FIR 可以完全并行执行
  • 每个内核可以运行在独立的 AIE 核心上
  • 符合 AIE 架构的众核并行设计理念

代价：

  • 内核间流通信引入延迟（约几个周期）
  • 增加了资源占用（5 个核心 vs 1 个核心）
  • 功耗增加

适用场景：当采样率要求超过单核处理能力时，这种拆分是必要的。对于较低采样率，farrow_baseline_graph 的单核实现可能更合适。

2. 定点数 vs 浮点数

决策：全程使用定点数（cint16, int16, cacc48/64）

收益：

  • AIE 核心的定点 SIMD 单元效率远高于浮点
  • 功耗更低
  • 确定性时序行为（无浮点异常）

代价：

  • 需要仔细管理量化噪声（DNSHIFT 参数）
  • 动态范围受限
  • 设计验证更复杂

权衡依据：Farrow 滤波器通常用于通信系统，这些系统本身就是定点的（ADC/DAC 输出），因此内部使用定点不会引入额外量化误差。

3. 状态管理策略

决策：farrow_kernel1 使用成员数组保存状态，farrow_kernel2 使用成员数组保存中间结果

对比：

特性 kernel1 (FIR) kernel2 (Horner)
状态类型 输入历史（8 样本） 中间计算结果（1024 样本）
生命周期 跨调用持久 单次调用内有效
初始化 构造函数清零 不需要
对齐要求 32 字节 32 字节

关键风险：farrow_kernel2 的 z 缓冲区大小为 BUFFER_SIZE（1024），如果输入缓冲区大小超过此值，会发生缓冲区溢出。当前代码假设输入严格为 1024 样本，这一契约由调用者保证。

4. 循环结构选择

决策：kernel2 使用三个独立循环而非嵌套循环

// 实际实现：三个顺序循环
for (rr=0; rr < BUFFER_SIZE/16; rr++) { /* 计算 z */ }
for (rr=0; rr < BUFFER_SIZE/16; rr++) { /* 计算 z' */ }
for (rr=0; rr < BUFFER_SIZE/16; rr++) { /* 计算输出 */ }

替代方案（未采用）：

// 理论上的单层循环（需要双缓冲）
for (rr=0; rr < BUFFER_SIZE/16; rr++) {
    // 同时处理三个阶段，需要复杂的索引管理
}

选择理由：

  • 代码可读性更高
  • 编译器更容易优化（明确的阶段边界）
  • 缓冲区访问模式更简单（顺序遍历）

潜在优化：如果内存带宽成为瓶颈，可以考虑融合循环，但需要引入双缓冲机制来解耦各阶段的依赖关系。

新贡献者必读：陷阱与注意事项

1. 缓冲区大小契约

危险：所有内核都硬编码了 BUFFER_SIZE = 1024

static constexpr unsigned BUFFER_SIZE = 1024;

如果上层应用传入不同大小的缓冲区，会导致：

  • 未处理的数据残留
  • 内存越界访问
  • 静默的错误结果

建议：在生产代码中添加断言检查：

assert(sig_i.size() == BUFFER_SIZE);  // 运行时检查

2. 状态持久性陷阱

farrow_kernel1 的状态数组 f_state 在构造函数中初始化为零：

for(int i=0; i<STATE_LEN; i++) {
    f_state[i] = (cint16){0,0};
}

但在实际运行中，状态会在 run() 调用之间持续：

*(v8cint16*)f_state = v_buff.extract<8>(3);  // 保存最后 8 个样本

陷阱：如果在仿真中重复使用同一个内核实例处理多个不相关的数据块，前一块的结尾会影响后一块的开头（这在连续流处理中是正确行为，但在分块测试中可能是意外）。

3. 延迟参数的格式

farrow_kernel2 期望的延迟参数类型是 int32：

typedef int32 TT_DEL;

但实际使用时被转换为 int16：

del = aie::vector_cast<int16>(*p_del_i++);

隐含契约：延迟参数必须已经缩放到合适的定点格式（通常是 Q15 格式，即 16 位有符号小数）。如果传入未经缩放的原始浮点延迟值，结果将完全错误。

4. 舍入模式的全局影响

构造函数中设置了全局舍入和饱和模式：

aie::set_rounding(aie::rounding_mode::symmetric_inf);
aie::set_saturation(aie::saturation_mode::saturate);

副作用：这些设置是内核级别的全局状态，可能影响同一内核内的其他运算。虽然在这个简单设计中只有一个运算类型，但在更复杂的内核中，这种全局设置可能导致意外的数值行为。

5. __restrict 关键字的重要性

所有缓冲区指针都标记了 __restrict：

void run(input_buffer<TT_SIG,...>& __restrict sig_i,
         output_buffer<TT_SIG,...>& __restrict y);

含义：向编译器保证该指针是访问对应内存的唯一方式（无别名）。这使得编译器可以激进地优化加载/存储指令的顺序，例如提前预取或延迟写回。

违反后果：如果调用者传入重叠的缓冲区（如输入输出指向同一块内存），优化后的代码可能产生错误结果。不要移除这些 __restrict 标记，除非你完全理解别名分析的影响。

6. 仿真与硬件的行为差异

dut_graph 中的条件编译：

#ifdef AIE_SIM_ONLY
    // 文件 I/O 模式
#else
    // 裸 PLIO 模式
#endif

常见错误：在仿真验证通过后直接部署到硬件，却忘记定义 AIE_SIM_ONLY 宏，导致尝试打开不存在的文件而崩溃。

最佳实践：在 Makefile 中明确区分仿真目标和硬件目标，不要依赖手动定义宏。

性能考量

吞吐能力估算

基于代码结构和 AIE-ML 架构特性：

  • FIR 子滤波器（kernel1）：

    • 每次循环处理 32 个样本（2 次 16 样本向量操作）
    • 循环次数：1024/32 = 32 次
    • 估计周期：约 100-200 周期/调用（高度依赖内存延迟）

  • Horner 合并核（kernel2）：

    • 三轮循环，每轮处理 16 个样本
    • 循环次数：1024/16 = 64 次 × 3 轮 = 192 次迭代
    • 估计周期：约 300-500 周期/调用

瓶颈分析：Horner 核是吞吐瓶颈，因为它必须串行执行三轮遍历。如果采样率要求极高，可能需要考虑：

  1. 进一步拆分 Horner 计算（空间换时间）
  2. 使用双缓冲重叠通信与计算
  3. 降低 BUFFER_SIZE 以减少延迟

资源占用

  • AIE 核心：5 个
  • 程序内存：每个内核约 1-2 KB
  • 数据内存：
    • kernel1：8×4 = 32 字节（状态）+ 8×2 = 16 字节（系数引用）
    • kernel2：1024×4 = 4 KB（中间缓冲区 z）
  • 流通道：5 条（4 条 FIR→Horner，1 条 输出）

总结

farrow_opt_2 展示了 AIE 架构下 Farrow 滤波器的一个经典优化路径：识别并行性 → 拆分计算图 → 优化数据局部性。它的核心价值在于：

  1. 教学意义：清晰展示了如何将数学结构（Farrow/Horner）映射到硬件架构
  2. 实用价值：在保持代码可读性的同时实现了显著的并行加速
  3. 演进基础：为后续进一步优化（如对称系数利用、循环融合）奠定了结构基础

对于新加入团队的开发者，建议按以下顺序深入理解：

  1. 先阅读 farrow_baseline_graph 了解单核实现
  2. 对比本模块的并行拆分策略
  3. 最后研究 final_aie_graph_implementation 看如何进一步优化
