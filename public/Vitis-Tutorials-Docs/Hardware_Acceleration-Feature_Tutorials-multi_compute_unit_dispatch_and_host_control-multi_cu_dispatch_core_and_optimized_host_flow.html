Multi-CU Dispatch Core and Optimized Host Flow

概述

本模块展示了如何通过乱序执行（Out-of-Order Execution）和多计算单元（Compute Unit, CU）调度，将 FPGA 加速器从"串行任务处理器"转变为"并行流水线工厂"。想象一下：你经营着一家餐厅，只有一个厨师（单 CU）时，客人必须排队等待；而当你雇佣多个厨师（多 CU）并引入智能订单调度系统（乱序命令队列），多位客人的订单可以同时被不同厨师处理——这就是本模块的核心思想。

在硬件加速场景中，FPGA 内核（Kernel）的执行通常涉及三个阶段：数据上传（Host→Device）→ 内核计算 → 结果回传（Device→Host）。如果采用传统的顺序执行模式，每个阶段都必须等待前一个完全完成才能开始，这就像让厨师做完一道菜、洗完手、再开始做下一道——效率极低。本模块通过 OpenCL 的乱序命令队列（CL_QUEUE_OUT_OF_ORDER_EXEC_MODE_ENABLE），实现了这三个阶段的流水线化重叠，配合多 CU 架构，最大化硬件利用率。

核心问题空间

为什么要做这个优化？

在典型的图像处理工作负载中（如 YUV 格式的 2D 滤波），一个完整的帧需要分别处理 Y、U、V 三个平面。如果没有优化：

  1. 顺序瓶颈：Y 平面的"上传→计算→回传"全部完成后，才能开始 U 平面
  2. CU 闲置：即使 FPGA 上有多个 CU，它们也只能串行工作
  3. 内存带宽浪费：数据传输和计算无法重叠，总线空闲与内核忙碌交替出现

本模块解决的问题可以量化为：如何将 \(N\) 个独立任务的执行时间从 \(N \times T_{sequential}\) 降低到接近 \(\max(T_{upload}, T_{compute}, T_{download})\)？

关键组件解析

1. Filter2DDispatcher —— "智能订单调度员"

这是整个系统的核心调度器，封装了 OpenCL 资源管理和命令提交逻辑。它的设计遵循工厂模式——每次调用 operator() 都会生成一个新的 Filter2DRequest，代表一个独立的处理任务。

为什么这样设计？

想象餐厅的订单系统：服务员（Dispatcher）接收顾客订单，将其分解为"备料→烹饪→上菜"三个步骤，并为每张订单创建一个追踪号（Request ID）。后厨（OpenCL Runtime）可以根据厨师空闲情况乱序执行，但顾客可以通过追踪号查询自己的订单状态。

// 构造函数：初始化 OpenCL 资源
Filter2DDispatcher(cl_device_id &Device, cl_context &Context, cl_program &Program) {
    mKernel = clCreateKernel(Program, "Filter2DKernel", &mErr);
    // 关键差异：host-final.cpp 启用了 CL_QUEUE_OUT_OF_ORDER_EXEC_MODE_ENABLE
    mQueue = clCreateCommandQueue(Context, Device, 
                                  CL_QUEUE_PROFILING_ENABLE | 
                                  CL_QUEUE_OUT_OF_ORDER_EXEC_MODE_ENABLE, &mErr);
}

2. Filter2DRequest —— "异步任务句柄"

这是一个轻量级的追踪结构，包含三个 OpenCL 事件对象，分别对应：

  • mEvent[0]：输入数据迁移完成（Coeffs + Src）
  • mEvent[1]：内核执行完成
  • mEvent[2]：输出数据回传完成
struct Filter2DRequest {
    cl_event mEvent[3];  // 事件链：Write → Execute → Read
    int      mId;        // 调试追踪 ID
    
    void sync() {
        // 只等待最后一个事件（Read 完成）即可确保整条链完成
        clWaitForEvents(1, &mEvent[2]);
        // 显式释放事件对象，避免内存泄漏
        clReleaseEvent(mEvent[0]);
        clReleaseEvent(mEvent[1]);
        clReleaseEvent(mEvent[2]);
    }
};

注意：这里的 sync() 设计体现了级联等待的思想——由于事件之间存在依赖关系（通过 clEnqueueTask 和 clEnqueueMigrateMemObjects 的参数指定），等待最后一个事件实际上保证了整个链条的完成。

3. 三阶段流水线命令提交

// Stage 1: 上传系数和源数据（Host → Device）
clEnqueueMigrateMemObjects(mQueue, 2, mSrcBuf, 0, 0, nullptr, &req->mEvent[0]);

// Stage 2: 执行内核（依赖 Stage 1 完成）
clEnqueueTask(mQueue, mKernel, 1, &req->mEvent[0], &req->mEvent[1]);

// Stage 3: 回传结果（依赖 Stage 2 完成，标记为回传到 Host）
clEnqueueMigrateMemObjects(mQueue, 1, mDstBuf, CL_MIGRATE_MEM_OBJECT_HOST, 
                           1, &req->mEvent[1], &req->mEvent[2]);

// 注册回调函数用于调试输出
clSetEventCallback(req->mEvent[1], CL_COMPLETE, event_cb, &req->mId);

这里的依赖关系形成了一个有向无环图（DAG）：

\[ \text{Write} \rightarrow \text{Execute} \rightarrow \text{Read} \]

OpenCL Runtime 可以在满足依赖的前提下自由重排命令，实现最大程度的并行。

关键设计决策与权衡

1. 乱序执行 vs 顺序执行

特性 host.cpp (顺序) host-final.cpp (乱序)
命令队列标志 CL_QUEUE_PROFILING_ENABLE CL_QUEUE_PROFILING_ENABLE | CL_QUEUE_OUT_OF_ORDER_EXEC_MODE_ENABLE
执行模型 FIFO，命令按提交顺序执行 DAG 调度，满足依赖即可执行
适用场景 单 CU、简单依赖 多 CU、复杂流水线
复杂度 低，易于调试 高，需要正确设置事件依赖

为什么选择乱序？

在多 CU 场景下，乱序执行允许 Runtime 将一个 CU 上的计算与另一个 CU 的数据传输重叠。例如：

  • CU1 正在执行第 N 个任务的计算
  • 同时 CU2 可以开始第 N+1 个任务的数据上传
  • 同时 CU3 可以回传第 N-1 个任务的结果

这种三级流水线可以将有效吞吐量提升近 3 倍（理想情况下）。

2. 同步策略：显式 Sync vs 隐式依赖

代码中采用了延迟同步策略——所有请求先提交到队列，最后再统一等待：

for(int xx=0; xx<numRuns; xx++) {
    // 连续提交 3 个平面（Y/U/V）的处理请求
    request[xx*3+0] = Filter(...);  // Y plane
    request[xx*3+1] = Filter(...);  // U plane
    request[xx*3+2] = Filter(...);  // V plane
    
    // 等待本轮全部完成
    request[xx*3+0]->sync();
    request[xx*3+1]->sync();
    request[xx*3+2]->sync();
}

权衡分析：

  • 优点：最大化并行度，Runtime 可以自由调度 3 个请求到不同的 CU
  • 风险：如果 numRuns 很大，会累积大量未完成的请求，可能导致内存压力
  • 替代方案：可以使用更细粒度的流控制（如滑动窗口），但会增加代码复杂度

3. 内存管理：CL_MEM_USE_HOST_PTR

代码使用了 CL_MEM_USE_HOST_PTR 标志创建缓冲区，这意味着：

mSrcBuf[1] = clCreateBuffer(mContext, 
                            CL_MEM_USE_HOST_PTR | CL_MEM_READ_ONLY,  
                            nbytes, src, &mErr);

含义：OpenCL 直接使用主机内存指针 src，而不是分配新的设备内存。这要求：

  • src 必须是 4KB 对齐（代码中使用 aligned_allocator<uchar>）
  • 数据迁移时可能涉及零拷贝（Zero Copy）或 DMA 传输，取决于平台

权衡：

  • 优点：减少一次内存拷贝，降低延迟
  • 约束：主机内存必须保持有效直到对应的 cl_mem 对象被释放
  • 可移植性：某些平台可能对 USE_HOST_PTR 的支持有限

4. 事件生命周期管理

void sync() {
    clWaitForEvents(1, &mEvent[2]);
    clReleaseEvent(mEvent[0]);
    clReleaseEvent(mEvent[1]);
    clReleaseEvent(mEvent[2]);
}

设计考量：

  • OpenCL 事件是引用计数的对象，必须通过 clReleaseEvent 显式释放
  • 在 sync() 中集中释放简化了调用者的责任
  • 如果忘记释放，会导致内存泄漏（虽然单次泄漏很小，但在高频循环中会累积）

C++ 实现细节分析

内存所有权模型

本模块采用了一种混合所有权策略，结合了 RAII 和手动资源管理：

1. Filter2DDispatcher —— RAII 容器

class Filter2DDispatcher {
private:
    cl_kernel         mKernel;      // 由 Dispatcher 拥有，析构时释放
    cl_command_queue  mQueue;       // 由 Dispatcher 拥有，析构时释放
    cl_context        mContext;     // 借用（外部传入，不释放）
    cl_mem            mSrcBuf[2];   // 由 Dispatcher 拥有，但复用于多个请求
    cl_mem            mDstBuf[1];   // 由 Dispatcher 拥有，但复用于多个请求
    // ...
public:
    ~Filter2DDispatcher() {
        clReleaseCommandQueue(mQueue);  // 释放拥有的资源
        clReleaseKernel(mKernel);
        // 注意：mSrcBuf/mDstBuf 在这里没有释放！
    }
};

所有权分析：

成员 所有权 释放位置 风险点
mKernel 独占 ~Filter2DDispatcher() 安全
mQueue 独占 ~Filter2DDispatcher() 安全
mContext 借用 外部管理 需确保外部生命周期更长
mSrcBuf/mDstBuf 未释放！ N/A ⚠️ 潜在泄漏

重要发现：当前代码中 mSrcBuf 和 mDstBuf 在 ~Filter2DDispatcher() 中没有释放！这是一个资源泄漏缺陷。正确的实现应该添加：

~Filter2DDispatcher() {
    clReleaseMemObject(mSrcBuf[0]);
    clReleaseMemObject(mSrcBuf[1]);
    clReleaseMemObject(mDstBuf[0]);
    clReleaseCommandQueue(mQueue);
    clReleaseKernel(mKernel);
}

2. Filter2DRequest —— 手动生命周期管理

struct Filter2DRequest {
    cl_event mEvent[3];  // 原始句柄，非智能指针
    int      mId;
    
    void sync();  // 等待并释放事件
};

为什么不用智能指针？

OpenCL 对象（cl_event, cl_mem 等）是 C API 的不透明句柄，不是 C++ 对象。虽然可以封装 std::unique_ptr 配合自定义删除器，但代码选择了更直接的手动管理，原因包括：

  • 与 OpenCL C API 的风格保持一致
  • 避免额外的抽象层带来的性能开销（虽然微乎其微）
  • 明确表达资源管理的语义（创建→使用→等待→释放的显式流程）

3. 主机内存缓冲区 —— STL 容器管理

std::vector<uchar, aligned_allocator<uchar>> y_src(nbytes);

这里使用了自定义分配器的 std::vector：

  • 所有权：std::vector 完全拥有缓冲区，遵循 RAII
  • 对齐保证：aligned_allocator 确保 4KB 对齐，满足 CL_MEM_USE_HOST_PTR 的要求
  • 生命周期：向量超出作用域时自动释放，无需手动 free

关键契约：当使用 CL_MEM_USE_HOST_PTR 创建 cl_mem 时，主机缓冲区必须保持有效直到 cl_mem 被释放。在本模块中：

  • y_src 等向量在 main() 的整个执行期间都有效
  • cl_mem 对象（mSrcBuf[1]）在 Filter 调用时创建，但没有被显式释放

这实际上是一个悬垂引用风险：如果 Filter2DDispatcher 被设计为长期存活、多次调用，而主机向量被提前销毁，将导致未定义行为。

错误处理策略

本模块采用了**快速失败（Fail-Fast）**策略：

// xclbin_helper.cpp
if (err != CL_SUCCESS) {
    std::cout << "ERROR: failed to get device_id\n";
    exit(1);  // 直接退出，无恢复
}

权衡分析：

  • 优点：简化控制流，错误不会静默传播
  • 缺点：不适合生产环境的健壮性要求（如需要清理资源、记录日志、尝试备用方案等）
  • 改进方向：在生产代码中，应使用异常或返回码机制，确保析构函数被调用

Const-Correctness 与可变性

代码在 const-correctness 方面较为宽松：

// 当前实现：参数非常量
Filter2DRequest* operator()(short *coeffs, unsigned char *src, ...);

// 更严格的接口应该标记输入为 const
Filter2DRequest* operator()(const short *coeffs, const unsigned char *src, ...);

代码演进：host.cpp vs host-final.cpp

两个文件几乎完全相同，唯一的区别在于命令队列的创建：

// host.cpp - 顺序执行
mQueue = clCreateCommandQueue(Context, Device, CL_QUEUE_PROFILING_ENABLE, &mErr);

// host-final.cpp - 乱序执行
mQueue = clCreateCommandQueue(Context, Device, 
                              CL_QUEUE_PROFILING_ENABLE | 
                              CL_QUEUE_OUT_OF_ORDER_EXEC_MODE_ENABLE, &mErr);

这个微小的改动（增加一个标志位）带来了架构层面的变化：

  • host.cpp：适用于验证功能正确性，调试更简单（命令按序执行，行为确定）
  • host-final.cpp：生产环境使用，最大化性能潜力

这种渐进式优化的策略体现了 FPGA 开发的最佳实践：先保证正确性，再追求性能。

  • xclbin_helper.h（本模块内）：封装了 XCLBIN 文件加载、平台选择、设备初始化的标准流程。这是所有 Vitis 主机应用的通用基础设施。

  • coefficients.h / filter2d.h（本模块内）：定义了 15×15 的 2D 滤波器核系数和参考软件实现。filterCoeffs 数组包含 Identity、Blur、Motion Blur、Emboss 四种预设滤波器。

平行变体

  • opencv_integrated_dispatch_variant：host_opencv.cpp 展示了如何将相同的调度模式与 OpenCV 图像 I/O 集成，支持真实图片（BMP 格式）的处理。这在本模块的基础上增加了图像格式转换层（IplImage2Raw / Raw2IplImage）。

下游扩展

  • host_timing_and_logging_support：可能提供更精细的性能分析工具和结构化日志记录机制，用于深度优化场景。

横向关联

  • mixing_c_and_rtl_kernels_integration：展示了如何在同一设计中混合使用 HLS C/C++ 内核和 RTL 内核。本模块的调度模式可以应用于这种混合架构。

  • dataflow_debug_and_deadlock_analysis：当流水线并行度提高时，死锁和性能瓶颈分析变得至关重要。该模块提供的调试技术可用于诊断本模块的调度问题。

新贡献者注意事项

1. 事件依赖链的正确性

最容易出错的地方是事件依赖的设置。检查以下参数：

// 错误示例：Stage 2 不等待 Stage 1
clEnqueueTask(mQueue, mKernel, 0, nullptr, &req->mEvent[1]);  // ❌

// 正确：Stage 2 明确等待 Stage 1 的事件
clEnqueueTask(mQueue, mKernel, 1, &req->mEvent[0], &req->mEvent[1]);  // ✓

如果依赖设置错误，在乱序队列中可能导致竞态条件（Race Condition），表现为间歇性的数据损坏或程序崩溃。

2. 内存对齐要求

std::vector<uchar, aligned_allocator<uchar>> y_src(nbytes);

如果不使用 aligned_allocator，CL_MEM_USE_HOST_PTR 可能失败或导致性能下降。始终确保：

  • 主机缓冲区 4KB 对齐
  • 缓冲区大小是页面大小的整数倍（如果需要）

3. 回调函数的线程安全

void event_cb(cl_event event, cl_int cmd_status, void *id) {
    if (getenv("XCL_EMULATION_MODE") != NULL) {
        std::cout << "  kernel finished processing request " << *(int *)id << std::endl;
    }
}

回调函数由 OpenCL Runtime 的内部线程调用，不要在其中执行耗时操作或修改共享状态（除非使用适当的同步）。当前实现仅做打印，是安全的。

4. 资源泄漏检查

确保所有 clCreate* 都有对应的 clRelease*：

创建 释放 位置
clCreateKernel clReleaseKernel ~Filter2DDispatcher()
clCreateCommandQueue clReleaseCommandQueue ~Filter2DDispatcher()
clCreateBuffer clReleaseMemObject 隐式（缓冲区复用）
clEnqueue*(..., &event) clReleaseEvent Filter2DRequest::sync()

注意：mSrcBuf 和 mDstBuf 在 Filter2DDispatcher 中是成员变量，被复用于多个请求。这种设计假设同一时刻只有一个请求在使用这些缓冲区——如果有并发请求，需要为每个请求单独分配缓冲区。

5. 性能验证方法

代码内置了 CPU 参考实现对比和性能报告：

if (getenv("XCL_EMULATION_MODE") == NULL) {
    // 仅在硬件运行模式下报告性能
    std::chrono::duration<double> fpga_duration = fpga_end - fpga_begin;
    std::cout << "FPGA Throughput: " 
              << (double) numRuns*3*nbytes / fpga_duration.count() / (1024.0*1024.0)
              << " MB/s" << std::endl;
}

重要：模拟模式（Emulation）下的计时没有意义，因为模拟器执行速度远慢于实际硬件。

总结

本模块展示了 FPGA 异构计算中的核心优化技术——乱序流水线调度。关键要点：

  1. 抽象层设计：Filter2DDispatcher 封装了复杂的 OpenCL 命令管理，对外提供简单的函数调用接口
  2. 异步模式：通过 Filter2DRequest 实现"提交-分离-同步"的异步编程模型
  3. 事件驱动依赖：利用 OpenCL 事件机制构建执行依赖图，让 Runtime 自动优化调度
  4. 渐进优化：从顺序执行（host.cpp）到乱序执行（host-final.cpp）的演进路径清晰可控

理解这个模块的设计，将帮助你掌握 FPGA 加速器编程的核心范式：把计算切分为可重叠的阶段，用依赖而非顺序表达执行约束，让底层 Runtime 发挥硬件的最大潜力。
