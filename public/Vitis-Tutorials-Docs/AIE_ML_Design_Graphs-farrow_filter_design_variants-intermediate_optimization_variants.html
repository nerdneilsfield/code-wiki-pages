中间优化变体 (Intermediate Optimization Variants)

一句话概括

这是一个展示如何通过计算并行化和资源扩展将 Farrow 滤波器性能从 371 MSPS 提升到 1 GSPS 的 AIE-ML 架构优化教程——核心思想是"一个核做一件事，用空间换时间"。

问题背景：为什么需要这个模块？

Farrow 滤波器的本质

Farrow 滤波器是一种分数延迟滤波器，广泛应用于数字接收机的时序同步。它的核心数学原理是将分数延迟 \(\mu\) 表示为多项式：

\[y(n) = \sum_{k=0}^{M} \mu^k \cdot y_k(n)\]

其中 \(y_k(n)\) 是通过不同子滤波器（\(f_0, f_1, f_2, f_3\)）处理后的中间结果。

AIE → AIE-ML 迁移的挑战

当从 AIE 架构迁移到 AIE-ML 架构时，遇到了一个关键障碍：

特性 AIE AIE-ML
预加器 (Pre-adder) ✅ 支持 ❌ 不支持
对称滤波优化 API sliding_mul_sym_xy_ops<> 不可用

AIE 架构利用预加器只需存储一半抽头系数（利用对称性计算另一半），但 AIE-ML 没有这一硬件特性。这意味着：

  1. 必须使用完整系数 —— 内存带宽和计算量增加
  2. 必须更换 API —— 从 sliding_mul_sym_xy_ops<8,8,...> 切换到 sliding_mul_ops<16,8,...>
  3. 性能下降 —— 初始移植后仅能达到 371.5 MSPS，远低于 1 GSPS 目标

优化的核心问题

如何用更多的 AIE-ML 计算单元来弥补单核算力的不足？

这就是 intermediate_optimization_variants 模块要回答的问题。

心智模型：流水线工厂

想象 Farrow 滤波器是一个汽车装配工厂：

┌─────────────────────────────────────────────────────────────┐
│                    Farrow 滤波器 "工厂"                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  原材料输入                                                    │
│  ┌─────────┐                                                │
│  │ sig_i   │ ──► 冲压车间 (4个子滤波器 f0,f1,f2,f3)              │
│  └─────────┘        │                                        │
│                     ▼                                        │
│            ┌─────────────────┐                              │
│            │ 子滤波器输出     │ y0, y1, y2, y3               │
│            └─────────────────┘                              │
│                     │                                        │
│                     ▼                                        │
│            ┌─────────────────┐                              │
│            │ 总装车间         │ 按延迟系数 μ 加权求和           │
│            │ (farrow_kernel2) │                              │
│            └─────────────────┘                              │
│                     │                                        │
│                     ▼                                        │
│            ┌─────────────────┐                              │
│            │ 成品输出 sig_o   │                              │
│            └─────────────────┘                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘

初始方案（farrow_port_initial）的问题：

  • 一个工人（kernel）负责全部 4 个冲压工序
  • 每个循环迭代要做 4 次滤波，II（Initiation Interval）高达 105 周期
  • 相当于一个工人手忙脚乱地同时操作 4 台机器

优化思路：

  • Opt 1: 两个工人，每人负责 2 个冲压工序
  • Opt 2: 四个工人，每人专精 1 个冲压工序

这就是本模块展示的两种中间优化方案。

数据流详解

farrow_opt_1 的数据流

sig_i (cint16) ──┬──► farrow_kernel1_ins ──┬──► y3 ──┐
                 │   [taps1_2: 16 coeffs]  └──► y2 ──┤
                 │                                    │
                 └──► farrow_kernel2_ins ──┬──► y1 ──┼──► farrow_kernel3_ins ──► sig_o
                     [taps3_4: 16 coeffs]  └──► y0 ──┘
                                                     ▲
del_i (int32) ───────────────────────────────────────┘

关键设计点：

  • farrow_kernel1 被实例化两次，分别处理 (f0,f1) 和 (f2,f3)
  • 每个实例处理 16 个抽头系数（完整的两组 8-tap 滤波器）
  • 输出端口从 4 个减少到 2 个（每 kernel 输出 2 路）

farrow_opt_2 的数据流

sig_i (cint16) ──┬──► kernel1_ins ──► y3 ──┐
                 ├──► kernel2_ins ──► y2 ──┤
                 ├──► kernel3_ins ──► y1 ──┼──► kernel5_ins ──► sig_o
                 └──► kernel4_ins ──► y0 ──┘
                                             ▲
del_i (int32) ──────────────────────────────┘

关键设计点：

  • farrow_kernel1 被实例化四次，每个只处理 8 个抽头（单滤波器）
  • 输出端口简化为 1 个
  • 实现了真正的"一核一功能"极致并行

核心组件详解

1. dut_graph 类（应用层封装）

位于 farrow_app.cpp 中的顶层封装，负责：

  • PLIO 接口管理：创建与可编程逻辑 (PL) 的数据通道
  • 仿真/硬件模式切换：通过 AIE_SIM_ONLY 宏区分仿真数据文件和真实硬件接口
  • Graph 连接：将外部 PLIO 连接到内部 farrow_graph
class dut_graph : public graph {
  farrow_graph farrow;  // 内部实际计算图
public:
  std::array<input_plio,1> sig_i;   // 信号输入
  std::array<input_plio,1> del_i;   // 延迟输入
  std::array<output_plio,1> sig_o;  // 信号输出
  // ...
};

2. farrow_graph 类（计算图定义）

这是真正的计算拓扑定义，展示了两种优化策略的差异：

Opt 1 版本（3 个 Kernel）

// farrow_opt_1/farrow_graph.h
class farrow_graph : public graph {
  kernel farrow_kernel1_ins;  // 处理 taps1_2 (f0,f1)
  kernel farrow_kernel2_ins;  // 处理 taps3_4 (f2,f3)
  kernel farrow_kernel3_ins;  // 最终计算
  
  // 抽头系数：每组包含两个滤波器的完整系数
  std::vector<int16> taps1_2 = { /* 16 values for f0 & f1 */ };
  std::vector<int16> taps3_4 = { /* 16 values for f2 & f3 */ };
};

Opt 2 版本（5 个 Kernel）

// farrow_opt_2/farrow_graph.h
class farrow_graph : public graph {
  kernel farrow_kernel1_ins;  // f0 only
  kernel farrow_kernel2_ins;  // f1 only
  kernel farrow_kernel3_ins;  // f2 only
  kernel farrow_kernel4_ins;  // f3 only
  kernel farrow_kernel5_ins;  // 最终计算
  
  // 抽头系数：每组仅 8 个值（单滤波器）
  std::vector<int16> taps1 = { /* 8 values for f0 */ };
  std::vector<int16> taps2 = { /* 8 values for f1 */ };
  // ...
};

3. farrow_kernel1（FIR 滤波器核）

执行实际的 FIR 滤波运算，关键实现细节：

构造函数

farrow_kernel1( int16 (&taps)[8] ) : f_taps(taps)
{
  // AIE-ML 特定的舍入和饱和模式设置
  aie::set_rounding(aie::rounding_mode::symmetric_inf);
  aie::set_saturation(aie::saturation_mode::saturate);
  
  // 状态初始化（用于保持滤波器状态）
  for(int i=0;i<STATE_LEN;i++) {
    f_state[i] = (cint16){0,0};
  }
}

Run 方法核心逻辑

void run(input_buffer<TT_SIG>& sig_i, output_buffer<TT_SIG>& y)
{
  // 使用 AIE-ML 优化的滑动乘法
  acc_f3 = aie::sliding_mul_ops<16, 8, 1, 1, 1, int16, cint16>::mul(
      f_coeffs, 0, v_buff, 25);
  
  // 结果右移 14 位（定点数归一化）
  *p_y++ = acc_f3.to_vector<TT_SIG>(DNSHIFT);
}

类型定义说明：

类型别名 实际类型 含义
TT_SIG cint16 复数信号样本（16-bit I/Q）
TT_ACC cacc64 64-bit 复数累加器（AIE-ML 支持）
TT_DEL int32 延迟参数（32-bit 整数）
BUFFER_SIZE 1024 每批次处理样本数
DNSHIFT 14 定点数归一化移位量

4. farrow_kernel2（多项式插值核）

执行最终的 Horner 形式多项式计算：

z2 = y2 + μ·y3
z1 = y1 + μ·z2
sig_o = y0 + μ·z1

使用 aie::mac（乘累加）指令高效实现：

acc_x = aie::mac(aie::from_vector<TT_ACC>(*p_y2++, DNSHIFT), 
                 *p_y3++, del);

注意：使用了内部缓冲区 z[BUFFER_SIZE] 存储中间结果 z2 和 z1。

关键设计决策与权衡

决策 1：从对称优化到全系数计算

方面 AIE (原设计) AIE-ML (本模块)
API sliding_mul_sym_xy_ops<8,8,...> sliding_mul_ops<16,8,...>
系数存储 8 个（利用对称性） 16 个（完整存储）
计算效率 高（预加器辅助） 中（纯 MAC）
内存占用 低 高

为什么选择接受更高的内存开销？

因为 AIE-ML 的硬件限制使对称优化不可行。与其强行模拟对称计算（会增加复杂度和 II），不如直接采用全系数方案，换取更可预测的性能和更简单的代码。

决策 2：渐进式并行化策略

初始移植:  1 Tile × 4 Filters  = 371.5 MSPS  (II=105)
    ↓
Opt 1:     2 Tiles × 2 Filters = 667.1 MSPS  (II=58)
    ↓
Opt 2:     4 Tiles × 1 Filter  = 1011.8 MSPS (II=29)

为什么是渐进式而不是一步到位？

  1. 教学目的：展示性能瓶颈如何随并行度变化
  2. 资源评估：帮助理解 Tile 数量与性能的近似线性关系
  3. 工程实践：在实际项目中，通常需要在性能和资源消耗之间找到平衡点

决策 3：保持 kernel 代码复用

注意到 farrow_kernel1 在 Opt 1 和 Opt 2 中是同一个类，只是通过模板参数和构造函数实现不同的行为：

  • Opt 1: 传入 16 个系数，输出 2 路结果
  • Opt 2: 传入 8 个系数，输出 1 路结果

这种设计的优势：

  • ✅ 减少代码重复
  • ✅ 便于维护和调试
  • ⚠️ 需要仔细验证不同配置下的边界条件

性能数据对比

设计方案 Tile 数量 Throughput II (cycles) 备注
farrow_port_initial 2 371.5 MSPS 105 基线移植
farrow_opt_1 3 667.1 MSPS 58 第一次优化
farrow_opt_2 5 1011.8 MSPS 29 满足 1 GSPS 目标

关键洞察：

  • 每增加一个专用 Filter Tile，II 大约减半
  • 最终 II=29 接近理论最优（每样本 1 周期 @ 1.25GHz = 1.25 GSPS）
  • 实际吞吐量略低于理论值 due to overhead

新贡献者必读：陷阱与注意事项

1. API 兼容性问题

⚠️ 不要在 AIE-ML 上使用 sliding_mul_sym_xy_ops

编译错误示例：

error: constraints not satisfied for alias template 'sliding_mul_sym_xy_ops'
note: because 'arch::is(arch::AIE)' evaluated to false

解决方案：始终使用 sliding_mul_ops<16, 8, 1, 1, 1, int16, cint16>

2. 抽头系数长度匹配

确保传递给 kernel 的系数数组长度与代码中的访问模式一致：

// Opt 2: 8 个系数
std::vector<int16> taps1 = {206,-1264,6606,-14835,14835,-6606,1264,-206};
farrow_kernel1_ins = kernel::create_object<farrow_kernel1>(taps1);

// kernel 内部使用 load_v<8>
aie::vector<int16,8> f_coeffs = aie::load_v<8>(f_taps);

不匹配会导致未定义行为或编译错误。

3. Buffer Size 一致性

所有 kernel 的 BUFFER_SIZE 必须一致（本设计中为 1024）：

  • Graph 中的 buffer 声明
  • Kernel 的 input_buffer/output_buffer 声明
  • Host 端的 DMA 传输大小

4. 舍入和饱和模式

AIE-ML 要求显式设置算术模式：

aie::set_rounding(aie::rounding_mode::symmetric_inf);
aie::set_saturation(aie::saturation_mode::saturate);

忘记设置可能导致数值精度问题。

5. Runtime Ratio 配置

runtime<ratio>(farrow_kernel1_ins) = 0.9;

这表示该 kernel 占用 Tile 90% 的计算资源。如果多个 kernel 共享一个 Tile，需要调整此值以避免资源冲突。

总结

intermediate_optimization_variants 模块展示了 AIE-ML 架构下调试性能的经典方法论：

  1. 识别瓶颈：通过 II 分析找到限制吞吐量的关键 kernel
  2. 任务拆分：将重负载 kernel 的工作拆分到多个 Tile
  3. 渐进验证：每一步都验证功能和性能，确保方向正确
  4. 资源权衡：用更多的 Tile 换取更高的并行度
