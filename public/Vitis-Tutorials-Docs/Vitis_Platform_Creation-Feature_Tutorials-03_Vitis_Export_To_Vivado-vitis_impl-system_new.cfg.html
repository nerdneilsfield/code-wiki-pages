Vitis Export to Vivado: 多实例 AIE Graph 连接配置

概述

本模块展示了如何在 Vitis Export to Vivado 流程中支持多实例 AIE Graph的复杂连接场景。通过 system_new.cfg 配置文件，系统定义了两个独立的 AIE 处理流水线（clipgraph1 和 clipgraph2），每个流水线包含完整的 interpolator → polar_clip → classifier 数据处理链。

该设计模式的核心价值在于：在单一平台上实现计算资源的横向扩展，同时保持各实例间的数据隔离和独立控制。这解决了大规模信号处理应用中"单一路径吞吐量不足"和"需要并行处理多路独立数据流"的工程需求。

内核实例化映射

# 内核数量声明 - nk=<kernel>:<count>:<instance_names>
nk=aie_stream_in:1:aie_stream_in      # 共享数据源
nk=stream_in:2:stream_in1.stream_in2  # 两个输出DMA
nk=polar_clip:2:polar_clip1.polar_clip2  # 两个PL处理单元

这种命名约定遵循 <basename><index> 模式，使得：

  • 同一类型的多个实例具有可预测的标识符
  • 主机代码可以通过 {kernel}:{instance} 语法精确寻址特定实例
  • 连接配置可以清晰区分不同实例的端口

关键设计决策

1. 时钟域规划

freqhz=200MHz:stream_in1.ap_clk
freqhz=200MHz:stream_in2.ap_clk

[clock]
freqHz=100000000:polar_clip1.ap_clk
freqHz=100000000:polar_clip2.ap_clk

设计考量：

  • stream_in 内核运行在 200MHz，匹配 AIE PLIO 的典型速率
  • polar_clip 内核运行在 100MHz，反映其 CORDIC 计算的迭代特性
  • 同一类型的所有实例共享相同时钟频率，简化时序收敛

权衡分析：

方案 优点 缺点
统一时钟 简化 CDC 设计，减少时钟资源 无法针对各内核优化频率
独立时钟 最大化各内核性能 增加跨时钟域复杂性
当前方案 按功能分组时钟，平衡性能与复杂度 同组内核需满足最低要求

2. 连接命名空间设计

AIE 端口的实例化命名采用 _<index> 后缀约定：

sc=ai_engine_0.clip_in_1:polar_clip1.in_sample   # 实例1的输入
sc=ai_engine_0.clip_in_2:polar_clip2.in_sample   # 实例2的输入

这与单实例配置的命名形成对比：

  • 单实例：clip_in, clip_out, DataIn1, DataOut1
  • 多实例：clip_in_1, clip_out_1, DataIn_1, DataOut_1

关键洞察：下划线位置的变化（DataIn1 vs DataIn_1）反映了不同层次的命名空间——前者是 Vitis 自动生成的端口名，后者是显式配置的实例化标识。

数据流时序分析

端到端延迟模型

对于每个处理实例，数据流经以下阶段：

\[ T_{total} = T_{source} + T_{interp} + T_{cordic} + T_{classify} + T_{sink} \]

其中：

  • \(T_{source}\): aie_stream_in 从 DDR 读取的延迟 (~μs 级)
  • \(T_{interp}\): 27-tap FIR 插值器处理时间 (~256 样本 × 2x 插值)
  • \(T_{cordic}\): 6 步 CORDIC 迭代 + 阈值判断
  • \(T_{classify}\): 逐样本象限分类
  • \(T_{sink}\): stream_in 写回 DDR 的延迟

流水线并发性

由于两个实例共享同一个 aie_stream_in 数据源，实际并发模型为：

时间轴 →

Instance 1: [读取][FIR][CORDIC][分类][写入]
Instance 2:       [读取][FIR][CORDIC][分类][写入]
                      ↑
                 数据源时分复用

这意味着理论峰值吞吐量受限于数据源带宽，而非计算资源。在实际部署中，如果两路处理需要完全并行，应考虑：

  1. 使用独立的 DMA 源（如 stream_out RTL IP 替代 aie_stream_in）
  2. 增加 AIE 阵列中的数据缓冲深度
  3. 采用 ping-pong 缓冲区实现真正的双缓冲

集成注意事项

1. VMA 导出与导入顺序

当修改为多实例配置后，必须遵循正确的工具流顺序：

# 1. 重新编译 AIE Graph（使用 multi_graph.cpp）
make TARGET=hw_emu XSA=hw_emu.xsa graph

# 2. 重新编译新增的内核（aie_stream_in）
make TARGET=hw_emu XSA=hw_emu.xsa kernels

# 3. 导出新的 VMA
make TARGET=hw_emu XSA=hw_emu.xsa export

# 4. 在 Vivado 中移除旧 VMA 并导入新 VMA
vivado -mode batch -source viv_changes.tcl

常见错误：忘记更新 KERNEL_XO 列表或 CONFIG 文件路径，导致链接阶段使用旧的单实例配置。

2. 主机代码适配

多实例配置要求主机代码显式管理多个 graph 句柄：

// 获取两个独立的 graph 句柄
auto cghdl_1 = xrt::graph(device, xclbin_uuid, "clipgraph1");
auto cghdl_2 = xrt::graph(device, xclbin_uuid, "clipgraph2");

// 分别启动
cghdl_1.run(1);
cghdl_2.run(1);

// 等待完成
cghdl_1.end();
cghdl_2.end();

注意 kernel 实例的命名语法：stream_in:{stream_in1} 用于指定具体实例。

3. 资源利用率预估

双实例配置的资源开销约为单实例的 1.8-1.9x（而非 2x），因为：

  • AIE 阵列资源：完全复制（2x）
  • PL 逻辑：接近 2x（polar_clip ×2, stream_in ×2）
  • NoC/DDR 带宽：取决于访问模式，可能 <2x

调试与验证建议

  1. 连接完整性检查：使用 v++ --link 的 --save-temps 选项保留中间连接报告
  2. 时序验证：确保 200MHz/100MHz 时钟域 crossing 已正确约束
  3. 数据一致性：比较两个实例的输出 golden 数据，验证功能等价性
  4. 性能基准：测量单实例 vs 双实例的总处理时间，验证并行效率
