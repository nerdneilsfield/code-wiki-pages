RTP 重配置流 (rtp_reconfiguration_flows)

一句话概括

本模块展示了如何在 AI Engine (AIE) 图(Graph)运行期间动态重配置运行时参数(Run-Time Parameters, RTP)，支持同步/异步、标量/数组、写入/回读等多种模式，实现无需停止硬件即可调整滤波器系数、增益值等关键参数。

问题空间：为什么需要 RTP 重配置？

想象你正在运行一个 5G 基带处理链，其中 AIE 实现了一个数字下变频器(DDC)。在运行过程中，你可能需要：

  1. 自适应调整：根据信道质量动态改变滤波器抽头系数
  2. 模式切换：从单载波模式切换到多载波模式，改变处理参数
  3. 校准补偿：根据温度传感器读数调整补偿增益

传统方案的局限：

  • 编译时固定：参数在编译 AIE 图时确定，运行时无法更改
  • 停止-重载：必须停止 AIE 图、重新编译、重新加载比特流，耗时数秒到数分钟
  • 多实例浪费：为每种参数组合维护独立的 AIE 图实例，消耗大量片上存储和计算资源

RTP 的解决方案：

  • 将关键参数标记为 Run-Time Parameter (RTP)
  • AIE 图运行期间，通过专用控制路径动态更新这些参数
  • 支持同步更新（阻塞直到 AIE 确认接收）或异步更新（写入后立即返回，稍后查询状态）
  • 支持标量 RTP（单个值）和数组 RTP（批量更新）
  • 支持回读（从 AIE 读取当前 RTP 值，用于调试或状态监控）

核心概念与心智模型

类比：剧院的舞台控制系统

想象 AIE 图是一个正在演出的剧院舞台：

  • 演员 (AIE 核)：正在执行计算（唱歌、跳舞）
  • 舞台布景 (RTP)：背景幕布、灯光颜色、音响效果参数
  • 舞台监督 (主机程序)：观察演出状态，决定何时调整布景

同步更新 (Sync RTP)： 舞台监督举起"暂停"牌子，演员完成当前动作后暂停，工作人员更换布景，演员看到新布景后继续演出。这是确定性的，但会引入短暂停顿。

异步更新 (Async RTP)： 舞台监督通过轨道系统推送新布景，演员在后台准备的瞬间滑入新布景，无需暂停演出。吞吐不间断，但存在新旧布景切换的短暂过渡期。

数组 RTP (Array RTP)： 不是更换单个道具，而是同时更换整套舞台布景（背景、灯光、道具）。适合场景切换（如从森林场景切换到城堡场景）。

回读 (Readback)： 舞台监督通过监控摄像头查看当前舞台上的实际布景，确认之前的更换命令是否生效，或检查演员是否意外移动了道具。

数据流详解

1. 初始化阶段

Host (PS)                                    PL (Programmable Logic)              AIE Array
┌──────────────┐                           ┌──────────────┐                    ┌──────────────┐
│  加载 XCLBIN  │                           │              │                    │              │
│  初始化 XRT   │                           │  s2mm (等待)  │                    │ AIE 图等待   │
│  实例化 graph │                           │  datagen     │                    │ RTP 初始化   │
└──────┬───────┘                           └──────┬───────┘                    └──────┬───────┘
       │                                          │                                  │
       └──────────────────────────────────────────┴──────────────────────────────────┘
                                          流式连接建立
                        datagen.out → ai_engine_0.Datain0
                        ai_engine_0.Dataout0 → s2mm.s

关键步骤：

  1. 主机加载 XCLBIN，初始化 xrt::device 和 xrt::uuid
  2. 实例化 AIE 图 (ai_engine_0) 和 PL 内核 (datagen、s2mm)
  3. 系统链接器已根据 system.cfg 建立 AXI-Stream 连接

2. 运行阶段：数据流与 RTP 控制的交织

正常数据流（独立于 RTP）：

datagen (PL) ──AXI4-Stream──▶ AIE Input ──▶ AIE Kernel ──▶ AIE Output ──AXI4-Stream──▶ s2mm (PL) ──▶ DDR

RTP 控制流（与数据流正交）：

Host (xrt::graph::update) ──▶ XRT Runtime ──▶ AIE RTP Port ──▶ Kernel Parameter

四种模式的关键差异：

模式 数据流时序 RTP 控制时序 关键观察
Sync RTP run(1) → 处理1帧 update() 在帧边界生效 内核在 run() 之间等待 RTP
Async RTP run() 连续运行 update() 立即写入内存 内核自主决定何时读取
Async Array RTP run() 连续运行 update() 触发 DMA 传输 整组原子替换，有 DMA 延迟
Async Array RTP Readback run() 连续运行 read() 采样当前值 读取的是快照，非精确同步

3. 收尾阶段

1. Host 调用 graph.end() 或等待自然完成
2. s2mm 将最后的数据写入 DDR
3. Host 从 DDR 读取结果（通过 xrt::bo::read 或 memcpy）
4. 验证结果，释放资源

组件深度剖析

1. HLS 内核配置 (config.cfg)

每个示例都包含 PL 内核的 HLS 配置。以 s2mm (Stream-to-Memory) 为例：

[hls]
freqhz=400000000          # 目标频率：400MHz
syn.top=s2mm              # 顶层函数名
syn.file=s2mm.cpp         # 源文件
flow_target=vitis         # 目标流程：Vitis
package.output.format=xo  # 输出格式：Xilinx Object
package.output.file=s2mm.xo

设计意图：

  • 400MHz 频率选择：这是 Versal AI Engine 与 PL 之间 AXI-Stream 接口的典型频率，平衡了时序收敛难度与带宽需求。
  • XO 格式：Vitis 链接器期望的输入格式，包含 RTL 包装器和元数据。

2. 系统连接配置 (system.cfg)

这是 Vitis 系统链接器的配置，定义了完整的系统拓扑：

[connectivity]
nk=s2mm:1:s2mm                    # 实例化 1 个 s2mm，命名为 s2mm
nk=datagen:1:datagen              # 实例化 1 个 datagen，命名为 datagen
stream_connect=ai_engine_0.Dataout0:s2mm.s    # AIE 输出 → s2mm
stream_connect=datagen.out:ai_engine_0.Datain0 # datagen → AIE 输入

关键概念：

  • nk 指令："num kernels"，实例化 RTL/IP 内核。格式为 nk=<kernel_type>:<num_instances>:<instance_name_prefix>。
  • stream_connect：建立 AXI4-Stream 连接。格式为 <source>:<sink>。
  • 隐式 AIE 图实例：ai_engine_0 是 AIE 图的默认实例名，其端口 Dataout0 和 Datain0 由 AIE 内核的 connect 语句定义。

3. 四种 RTP 模式的比较

模式 特点 适用场景 关键 API
Sync RTP (sync_rtp) 确定性更新，在图迭代边界应用 需要严格时序控制 graph.run(1) + graph.update()
Async RTP (async_rtp) 低延迟，随时更新 快速响应外部事件 graph.run() (async) + graph.update()
Async Array RTP (async_array_rtp) 批量原子更新 FIR/IIR 系数更新 graph.update(rtp, array, size)
Async Array RTP Readback (async_array_rtp_read) 更新+验证 调试和校准 graph.read(rtp, array, size)

设计权衡分析：

  1. 同步 vs 异步：

    • 同步模式简化了应用逻辑（无竞态条件），但最大更新频率受限于图迭代周期。
    • 异步模式提供了理论上的即时更新，但要求应用理解 RTP 更新与 AIE 内核执行之间的时序关系。

  2. 标量 vs 数组：

    • 标量 RTP 适合增益、阈值等单值参数。
    • 数组 RTP 通过单次事务更新多个相关值，保证了一组系数的原子性切换（避免滤波器在更新过程中使用混合的新旧系数）。

  3. 读回功能：

    • 看似简单的功能，实则涉及 AIE 内存与主机之间的 DMA 传输，会消耗总线带宽。
    • 主要用于调试和校准阶段，生产环境通常禁用。

设计决策与权衡

1. 为什么使用 PL 数据生成器/接收器？

你可能会问：为什么需要 datagen 和 s2mm 这样的 PL 内核？AIE 不能自己生成和消耗数据吗？

关键洞察：这是教程设计，而非生产必需。原因如下：

  • 可控的测试向量：datagen 可以精确生成已知模式（如线性递增、伪随机、脉冲序列），便于验证 AIE 内核输出是否符合预期。
  • 与主机内存交互：s2mm 将 AIE 的 AXI-Stream 输出写入 DDR，主机随后可以读取并分析结果。
  • 演示完整的端到端数据流：从主机 → PL → AIE → PL → 主机，涵盖了实际系统的所有关键路径。

在生产系统中，AIE 可能直接与 RF-ADC/DAC 接口，或者与 PCIe/DMA 引擎通信，不需要这些 PL 测试内核。

2. 配置分离：HLS config.cfg vs 系统 system.cfg

你可能会注意到配置被拆分为两个文件。这是有意为之的关注点分离：

  • config.cfg（HLS 阶段）：定义单个内核的实现细节——时钟频率、源文件、顶层函数名。这是一个"内核蓝图"。
  • system.cfg（系统链接阶段）：定义系统拓扑——实例化哪些内核、如何连接、使用什么时钟。这是"系统集成图"。

为什么这样分离？ 可重用性。同一个 s2mm 内核（由 config.cfg 定义）可以在完全不同的系统中使用（不同的 system.cfg 连接它到不同的 AIE 图或 DDR 控制器）。这种分离遵循了硬件设计中的标准 IP 重用模式。

3. AXI-Stream vs AXI-MM 的选择

观察 system.cfg 中的连接：datagen.out → ai_engine_0.Datain0 和 ai_engine_0.Dataout0 → s2mm.s。这些都是 AXI4-Stream 连接（单比特 TVALID/TREADY/TDATA 握手协议），而不是 AXI4-MM（内存映射，含地址线）。

为什么用 Stream？

  • 流式数据特性：信号处理本质上是流式的——采样点按时间顺序到达，不需要随机地址访问。
  • 确定性时序：AXI-Stream 的握手协议在硬件中实现简单，时序可预测，适合高吞吐量的流水线。
  • 零地址开销：不需要传输地址，节省总线带宽和逻辑资源。

何时用 AXI-MM？ 当需要主机通过指针随机访问数据（如从 DDR 读取大量历史样本做 FFT）时。本教程的 s2mm 内部可能使用 m_axi 接口将接收到的流数据写入 DDR，但对外与 AIE 的接口是 axis。

新贡献者必读：边缘情况与陷阱

陷阱 1：RTP 更新时序竞态

问题：在异步模式下，如果在 AIE 内核读取 RTP 的同时主机更新它，会发生什么？

答案：AIE 架构保证 RTP 读写的原子性——一个 32-bit 标量 RTP 的读写是原子的。但是，对于数组 RTP，如果没有使用正确的 API，内核可能读到部分更新的数据。

正确做法：

// 错误：多次单独更新可能导致内核读到中间状态
for (int i = 0; i < N; i++) {
    graph.update("coeff", i, value[i]);  // 假设有这种 API
}

// 正确：一次性原子更新整个数组
graph.update("coeff", values.data(), N);

陷阱 2：同步模式的"一帧延迟"陷阱

问题：在同步 RTP 模式下，我调用 graph.update() 后立即读取输出，为什么新参数没有生效？

答案：同步模式下，graph.update() 的参数在下一次 graph.run() 时才生效。这是设计上的确定性保证——当前迭代使用的参数在迭代开始时就已经锁定。

graph.run(1);           // 第 1 帧，使用初始参数
graph.update(param, v1); // 为第 2 帧准备新参数
graph.run(1);           // 第 2 帧，使用 v1
// 输出反映的是 v1 的效果

陷阱 3：PL 内核与 AIE 的时钟域交叉

问题：datagen 运行在 400MHz，ai_engine_0 运行在 1GHz，它们之间的 AXI-Stream 连接可靠吗？

答案：可靠，因为 AXI-Stream 协议内置了握手机制 (TVALID/TREADY) 来处理时钟域交叉 (CDC)。但是，FIFO 深度需要足够容纳跨时钟域的延迟。

在 system.cfg 中定义的连接：

stream_connect=datagen.out:ai_engine_0.Datain0

这实际上会经过 AXI4-Stream Interconnect，它会自动处理 CDC 和 FIFO 缓冲。但如果在 HLS 代码中使用 hls::stream 直接连接，你需要显式指定深度：

// 在 datagen 中（400MHz）
hls::stream<ap_uint<32>> out("out");
#pragma HLS stream variable=out depth=32  // 足够缓冲跨时钟域延迟

// 在 AIE 边界，1GHz 侧读取时会有自然的速率匹配

陷阱 4：数组 RTP 的最大尺寸限制

问题：我可以通过 RTP 传递多大的数组？

答案：AIE 架构对 RTP 传输有特定限制：

  • 标量 RTP：32-bit（int、float）或 64-bit（long long、double）
  • 数组 RTP：通常限制在 4KB 以内（具体取决于 AIE 代际）

如果你需要传递更大的参数块（如 1024-tap FIR 的所有系数），你需要：

  1. 使用 GMIO (Global Memory I/O) 代替 RTP
  2. 或者将系数分成多个 4KB 块，分批 RTP 更新

陷阱 5：HLS s2mm 的内存对齐要求

问题：s2mm 内核写入 DDR 时有什么对齐要求？

答案：s2mm 通常使用 m_axi 接口写入 DDR，默认情况下需要 64-byte 对齐（AXI4 协议要求）。在主机代码中分配缓冲区时：

// 错误：普通 new/malloc 可能不对齐
float* buffer = new float[1024];  // 可能 4-byte 对齐，不是 64-byte

// 正确：使用 XRT 的对齐分配器
auto buffer = xrt::bo(device, 4096, XCL_BO_FLAGS_HOST_ONLY);
void* ptr = buffer.map();

如果地址不对齐，HLS 生成的 m_axi 逻辑会处理它，但会损失性能（可能需要两次传输）。XRT 的 xrt::bo 自动保证 4KB 页对齐，满足所有 AXI 要求。

总结：什么时候用什么模式？

场景 推荐模式 原因
实时控制系统，参数更新必须与时序严格对齐 Sync RTP 确定性保证，无竞态条件
自适应算法，需要快速响应外部事件 Async RTP 低延迟，立即生效
滤波器系数更新，多个系数必须同时生效 Async Array RTP 原子性批量更新
算法调试，需要验证内核看到的参数值 Async Array RTP Readback 可验证 RTP 传输正确性

理解这些模式的设计哲学——同步保证确定性、异步追求低延迟、数组保证原子性——你就能够根据应用需求做出正确的架构选择。RTP 是 AIE 平台区别于传统 FPGA 的关键特性之一，掌握它意味着你能够构建真正的自适应系统——在运行的同时进化。

四种 RTP 模式对比

1. 同步标量 RTP (sync_rtp)

核心特性： 每次内核迭代必须等待一次 RTP 更新

// graph.h: 同步连接（默认）
adf::connect<adf::parameter>(value, k.in[1]);
// 等价于: adf::connect<adf::parameter>(value, sync(k.in[1]));

// graph.cpp: 每次迭代前更新
gr.run(4);
gr.update(gr.value, 10);   // 第1次迭代
gr.update(gr.value, 100);  // 第2次迭代
gr.update(gr.value, 1000); // 第3次迭代
gr.update(gr.value, 10000);// 第4次迭代

适用场景：

  • 需要严格按顺序执行的控制流程
  • 每个数据块需要不同的处理参数
  • 算法需要精确知道参数何时生效

设计权衡：

  • ✅ 确定性：每次迭代使用哪个参数值完全确定
  • ❌ 延迟敏感：内核会阻塞等待 RTP 更新，可能降低吞吐量

2. 异步标量 RTP (async_rtp)

核心特性： 内核读取最新 RTP 值，不等待更新

// graph.h: 异步连接
adf::connect<adf::parameter>(value, adf::async(k.in[1]));

// graph.cpp: 只需一次更新（或设置初始值）
gr.run(4);
gr.update(gr.value, 10);  // 所有迭代都使用最新值

适用场景：

  • 全局配置参数（如增益、阈值）
  • 可以容忍一定延迟的反馈控制
  • 需要频繁更新但不需要每次迭代都不同的参数

设计权衡：

  • ✅ 高吞吐量：内核不会因等待 RTP 而停滞
  • ❌ 不确定性：无法精确知道哪次迭代使用了哪个参数值
  • ⚠️ 必须至少更新一次：否则内核会永远等待初始值

绕过初始更新：

// 在 graph.h 中设置初始值
initial_value(k.in[1]) = 10;

3. 异步数组 RTP (async_array_rtp)

核心特性： 数组参数作为 RTP，整组原子更新

// kernel.h: 数组参数签名
void vect_add(..., const int32 (&value)[NUM]);

// graph.h: 异步数组连接
adf::connect<adf::parameter>(value, async(k.in[1]));

// graph.cpp: 更新整个数组
int value[256];
for(int i = 0; i < 256; i++) value[i] = i;
gr.update(gr.value, value, 256);  // 整组更新

关键约束：

  • 整组更新：不能只更新数组的一部分
  • 原子性：整个数组要么全部更新，要么全部不更新

适用场景：

  • FIR 滤波器系数更新
  • 查找表 (LUT) 重配置
  • 任何需要一组相关参数同时生效的场景

4. 异步数组 RTP 带读回 (async_array_rtp_read)

核心特性： Inout RTP 端口，支持双向数据流

// kernel.h: Inout 参数签名
void vect_add(..., const int32 (&value)[NUM], int32 &iter);

// graph.h: Inout 端口定义
adf::port<adf::direction::inout> iter;

// 异步 Inout 连接（默认就是异步）
adf::connect<adf::parameter>(async(k.inout[0]), iter);

// graph.cpp: 读取内核状态
int current_iter = -1;
gr.read(gr.iter, current_iter);  // 读取迭代计数

适用场景：

  • 探查 AIE 内部状态（迭代计数、错误标志等）
  • 最小数据量的状态反馈（不适合高吞吐）
  • 调试和监控

设计权衡：

  • ✅ 非侵入式：不需要停止内核即可读取状态
  • ⚠️ 非实时：读取的是采样时刻的状态，不是精确的时序同步
  • ⚠️ 2024.2 已知问题：异步数组 RTP 更新与读回存在硬件执行问题

设计决策与权衡

1. 同步 vs 异步的哲学

同步 (sync) 设计哲学：

"我确切地知道每个参数何时生效，吞吐量是可以接受的代价。"

  • 选择同步当：需要可重现性、确定性、严格的因果关系
  • 牺牲：峰值吞吐量、延迟

异步 (async) 设计哲学：

"我不在乎具体哪次迭代用了哪个值，我要的是最高的吞吐量和最低的延迟。"

  • 选择异步当：参数变化缓慢、需要持续高吞吐、可以接受一定不确定性
  • 牺牲：精确的可重现性、需要额外机制保证参数生效

2. 标量 vs 数组的权衡

标量 RTP：

  • ✅ 轻量级（单周期传输）
  • ✅ 适合频繁更新的控制信号
  • ❌ 无法表示复杂配置（如 FIR 系数）

数组 RTP：

  • ✅ 可以传输复杂配置
  • ✅ 原子性保证（整组同时生效）
  • ❌ 带宽占用高（需要 DMA 传输）
  • ❌ 更新延迟大

3. Inout 端口的设计意图

为什么需要 Inout RTP？

AIE 内核是数据流驱动的——它们持续处理输入流，很难在不停机的情况下"询问"当前状态。Inout RTP 提供了一种非侵入式探查机制：

  • 不需要停止内核
  • 不需要额外的调试接口
  • 可以在正常数据流中穿插状态采样

但是要注意：读取的是采样时刻的状态，不是精确的时钟同步状态。

实现细节与工程实践

HLS 数据移动器 (s2mm 和 datagen)

s2mm (Stream-to-Memory-Mapped)：

void s2mm(ap_int<32>* mem, hls::stream<ap_axis<32, 0, 0, 0>>& s, int size) {
    // AXI4-Full 接口：高效批量写入 DDR
    #pragma HLS INTERFACE m_axi port=mem offset=slave bundle=gmem
    // AXI4-Stream 接口：接收 AIE 输出
    #pragma HLS interface axis port=s
    // AXI4-Lite 接口：Host 配置
    #pragma HLS INTERFACE s_axilite port=mem bundle=control
    
    for(int i = 0; i < size; i++) {
        #pragma HLS PIPELINE II=1
        ap_axis<32, 0, 0, 0> x = s.read();
        mem[i] = x.data;
    }
}

关键设计决策：

  • II=1：每个周期处理一个样本，匹配 AIE 的吞吐
  • m_axi 接口：使用突发传输最大化 DDR 带宽
  • axis 接口：与 AIE 的 PLIO 直接流式连接

datagen (Data Generator)：

void datagen(hls::stream<ap_axis<32,0,0,0>> & out, int size, int init_val=0) {
    #pragma HLS INTERFACE axis port=out
    #pragma HLS INTERFACE s_axilite port=return bundle=control
    // ...
    for(int i=0; i<size; i++){
        ap_axis<32,0,0,0> tmp;
        tmp.data = i + init_val;
        tmp.keep = -1;  // 所有字节有效
        tmp.last = 0;   // 非最后一个（包模式）
        out.write(tmp);
    }
}

关键设计决策与权衡

决策 1：同步 vs 异步的默认行为

问题：应该让 RTP 默认同步还是异步？

选择：默认是同步（sync），需要显式指定 async。

// 默认：同步
adf::connect<adf::parameter>(value, k.in[1]);
// 等价于:
adf::connect<adf::parameter>(value, sync(k.in[1]));

// 显式：异步
adf::connect<adf::parameter>(value, async(k.in[1]));

权衡分析：

方面 默认同步的优势 如果默认异步会怎样
安全性 新用户得到的是确定性行为，不易出错 新用户可能惊讶于参数没有"生效"
调试 死锁或停滞立即暴露问题 错误可能静默传播，难以调试
性能 明确知道需要优化时才用 async 可能无意中用 async 破坏了确定性

哲学：显式优于隐式，安全优于便利。让 "性能优化" 成为一个有意识的选择。

决策 2：数组 RTP 的原子性保证

问题：如果数组 RTP 只更新了一半，内核就读取了会怎样？

保证：AI Engine 架构保证数组 RTP 更新是原子的——要么整个数组是新值，要么整个是旧值，不会出现"混合状态"。

实现机制：

  • 数组 RTP 通过 DMA 传输到 AIE 的专用 RTP 内存
  • 使用"双缓冲"机制：更新写入缓冲区 A，读取从缓冲区 B；完成后交换指针
  • Host 通过 update() 调用触发 DMA 传输

权衡：

  • ✅ 数据一致性：不会出现半新半旧的错误状态
  • ❌ 内存开销：需要 2x 数组大小的内存用于双缓冲
  • ❌ 更新延迟：DMA 传输需要时间，不是瞬时完成

决策 3：Inout RTP 的非实时性设计

问题：从 Host 读取 AIE 状态时，读取的是哪个时刻的状态？

现实：读取的是采样时刻的快照，不是精确的硬件同步状态。

原因：

  1. 跨时钟域：Host (ARM) 和 AIE 运行在不同的时钟域
  2. 总线延迟：通过 NOC (Network-on-Chip) 传输需要时间
  3. 软件开销：XRT API 调用到实际读取之间有不确定的延迟

设计意图：

  • Inout RTP 不是用于精确时序同步的机制
  • 它是为了监控、调试、慢速控制而设计的
  • 如果需要精确的 AIE-to-Host 同步，应该使用中断或流式数据输出

权衡：

  • ✅ 简单易用：不需要额外的同步逻辑
  • ✅ 低开销：不会阻塞 AIE 执行
  • ❌ 时间不确定性：无法确定读取的是哪个时钟周期的状态
  • ❌ 不适用于实时控制：不能用于需要微秒级响应的场景

新贡献者必读：陷阱与最佳实践

常见错误 1：忘记初始 RTP 更新

错误代码：

gr.run(4);
// 忘记调用 gr.update()！
gr.end();

现象：内核永远等待 RTP 值，程序挂死。

解决方案：

// 方法 1: 在 graph.h 中设置初始值
initial_value(k.in[1]) = 10;

// 方法 2: 运行前确保至少一次 update
gr.run(4);
gr.update(gr.value, 10);  // 必须有！

常见错误 2：混淆 sync 和 async 的行为

期望 sync 的行为却用了 async：

// graph.h: 错误地使用了 async
adf::connect<adf::parameter>(value, async(k.in[1]));

// graph.cpp: 期望每次迭代使用不同值
gr.run(4);
gr.update(gr.value, 10);
gr.update(gr.value, 100);
gr.update(gr.value, 1000);
gr.update(gr.value, 10000);

现象：内核可能多次使用同一个值，因为 async 模式下内核自主决定何时读取。

解决方案：

// 如果需要确定性的每次迭代不同值，使用 sync
adf::connect<adf::parameter>(value, k.in[1]);  // 默认 sync

常见错误 3：部分更新数组 RTP

错误尝试：

int value[256];
for(int i = 0; i < 256; i++) value[i] = i;

gr.update(gr.value, value, 128);  // 错误：只更新前128个！

问题：虽然代码能运行，但数组的后半部分是未定义的（可能是旧值或垃圾值）。

正确做法：

// 必须更新整个数组
gr.update(gr.value, value, 256);  // 正确：完整的256个元素
