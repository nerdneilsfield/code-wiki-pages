Buffer AIE Packet Switching Pipeline 技术深度解析

一句话概括

这是一个展示如何在 Versal ACAP 架构上使用**包交换（Packet Switching）**机制，将多个独立的数据流通过单一的物理通道传输到 AIE（AI Engine）阵列进行并行处理的参考设计。它像是一个"数据多路复用器"——让 4 条独立的数据流共享同一条"高速公路"，在 AIE 端再智能地分发到 4 个独立的处理核心。

问题空间：为什么需要这个模块？

硬件约束的现实

Versal ACAP 的 AIE 阵列拥有数百个计算核心，但 PL（Programmable Logic）与 AIE 之间的物理连接资源是有限的。在实际系统中，你经常会遇到这样的场景：

  • 多路数据源：4 个独立的 DMA 通道需要从 DDR 读取数据
  • 单一物理接口：PL 到 AIE 的物理连接只有一条 AXI4-Stream 通道
  • 并行处理需求：AIE 内部有 4 个核心可以并行处理这 4 路数据

如果没有包交换机制，你需要为每路数据单独建立物理连接，这在资源受限的 FPGA 上是不现实的。

类比理解

想象一个快递分拣中心：

  • 传统方式：每个发货仓库都修一条专用公路到分拣中心（资源浪费）
  • 包交换方式：所有仓库共用一条高速公路，每个包裹贴上目的地标签，到达后自动分拣（资源共享 + 智能路由）

这就是 buffer_aie_packet_switching_pipeline 要解决的核心问题：如何用有限的物理连接，传输多路逻辑数据流。

数据流说明：

  1. Host → PL: Host 初始化 4 个独立的输入缓冲区，通过 XRT API 同步到 DDR
  2. PL Input: 4 个 mm2s DMA 内核从 DDR 读取数据，hls_packet_sender 将数据打包并添加路由 header
  3. PL → AIE: 单一 pktstream 通道传输所有数据包
  4. AIE Internal: pktsplit<4> 根据 packet ID 硬件分发到 4 个核心，处理完成后 pktmerge<4> 合并输出
  5. AIE → PL: hls_packet_receiver 解析 header 并解包分发到 4 个 s2mm DMA 内核
  6. PL → Host: 数据写回 DDR，Host 验证结果

核心抽象与心智模型

1. 包（Packet）——数据的集装箱

每个包由两部分组成：

  • Header（头部）：32 位元数据，包含目的地址（packet ID）、源位置、校验位等
  • Payload（载荷）：8 个 32-bit 数据字（PACKET_LEN=8）

可以把包想象成快递包裹：面单上的信息告诉系统"这个包裹要送到哪里"，里面的内容才是实际货物。

2. pktsplit/pktmerge —— AIE 内部的路由器

这是 AIE 特有的硬件原语：

  • pktsplit<4>：将一个输入包流根据 header 中的 ID 分发到 4 个输出端口
  • pktmerge<4>：将 4 个输入流合并成一个输出流，自动为每个包添加正确的 header

它们是硬件实现的多路选择器/合并器，零开销完成数据路由。

3. PL 端的包处理器 —— 软件定义的分拣机

由于 PL 端没有硬件 pktsplit/pktmerge，需要用 HLS 编写：

  • hls_packet_sender：从 4 个输入流读取数据，打包后发送到 AIE
  • hls_packet_receiver：从 AIE 接收包流，根据 header 解包分发到 4 个输出流

数据流全链路追踪

让我们跟随一个数据字的完整旅程：

阶段 1：Host → PL（数据准备）

// host.cpp - 初始化输入数据
for(int i=0; i<mem_size/sizeof(int); i++){
    *(host_in1+i) = i;      // 通道 1: 0, 1, 2, 3...
    *(host_in2+i) = 2*i;    // 通道 2: 0, 2, 4, 6...
    *(host_in3+i) = 3*i;    // 通道 3: 0, 3, 6, 9...
    *(host_in4+i) = 4*i;    // 通道 4: 0, 4, 8, 12...
}
// 同步到 DDR
xrtBOSync(in_bo1, XCL_BO_SYNC_BO_TO_DEVICE, mem_size, 0);

关键点：Host 分配了 4 个独立的 buffer，分别对应 4 个逻辑数据流。

阶段 2：PL DMA → Packet Sender（打包）

// mm2s.cpp - 从 DDR 读取数据到 AXI Stream
void mm2s(ap_uint<32>* mem, hls::stream<ap_axiu<32,0,0,0>>& s, int size) {
    for(int i = 0; i < size; i++) {
        #pragma HLS PIPELINE II=1
        ap_axiu<32,0,0,0> x;
        x.data = mem[i];
        x.keep = -1;  // 所有字节有效
        x.last = 0;   // 不是最后一个数据
        s.write(x);
    }
}

4 个 mm2s 实例（mm2s_1 到 mm2s_4）同时工作，各自产生一个独立的 AXI Stream。

// hls_packet_sender.cpp - 多路复用 + 打包
void hls_packet_sender(hls::stream<ap_axiu<32,0,0,0>> &s0, &s1, &s2, &s3,
                       hls::stream<ap_axiu<32,0,0,0>> &out, 
                       const unsigned int num) {
    for(unsigned int iter=0; iter<num; iter++) {
        for(int i=0; i<PACKET_NUM; i++) {  // 遍历 4 个通道
            // 1. 生成并发送 Header
            ap_uint<32> header = generateHeader(pktType, packet_ids[i]);
            tmp.data = header;
            out.write(tmp);
            
            // 2. 发送 8 个数据字（Payload）
            for(int j=0; j<PACKET_LEN; j++) {
                switch(i) {
                    case 0: tmp = s0.read(); break;
                    case 1: tmp = s1.read(); break;
                    case 2: tmp = s2.read(); break;
                    case 3: tmp = s3.read(); break;
                }
                if(j == PACKET_LEN-1) tmp.last = 1;  // 包结束标记
                out.write(tmp);
            }
        }
    }
}

关键设计决策：

  • 轮询（Round-robin）方式从 4 个输入流各取 8 个字，组成一个完整的包
  • 每个包以 Header 开头，以 TLAST=1 的数据字结尾
  • Header 中嵌入目标 AIE core 的 ID（packet_ids[i]）

阶段 3：AIE 内部路由（硬件分拣）

// graph.h - AIE 图定义
class mygraph: public adf::graph {
    adf::pktsplit<4> sp;   // 包拆分器：1 入 → 4 出
    adf::pktmerge<4> mg;   // 包合并器：4 入 → 1 出
    adf::kernel core[4];   // 4 个处理核心
    
    mygraph() {
        // 创建 4 个核心，分别执行 aie_core1~4
        for(int i=0; i<4; i++) {
            adf::connect<> (sp.out[i], core[i].in[0]);   // 分发到对应核心
            adf::connect<> (core[i].out[0], mg.in[i]);   // 收集输出
        }
        // 包流连接（使用 pktstream 类型）
        adf::connect<adf::pktstream> (in.out[0], sp.in[0]);
        adf::connect<adf::pktstream> (mg.out[0], out.in[0]);
    }
};

硬件魔法在这里发生：

  • pktsplit<4> 自动解析每个包的 header，根据 ID 字段将数据路由到对应的 core[i]
  • 每个核心看到的是独立的 buffer 接口，完全感知不到包的存在
  • 处理完成后，pktmerge<4> 自动为输出包添加正确的 header

阶段 4：AIE Core 处理（业务逻辑）

// aie_core1.cpp - 示例处理核
void aie_core1(input_buffer<int32,extents<8>> &in,
               output_buffer<int32,extents<8>> &out) {
    auto inIter = aie::begin(in);
    auto outIter = aie::begin(out);
    for(int i=0; i<8; i++) {
        *outIter++ = *inIter++ + 1;  // 每个数加 1
    }
}

4 个核心的区别仅在于加的数值不同（+1, +2, +3, +4），用于验证数据路由的正确性。

阶段 5：Packet Receiver → PL DMA（解包）

// hls_packet_receiver.cpp - 解复用 + 解包
void hls_packet_receiver(hls::stream<ap_axiu<32,0,0,0>> &in,
                         hls::stream<ap_axiu<32,0,0,0>> &out0, &out1, &out2, &out3,
                         const unsigned int total_num_packet) {
    for(unsigned int iter=0; iter<total_num_packet; iter++) {
        // 1. 读取 Header，提取目标通道 ID
        ap_axiu<32,0,0,0> tmp = in.read();
        unsigned int ID = getPacketId(tmp.data);
        unsigned int channel = packet_ids[ID];
        
        // 2. 读取 8 个数据字，根据 ID 分发到对应输出流
        for(int j=0; j<PACKET_LEN; j++) {
            tmp = in.read();
            switch(channel) {
                case 0: out0.write(tmp); break;
                case 1: out1.write(tmp); break;
                case 2: out2.write(tmp); break;
                case 3: out3.write(tmp); break;
            }
        }
    }
}

阶段 6：PL → Host（结果回传）

// s2mm.cpp - AXI Stream 写入 DDR
void s2mm(ap_uint<32>* mem, hls::stream<ap_axiu<32,0,0,0>>& s, int size) {
    for(int i = 0; i < size; i++) {
        #pragma HLS PIPELINE II=1
        ap_axiu<32,0,0,0> x = s.read();
        mem[i] = x.data;
    }
}

4 个 s2mm 实例将数据写回 4 个独立的 DDR buffer，Host 进行最终验证。

关键设计决策与权衡

1. 包大小选择：PACKET_LEN=8

为什么是 8？

  • AIE 核心使用 SIMD 向量处理，8 个 32-bit 字正好匹配 AIE 的向量宽度
  • 较小的包减少延迟，较大的包减少 header 开销；8 是吞吐和延迟的平衡点
  • 如果业务需要更大的数据块，可以增加此值，但要确保 AIE buffer 能容纳

2. 轮询调度 vs 优先级调度

当前实现采用简单的轮询（round-robin）：

for(int i=0; i<PACKET_NUM; i++) {  // 0 → 1 → 2 → 3 → 0 → ...
    // 从第 i 个流读取 PACKET_LEN 个字
}

权衡分析：

  • ✅ 简单可靠：不会出现某个流饿死的情况
  • ✅ 时序确定：便于分析和验证
  • ❌ 非最优吞吐：如果某个流数据不足，会阻塞其他流

替代方案：动态优先级或信用制（credit-based）调度，但需要更复杂的握手逻辑。

3. PL 端包处理器的纯串行实现

注意到 hls_packet_sender/receiver 中没有使用 #pragma HLS DATAFLOW 或 PIPELINE：

原因：

  • 这些 kernel 的主要工作是数据搬运和格式转换，计算量极小
  • 引入流水线会增加资源消耗和复杂度，收益有限
  • 瓶颈通常在内存带宽（DDR → PL）或 AIE 处理能力，不在包处理器

4. Header 格式的兼容性设计

ap_uint<32> generateHeader(unsigned int pktType, unsigned int ID) {
    ap_uint<32> header = 0;
    header(4,0) = ID;           // 低 5 位：Packet ID（最多 32 个目标）
    header(14,12) = pktType;    // 类型字段
    header(20,16) = -1;         // 源 Row（未使用）
    header(27,21) = -1;         // 源 Column（未使用）
    header[31] = header(30,0).xor_reduce() ? 0 : 1;  // 奇偶校验位
    return header;
}

设计考量：

  • 与 AIE 硬件 pktsplit/pktmerge 的 header 格式保持一致
  • 预留了源位置字段，支持更复杂的路由场景
  • 奇偶校验位用于基本的完整性检查

新贡献者必读：陷阱与注意事项

⚠️ 隐式契约 1：Packet ID 的一致性

packet_ids_c.h 中的宏定义（如 Datain0_0, Dataout0_1）是由 AIE 编译器根据 graph 连接关系自动生成的。不要手动修改这些值，否则会导致 AIE 硬件路由错误。

// packet_ids_c.h（自动生成示例）
#define Datain0_0  0  // 对应 core[0]
#define Datain0_1  1  // 对应 core[1]
#define Datain0_2  2  // 对应 core[2]
#define Datain0_3  3  // 对应 core[3]

⚠️ 隐式契约 2：数据量的对齐要求

Host 代码中的 mem_size 计算：

int packet_num = 2;                    // 每个通道发送 2 个包
int total_packet_num = 2 * 4;          // 总共 8 个包（4 个通道 × 2）
int mem_size = packet_num * 32;        // 每个包 8 字 × 4 字节 = 32 字节

必须满足：

  • 每个通道的数据量必须是 PACKET_LEN 的整数倍
  • Host 设置的 packet_num 必须与 PL kernel 的 num 参数一致
  • 不一致会导致死锁（某一方等待永远不会到达的数据）

⚠️ 启动顺序的重要性

// host.cpp - 正确的启动顺序
// 1. 先启动输出链路的 kernel（S2MM 和 packet_receiver）
xrtRunStart(s2mm_r1);  // ... r2, r3, r4
xrtRunStart(hls_packet_receiver_r);

// 2. 再启动输入链路的 kernel（MM2S 和 packet_sender）
xrtRunStart(mm2s_r1);  // ... r2, r3, r4
xrtRunStart(hls_packet_sender_r);

// 3. 最后启动 AIE graph
gr.run(2);

原因：如果先启动输入链路，数据可能到达时输出链路还没准备好，导致 AXI Stream 反压（back-pressure）或数据丢失。

⚠️ 时钟域 crossing

[clock]
defaultFreqHz=250000000  # PL 运行频率

# config.cfg 中
freqhz=400000000         # HLS kernel 综合目标频率

注意 PL 和 AIE 可能运行在不同的时钟域，AXI Stream 接口会自动处理异步 FIFO，但性能分析时要考虑跨时钟域的延迟。

⚠️ 仿真 vs 硬件的差异

// graph.cpp
#if defined(__AIESIM__) || defined(__X86SIM__)
int main(int argc, char ** argv) {
    gr.init();
    gr.run(2);  // 仿真模式下只运行 2 次迭代
    gr.end();
    return 0;
}
#endif

AIE 仿真器和实际硬件的行为基本一致，但仿真速度较慢，大数据量测试建议直接在硬件上运行。

⚠️ HLS Kernel 的接口约束

// mm2s.cpp / s2mm.cpp
#pragma HLS INTERFACE m_axi port=mem offset=slave bundle=gmem
#pragma HLS interface axis port=s
#pragma HLS INTERFACE s_axilite port=mem bundle=control
#pragma HLS INTERFACE s_axilite port=size bundle=control
#pragma HLS interface s_axilite port=return bundle=control

接口说明：

  • m_axi: 连接到 DDR 内存，支持突发传输（burst）
  • axis: AXI4-Stream 接口，用于流式数据传输
  • s_axilite: 控制寄存器接口，Host 通过这里设置参数（如 size）

注意：offset=slave 表示内核作为 AXI 总线的从设备，由 DMA 控制器发起传输。

性能特征与时序分析

理论吞吐计算

组件 频率 数据宽度 理论峰值带宽
PL Kernels 400 MHz 32-bit 1.6 GB/s per stream
AIE-PL Interface 250 MHz (默认) 32-bit 1.0 GB/s
AIE Core 1 GHz 128-bit (vector) 16 GB/s internal

瓶颈分析：

  1. PL → AIE 通道: 单一 pktstream 通道是潜在瓶颈。4 个输入流合并后，如果每个流都满载，需要 4 × 1.6 = 6.4 GB/s，远超 1.0 GB/s 的物理限制。
  2. 实际场景: 本设计中每个包包含 9 个 words（1 header + 8 data），header 开销为 11%，实际有效带宽约为理论的 89%。
  3. AIE 处理能力: 4 个核心并行处理，每个核心处理 8 个整数只需约 8 个时钟周期（考虑流水线），远快于数据到达速率。

延迟分解

一个数据字从 Host 到 Host 的完整旅程延迟：

┌─────────────┬────────────────────────────────────────────────────┐
│ 阶段        │ 延迟来源                                           │
├─────────────┼────────────────────────────────────────────────────┤
│ Host→DDR    │ PCIe DMA 延迟 (~1-10 μs，取决于系统负载)           │
│ DDR→PL      │ MM2S 读取延迟 (~几拍，可忽略)                      │
│ PL Packing  │ hls_packet_sender 轮询等待 (~0-3 包周期)           │
│ PL→AIE      │ AXI Stream 传输 + FIFO 缓冲 (~几十拍)              │
│ AIE Routing │ pktsplit 硬件解析 (~1-2 拍)                        │
│ AIE Compute │ 核心处理 (~8-16 拍)                                │
│ AIE→PL      │ pktmerge + Stream 传输 (~几十拍)                   │
│ PL Unpack   │ hls_packet_receiver 解析 (~1 拍)                   │
│ PL→DDR      │ S2MM 写入 (~几拍)                                  │
│ DDR→Host    │ PCIe DMA 回传 (~1-10 μs)                           │
└─────────────┴────────────────────────────────────────────────────┘

关键洞察：PCIe 传输是整个链路的主要延迟来源（~20 μs 往返），片上处理延迟（<1 μs）可以忽略不计。这意味着：

  • 批处理（batch processing）比逐元素处理更高效
  • 增大 packet_num 可以摊平 PCIe 开销

子模块详解

本模块由以下三个子模块组成，每个子模块负责不同的功能层面：

1. PL Kernels - 包处理引擎

文件: pl_kernels/hls_packet_sender.cpp, pl_kernels/hls_packet_receiver.cpp, pl_kernels/mm2s.cpp, pl_kernels/s2mm.cpp

PL 端的内核负责在标准 AXI4-Stream 和带路由信息的 packet stream 之间进行转换。这是软件定义的数据多路复用/解复用层。

关键设计要点：

  • hls_packet_sender: 轮询 4 个输入流，为每个数据块添加 32-bit header（包含目标 AIE core ID）
  • hls_packet_receiver: 解析 header，根据 ID 将数据分发到对应的输出流
  • mm2s/s2mm: 标准的内存到流/流到内存 DMA，II=1 流水线保证吞吐

详细文档：PL Kernels Packet Processing

2. AIE Graph & Kernels - 计算阵列

文件: aie/graph.h, aie/graph.cpp, aie/kernels.h, aie/aie_core*.cpp

AIE 层使用 ADF (Adaptive Data Flow) 框架定义计算图，利用硬件原语 pktsplit<4> 和 pktmerge<4> 实现零开销的包路由。

关键设计要点：

  • pktsplit<4> / pktmerge<4>: AIE 特有的硬件包交换原语，自动处理 header 解析和生成
  • 4 个独立的 AIE 核心 (aie_core1~4)，每个执行简单的 +N 运算用于验证
  • Buffer-based kernel 接口：核心看到的是普通 buffer，完全感知不到 packet 的存在

3. Host Control - 系统编排

文件: sw/host.cpp

Host 代码负责系统初始化、内核启动顺序控制和结果验证。这是整个系统的"指挥家"。

关键设计要点：

  • XRT API 用于设备管理和缓冲区分配
  • 严格的启动顺序：先启动接收端（S2MM, packet_receiver），再启动发送端（MM2S, packet_sender），最后启动 AIE graph
  • 双缓冲机制：输入/输出各 4 个独立 buffer，对应 4 个数据通道

buffer_aie_packet_switching_pipeline 是一个教学性质的参考设计，展示了 Versal ACAP 上资源复用的经典模式。它通过软硬件协同的方式，在有限的物理连接上实现了多路数据流的并行处理。

关键技术一览

层面 关键技术 作用 替代方案
PL 端 HLS 包处理器 软件定义的多路复用/解复用 专用 DMA 控制器（更复杂）
AIE 边界 AXI4-Stream pktstream 带路由信息的流传输 多条独立 stream（资源受限）
AIE 内部 pktsplit<4> / pktmerge<4> 硬件加速的包路由 软件路由（性能差）
AIE 核心 Buffer-based kernels 专注于业务逻辑，无感知路由细节 Stream-based kernels（需处理 back-pressure）

何时使用此模式

✅ 适用场景：

  • PL-AIE 接口数量有限，但需要传输多路独立数据流
  • 各路数据流的处理延迟相近（避免某一路阻塞其他路）
  • 需要动态路由或负载均衡的场景

❌ 不适用场景：

  • 单路数据流已占满带宽（包交换带来额外开销）
  • 对延迟极度敏感（包头解析增加少量延迟）
  • 数据量极小（header 开销占比过高）

学习路径建议

对于新加入团队的工程师，建议按以下顺序深入理解本模块：

  1. 先跑通仿真：使用 graph.cpp 中的 x86sim/aiesim 模式，观察数据流
  2. 阅读 Host 代码：理解 XRT API 的使用和启动顺序的重要性
  3. 分析 PL kernel：重点关注 hls_packet_sender/receiver 的状态机设计
  4. 修改并实验：尝试改变 PACKET_LEN 或通道数，观察对性能和资源的影响
  5. 对比其他变体：阅读 Buffer AIE Mixed Datatype 和 Pktstream Pipeline 了解不同设计选择

理解这个设计的精髓，你就能在更复杂的系统中灵活运用包交换机制，解决 PL-AIE 连接资源受限的问题。
