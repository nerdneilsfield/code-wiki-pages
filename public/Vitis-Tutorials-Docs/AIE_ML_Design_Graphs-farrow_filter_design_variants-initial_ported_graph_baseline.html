Initial Ported Graph Baseline (Farrow Filter) — 技术深度解析

一句话概括

这个模块是 Farrow 滤波器在 AIE-ML 架构上的"初版移植" —— 它将一个经典的多相分数延迟滤波算法从理论模型转化为可运行的 AI Engine 图结构。你可以把它想象成一个精密装配流水线：上游的 farrow_kernel1 负责生产四个并行的滤波子带（y3/y2/y1/y0），下游的 farrow_kernel2 则像一位精算师，根据实时输入的延迟参数 del_i，通过多项式插值将这些子带合成为最终的时移信号。整个设计的核心挑战在于：如何在 AIE-ML 的 SIMD 向量单元、固定内存布局和流式数据接口之间，找到一个既保持算法正确性又能榨取硬件性能的甜蜜点。

问题空间：为什么需要这个模块？

Farrow 滤波器的本质

Farrow 滤波器解决的是分数采样率转换和任意延迟调整问题。传统 FIR 滤波器只能处理整数倍延迟，而 Farrow 结构通过拉格朗日多项式插值，将滤波器系数表示为延迟参数 \(\mu\) 的多项式：

\[H(z, \mu) = \sum_{k=0}^{N-1} h_k(\mu) \cdot z^{-k}\]

其中 \(h_k(\mu)\) 是关于 \(\mu\) 的多项式。这样，对于每个输入样本，我们只需要根据当前的 \(\mu\) 值计算一组新的系数，而不需要存储大量预计算的滤波器组。

为什么直接移植行不通？

将这个概念性的 DSP 算法搬到 AIE-ML 上，你会立即撞上三道墙：

  1. 向量化鸿沟：AIE-ML 的核心是 512-bit SIMD 向量单元（支持 32x16-bit 复数操作），但传统 Farrow 实现通常是标量或轻度并行化的。你需要重新思考循环结构，让每次迭代处理 8 或 16 个样本。

  2. 内存墙与状态管理：Farrow 滤波器是有状态的（需要维护前几个样本的历史）。在 AIE-ML 上，这意味着要小心管理 f_state 数组的对齐（alignas(32)）和生命周期，确保它在 kernel 调用之间持久化。

  3. 数据流编排复杂度：算法需要两个独立的输入流（信号 sig_i 和延迟 del_i），以及四个中间子带流。如何将这些流正确地连接到两个 kernel，避免死锁或背压，是一个非平凡的图结构设计问题。

设计洞察：分层解耦

这个 baseline 版本的关键设计决策是将 Farrow 滤波器拆分为两个独立的 kernel，而不是试图在一个 kernel 内完成所有计算。这种分层类似于编译器中的"前端/后端"分离：

  • Kernel1（前端）：专注于滤波子带生成。它只关心输入信号和固定的滤波器系数，输出四个并行的子带流。这部分计算密集、高度规则，非常适合 AIE-ML 的 MAC 阵列。

  • Kernel2（后端）：专注于延迟驱动的多项式求值。它接收延迟参数和四个子带，执行三次逐级的 MAC 运算（Horner 法则形式），输出最终结果。这部分有数据依赖（每级依赖前一级的结果），但通过巧妙的双缓冲（z 数组）实现了流水线化。

心智模型：理解这个模块的三种视角

视角二：计算流水线（Pipeline View）

从微架构角度看，这是一个三级展开的向量化流水线：

  1. Load/Shuffle 阶段：v_buff.insert() 操作将新样本滑入向量寄存器，同时保留历史状态（f_state）。这类似于滑动窗口操作。

  2. MAC 计算阶段：aie::sliding_mul_ops 执行 16x8 的滑动乘加，利用 AIE-ML 的专用乘法器阵列。注意这里用了四个独立的 accumulator（acc_f3/f2/f1/f0），允许指令级并行。

  3. Store/Shift 阶段：to_vector<DNSHIFT>() 将累加结果右移 14 位（量化补偿），写回输出缓冲区。

视角三：状态机（State Machine View）

每个 kernel 内部都有一个隐式的状态机：

  • Kernel1：维护 f_state[8] 作为 FIR 滤波器的历史状态。每次调用消耗 1024 个新样本，但保留最后 8 个样本用于下次调用的重叠相加。

  • Kernel2：使用 z[BUFFER_SIZE] 作为双缓冲工作区。第一遍计算将中间结果存入 z，第二遍读取 z 并继续计算，第三遍输出最终结果。这种设计避免了在同一个循环内读写冲突，同时允许编译器进行流水线优化。

组件深度剖析

1. dut_graph —— 测试封装层

文件: farrow_app.cpp

这是整个设计的入口点和测试脚手架。它的存在是为了让同一个二进制既能跑 AIE 仿真（AIE_SIM_ONLY 模式），又能在真实硬件上运行。

关键设计：条件编译的 PLIO 配置

#ifdef AIE_SIM_ONLY
    sig_i[0] = input_plio::create("PLIO_i_0", plio_64_bits, "data/sig_i.txt");
    // ... 其他 PLIO 绑定到文本文件
#else
    sig_i[0] = input_plio::create("PLIO_i_0", plio_64_bits);
    // ... 纯端口定义，无文件绑定
#endif

为什么是这种设计？

在 AIE 仿真环境中，我们需要从文本文件注入测试向量；而在实际硬件部署时，PLIO 会连接到真实的 DMA 或逻辑单元。通过宏控制，同一份代码可以服务两种场景，避免了维护两份几乎相同的图定义。

连接拓扑

connect<stream>(sig_i[0].out[0], farrow.sig_i[0]);
connect<stream>(del_i[0].out[0], farrow.del_i[0]);
connect<stream>(farrow.sig_o[0], sig_o[0].in[0]);

注意这里使用了 connect<stream> 而非默认的缓冲区连接。这意味着数据以流式方式传递，kernel 之间通过 FIFO 直接握手，没有中间的乒乓缓冲。这对延迟敏感型应用很重要，但也意味着 producer 和 consumer 必须同步推进。

2. farrow_graph —— 核心计算图

文件: farrow_graph.h

这是算法的物理实现蓝图。它继承自 adf::graph，使用 AMD/Xilinx 的 AI Engine DSL 来描述硬件拓扑。

Kernel 实例化与属性配置

farrow_kernel1_ins = kernel::create_object<farrow_kernel1>();
source(farrow_kernel1_ins) = "farrow_kernel1.cpp";
runtime<ratio>(farrow_kernel1_ins) = 0.9;

runtime<ratio>(0.9) 的含义：

这告诉调度器，该 kernel 每个周期可以使用 90% 的处理器时间。保留 10% 的余量是为了给内存访问和其他开销留出空间。在后续的优化版本中（如 farrow_opt_1、farrow_opt_2），你可能会看到不同的数值，这是性能调优的重要杠杆。

location<stack>() 的意图：

location<stack>(farrow_kernel1_ins) = location<kernel>(farrow_kernel1_ins);

这强制 stack 内存与 kernel 本身位于同一 tile。在 AIE 架构中，跨 tile 的内存访问有额外延迟。这个约束确保了函数调用和局部变量的最高效访问。

数据流连接详解

// 外部输入 → Kernel1
connect<> (sig_i[0], farrow_kernel1_ins.in[0]);

// 外部延迟 → Kernel2
connect<> (del_i[0], farrow_kernel2_ins.in[0]);

// Kernel1 → Kernel2（四路子带）
connect<> (farrow_kernel1_ins.out[0], farrow_kernel2_ins.in[1]); // y3
connect<> (farrow_kernel1_ins.out[1], farrow_kernel2_ins.in[2]); // y2
connect<> (farrow_kernel1_ins.out[2], farrow_kernel2_ins.in[3]); // y1
connect<> (farrow_kernel1_ins.out[3], farrow_kernel2_ins.in[4]); // y0

// Kernel2 → 外部输出
connect<> (farrow_kernel2_ins.out[0], sig_o[0]);

端口号的设计逻辑：

  • Kernel1 的 in[0] 接收信号，out[0..3] 输出四个子带
  • Kernel2 的 in[0] 接收延迟，in[1..4] 接收四个子带，out[0] 输出结果

这种编号约定不是强制的，但在整个 Farrow 教程系列中保持一致，便于读者跟踪数据流。

3. farrow_kernel1 —— 子带生成引擎

文件: farrow_kernel1.h, farrow_kernel1.cpp

这是算法的计算密集型前端。它的任务是用四组不同的滤波器系数对输入信号进行滤波，产生四个并行的子带输出。

类型系统与精度设计

typedef cint16  TT_SIG;   // 复数信号：16-bit 实部 + 16-bit 虚部
typedef cacc64  TT_ACC;   // 复数累加器：48-bit 实部 + 48-bit 虚部（扩展到 64-bit）

为什么是 cint16 和 cacc64？

  • AIE-ML 的 SIMD 单元原生支持 16-bit 复数乘法（即一次指令完成 4 个实数乘加）。
  • 累加器需要更宽的位宽来防止溢出。FIR 滤波器是线性卷积，最坏情况下需要 \(\log_2(N) + \text{input\_bits}\) 的位宽增长。这里用 64-bit 累加器处理 8-tap 滤波，提供了充足的安全边际。

滤波器系数的布局奥秘

alignas(32) static constexpr int16 f_taps[32] = {
    // f3 taps (anti-sym): 206,-1264,6606,-14835,14835,-6606,1264,-206
    // f2 taps (sym):      906,-3543,10352,-7628,-7628,10352,-3543,906
    // f1 taps (anti-sym): -51,316,-1652,20093,-20093,1652,-316,51
    // f0 taps (sym):      -226,886,-2588,10099,10099,-2588,886,-226
};

对齐要求：alignas(32) 确保系数数组起始于 256-bit（32-byte）边界，这是 AIE-ML 向量加载指令（aie::load_v<32>）的硬性要求。未对齐访问会导致运行时异常或静默的数据损坏。

交错布局：四个滤波器的系数被交错存储在一个数组中，而非四个独立数组。这使得 sliding_mul_ops 可以通过偏移量（0, 8, 16, 24）快速访问不同子带的系数，利用了向量寄存器的广播能力。

核心循环的微架构解读

for (unsigned rr = 0; rr < BUFFER_SIZE/32; rr++)
    chess_loop_range(1,0)
    chess_prepare_for_pipelining
{
    // 第一阶段：加载 16 个样本，计算所有子带
    v_buff.insert(0, *p_sig_i++);
    acc_f3 = aie::sliding_mul_ops<16, 8, 1, 1, 1, int16, cint16>::mul(f_coeffs, 0, v_buff, 25);
    *p_y3++ = acc_f3.to_vector<TT_SIG>(DNSHIFT);
    // ... 类似地计算 y2, y1, y0

    // 第二阶段：再加载 16 个样本，计算剩余部分
    v_buff.insert(1, *p_sig_i++);
    acc_f3 = aie::sliding_mul_ops<16, 8, 1, 1, 1, int16, cint16>::mul(f_coeffs, 0, v_buff, 9);
    *p_y3++ = acc_f3.to_vector<TT_SIG>(DNSHIFT);
    // ...
}

chess_loop_range(1,0)：这是给 AIE 编译器的提示，表示循环次数至少为 1，上限未知（0 表示无上限）。这让编译器可以展开循环并进行软件流水线优化。

sliding_mul_ops<16, 8, 1, 1, 1, ...> 的参数含义：

  • 16：输出向量长度（16 个复数结果）
  • 8：每个输出的 tap 数（8-tap FIR）
  • 1, 1, 1：各种步长和重复参数，控制滑动窗口的行为
  • v_buff, 25：从向量缓冲区的第 25 个位置开始读取（配合 insert 操作形成滑动窗口）

状态保存：

*(v8cint16*)f_state = v_buff.extract<8>(3);

每次调用结束时，提取 v_buff 的最后 8 个元素保存到 f_state，供下次调用时使用。这是实现"重叠相加" FIR 的关键。

4. farrow_kernel2 —— 多项式求值引擎

文件: farrow_kernel2.h, farrow_kernel2.cpp

这是算法的延迟驱动后端。它实现了 Farrow 结构的核心数学：根据延迟参数 \(\mu\)，对四个子带进行多项式求值。

数学原理：Horner 法则的三级展开

Farrow 滤波器的输出可以表示为：

\[y(n) = y_0 + \mu \cdot (y_1 + \mu \cdot (y_2 + \mu \cdot y_3))\]

这就是 Horner 法则的形式，将四次多项式求值转化为三次逐级 MAC 运算。

双缓冲流水线的实现

// 第一遍：计算 y2 + mu*y3 -> 存入 z
for (unsigned rr = 0; rr < BUFFER_SIZE/16; rr++) {
    del = aie::vector_cast<int16>(*p_del_i++); *p_del_i++;
    acc_x = aie::mac(aie::from_vector<TT_ACC>(*p_y2++, DNSHIFT), *p_y3++, del);
    *p_z2++ = acc_x.to_vector<TT_SIG>(DNSHIFT);
    // ... 第二次迭代
}

// 第二遍：计算 y1 + mu*z -> 存入 z（另一区域）
p_del_i = aie::begin_vector<4>(del_i);  // 重置延迟指针！
for (...) {
    acc_y = aie::mac(aie::from_vector<TT_ACC>(*p_y1++, DNSHIFT), *p_z2++, del);
    *p_z1++ = acc_y.to_vector<TT_SIG>(DNSHIFT);
}

// 第三遍：计算 y0 + mu*z -> 输出
p_del_i = aie::begin_vector<4>(del_i);  // 再次重置！
for (...) {
    acc_z = aie::mac(aie::from_vector<TT_ACC>(*p_y0++, DNSHIFT), *p_z1++, del);
    *p_sig_o++ = acc_z.to_vector<TT_SIG>(DNSHIFT);
}

关键观察：

  1. 延迟指针的三次重置：del_i 在每个阶段开始前都被重置到起点。这是因为每个输出样本都需要完整的延迟序列参与计算。这种设计选择意味着延迟数据会被读取三次，换取了计算结构的规整性。

  2. z 数组的双重角色：p_z2 和 p_z1 都指向同一个 z 数组，但在不同阶段写入不同区域。第一遍写入前半部分，第二遍从前半部分读取并写入后半部分，第三遍从后半部分读取。

  3. 向量类型转换：aie::vector_cast<int16>() 将 32-bit 延迟值转换为 16-bit，以便与 16-bit 信号进行乘法。这隐含了一个假设：延迟值已经被预先缩放到合适的动态范围。

数据流全景追踪

让我们跟随一个样本从进入到离开的全过程：

1. 主机/测试平台
   └── 从 data/sig_i.txt 读取 1024 个复数样本
   └── 从 data/del_i_optimized.txt 读取 1024 个延迟值
   
2. dut_graph (farrow_app.cpp)
   └── PLIO 端口将文件数据转换为 AIE 流格式
   └── sig_i[0] → farrow.sig_i[0]
   └── del_i[0] → farrow.del_i[0]
   
3. farrow_graph (farrow_graph.h)
   └── sig_i[0] → farrow_kernel1_ins.in[0]
   └── del_i[0] → farrow_kernel2_ins.in[0]
   
4. farrow_kernel1 (farrow_kernel1.cpp)
   └── 每轮处理 32 个样本（分成两组 16 个）
   └── 对每个样本位置，计算 4 个 8-tap FIR
   └── 输出 y3, y2, y1, y0 四个子带流
   └── 保留最后 8 个样本到 f_state
   
5. farrow_graph (连接层)
   └── y3 → farrow_kernel2_ins.in[1]
   └── y2 → farrow_kernel2_ins.in[2]
   └── y1 → farrow_kernel2_ins.in[3]
   └── y0 → farrow_kernel2_ins.in[4]
   
6. farrow_kernel2 (farrow_kernel2.cpp)
   └── 第一遍：z = y2 + mu*y3（使用 del_i）
   └── 第二遍：z = y1 + mu*z（重用 del_i）
   └── 第三遍：output = y0 + mu*z（重用 del_i）
   
7. farrow_graph
   └── sig_o[0] ← farrow_kernel2_ins.out[0]
   
8. dut_graph
   └── sig_o[0] → PLIO 输出
   
9. 主机/验证
   └── 对比 data/sig_o.txt 与仿真输出
   └── check_sim_output.m 执行数值验证

设计权衡与决策分析

1. 拆分 vs 合并 Kernel

选择：将 Farrow 滤波器拆分为两个 kernel

权衡分析：

方案 优点 缺点
拆分（当前） 职责清晰；Kernel1 可被其他变体复用；更容易分别优化 增加核间通信开销；需要额外的 buffer/流资源
合并 减少数据传输；可能更低的端到端延迟 单个 kernel 复杂度更高；难以针对子任务调优

为什么选拆分：这个 baseline 版本的目标是可理解性和可扩展性。后续优化版本（如 farrow_opt_1、farrow_opt_2）会在此基础上进行激进的融合和向量化，但 baseline 保持简单以便于教学。

2. Stream vs Buffer 连接

选择：使用 connect<stream> 而非默认 buffer

理由：

  • Farrow 滤波器是流式算法，每个输出样本只依赖当前和历史的输入样本
  • Stream 连接天然支持反压（backpressure），当 consumer 跟不上时自动 throttle producer
  • 减少了 ping-pong buffer 的内存占用

代价：Producer 和 consumer 必须在时间上同步，不能独立调度。这在 farrow_kernel2 中体现为三次重置 del_i 指针的需求。

3. 定点数缩放策略

选择：统一的 DNSHIFT = 14 右移操作

背景：AIE-ML 使用定点数而非浮点数。每次 MAC 运算后，累加器的位宽会增长，需要通过右移来归一化。

为什么是 14：

  • 滤波器系数是 Q15 格式（16-bit 有符号，范围 [-1, 1)）
  • 8-tap FIR 的最大增益约为 8（即 \(2^3\)）
  • 为了保持输出与输入相同的 Q15 格式，需要右移约 14-15 位
  • 具体值通过 MATLAB/Python 仿真确定，以最小化量化误差

4. 延迟数据的重读设计

选择：farrow_kernel2 中三次从头读取 del_i

替代方案：可以将延迟值缓存到本地数组，避免重复读取

为什么选择重读：

  • 简化代码：不需要额外的 buffer 管理逻辑
  • 利用数据局部性：del_i 通常很小（1024 个 32-bit 整数 = 4KB），可以驻留在 L1 cache
  • 为后续优化留空间：这是明显的优化点，留给后续版本展示改进效果

内存所有权与生命周期（C++ 分析）

Kernel 对象模型

// farrow_graph.h
kernel farrow_kernel1_ins;
kernel farrow_kernel2_ins;

这里的 kernel 是 adf 命名空间中的句柄类型，不代表实际的对象所有权。真正的 kernel 实例由 AIE 编译器和运行时管理，farrow_kernel1_ins 只是一个用于图构建阶段的引用。

状态数组的生命周期

// farrow_kernel1.h
alignas(32) TT_SIG f_state[STATE_LEN];  // STATE_LEN = 8

// farrow_kernel2.h
alignas(32) TT_SIG z[BUFFER_SIZE];      // BUFFER_SIZE = 1024

这些成员数组的生存期与 kernel 实例绑定：

  • 分配：由 AIE 运行时在每个 AIE tile 的本地数据内存（DM）中分配
  • 初始化：在 main() 调用 aie_dut.init() 时，执行 kernel 的构造函数
  • 持久化：在多次 run() 调用之间保持状态（这是 FIR 滤波器正确工作的前提）
  • 销毁：在 aie_dut.end() 时随 tile 释放

缓冲区访问契约

void run(input_buffer<TT_SIG, adf::extents<BUFFER_SIZE>>& __restrict sig_i,
         output_buffer<TT_SIG, adf::extents<BUFFER_SIZE>>& __restrict y3, ...)
  • input_buffer / output_buffer：由 AIE 运行时分配的乒乓缓冲，对 kernel 代码表现为连续的内存区域
  • __restrict：承诺指针不别名，允许编译器激进优化
  • adf::extents<BUFFER_SIZE>：编译时已知的大小，用于边界检查和向量化

重要：kernel 代码不拥有这些缓冲区的所有权，也不应该尝试释放它们。它们由 graph 的连接声明隐式创建和管理。

并发与线程安全

单线程设计

整个 initial_ported_graph_baseline 模块是单线程的。两个 kernel 虽然可以概念上并行执行，但在每个 AIE tile 上是顺序调度的：

  1. farrow_kernel1 运行直到完成（处理 1024 个样本）
  2. 数据通过 stream 传递到 farrow_kernel2
  3. farrow_kernel2 运行直到完成

这种设计简化了推理，但限制了吞吐量。后续优化版本会引入流水线并行（pipelining），让两个 kernel 交替处理数据块。

无锁数据传递

Stream 连接使用 AIE 硬件的FIFO 机制，天然是无锁的：

  • Producer 写入直到 FIFO 满，然后阻塞
  • Consumer 读取直到 FIFO 空，然后阻塞
  • 硬件保证操作的原子性（以 32-bit 或 128-bit 为单位）

边缘情况与陷阱

1. 状态初始化陷阱

farrow_kernel1::farrow_kernel1(void) {
    for(int i = 0; i < STATE_LEN; i++) {
        f_state[i] = (cint16){0, 0};
    }
}

陷阱：如果忘记清零 f_state，首次调用会产生错误输出（包含随机历史状态）。

缓解：构造函数显式初始化。但在某些优化版本中，可能会为了提高性能而省略这一步——需要仔细检查。

2. 延迟值范围假设

del = aie::vector_cast<int16>(*p_del_i++);

隐含假设：del_i 的值已经被预处理到适合 16-bit 有符号整数的范围（并且已经按 DNSHIFT 缩放）。

后果：如果输入延迟值过大，会发生溢出或饱和（取决于 set_saturation 设置），导致错误的插值结果。

3. Buffer Size 对齐要求

static constexpr unsigned BUFFER_SIZE = 1024;
// ...
for (unsigned rr = 0; rr < BUFFER_SIZE/32; rr++)

要求：BUFFER_SIZE 必须是 32 的倍数（因为每次循环处理 32 个样本）。

后果：如果修改 BUFFER_SIZE 为不被 32 整除的值，末尾样本会被静默丢弃或产生越界访问。

4. 舍入模式的全局影响

aie::set_rounding(aie::rounding_mode::symmetric_inf);
aie::set_saturation(aie::saturation_mode::saturate);

副作用：这些设置是全局的，影响整个 AIE tile 的行为。

风险：如果多个 kernel 共享一个 tile（在这个 baseline 中不会，但优化版本可能会），它们的舍入/饱和设置会互相干扰。

5. 文件路径硬编码

sig_i[0] = input_plio::create("PLIO_i_0", plio_64_bits, "data/sig_i.txt");

限制：路径是相对于工作目录的。如果从错误的位置启动仿真，会找不到输入文件。

最佳实践：使用环境变量或命令行参数传递数据目录，但这个 baseline 为了简洁而硬编码。

构建与运行

Makefile 关键选项

AIE_SIM_ONLY      ?= true
PLATFORM_USE      := xilinx_vek280_base_202420_1
OTHER_OPTS        = --aie.pl-freq=625.0 --aie.Xchess="-std=c++17"
AIE_FLAGS         := --aie.verbose ... --aie.Xmapper=BufferOptLevel9 --aie.xlopt=1

关键参数解释：

  • --aie.pl-freq=625.0：PL（Programmable Logic）侧时钟频率 625 MHz，影响 PLIO 的带宽计算
  • --aie.Xmapper=BufferOptLevel9：启用最高级别的 buffer 优化，会自动插入必要的乒乓缓冲
  • --aie.xlopt=1：启用跨层优化，允许编译器在 kernel 之间进行融合和调度优化

给新贡献者的建议

推荐阅读顺序

  1. 先通读 farrow_graph.h，理解整体拓扑
  2. 再看 farrow_kernel1.h/.cpp，这是计算最密集的部分，也是 AIE API 的典型用法
  3. 最后看 farrow_kernel2.h/.cpp，理解多项式求值的实现技巧
  4. 对比 farrow_app.cpp 中的 AIE_SIM_ONLY 宏，理解仿真 vs 硬件的差异

修改前的检查清单

  • [ ] 修改 BUFFER_SIZE 时，确认它是 32 的倍数（Kernel1）和 16 的倍数（Kernel2）
  • [ ] 修改滤波器系数时，更新 f_taps 数组并保持 alignas(32) 对齐
  • [ ] 添加新的 kernel 端口时，同步更新 farrow_graph.h 中的连接声明
  • [ ] 调整 runtime<ratio> 时，参考编译器报告的利用率报告，避免超过 1.0

调试技巧

  1. 数值不匹配：先用 MATLAB/Python 生成参考输出，再用 check_sim_output.m 对比。常见问题包括：

    • 舍入模式不一致（MATLAB 默认 round-to-nearest，AIE 默认 symmetric-inf）
    • 延迟值缩放因子错误

  2. 死锁/挂起：检查 stream 连接是否形成环，或者 consumer 是否比 producer 慢导致 FIFO 溢出

  3. 性能不达标：使用 make get_II 查看循环的 Initiation Interval。理想情况下，向量化循环的 II 应该为 1（每个周期处理一个向量）。

总结

initial_ported_graph_baseline 是一个教学优先、正确性优先的参考实现。它没有追求极致的性能，而是通过清晰的模块划分、显式的状态管理和详尽的注释，为读者建立 Farrow 滤波器在 AIE-ML 上实现的 mental model。

理解了这个 baseline，你就掌握了：

  • 如何用 adf::graph 描述多 kernel 数据流
  • 如何用 aie_api 进行向量化 FIR 滤波
  • 如何处理有状态 kernel 的跨调用持久化
  • 如何组织条件编译以支持仿真和硬件两种模式

这些都是后续优化版本的基础。当你阅读 farrow_opt_1 或 farrow_final_aie 时，可以时刻回顾这个 baseline，看看哪些结构被保留（因为它们反映了算法的本质），哪些被重构（为了性能而做的妥协）。
