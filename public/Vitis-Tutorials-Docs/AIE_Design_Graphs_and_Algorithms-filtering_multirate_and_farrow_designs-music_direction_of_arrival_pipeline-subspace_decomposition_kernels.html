子空间分解内核 (Subspace Decomposition Kernels)

一句话概括

这个模块是 MUSIC 方向估计算法的"数学心脏" —— 它将接收到的多通道信号数据分解为信号子空间和噪声子空间，就像用一把精密的手术刀将混合在一起的不同频率成分分离出来，让后续的谱峰搜索能够准确找到信号的来波方向。

问题空间：为什么需要子空间分解？

想象你站在一个嘈杂的十字路口，周围有多个喇叭在播放不同的音乐。你的任务是：不仅要听到这些音乐，还要精确判断每个喇叭来自哪个方向。这就是 MUSIC (Multiple Signal Classification) 算法要解决的实际问题。

核心挑战

  1. 信号混叠：多个信号源的信号在传感器阵列处叠加，形成复杂的混合信号
  2. 噪声干扰：环境噪声与目标信号混在一起，直接分析会被噪声淹没
  3. 高维数据处理：现代雷达/通信系统可能有数十个接收通道，数据维度很高
  4. 实时性要求：方向估计需要在毫秒级完成，不能有太多延迟

传统方法的局限

  • 傅里叶变换法：分辨率受限于阵列孔径，无法区分角度相近的信号源
  • 波束形成法：需要预先知道信号数量，且旁瓣会干扰主瓣检测
  • 最大似然法：计算复杂度太高，不适合实时应用

MUSIC 算法的洞察

MUSIC 算法的核心洞察是：信号子空间和噪声子空间是正交的。如果我们能把接收信号的协方差矩阵分解成这两个正交子空间，就可以利用这种正交性来构造"空间谱"——在真实信号方向上会出现尖锐的峰值。

这就像是：如果你知道噪音的特征（噪声子空间），你就可以设计一个滤波器，在任何地方都让它通过，唯独在信号来的方向上把它完全阻断——这样信号方向就会凸显为一个深谷（或峰值，取决于定义）。

模块架构：如何分解子空间？

子空间分解的核心数学操作是 特征值分解 (EVD) 或等价的 奇异值分解 (SVD)。本模块提供了两种实现路径：

两条技术路线的选择

特性 QRD 路径 SVD 路径
算法复杂度 O(n³)，但常数较小 O(n³)，数值更稳定
数值稳定性 良好，适合良态矩阵 优秀，适合病态矩阵

实现策略 分阶段 QR 分解 并行 SVD 迭代

核心组件详解

1. QRD 图 (qrd_app.cpp)

职责定位

QRD (QR Decomposition) 路径采用 QR 分解 作为子空间分解的基础。它通过将协方差矩阵分解为正交矩阵 Q 和上三角矩阵 R，然后对 R 进行进一步处理来提取子空间信息。

关键代码解析

class dut_graph : public graph {
public:
  qrd_graph   qrd;           // QRD 计算子图
  input_plio  sig_i;         // 输入 PLIO 接口
  output_plio sig_o;         // 输出 PLIO 接口

  dut_graph( void )
  {
    // 创建 64-bit PLIO 接口，连接外部数据源
    sig_i =  input_plio::create("PLIO_i", plio_64_bits, "data/sig_i.txt");
    sig_o = output_plio::create("PLIO_o", plio_64_bits, "data/sig_o.txt");
    
    // 连接数据流
    connect<>( sig_i.out[0], qrd.sig_i );
    connect<>( qrd.sig_o, sig_o.in[0] );

    // 布局策略：从第 12 列开始，每核占一列
    static constexpr unsigned COL_START = 12;
    for (unsigned i = 0, col = COL_START; i < qrd_graph::NUM_QRD_KERNELS; ++i, ++col) {
      adf::location<adf::kernel>(qrd.qrd_kernel[i]) = adf::tile(col, ROW_0);
    }
  }
};

内存所有权与生命周期

  • PLIO 对象：由 dut_graph 构造函数创建，生命周期与图实例绑定
  • Kernel 位置：通过 adf::location 显式绑定到物理 AIE tile，编译期确定
  • 数据缓冲区：由 AIE 运行时自动管理，遵循 ADF 图的流式语义
  • 栈空间：注释掉的栈位置代码表明可以显式控制栈分配，但当前使用默认策略

2. SVD 图 (svd_app.cpp)

职责定位

SVD (Singular Value Decomposition) 路径直接对数据矩阵进行奇异值分解：

A = U × Σ × Vᴴ

其中 Σ 对角线上的奇异值可以直接区分信号和噪声——大奇异值对应信号，小奇异值对应噪声。

class dut_graph : public graph {
public:
  svd_graph   dut;           // SVD 计算子图（注意命名差异）
  input_plio  sig_i;
  output_plio sig_o;

  dut_graph( void )
  {
    sig_i =  input_plio::create("PLIO_i", plio_64_bits, "data/sig_i.txt");
    sig_o = output_plio::create("PLIO_o", plio_64_bits, "data/sig_o.txt");
    connect<>( sig_i.out[0], dut.sig_i );
    connect<>( dut.sig_o, sig_o.in[0] );

    // 布局策略：从第 10 列开始，共 38 个 tiles

    for (unsigned i = 0, col = COL_START; i < svd_graph::NUM_SVD_KERNELS; ++i, ++col) {
      adf::location<adf::kernel>(dut.svd_kernel[i]) = adf::tile(col, ROW_0);
      // 显式指定栈位置与核位置相同
      adf::location<adf::stack >(dut.svd_kernel[i]) = adf::location<adf::kernel>(dut.svd_kernel[i]);
    }
  }
};

设计决策与权衡

1. 为什么提供两种分解路径？

决策：同时支持 QRD 和 SVD 两种子空间分解方法。

权衡分析：

        精度需求高 ──────────────────────► 精度需求低
        │                                    │
   SVD  │◄────────── 灵活选择区域 ──────────►│  QRD
  路径  │                                    │  路径
        │         根据应用场景动态选择          │
        ▼                                    ▼
   资源消耗大                           资源消耗小
   (38 tiles)                          (较少 tiles)

原因：

  • SVD 是子空间分解的"黄金标准"，数值稳定性最好，但计算和资源开销大
  • QRD 在某些条件下可以等价地提取子空间信息，且更适合资源受限场景
  • 允许用户根据 精度要求、资源预算、实时性约束 做工程权衡

2. 显式物理布局 vs 自动布局

决策：使用 adf::location 显式指定每个 kernel 的 tile 位置。

权衡分析：

方案 优点 缺点
显式布局（当前） 可预测性能、便于调试、满足时序约束 可移植性差、手动调优成本高
自动布局 可移植性好、开发效率高 可能不满足特定时序、难以调试

原因：

  • MUSIC 算法是计算密集型应用，对数据局部性和通信延迟敏感
  • 显式布局确保相邻计算单元在物理上也相邻，最小化片上网络 (NoC) 延迟
  • 便于与硬件团队协同优化，针对特定 Versal 器件微调

3. PLIO 64-bit 接口选择

决策：使用 plio_64_bits 作为 I/O 位宽。

权衡分析：

  • 64-bit 位宽在 带宽 和 资源 之间取得平衡
  • 对于复数数据（实部+虚部各 32-bit），64-bit 正好传输一个样本
  • 相比 128-bit，减少了 PL 侧的逻辑资源；相比 32-bit，减少了一半的传输次数

新贡献者必读：陷阱与注意事项

1. 核数量硬编码风险

// ⚠️ 危险：如果修改了 svd_graph.h 中的 NUM_SVD_KERNELS
// 但没有同步修改这里的循环边界，会导致布局错误
for (unsigned i = 0, col = COL_START; i < svd_graph::NUM_SVD_KERNELS; ++i, ++col)

建议：确保 svd_graph.h / qrd_graph.h 中的宏与布局代码保持一致。

2. Tile 冲突检查

// SVD 使用列 10-47
static constexpr unsigned COL_START = 10;  // 38 kernels

// QRD 使用列 12+
static constexpr unsigned COL_START = 12;  // NUM_QRD_KERNELS 个核

注意：如果同时实例化两个图，列 12+ 会发生冲突！实际部署时应该：

  • 只选择一种分解方式
  • 或调整其中一个的起始列避免重叠

3. 栈空间管理

// SVD 显式指定栈位置
adf::location<adf::stack>(dut.svd_kernel[i]) = adf::location<adf::kernel>(dut.svd_kernel[i]);

// QRD 注释掉了栈位置设置（使用默认）
// adf::location<adf::stack >(qrd.qrd_kernel[i]) = ...

潜在问题：

  • 默认栈分配可能与数据内存竞争
  • 如果 kernel 递归调用或使用大量局部数组，可能导致栈溢出
  • 建议：生产环境应像 SVD 一样显式控制栈位置

4. 文件路径依赖

input_plio::create("PLIO_i", plio_64_bits, "data/sig_i.txt");

注意：

  • 相对路径 data/sig_i.txt 依赖于工作目录
  • 仿真和实际部署时的文件系统视图可能不同
  • 建议：使用绝对路径或通过预处理器宏配置

5. 运行次数硬编码

int main(void) {
  aie_dut.init();
  aie_dut.run(4);  // ⚠️ 只运行 4 次迭代
  aie_dut.end();
  return 0;
}

注意：run(4) 表示执行 4 次图迭代。在实际应用中，这应该：

  • 由 host 代码动态控制
  • 或改为 run(-1) 无限运行直到显式停止

总结

子空间分解内核模块是 MUSIC 算法中最核心的数学计算部分。它通过两种技术路线（QRD 和 SVD）提供了灵活的工程选择，让开发者可以在 精度、资源、实时性 三者之间做权衡。

理解这个模块的关键心智模型是：

把高维信号空间看作一个房间，信号子空间是房间里几个人说话的声音方向，噪声子空间是背景杂音的方向。子空间分解就是找出这两个方向的正交基，然后利用它们的正交性来"屏蔽"噪声，突出信号。

对于新加入团队的工程师，建议按以下顺序深入：

  1. 先理解 MUSIC 算法的数学原理（信号/噪声子空间的正交性）
  2. 再研究 QRD 和 SVD 的具体实现（阅读对应的 kernel 代码）
  3. 最后掌握布局优化技巧（tile 分配、内存 banking、流水线平衡）
