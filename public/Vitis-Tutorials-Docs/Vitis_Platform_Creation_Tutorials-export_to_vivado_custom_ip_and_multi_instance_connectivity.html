Export to Vivado 自定义 IP 与多实例连接

一句话概述

本模块展示 Vitis Export to Vivado 流程的高级扩展模式——通过 system_custom_ip.cfg 演示如何集成第三方 RTL IP 替代标准 DMA 引擎，以及通过 system_new.cfg 展示多实例并行流水线架构，实现吞吐量的水平扩展。

双配置架构对比

┌─────────────────────────────────────────────────────────────────┐
│                     配置 1: 自定义 IP 集成                         │
│              system_custom_ip.cfg                               │
├─────────────────────────────────────────────────────────────────┤
│  特点:                                                          │
│  • 移除标准 stream_out (MM2S)，假设数据来自外部自定义 IP            │
│  • 引入抽象接口 AIE_IN，代表外部数据源                             │
│  • 保留标准 stream_in (S2MM) 作为数据汇                           │
│  • 展示如何将 Vitis 生成的模块集成到更大的 Vivado 项目             │
│                                                                 │
│  数据流: 外部 IP → AIE → polar_clip → AIE → S2MM → DDR          │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                     配置 2: 多实例扩展                           │
│              system_new.cfg                                     │
├─────────────────────────────────────────────────────────────────┤
│  特点:                                                          │
│  • 双实例并行流水线，展示水平扩展架构                              │
│  • polar_clip1 + stream_in1 (第一通道)                           │
│  • polar_clip2 + stream_in2 (第二通道)                           │
│  • 独立 AIE 接口: DataIn_1/DataOut_1 和 DataIn_2/DataOut_2        │
│  • aie_stream_in 提供第二通道的独立数据源                          │
│                                                                 │
│  数据流: 双通道并行处理                                           │
│  通道 1: AIE_IN → AIE(DataIn_1) → polar_clip1 → AIE → stream_in1 │
│  通道 2: aie_stream_in → AIE(DataIn_2) → polar_clip2 → AIE → stream_in2 │
└─────────────────────────────────────────────────────────────────┘

system_custom_ip.cfg 详解

配置内容

# 频率约束：stream_in 运行在 200MHz
freqhz=200MHz:stream_in.ap_clk

[connectivity]
# 内核实例化：仅实例化 stream_in 和 polar_clip
# 注意：没有 stream_out！数据源假设来自外部自定义 IP
nk=stream_in:1:stream_in
nk=polar_clip:1:polar_clip

# 流连接：
sc=AIE_IN:ai_engine_0.DataIn1
# ↑ 关键差异：使用 AIE_IN 而非 stream_out.s
#   AIE_IN 是抽象占位符，表示数据来自外部 RTL IP
#   实际连接在 Vivado 中通过 RTL 实例化完成

sc=ai_engine_0.clip_in:polar_clip.in_sample
sc=polar_clip.out_sample:ai_engine_0.clip_out
sc=ai_engine_0.DataOut1:stream_in.s
# 其余连接与基础配置相同

[clock]
freqHz=100000000:polar_clip.ap_clk
# polar_clip 保持 100MHz

└────────────────────────────────────────────────────────────────────┘

关键理解点：
1. Vitis 导出的模块包含 "AIE_IN" 抽象端口，没有实际驱动
2. 在 Vivado 中，开发者手动实例化自定义 RTL IP
3. 通过 RTL 连线将 custom_dma → AIE_IN 连接起来
4. 这种方式允许集成任何符合 AXI4-Stream 协议的 RTL IP

集成步骤

# 步骤 1: 在 Vitis 中导出设计
# 注意：system.cfg 中使用 AIE_IN 占位符
v++ -l --config system.cfg ... -o export.xsa

# 步骤 2: 在 Vivado 中导入并集成
vivado -mode tcl

# 读取导出的 XSA
read_xsa export.xsa

# 实例化自定义 RTL IP (假设已打包为 IP)
create_ip -name custom_dma -vendor user -library user -version 1.0 -module_name custom_dma_inst

# 手动连接 custom_dma 的 M_AXIS 到 AIE_IN 端口
# (通过 Vivado IP Integrator 图形界面或 Tcl 命令)

# 步骤 3: 实现并生成比特流
launch_runs impl_1 -to_step write_bitstream -jobs 8

system_new.cfg 详解

多实例配置内容

# 双通道时钟约束
freqhz=200MHz:stream_in1.ap_clk
freqhz=200MHz:stream_in2.ap_clk

[connectivity]
# 实例化 2 个 stream_in 和 2 个 polar_clip
# 实例化 1 个 aie_stream_in (第二通道数据源)
nk=aie_stream_in:1:aie_stream_in
nk=stream_in:2:stream_in1.stream_in2
nk=polar_clip:2:polar_clip1.polar_clip2

# ========== 通道 1 连接 ==========
sc=AIE_IN:ai_engine_0.DataIn_1
sc=ai_engine_0.clip_in_1:polar_clip1.in_sample
sc=polar_clip1.out_sample:ai_engine_0.clip_out_1
sc=ai_engine_0.DataOut_1:stream_in1.s

# ========== 通道 2 连接 ==========
sc=aie_stream_in.s:ai_engine_0.DataIn_2
sc=ai_engine_0.clip_in_2:polar_clip2.in_sample
sc=polar_clip2.out_sample:ai_engine_0.clip_out_2
sc=ai_engine_0.DataOut_2:stream_in2.s

[clock]
# 双通道 polar_clip 都运行在 100MHz
freqHz=100000000:polar_clip1.ap_clk
freqHz=100000000:polar_clip2.ap_clk

关键设计特点：
1. 完全独立的两个数据通路，互不干扰
2. 每个通道有自己的 AIE 接口 (DataIn_1/2, DataOut_1/2)
3. 每个通道有自己的 polar_clip 和 stream_in 实例
4. 通道 1 数据源是 AIE_IN（外部/AIE），通道 2 是 aie_stream_in（独立 DMA）
5. 两个 polar_clip 都运行在独立的 100MHz 时钟域
6. 两个 stream_in 运行在独立的 200MHz 时钟域

扩展策略分析

为什么选择双通道而非单通道 2× 频率？

扩展策略 双通道 (本设计) 单通道 2× 频率
时钟域 保持 100MHz/200MHz 需提升至 200MHz/400MHz
时序收敛难度 低（宽松约束） 高（接近工艺极限）
AIE 利用率 同时利用两组 PLIO 单组 PLIO 满载
PL 资源消耗 2×（两个 polar_clip） 1×（但流水线更深）
功耗 较高（并行工作） 较低（单通道）
延迟 较低（并行处理） 较高（流水线深度）

设计权衡结论：

  • 选择双通道是因为 AIE 的 PLIO 带宽是瓶颈，而非 PL 逻辑频率
  • 即使将 polar_clip 提升到 200MHz，AIE 到 PL 的接口速率受限（通常 1-2 GB/s 每 PLIO）
  • 双通道充分利用了 AIE 的两组独立 PLIO 接口，整体吞吐量翻倍
  • 100MHz 的保守时钟降低了时序收敛难度，有利于快速开发迭代

新贡献者指南

添加第三通道的步骤

若要将 system_new.cfg 扩展为三通道：

# 1. 添加时钟约束
freqhz=200MHz:stream_in3.ap_clk

[connectivity]
# 2. 实例化新内核
nk=stream_in:3:stream_in1.stream_in2.stream_in3
nk=polar_clip:3:polar_clip1.polar_clip2.polar_clip3

# 3. 添加通道 3 的连接（假设 AIE 支持 DataIn_3/DataOut_3）
sc=AIE_IN_3:ai_engine_0.DataIn_3
sc=ai_engine_0.clip_in_3:polar_clip3.in_sample
sc=polar_clip3.out_sample:ai_engine_0.clip_out_3
sc=ai_engine_0.DataOut_3:stream_in3.s

[clock]
# 4. 添加时钟配置
freqHz=100000000:polar_clip3.ap_clk

注意事项：

  1. AIE PLIO 限制：Versal 器件的 AIE 有固定的 PLIO 数量（通常 4-8 个），需确认硬件支持
  2. NoC 带宽：多通道共享 NoC（Network on Chip）带宽，需确保总带宽不超限额
  3. DDR 带宽：多通道的 S2MM 同时写 DDR 可能造成带宽瓶颈
  4. 布局布线：三实例可能导致 PL 资源紧张，需关注时序收敛

调试多通道配置的技巧

常见问题 1：通道间数据串扰

  • 现象：通道 1 的数据出现在通道 2 的输出
  • 原因：sc= 连接配置错误，如将 polar_clip1.out_sample 连到了 ai_engine_0.clip_out_2
  • 解决：仔细检查每个 sc= 行的端口索引

常见问题 2：某通道无输出

  • 现象：通道 1 正常，通道 2 始终无数据
  • 原因：时钟约束错误，如 polar_clip2 的时钟未绑定或绑定了不存在的时钟源
  • 解决：检查 [clock] 段的频率配置，确保每个实例都有时钟

常见问题 3：性能不达预期

  • 现象：双通道吞吐量不是单通道的 2 倍，而是 1.5 倍左右
  • 原因：AIE 的 PLIO 共享瓶颈，或 DDR 写带宽饱和
  • 解决：使用 Vivado ILA 抓取 AXI 事务，分析各通道的实际带宽利用率
